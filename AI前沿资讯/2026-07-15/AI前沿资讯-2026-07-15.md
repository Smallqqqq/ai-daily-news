## 1. Bonsai 27B（PrismML）——首个能在手机上跑的 27B 级模型

![[bonsai27b-demo.webp]]

**评分: 9.0/10 | 可操作性: 极高 | 热度: HN 443 points / 165 评论，X 全平台热议**

**是什么：**
PrismML 于 7 月 14 日发布的 Bonsai 27B，基于 Qwen3.6-27B 压缩而来，提供 1-bit 二进制（3.9GB）和 1.58-bit 三元（5.9GB）两个变体。通过端到端低比特架构，将 27B 参数的完整多模态能力（图文理解、工具调用、262K 上下文）压缩到可以在 iPhone 上运行的尺寸。iOS 限制单个 app 只能用约 6GB 内存，1-bit 版是首个通过这道门槛的 27B 级模型。

**怎么用：**
```bash
# 下载 GGUF 格式模型（任选一个）
# 1-bit 版（3.9GB，速度最快）
huggingface-cli download prism-ml/Bonsai-27B-gguf Bonsai-27B-Q1_0_g128.gguf --local-dir ./models

# 三元版（5.9GB，精度更高）
huggingface-cli download prism-ml/Ternary-Bonsai-27B-gguf Ternary-Bonsai-27B-Q2_0.gguf --local-dir ./models

# llama.cpp 本地推理
./llama-cli -m ./models/Bonsai-27B-Q1_0_g128.gguf -c 0 -p "Explain transformer attention."

# Apple Silicon 用 MLX
pip install mlx-lm
mlx_lm.generate --model prism-ml/Ternary-Bonsai-27B-mlx-2bit --prompt "解释量子计算"

# OpenAI 兼容 API（llama.cpp server 模式）
./llama-server -m ./models/Bonsai-27B-Q1_0_g128.gguf --port 8080
curl http://localhost:8080/v1/chat/completions -H "Content-Type: application/json" \
  -d '{"messages":[{"role":"user","content":"Hello"}],"tools":[...]}'
```

**为什么重要：**
在 15 项 benchmark 上，1-bit 版保留 FP16 版 90% 的平均性能（85.07 → 76.11）。关键指标：GSM8K 95.30→92.80，MATH-500 99.40→98.00，AIME25 93.29→88.75。M5 Max 上 1-bit 版达 87 tok/s，iPhone 17 Pro 实现可用推理速度。高级 AI 推理首次从云端走到端侧。CNBC 报道 Apple 已在与 PrismML 洽谈技术合作。

**讨论现场：**
- HN 热评："A 27B-class model that fits on a phone changes what 'local AI' can mean. Private by default, no network round trip, continuous operation on personal devices." — 置顶评论
- X 上 @prismml: "Intelligence density will be the defining axis of the next stage of AI progress. Raw capability determines what a model can do; density determines where it can do it." 获 890 转发
- 开发者实测反馈：Android 端偶有输出 "!!!!!!!!!!!!!!" 的 bug，社区已在 GitHub Issues 追踪修复

**来源：** https://prismml.com/news/bonsai-27b

## 2. Destructive Command Guard（dcg）——给 AI 编码 Agent 装上安全带

![[dcg-demo.png]]

**评分: 8.5/10 | 可操作性: 极高 | 热度: GitHub 473 stars/day，总星 3,900+**

**是什么：**
一个 Rust 编写的高性能命令行钩子，在 AI 编码 Agent 执行危险命令前拦截并阻止。覆盖 `git reset --hard`、`rm -rf`、`git push --force`、数据库删除操作等。支持 Claude Code、Codex CLI、Gemini CLI、Copilot、Cursor、Hermes Agent 等主流 Agent。采用白名单优先架构，亚毫秒级延迟，对正常开发流程零影响。7 月 13 日发布 v0.6.6。

**怎么用：**
```bash
# 安装（Rust 编译或下载预编译二进制）
cargo install destructive_command_guard

# 或直接下载
curl -L -o dcg https://github.com/Dicklesworthstone/destructive_command_guard/releases/latest/download/dcg-x86_64-linux
chmod +x dcg && sudo mv dcg /usr/local/bin/

# 验证安装
dcg --version

# 测试拦截效果（不会真正执行）
dcg test "git reset --hard HEAD~1"
# 输出: BLOCKED core.git:reset-hard → 建议 git stash / git reset --soft

# 测试安全命令
dcg test "git status"
# 输出: ALLOWED

# 管理白名单
dcg allow "rm -rf ./build" --reason "安全的构建目录清理" --expires 30d
dcg prune  # 清理过期的白名单条目

# 扫描仓库中的危险命令
dcg scan ./src
```

**为什么重要：**
使用 AI Agent 写代码的人都遇到过这个场景：Agent 为了"清理干净"直接执行 `git reset --hard` 或 `rm -rf ./src`，几小时的工作瞬间消失。dcg 在 shell 执行前完成拦截——不是事后恢复，而是事前阻止。与 AI-based 安全方案不同，dcg 是确定性的规则引擎，不存在模型自己的盲区。在 Agent 越来越自主的时代，这种"模型之外的安全层"正在成为刚需。

**讨论现场：**
- HN 评论："This should be pre-installed with every AI coding agent. The model doesn't feel the loss of three hours of work. You do." — +67 points
- X 上 @GabiDev98: "Protect your work against destructive commands when working with AI. A seatbelt, not a threat model."
- GitHub Issues 中社区积极贡献新的危险命令模式和语言支持

**来源：** https://github.com/Dicklesworthstone/destructive_command_guard

## 3. mattpocock/skills——TypeScript 大神的生产级 Agent 技能集

**评分: 8.3/10 | 可操作性: 极高 | 热度: GitHub 1,679 stars/day，总星 84,000+**

**是什么：**
TypeScript 教育家 Matt Pocock 将其个人 `.claude` 目录中的 20+ 工程技能公开发布。涵盖三个层次：(1) 用户调用技能——`/grill-me`（需求审讯）、`/to-prd`（生成 PRD）、`/to-issues`（拆分任务）、`/caveman`（削减 65% token 消耗）；(2) 模型调用技能——诊断 bug、TDD、领域建模、代码库设计；(3) 基础设施——会话管理、环境设置。7 月 13-14 日活跃更新，新增 Codex 插件兼容。

**怎么用：**
```bash
# 安装全部技能
npx skills@latest add mattpocock/skills

# 或只装单个技能
npx skills@latest add mattpocock/skills --skill caveman

# 使用 caveman 模式（削减 Claude 输出 token 约 65%）
# 在 Claude Code 对话中输入：
/caveman lite    # 轻度压缩
/caveman full    # 完全压缩（推荐）
/caveman ultra   # 极致压缩
# 恢复正常：/caveman off

# 需求审讯
/grill-me
# Claude 会连续追问 20+ 个问题，迫使你在写代码前想清楚所有边界条件
```

**为什么重要：**
Pocock 的技能集代表了 AI 编码的最佳实践方向：小粒度、可组合、编码工程纪律而非隐藏它。`/caveman` 技能实测能将 Claude Opus 4.7 的输出 token 削减 65%，长会话中效果更显著。与 Addy Osmani 的 agent-skills（侧重代码质量）不同，Pocock 的技能更侧重开发流程的工程化——从需求、设计到测试、部署的完整链路。

**讨论现场：**
- Developer's Digest 评价："Small, readable skills that encode workflow discipline without taking over the whole process. This is the better path beyond vibe coding."
- X 上用户反馈："/grill-me asked me 24 questions in a row. I sat there for an hour. That hour is the work."
- GitHub 上 536+ PR 已合并，社区活跃贡献新模式

## 4. Juggler——JUCE 之父做了一款"反 Electron"的 GUI 编码 Agent

![[juggler-demo.png]]

**评分: 7.8/10 | 可操作性: 高 | 热度: HN 184 points / 80 评论**

**是什么：**
Julian Storer（JUCE C++ 框架和 Tracktion DAW 的创建者）于 7 月 14 日在 HN Show 发布的开源 GUI 编码 Agent。Go 后端 + Wails 窗口（不是 Electron），插件化架构——所有上下文操作（读写/bash 等）、LLM 循环策略、斜杠命令和 UI 都是 JavaScript 插件，可检查、fork 和替换。支持 Claude、OpenAI/Codex、Gemini、Ollama、OpenRouter、DeepSeek 等 BYOK。AGPLv3 协议，插件 SDK 和内置扩展用 Apache 2.0。

**怎么用：**
```bash
# 克隆并运行
git clone https://github.com/juggler-ai/juggler.git
cd juggler

# 安装依赖（Go + Wails）
go mod download

# 开发模式运行
wails dev

# 或下载预编译版本
# Releases: https://github.com/juggler-ai/juggler/releases
```

**为什么重要：**
在 CLI Agent（Claude Code、Gemini CLI）和 IDE Agent（Cursor、Copilot）之外，Juggler 提供了第三种方案：独立的 GUI Agent。核心差异化：(1) 不需要 Electron——Wails 占用内存远小于 VS Code/Cursor；(2) 完全可扩展的插件系统——你不是在配置一个工具，而是在定制一个 Agent 平台；(3) 分离架构——headless server 跑在代码所在的机器上，GUI 从任意设备连接。

**讨论现场：**
- HN 上 @julesrms（作者）:"I spent too many hours loving what the models could do, but hating the CLI experience. Juggler is my attempt at a better UX."
- HN 热评："The plugin architecture is the differentiator. Every item in context, LLM loop strategies, slash commands are all inspectable JS plugins. This is what open-source agent UX should look like." — +42 points
- JUCE 社区讨论热烈："The guy who made JUCE is now making AI coding tools. The audio dev world is watching."

**来源：** https://github.com/juggler-ai/juggler

## 5. TencentDB Agent Memory——让 Agent 记住一切的 4 层记忆引擎

**评分: 7.5/10 | 可操作性: 高 | 热度: GitHub 8,200+ 总星，社区持续活跃**

**是什么：**
腾讯云数据库团队开源的分层记忆引擎，为 AI Agent 提供 L0-L3 四级渐进式记忆：L0 工具日志卸载、L1 上下文压缩（Mermaid 画布）、L2 会话记忆、L3 跨会话长期记忆。纯本地运行，零外部 API 依赖，MIT 协议。配合 OpenClaw 插件使用后，token 消耗降低最高 61.38%，任务通过率相对提升 51.52%。

**怎么用：**
```bash
# 安装（作为 OpenClaw 插件）
openclaw plugin install @tencentdb-agent-memory/memory-tencentdb

# 或独立使用
git clone https://github.com/TencentCloud/TencentDB-Agent-Memory.git
cd TencentDB-Agent-Memory
npm install

# 配置
# 在 OpenClaw 配置文件中启用：
# plugins:
#   memory-tencentdb:
#     l0_enabled: true    # 工具日志卸载
#     l1_enabled: true    # 上下文压缩
#     l2_enabled: true    # 会话记忆
#     l3_enabled: true    # 长期记忆
```

**为什么重要：**
大多数 Agent 记忆方案的问题不是"记不住"，而是"什么都记，没有任何优先级"。TencentDB Agent Memory 的核心创新是分层——短期记忆用工具日志 + Mermaid 画布（不占 context window），长期记忆用向量检索 + 符号化存储。PersonaMem 基准测试从 48% 跃升至 76%，说明"记住什么"比"记住多少"更重要。在超长任务（超过 100 轮对话）场景下，这个 61% 的 token 节省直接转化为推理成本下降。

**讨论现场：**
- Medium 深度评测："The real problem isn't that agents forget — it's that they remember everything, flatly, with no sense of what matters."
- LinkedIn 开发者反馈："After integrating, my agent stopped confusing project A's config with project B's config after 50+ turns."
- GitHub Discussions 中社区最活跃的话题是如何调参（L0-L3 的比例分配）

**来源：** https://github.com/TencentCloud/TencentDB-Agent-Memory

## 7. grok2api——多账号 Grok API 网关，绕开速率限制

**评分: 7.0/10 | 可操作性: 高 | 热度: GitHub 186 stars/day**

**是什么：**
Go 语言编写的多账号 API 网关，聚合多个 Grok（xAI）账号的 API 配额，提供统一的 OpenAI 兼容接口。支持 Grok Build、Grok Web、Grok Console 三种后端。自动负载均衡、速率限制管理和账号轮换。对于需要大量调用 Grok 4.5 但受限于单账号配额的开发者来说，这是直接的解决方案。

**怎么用：**
```bash
# 安装
go install github.com/chenyme/grok2api@latest

# 或下载预编译二进制
# Releases: https://github.com/chenyme/grok2api/releases

# 配置账号（编辑 config.yaml）
# accounts:
#   - name: "account1"
#     token: "xai-token-1"
#   - name: "account2"
#     token: "xai-token-2"

# 启动网关
grok2api serve --port 8080

# 使用 OpenAI 兼容接口调用
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"grok-4.5","messages":[{"role":"user","content":"Hello"}]}'
```

**为什么重要：**
Grok 4.5 发布后（7 月 9 日），开发者社区发现其 API 虽然兼容 OpenAI SDK，但单账号速率限制较严格。grok2api 巧妙地解决了这个问题——不是绕过限制，而是通过多账号聚合来扩展总配额。这种模式在 OpenAI API 早期出现过（如 ChatGPT-Next-Web 的多账号轮换），现在 Grok 生态也开始需要类似工具了。

**来源：** https://github.com/chenyme/grok2api

## 8. LeMario——用 JEPA 世界模型让 AI 学会玩超级玛丽

**评分: 6.5/10 | 可操作性: 中 | 热度: HN 54 points**

**是什么：**
一个研究项目，使用 JEPA（Joint Embedding Predictive Architecture）世界模型在超级玛丽兄弟上训练。与传统强化学习不同，JEPA 不逐像素预测下一帧，而是在抽象表征空间中预测未来状态。模型学会理解游戏世界的物理规律（重力、碰撞、惯性），而不是死记硬背画面。代码开源。

**怎么用：**
```bash
git clone https://github.com/LeMario-ai/LeMario.git
cd LeMario
pip install -r requirements.txt

# 训练 JEPA 世界模型
python train_jepa.py --env super_mario_bros --epochs 100

# 可视化世界模型预测
python visualize.py --checkpoint ./checkpoints/jepa_model.pt
```

**为什么重要：**
JEPA 是 Yann LeCun 提出的"世界模型"路线的核心架构。LeMario 虽然只是一个游戏 demo，但它展示了 JEPA 在理解物理世界动态方面的潜力。相比传统 RL 需要数百万次试错，JEPA 在少量交互后就能建立对环境的内部模型。这个方向如果延伸到机器人控制、自动驾驶等领域，可能比当前端到端模仿学习的方法样本效率更高。

**讨论现场：**
- HN 评论："JEPA is the architecture behind LeCun's vision for AGI. Seeing it work on something as concrete as Mario is more convincing than any white paper."

**来源：** https://github.com/LeMario-ai/LeMario
