## 1. code-review-graph——本地优先代码智能图谱，MCP + CLI 双接口，1925 stars/day

**评分: 9.2/10 | 可操作性: 极高 | 热度: GitHub 1,925 stars/day，24.2k total，Trending #3**

**是什么：**
用 Tree-sitter 解析代码库，构建增量更新的结构化代码图谱，再通过 MCP 协议喂给 AI 编码工具。核心思路——让 Agent 只读"相关代码"而非"整个仓库"，大幅降低 Token 消耗。支持 Claude Code、Codex、Cursor、Gemini CLI、Antigravity 等主流工具，一条 `install` 命令自动检测已安装工具并配置 MCP。内置 GitHub Action 集成，CI 中自动更新图谱。Python 94% + TypeScript 6%，MIT 协议，90 位贡献者，28 个 release，最新 v2.3.7。

**怎么用：**
```bash
# 安装并自动配置所有已安装的 AI 编码工具
pip install code-review-graph
code-review-graph install
code-review-graph build

# 在 Claude Code 中手动注册 MCP
claude mcp add code-review-graph --scope user -- code-review-graph serve

# GitHub Action 集成（每次 push 自动更新图谱）
# 在 .github/workflows 中添加官方 action 即可
```

**为什么重要：**
AI 编码工具做 Code Review 时最大的 Token 浪费来自通读整个代码库。code-review-graph 用 Tree-sitter AST 解析替代文本扫描，将上下文窗口压缩到只含相关符号的精确片段。benchmark 显示在大型仓库的 review 任务中可减少 60-80% 上下文消耗。对比纯 grep/RAG 方案，AST 级别的符号追踪不会漏掉跨文件引用，也不会被同名变量误导。

**讨论现场：**
- GitHub Issues 中社区贡献了多语言 Tree-sitter 解析器支持，日语/韩语/印地语 README 翻译已合并
- 社区反馈"把代码分析和 MCP 协议结合，比直接往 prompt 里贴 grep 结果聪明一个数量级"
- v2.3.7 刚支持增量更新——只重解析变更文件而非全量重建

**来源：** https://github.com/tirth8205/code-review-graph | https://code-review-graph.com

## 2. i-have-adhd——给 AI 编码 Agent 的"防跑偏药"，1866 stars/day

**评分: 8.8/10 | 可操作性: 极高 | 热度: GitHub 1,866 stars/day，6.5k total，Trending #4**

**是什么：**
一个 Claude Code / Codex 技能插件，强制 AI 编码助手以"ADHD 友好"风格输出：行动优先、步骤编号、不埋答案、不说"Hope this helps!"。不是真的为 ADHD 患者设计，而是解决 AI Agent 的通病：回答啰嗦、把关键操作埋在 300 字段落末尾、用模糊的"你可以试试"替代精确命令。安装后 AI 的输出风格从"让我想想……有几件事你可以考虑……希望对你有帮助"变成"运行 `npm install jsonwebtoken@latest`，编辑 `src/auth.ts` 第 42-58 行"。

**怎么用：**
```bash
# Claude Code 安装（一条命令，自动从 GitHub 拉取并保持更新）
claude plugin marketplace add ayghri/i-have-adhd
claude plugin install i-have-adhd@i-have-adhd

# 之后在任何任务中输入 /i-have-adhd 即可激活

# Codex 安装
codex plugin marketplace add ayghri/i-have-adhd --ref main
codex plugin add i-have-adhd@i-have-adhd
# 输入 $i-have-adhd 激活
```

**为什么重要：**

**讨论现场：**

- 作者回应：不是所有场景都需要紧凑输出，学习/探索任务中展开解释是合理的——这个技能是给你"选风格"的开关，不是修 bug

## 8. llmfit——一条命令找出适合你硬件的模型，129 stars/day

**评分: 8.0/10 | 可操作性: 极高 | 热度: GitHub 129 stars/day，29.7k total**

**是什么：**
一个 Rust 构建的终端工具，自动检测你的 RAM/CPU/GPU 配置，对数百个模型按质量、速度、适配度、上下文窗口四个维度打分，告诉你哪些模型能在你的机器上跑到什么水平。支持多 GPU、MoE 架构、GGUF/MLX 格式、动态量化。内置交互式 TUI（默认）和经典 CLI 模式。特色功能 `llmfit bench --share`：在你的机器上实测 tok/s 并提交 benchmark 数据回馈社区——其他人用同款硬件时就能看到真实性能数据而非理论估算。

**怎么用：**
```bash
# 安装
cargo install llmfit
# 或
brew install llmfit

# 检测硬件并推荐模型
llmfit

# 实测性能并贡献数据
llmfit bench --share

# 只看能跑在 GPU 上的模型
llmfit --gpu-only

# 导出兼容 Ollama 的配置
llmfit export --format ollama
```

**为什么重要：**
本地 LLM 部署最大的痛点是选模型——同样是 7B 参数，一个 Q4 量化的 Qwen3 能在 8GB 显存上跑 40 tok/s，另一个 FP16 的需要 24GB。llmfit 把这个试错过程自动化了。29.7k stars 说明"我该跑哪个模型"是本地 AI 社区最大的未解决问题之一。bench --share 的社区贡献机制让数据从"理论值"变成"真实值"，对推动本地 AI 生态有长远价值。

**讨论现场：**
- 80 位贡献者中大量是提交 benchmark 数据的用户，形成正向循环
- 社区反馈"终于不用在 Reddit 上搜 'Can my RTX 3060 run Llama 4'" 了

**来源：** https://github.com/AlexsJones/llmfit
