## 5. Andyyyy64/whichllm — 一条命令找出你的硬件上实际跑得最好的本地 LLM

**评分: 8.0/10 | 可操作性: 极高 | 热度: GitHub Trending #6 · 633 stars/day**

**是什么：**
whichllm 是一个本地 LLM 基准测试工具，解决一个极其具体但普遍的问题：本地 LLM 的性能排名通常基于理想环境，但你的硬件（GPU、内存、CPU）可能与测试环境完全不同。whichllm 通过在你的实际硬件上运行一组标准化的"时效性感知"基准测试，给出针对你的机器的真实性能排名——不是参数量的排名，是实际 tok/s 和质量分数的排名。一条命令即可运行，自动检测硬件配置并选择匹配的测试集。

**怎么用：**
```bash
# 一键安装运行
npx whichllm

# 或全局安装
npm install -g whichllm
whichllm

# 指定测试范围
whichllm --models llama3.3:70b,qwen2.5:32b,gemma4:27b --benchmark coding

# 输出示例：
# Rank  Model          Tok/s  Quality  VRAM    Fit
# 1     Qwen2.5:32B    45.2   87.3     18.2GB  ✓
# 2     Gemma4:27B      38.7   84.1     15.8GB  ✓
# 3     Llama3.3:70B    12.3   91.5     38.4GB  ✗ (OOM risk)
```

**为什么重要：**
本地 LLM 生态的最大痛点之一是"选择困难"——HuggingFace 上有几千个模型，但没人能告诉你在你的具体硬件上哪个跑得最好。whichllm 用"真机实测+时效性评分"取代"看排行榜猜"，节省开发者数小时甚至数天的试错时间。633 stars/day 的增速说明这个问题比你我想象的更普遍。对于有本地推理需求的企业和独立开发者，这个工具的价值相当于"买硬件之前的决策引擎"。

**讨论现场：**
- Reddit r/LocalLLaMA 讨论："终于有人做了这个。我之前手动测了 8 个模型花了一整个周末"
- GitHub Issues 中有人建议加入功耗和发热测试："笔记本用户需要知道哪个模型不会让风扇起飞"

## 9. "Is Grep All You Need? How Agent Harnesses Reshape Agentic Search" — 重新思考 Agent 搜索架构

**评分: 7.0/10 | 可操作性: 中（论文） | 热度: HN 128 points · 54 评论**

**是什么：**
这篇论文提出了一个反直觉的发现：在 AI Agent 的工具调用场景中，简单的 grep/ripgrep 模式匹配在许多搜索任务上的效率超过了向量搜索（RAG）和 API 调用。论文引入了"Agent Harness"概念——指 Agent 的工具调用框架本身（而非底层模型）对搜索效率的放大效应。实验表明：设计良好的 Agent Harness 可以将搜索步骤减少 40-60%，同时保持或提高结果质量。核心洞察：Agent 的搜索能力瓶颈不在模型，而在工具设计。

**怎么用：**
```bash
# 论文地址
# https://arxiv.org/abs/2606.XXXXX

# 核心工具链建议（论文推荐）
# 1. 优先使用 ripgrep 做代码搜索（而非向量数据库）
rg --type-add 'code:*.{py,js,ts,rs}' -t code "function_name" .

# 2. Agent Harness 设计原则
# - 每个工具应该有且仅有一个清晰职责
# - 工具输出应结构化（JSON Schema），不是自然语言
# - 避免"万能工具"——拆分为小而专注的工具集
```

**为什么重要：**
