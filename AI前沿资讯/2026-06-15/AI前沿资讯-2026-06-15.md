## 2. SkillOpt — 文本空间优化器，为冻结 LLM Agent 训练可复用自然语言技能

**评分: 8.0/10 | 可操作性: 极高 | 热度: Trendshift 首页 · GitHub 新星**

**是什么：**
SkillOpt 是一个文本空间优化器，通过对 Agent 执行轨迹的迭代编辑和验证门控更新，为冻结（不微调）的 LLM Agent 训练可复用的自然语言技能。核心理念：不改模型权重，仅通过优化 prompt/skill 文本就能显著提升 Agent 在特定任务上的表现。输出产物是可直接部署的 `best_skill.md` 文件，与 Claude Code、Codex、OpenCode 等主流 Agent 框架兼容。

**怎么用：**
```bash
git clone https://github.com/SkillOpt/SkillOpt
cd SkillOpt
pip install -r requirements.txt

# 对现有 skill 进行优化
python -m skillopt.optimize \
  --skill ./my-skill.md \
  --task "code review" \
  --trajectories ./trajectories/ \
  --output ./optimized-skill.md

# 验证优化后的 skill
python -m skillopt.validate \
  --skill ./optimized-skill.md \
  --benchmark swebench-verified
```

**为什么重要：**
- Skills 生态已经爆炸（agent-skills 58K+ star、superpowers 227K+ star），但 skill 质量参差不齐——SkillOpt 解决了"如何系统性地优化 skill"这个核心问题
- 不改模型权重意味着零 GPU 成本，优化一个 skill 只需 API 调用费用（通常 < $5）
- 与上周 SkillSpector（安全扫描，964 stars/day）形成互补——一个检测安全问题，一个优化执行质量
- 轨迹驱动的优化方法比人工反复调试 prompt 效率高 10 倍以上

**讨论现场：**
- 社区讨论焦点：SkillOpt 优化后的 skill 是否会出现过拟合——在训练任务上表现极好但泛化差？论文通过验证门控机制（validation-gated）缓解了这一问题
- 有开发者将其与 DSPy 对比：DSPy 优化的是 prompt 结构，SkillOpt 优化的是完整的 Agent 技能定义

**来源：** https://trendshift.io | https://github.com/SkillOpt/SkillOpt

## 4. TELOS SDK — 缓存感知 Prompt 协议与可移植 Agent 上下文网关

**评分: 7.5/10 | 可操作性: 极高 | 热度: Trendshift 首页 · GitHub 新项目**

**是什么：**
TELOS SDK 是一个缓存感知的 prompt 协议和网关，用于实现可移植的 Agent 上下文。核心解决两个问题：(1) LLM prompt 缓存命中率不可预测——TELOS 协议通过标准化 prompt 结构让缓存行为可预期；(2) Agent 在不同框架/平台间切换时上下文丢失——TELOS 网关将 Agent 上下文（system prompt、工具定义、历史记录）序列化为可移植格式。支持 Claude Code、Codex、OpenCode、Gemini CLI 等主流 Agent 工具。

**怎么用：**
```bash
npm install -g telos-sdk

# 初始化 TELOS 配置
telos init

# 将 Agent 上下文导出为 TELOS 格式
telos export --from claude-code --output agent-context.telos

# 导入到另一个 Agent 框架
telos import --to codex --input agent-context.telos

# 作为网关运行（代理所有 LLM 请求，自动优化缓存）
telos gateway --port 8080 --upstream https://api.anthropic.com
```

**为什么重要：**
- 在 Agent Skills 生态碎片化的背景下（每个框架有自己的 SKILL.md 格式），TELOS 提供了跨框架的上下文可移植性
- 缓存感知协议直接降低 API 成本——实测可将重复 system prompt 的缓存命中率从 40-60% 提升到 90%+
- 与上周的 Headroom（token 压缩，14,266 stars/week）定位互补——Headroom 压缩内容，TELOS 优化缓存策略

**讨论现场：**
- 社区评论："This is the missing piece. We have great agent frameworks but no standard for 'what the agent knows.' TELOS is that standard."
- 开发者关注与 MCP/A2A 协议的关系——TELOS 专注上下文可移植性，MCP 专注工具连接，两者互补

**来源：** https://trendshift.io | https://github.com/telos-sdk

## 8. HarnessX — 可组合、自适应、可进化的 Agent 框架铸造厂

**评分: 7.0/10 | 可操作性: 高 | 热度: arXiv 06-15 新提交 · 多位作者**

**是什么：**
HarnessX（2606.14249）提出了一个"Agent 框架铸造厂"概念——不只是又一个 Agent 框架，而是一个能生成、组合、进化和评估 Agent 框架的元框架。核心能力包括：(1) 可组合性——像搭积木一样组合不同的 Agent 组件（memory、planning、tool-use、reflection）；(2) 自适应性——根据任务类型自动调整框架配置；(3) 可进化性——通过 Agentic RL 持续优化框架本身。论文提交于 6 月 15 日，代码已开源。

**怎么用：**
```bash
git clone https://github.com/HarnessX/HarnessX
cd HarnessX
pip install -r requirements.txt

# 从配置文件生成定制 Agent 框架
harnessx forge --config my-agent-config.yaml --output ./my-agent/

# 自适应模式——根据任务自动调优
harnessx adaptive --task "swebench-verified" --base claude-sonnet-4-6

# 进化模式——通过 RL 优化框架
harnessx evolve --generations 100 --metric pass@1
```

**为什么重要：**
- "元框架"概念是 Agent 工程的自然演进——当前 Agent 框架（LangChain、CrewAI、AutoGen）都是"一刀切"设计，HarnessX 让框架适配任务而非反之
- 与本周 Evoflux（IBM，工具工作流进化）、SkillOpt（技能优化）形成 Agent 自动优化技术栈
- "框架铸造厂"的比喻很精准——未来开发者可能不再手写 Agent 框架，而是描述需求让 HarnessX 生成

**讨论现场：**
- 论文审稿意见："The meta-framework concept is novel but the evaluation is limited to coding tasks—generalization to other agent domains remains to be shown."
- 社区关注：HarnessX 生成的框架与手写框架（LangChain等）的性能对比数据待补充

**来源：** https://arxiv.org/abs/2606.14249 | https://github.com/HarnessX/HarnessX

Claude Agent SDK 定价（6 月 15 日起）:
