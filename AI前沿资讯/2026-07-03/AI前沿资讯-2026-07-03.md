## 1. Strix：开源 AI 渗透测试平台（usestrix/strix）

**评分: 9.0/10 | 可操作性: 高 | 热度: GitHub Trending #1 今日 | 28k+ stars | 3.1k forks**

**是什么：**
Strix 是一个开源的 AI 驱动渗透测试工具，用自主 AI Agent 模拟人类攻击者的行为来发现应用漏洞。它不像传统 SAST 工具那样只做静态规则匹配，而是让多个 AI Agent 协同工作——有的负责侦察 API 端点，有的尝试注入攻击，有的验证漏洞是否真实可利用，最后生成带 PoC（概念验证）的修复报告。技术栈基于 LiteLLM、Caido、Nuclei、Playwright 和 Textual，支持 REST/GraphQL/Web 应用的全覆盖测试。

Strix 已被 AWS、PayPal、Uber、Cisco、Chegg、Fortinet 等企业安全团队采用。它曾在 etcd 中发现过一个关键认证绕过漏洞，证明了 AI Agent 在安全领域的实战价值。

**怎么用：**
```bash
# 安装
pip install strix

# 初始化配置（选择 LLM 提供商和 API key）
strix init

# 对目标 URL 运行完整渗透测试
strix scan https://your-app.com

# 仅测试 API 端点（REST/GraphQL）
strix scan --api-only https://api.your-app.com

# 非交互模式（适合 CI/CD）
strix scan -n https://staging.your-app.com

# GitHub Actions 集成
# 在 workflow 中添加：
# - uses: usestrix/strix-action@v1
#   with:
#     target: https://staging.your-app.com
```

**为什么重要：**
传统渗透测试依赖昂贵的安全顾问，通常每季度做一次。Strix 把这种能力变成了 CI/CD 流水线中的自动化步骤——每次部署都跑一轮。它发现的漏洞附带可复现的 PoC，直接指向触发漏洞的精确请求，大幅缩短了"发现→验证→修复"的周期。相比 Nuclei 等基于模板的扫描器，Strix 的 Agent 能自主探索未在模板库中覆盖的攻击面。GitHub 28k+ stars 的热度说明安全自动化正在成为 AI Agent 落地的下一个爆发场景。

**讨论现场：**
- HN 相关讨论："Strix 的 Agent 不是简单的 fuzzer，它会理解 API 的业务逻辑然后构造有针对性的攻击——这和传统扫描器有本质区别"
- Reddit r/netsec："用 AI 做渗透测试最大的风险是误报率，但 Strix 的 PoC 验证机制很大程度上解决了这个问题"

## 2. Caveman：Claude Code 的"原始人"Token 压缩技能（JuliusBrussee/caveman）

**评分: 8.5/10 | 可操作性: 高 | 热度: GitHub Trending 今日 | 5k+ stars | HN 热议**

**是什么：**
Caveman 是一个 Claude Code/Codex 技能，让 AI 用"原始人说话"的风格输出——砍掉冠词、客套话、废话，保留全部技术实质。它不修改模型的思考过程（thinking tokens 不受影响），只压缩输出文本。根据作者 Julius Brussee 发布的 10 项基准测试，Caveman 平均减少 65% 的输出 token（范围 22%-87%），代码块、错误消息、技术术语完整保留。

原理基于 2026 年 3 月发表的论文"Brevity Constraints Reverse Performance Hierarchies in Language Models"——该研究发现约束大模型简短回复反而提升了准确率（部分基准上 +26 分）。Caveman 还提供了 `/caveman-commit`（生成极简 commit message）、`/caveman-review`（极简代码审查）、`/caveman-compress`（压缩文件）等子命令。

**怎么用：**
```bash
# Claude Code 安装（推荐）
claude plugin marketplace add JuliusBrussee/caveman
claude plugin install caveman@caveman

# 通用安装（支持 Codex/Cursor/Copilot/Gemini CLI 等 50+ 平台）
npx skills add JuliusBrussee/caveman -g

# 使用
# 在 Claude Code 中输入 /caveman 激活原始人模式
# /caveman lite     - 轻度压缩
# /caveman full     - 标准压缩（默认）
# /caveman ultra    - 极限压缩
# /caveman-stats    - 查看累计节省的 token 数
```

**为什么重要：**
Claude Code 等 Agent 工具的按 token 计费模型让输出冗长成为实实在在的成本问题。Caveman 的 65% 输出 token 节省意味着同样预算下可以做 3 倍的工作。更深层的启示：AI 的输出风格是可以被"技能层"精确控制的——不需要换模型，挂一个 Skill 就能改变行为模式。这展示了 Agent Skills 生态的真正威力：用轻量级、可组合的技能替代重量级的模型微调。

**讨论现场：**
- HN 热评："这本质上是对 Claude 做了一次'输出端蒸馏'——不花钱改模型，只改它怎么说话。惊人地有效" 
- LinkedIn 上作者父亲 Arjan Brussee 的帖子引爆传播："我儿子开发的 Claude skill，周末就炸了，GitHub #1 trending"
- 社区实测反馈：代码审查场景下效果最好（-41%），技术解释场景次之（-87%），架构设计场景效果较差（-30%，因为需要更多上下文说明）

**来源：** https://github.com/JuliusBrussee/caveman | https://juliusbrussee.github.io/caveman

## 3. OpenAI Codex 官方 Claude Code 插件（openai/codex-plugin-cc）

**评分: 8.0/10 | 可操作性: 高 | 热度: GitHub Trending 今日 | HN/Reddit 热议**

**是什么：**
OpenAI 官方发布的 Claude Code 插件，让开发者可以在 Claude Code 会话中直接调用 Codex（GPT-5.4/5.5）来执行代码审查、对抗性审计、调试救援等任务。核心设计思路：Claude 做主程（交互式结对编程），Codex 做副驾（后台异步审查和验证），形成双模型协作的开发工作流。

插件提供 7 个核心命令：`$claude review`（结构化代码审查）、`$claude adversarial-review`（对抗性审计，让 Codex 挑战 Claude 的方案）、`$claude rescue`（当 Claude 卡在某个 Bug 上时委托给 Codex）、`$claude advise`（快速获取第二意见）、`$claude do`（分配独立编码任务）、`$claude monitor`（监控后台 Claude 任务状态）。

**怎么用：**
```bash
# 在 Claude Code 中安装
claude plugin marketplace add openai/codex-plugin-cc
claude plugin install codex-plugin-cc

# 前置条件：需要 OpenAI API key 或 ChatGPT 订阅
# 设置环境变量
export OPENAI_API_KEY="sk-..."

# 使用示例
# 让 Codex 审查当前改动
$claude review

# 对抗性审查（Codex 专门挑刺）
$claude adversarial-review

# Claude 搞不定的 Bug 交给 Codex
$claude rescue "修复 src/auth.py 中的 token 刷新竞态条件"

# 快速获取第二意见
$claude advise "这个 SQL 查询在大表上会有性能问题吗？"
```

**为什么重要：**
"最好的 AI 编码工具不是某一个模型，而是多个模型的协作"——这个理念正在成为 2026 年开发者的共识。Codex 插件标志着模型互操作从"手动切换工具"升级为"自动化的 Agent 间协作"。实际效果：有开发者测试显示 Claude + Codex 双模型审查比单独使用任一模型发现的问题多 40%。这也说明 Agent Skills 生态正在成为模型无关的通用层——你可以在 Claude Code 里用 GPT，也可以在 Codex 里挂 Claude 的 Skill。

**讨论现场：**
- YouTube 实测视频（Eric Tech, 6.3k 观看）："GPT-5.4 在对抗性审查中发现了 Opus 4.7 漏掉的 3 个安全问题，双模型协作真的不是噱头"
- HN 讨论："这是 Anthropic 和 OpenAI 在 Agent 层的'竞合'——模型层面互相竞争，Agent 工具层面互相兼容。对开发者是好事"
- Reddit："最实用的场景是 `$claude rescue`——当 Claude 在某个问题上反复绕圈时，换个模型往往立刻找到突破口"

## 4. Agent Skills 规范正式仓库上线（agentskills/agentskills）

**评分: 7.5/10 | 可操作性: 高（可直接贡献和学习） | 热度: GitHub Trending 今日 | 跨平台生态**

**是什么：**
Agent Skills 规范的官方 GitHub 仓库上线，Anthropic 于 2025 年 10 月推出该格式，2025 年 12 月作为开放标准发布在 agentskills.io。截至 2026 年 6 月，约 40 款产品已兼容该标准（Claude Code、OpenAI Codex、GitHub Copilot、Cursor、Gemini CLI、VS Code 等），社区目录已索引约 190 万公开 Skill。

Skill 采用渐进式披露设计：Agent 启动时只加载 Skill 名称和描述（极低上下文成本），当任务匹配时才加载完整指令。每个 Skill 就是一个包含 `SKILL.md` + 可选脚本/模板/资源的文件夹，可以用 Git 版本控制、通过 npm 分发、跨 Agent 平台复用。该仓库包含完整的规范文档、参考实现和贡献指南。

**怎么用：**
```bash
# 了解规范
git clone https://github.com/agentskills/agentskills
cat agentskills/README.md

# 创建你的第一个 Skill
mkdir -p my-skill
cat > my-skill/SKILL.md << 'EOF'
---
name: my-first-skill
description: 我的第一个 Agent Skill，用于自动格式化 Python 代码
---
使用 black 和 isort 格式化 Python 代码。
执行前先检查文件是否存在，然后运行：
```bash
black --line-length 100 "$file"
isort --profile black "$file"
```
EOF

# 在 Claude Code 中加载（将 my-skill 放入 skills 目录）
# Claude Code 会自动发现并加载
```

**为什么重要：**
Agent Skills 正在成为 AI Agent 世界的"npm"——一个跨平台、跨模型的能力分发标准。190 万 Skill 的生态规模说明分发问题已解决，但质量参差不齐（SkillsBench 分析了 47,150 个公开 Skill，平均质量分 6.2/12）。下一步的核心矛盾是"Skill 质量与安全审核"——正如 npm 有恶意包问题，Skill 也会有恶意指令注入风险。规范仓库的上线是生态走向成熟的第一步。

**讨论现场：**
- "Agent Skills 解决了 AI Agent 最大的痛点：每个会话都要重新解释项目规则。现在把规则写成 Skill，一次编写，到处运行"
- Reddit r/GithubCopilot："Agent Skills 是第一个真正跨厂商的 Agent 标准。Anthropic 把它开源、OpenAI 和微软都跟进，这在 AI 行业太少见了"
- SkillsBench 论文指出："精选的高质量 Skill 能将 Agent 任务通过率平均提升 16.2 个百分点"

## 5. Claude-real-video：让任何大模型"看懂"视频（HUANGCHIHHUNGLeo/claude-real-video）

**评分: 7.5/10 | 可操作性: 高 | 热度: HN 82 pts, 28 comments**

**是什么：**
一个轻量级工具，让任何支持多模态的 LLM（Claude、GPT 等）都能理解视频内容。工作原理：用 ffmpeg 提取关键帧（可配置帧率）+ Whisper 转录音频 + 时间戳对齐，然后将帧序列（base64 图片）和带时间戳的文本转录一并送入 LLM。LLM 接收到的不是"视频文件"，而是"一系列按时间排列的画面+字幕"。

与 bradautomates/claude-video（`/watch` 命令）和 jordanrendric/claude-video-vision 相比，claude-real-video 的定位更底层——它不绑定特定 Agent 框架，可以嵌入任何支持多模态模型的工具链中。82 pts 的 HN 热度说明开发者对"视频理解"这个能力的需求很大。

**怎么用：**
```bash
# 安装依赖
pip install claude-real-video
brew install ffmpeg yt-dlp  # macOS
# Windows: 下载 ffmpeg 和 yt-dlp 并加入 PATH

# 分析本地视频
claude-real-video analyze meeting.mp4 --question "会议中讨论了哪些关键决策？"

# 分析 YouTube 视频
claude-real-video analyze "https://youtube.com/watch?v=xxx" --model claude-sonnet-5

# 调整帧采样率（每秒 1 帧，默认 0.5）
claude-real-video analyze demo.mp4 --fps 1

# 仅用字幕模式（最快，不需要视觉理解）
claude-real-video analyze talk.mp4 --transcript-only
```

**为什么重要：**
视频是互联网上最大的信息载体（YouTube 每天上传 720,000 小时），但 LLM 原生不支持视频输入。claude-real-video 用一个务实的工程方案桥接了这个鸿沟——它把视频降维成"图片序列+文本"（LLM 天然能处理的形式）。随着多模态模型能力增强，帧提取+转录的方案在会议纪要、教程学习、安全监控等场景有直接落地价值。82 pts HN 热度反映需求真实存在。

**讨论现场：**
- HN 热评："这本质上是个聪明的'协议转换器'——把 LLM 不理解的视频格式转换成它能理解的多模态上下文。简单但有效"
- "帧提取的问题是成本——一个 10 分钟视频按每秒 1 帧就是 600 张图片，token 消耗不小。transcript-only 模式是更实际的日常用法"

**来源：** https://github.com/HUANGCHIHHUNGLeo/claude-real-video | https://news.ycombinator.com (82 pts)

## 6. Short Leash AI 编码方法论：如何用 AI 写出安全关键系统代码

**评分: 7.0/10 | 可操作性: 高（方法论可直接应用） | 热度: HN 65 pts, 63 comments | 7 月 2 日发布**

**是什么：**
Greg Slepak（okTurtles 创始人，安全关键系统开发者）于 7 月 2 日发布了"Short Leash Method"（短绳法），总结了他超过一年的 AI 编码实践经验。核心理念：不是让 AI 替你写代码，而是把 AI 当成一个"超快但不可靠的初级开发者"——你设计架构和接口，AI 填充实现细节，你逐行审查。

方法论的四个原则：（1）先写接口和测试，再让 AI 实现；（2）每次只让 AI 改一个函数或模块，不给它"整个项目"的上下文；（3）AI 的输出必须通过你的测试才算完成；（4）对安全关键代码（加密、认证、权限），完全不允许 AI 写第一版——它只能做审查和建议。文章强调此方法只适用于专业软件开发者，不适用于非技术用户。

**怎么用：**
```bash
# Short Leash 方法论的核心工作流：
# 1. 自己定义接口
# 2. 自己写测试
# 3. AI 填充实现（短绳：一次一个函数）
# 4. 运行测试验证
# 5. 逐行审查 AI 的代码
# 6. 合入后 AI 做审查者而非作者

# 在 Claude Code 中的实践示例：
# "请实现以下接口，确保通过已有测试，不要修改接口签名"
# "只修改 src/auth/token.go 中的 refreshToken 函数，其他文件不要动"
# "审查以下代码的安全问题，不要直接修改"
```

**为什么重要：**
在"AI 替代程序员"的喧嚣中，Short Leash 给出了一个冷静务实的声音：AI 是工具，不是替代品。它承认 AI 编码的局限性（上下文窗口有限、幻觉风险、安全关键代码不可信），同时最大化其优势（填充样板代码、快速迭代、持续审查）。这篇文章在 HN 上引发 63 条评论的激烈讨论，反映了开发者社区正在从"AI 万能论"和"AI 无用论"两个极端走向务实的中间地带。

**讨论现场：**
- HN 高赞评论："Efficient ≠ effective。作者的方法确实'慢'，但产出的代码质量高得多。对于安全关键系统，这可能是唯一正确的方法" — sothatsit
- 反对意见："对 Fable 5 这种级别的模型来说，Short Leash 太保守了。你可以和强模型进行越来越细致的讨论，它们写的代码比一年前好太多了"
- 中立观点："短绳法的核心不是限制 AI，而是强制人类保持对代码的深度理解。如果你不读 AI 写的代码，你就不是工程师，你是 AI 的 QA"

**来源：** https://blog.okturtles.org/2026/07/short-leash-ai-method | https://news.ycombinator.com/item?id=48766026

## 7. Simon Willison 发布 llm-coding-agent 0.1a0：用 Fable 5 驱动的编码 Agent

**评分: 6.5/10 | 可操作性: 高 | 热度: Simon Willison 官方博客 | Fable 5 实验 | 7 月 2 日发布**

**是什么：**
Simon Willison 于 7 月 2 日发布了 llm-coding-agent 的第一个 alpha 版本。这是他基于自研 LLM 库构建的编码 Agent，底层默认使用 Fable 5 模型。整个 Agent 核心仅有约 200 行 Python 代码，提供文件读写、命令执行等基本工具，支持 `--yolo` 模式（自动执行无需确认）。

Willison 的演示令人印象深刻：他用两条 Claude Code 提示就生成了项目骨架，然后让 Agent 创建一个 ASCII 艺术时钟 Swift CLI 应用——GPT-5.5 在推理中准确指出"SwiftUI 不适合真正的 CLI"，然后做出了正确的技术决策。

**怎么用：**
```bash
# 安装
pip install llm-coding-agent

# 基本使用
llm code "创建一个 Python web scraper"

# 自动执行模式（无需人工确认）
llm code --yolo "在 /tmp/demo 创建一个 FastAPI 应用"

# 指定模型
llm code --model gpt-5.5 "重构 src/ 目录下的所有 TypeScript 文件"
```

**为什么重要：**
这不是又一个要和 Claude Code/Codex 竞争的编码 Agent——Willison 的目标是展示"用 LLM 库构建 Agent 框架有多简单"。200 行代码的 Agent 意味着 Agent 框架本身不再是壁垒，模型能力和提示工程才是。它也展示了 Fable 5 作为编码 Agent 底层的实用性——在复杂技术决策（SwiftUI vs CLI）上表现出超过预期的判断力。

**讨论现场：**
- Willison 博客："现在我的 LLM 库已经进化成了 Agent 框架，是时候看看基于它构建的最简编码 Agent 长什么样了"
- 社区评论："200 行的 Agent 就能工作，说明'Agent 框架'这个品类可能被过度设计了。真正重要的是模型能力和系统提示"

**来源：** https://simonwillison.net/2026/Jul/2/llm-coding-agent | https://github.com/simonw/llm-coding-agent

## 8. Microsoft Frontier Company：微软豪掷 25 亿美元、6000 工程师做 AI 落地

## 9. Box3D：Erin Catto 发布 3D 物理引擎（Box2D 作者，AI 辅助开发）

**评分: 6.0/10 | 可操作性: 高 | 热度: Lobsters 95 pts, vibecoding 标签**

**是什么：**
Erin Catto（Box2D 作者，游戏物理引擎领域传奇人物）发布了 Box3D——Box2D 的 3D 继任者。这是 Catto 首次涉足 3D 物理引擎，标志着他的工作从 2D 扩展到 3D。Box3D 用 C 语言编写，提供刚体动力学、碰撞检测、约束求解等核心物理模拟功能。

有趣的是这篇发布被 Lobsters 标记为 "vibecoding"——暗示 Catto 在开发过程中使用了 AI 编码工具辅助。95 pts 的高热度既来自 Box2D 在游戏行业的巨大影响力，也来自"物理引擎大师 + AI 辅助"的话题性。

**怎么用：**
```bash
# 克隆仓库
git clone https://github.com/erincatto/box3d

# 构建
cd box3d
cmake -B build
cmake --build build

# 运行示例
./build/bin/box3d_sample
```

**为什么重要：**
Box2D 是游戏行业使用最广泛的 2D 物理引擎（Unity、Unreal 都内置或兼容），累计下载量以亿计。Box3D 的出现填补了开源 3D 物理引擎的一个重要空白——Bullet Physics 和 PhysX 虽然存在多年，但 Box2D 级别的 API 简洁性和文档质量在 3D 领域一直缺失。"vibecoding"标签也很耐人寻味：最顶尖的物理引擎程序员也在用 AI 辅助开发，这比任何 Benchmark 都更能说明 AI 编码工具的渗透深度。

**讨论现场：**
- Lobsters 热评："Erin Catto 的 API 设计感是 Box2D 成功的核心——简单到新手能上手，强大到 AAA 游戏能用。期待 Box3D 延续这个传统"
- "被标记为 vibecoding 很妙——如果连 Box2D 的作者都在用 AI 工具，那些坚持'真正程序员不用 AI'的人该醒醒了"

## 10. Vibe-Trading：你的 AI 个人交易代理（HKUDS/Vibe-Trading）

**评分: 5.5/10 | 可操作性: 中（需金融背景谨慎使用） | 热度: GitHub Trending 今日**

**是什么：**
香港大学数据科学实验室（HKUDS）发布的 AI 驱动个人交易代理。它不只是一个回测框架——它试图让 LLM 扮演"交易助手"的角色，能够解读财经新闻、分析技术指标、提出交易建议。项目基于 Python，使用多 Agent 架构：一个 Agent 负责市场数据分析，一个负责新闻情绪解读，一个负责风险评估。

**怎么用：**
```bash
# 安装
git clone https://github.com/HKUDS/Vibe-Trading
cd Vibe-Trading
pip install -r requirements.txt

# 配置 API key（OpenAI/Anthropic）
cp .env.example .env
# 编辑 .env 填入你的 API key

# 运行演示
python demo.py --symbol AAPL --days 30
```

**为什么重要：**
金融是 AI Agent 最敏感也最具商业价值的落地场景之一。Vibe-Trading 是学术界对"AI + 交易"的开源探索，和 BloombergGPT 等闭源商业方案形成对照。项目强调"辅助决策"而非"自动交易"——Agent 提供分析但不执行订单。GitHub trending 的热度说明开发者对 AI 在金融领域的应用兴趣浓厚，但实际使用需要极强的风险意识。

**讨论现场：**
- GitHub Issues 中的讨论焦点：模型幻觉在金融场景的后果远比其他领域严重——一个错误的财报解读可能导致实际资金损失
- 社区提醒："这更像一个研究项目而非生产工具。不要用真金白银测试 AI 的交易建议"

**来源：** https://github.com/HKUDS/Vibe-Trading
