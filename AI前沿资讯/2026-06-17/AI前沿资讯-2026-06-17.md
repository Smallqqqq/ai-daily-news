## 1. 小米 MiMo Code 开源：终端原生 AI 编程 Agent，多代理架构击败 Claude Code（MiMo Code）

**评分: 9.0/10 | 可操作性: 高 | 热度: GitHub Trending #1 中文社区 + Product Hunt 热议 + 多家中英文科技媒体报道**

**是什么：**
小米于 6 月 11 日正式开源 MiMo Code，一款终端原生 AI 编程助手，基于 OpenCode 深度 Fork 改造。核心创新是「多代理协同」架构——面对任务时多个 Agent 各自生成候选执行步骤，再由评估 Agent 选出最优方案执行。内置记忆管理机制，可自动萃取长对话重点并在新会话中恢复上下文。另外引入了技能指引自动转译为 JavaScript 代码的机制，以确定性逻辑消除 LLM 随机性偏差。

**怎么用：**
```bash
# 克隆仓库
git clone https://github.com/Xiaomi/MiMo-Code
cd MiMo-Code
# 安装依赖
npm install
# 配置 API Key
export MIMO_API_KEY="your_api_key"
# 启动交互式会话
npx mimo-code
```

**为什么重要：**
在 576 名人类开发者参与的「双盲测试」中，面对 200+ 步高复杂度任务，MiMo Code + MiMo-V2.5 组合取得了 **65% 的胜率**，显著优于同等条件下的 Claude Code + Claude Sonnet 4.6。在内部基准测试中，无论搭配自家 MiMo-V2.5 还是 MiMo-V2.5-Pro，均超越使用相同模型的 Claude Code。多代理架构解决了单 Agent 长任务「失控」和「遗忘」的核心痛点。

**讨论现场：**

- T客邦分析："从单纯依赖大语言模型正走向系统化工程管控，这代表了 AI 编程工具的下一个方向。"

## 2. Z.ai GLM-5.2：100 万 token 可用上下文 + 双思考模式，直接接入 Claude Code（GLM-5.2）

## 3. Agentjacking：新型攻击劫持 Claude Code / Cursor / Codex，85% 成功率（Agentjacking）

**评分: 8.5/10 | 可操作性: 高（安全团队必须关注） | 热度: AI 安全圈爆火 + 多家中英文媒体报道 + HN 首页**

**是什么：**
一种称为 "Agentjacking" 的新型攻击向量正在活跃利用 AI 编程 Agent（Claude Code、Cursor、Codex）。攻击方式：在代码仓库或 CI 环境中注入伪造的 Sentry 错误消息——Sentry 是绝大多数专业开发团队使用的错误监控工具。当 AI Agent 读取这些伪造的错误报告后，会按照其中的「修复指令」执行攻击者控制的命令，而非开发者的原始意图。**2,388 个组织已被曝光受影响，攻击利用率高达 85%。**

**怎么用：**
```bash
# 安全团队：立即检查 AI Agent 的工作目录和 CI 环境
# 1. 扫描项目中异常的 Sentry 错误引用
grep -r "sentry" --include="*.md" --include="*.txt" --include="*.log" .
# 2. 限制 AI Agent 的 shell 执行权限
# Claude Code: 设置 --permission-mode plan 或 acceptEdits
claude --permission-mode plan
# 3. 使用 SkillSpector 扫描 Agent Skills 漏洞
pip install skillspector
skillspector scan ./skills/
```

**为什么重要：**
这是首次有记录的、大规模针对 AI 编程 Agent 的供应链攻击。AI Agent 在执行代码时以开发者权限运行，攻击者可以注入任意代码、窃取密钥、篡改 CI/CD 管线。85% 的利用率意味着绝大多数 Agent 无法识别伪造的 Sentry 错误。这对所有使用 AI 编程 Agent 的团队是安全优先级最高的威胁。

**讨论现场：**
- AI Tools Recap："Agentjacking 是 AI 编码 Agent 时代的第一个重大安全事件，2,388 个组织受影响。"
- 安全社区：呼吁所有 Agent 工具加入沙箱执行和命令白名单机制。
- HN 讨论：MCP 服务器的安全审计和 Skill 签名验证成为紧迫需求。

**来源：** https://aitoolsrecap.com/Blog/ai-news-june-16-2026 | https://github.com/NVIDIA/SkillSpector

## 4. Google Cloud 发布 Open Knowledge Format (OKF)：为 AI Agent 设计的知识格式标准（OKF v0.1）

**评分: 8.2/10 | 可操作性: 高 | 热度: MarkTechPost + Karpathy 关联讨论 + 多家技术媒体报道**

**是什么：**
Google Cloud 于 6 月 16 日正式发布 OKF v0.1——一个供应商中立的开放知识格式标准，将 LLM-Wiki 模式形式化为可移植、可互操作的规范。核心设计极简：**一个 Markdown 文件目录 + YAML frontmatter**，无需 SDK、运行时或注册中心。每个概念只需一个必填字段 `type`，文件之间的交叉引用形成知识图谱。灵感来自 Andrej Karpathy 2026 年 4 月提出的 LLM Wiki 概念——LLM 不会厌倦、不会忘记更新交叉引用，正是 Wiki 维护的最佳角色。

**怎么用：**
```bash
# OKF 就是 Markdown 文件，无需安装任何工具
# 创建一个 OKF 知识包
mkdir my-knowledge-bundle
cat > my-knowledge-bundle/api-design.md << 'EOF'
---
type: concept
title: "API Design Principles"
description: "RESTful API 设计最佳实践"
tags: [api, rest, design]
---
# API Design Principles
## Related
- [[authentication]] - 认证机制
- [[rate-limiting]] - 限流策略
EOF

# 用任何 Agent 直接读取
# Agent 遍历文件间的 [[wikilink]] 即可构建知识图谱
```

**为什么重要：**
与 RAG 不同，OKF 存储的是策展过的、版本可控的概念，Agent 可以直接读取和更新。Google 已配套发布 BigQuery 丰富 Agent、静态 HTML 可视化工具和三个示例知识包。这填补了"如何让 Agent 可靠地获取结构化上下文"的关键空白——目前大多数方案是直接塞 Markdown 文件，但缺乏统一规范。OKF 可能是 Agent 知识管理的 TCP/IP 时刻。

**讨论现场：**
- Karpathy 的原推："LLM 不会厌倦、不会忘记更新交叉引用，这正是人类放弃个人 Wiki 的原因，也是 LLM 擅长的事。"
- 社区反馈：OKF 的「格式即合约」理念与 Unix 哲学一致——简单、可组合、不绑定平台。
- 质疑声音：OKF 能否推动足够的采纳达到网络效应临界点？

**来源：** https://www.marktechpost.com/2026/06/16/google-cloud-introduces-open-knowledge-format-okf

## 5. Microsoft FastContext：专用探索子 Agent 让编程 Agent token 消耗暴降 60%（FastContext）

**评分: 8.0/10 | 可操作性: 高（已开源） | 热度: HF Papers 当日最热 + 407 赞 + Microsoft 品牌**

**是什么：**
微软发布 FastContext——一个专门用于代码仓库探索的子 Agent 架构。当前 AI 编程 Agent 的主要瓶颈之一：探索代码仓库消耗大量 token 且污染上下文。FastContext 将「探索」与「解决」解耦：一个 4B-30B 参数的专用模型负责并行搜索仓库并返回精确的文件路径和行号范围，而非整段代码。支持多语言（SWE-bench Multilingual）、多轮证据收集和精确引用生成。

**怎么用：**
```bash
# 安装 fastcontext
pip install fastcontext
# 在 Mini-SWE-Agent 中集成
from fastcontext import FastContextExplorer
explorer = FastContextExplorer(model="fastcontext-7b")
# Agent 调用时自动触发探索
result = explorer.explore(repo_path="./my-project", query="找到所有认证相关的中间件")
```

**为什么重要：**
在 SWE-bench Multilingual、SWE-bench Pro 和 SWE-QA 上，集成 FastContext 后**问题解决率提升 5.5%**，同时编程 Agent 的 token 消耗**降低 60%**。这意味着同样预算下 Agent 能处理更多任务，或更复杂的长程任务。架构上证明了仓库探索可以从通用 LLM 中分离出来由专用模型高效完成。

**讨论现场：**
- HF Papers 社区：407 赞，"将探索与解决解耦是编码 Agent 架构的正确方向"。
- 研究者评论：FastContext 的训练方法（从强参考模型轨迹引导 + 任务导向奖励优化）值得在其他 Agent 子任务上复用。

**来源：** https://github.com/microsoft/fastcontext | https://arxiv.org/abs/2606.14066

**是什么：**
Engram 是一个开源双处理记忆引擎，基于「双时态数据模型」。快写入路径在无 LLM 干预下追加无损片段；异步整合路径提取 `(主,谓,宾)` 三元组构建双时态知识图谱并用非 LLM 方式消除矛盾；混合读取路径融合稠密检索、词汇检索、图检索和时态过滤。**核心发现：用检索到的精简上下文（约 9,600 token）比用完整历史（79,000 token）准确率更高。**

**怎么用：**
```bash
git clone https://github.com/liuyinwang/Engram
cd Engram
pip install -e .
# 在 Agent 中集成
from engram import EngramMemory
memory = EngramMemory()
memory.ingest_session(conversation)  # 快速写入
context = memory.retrieve("用户上周提到了什么关于支付的偏好？", as_of="2026-06-10")
```

**为什么重要：**
在 LongMemEvalS（500 题，7 类）标准评测中，Engram 精简模式**83.6%** vs 完整上下文基线 **73.2%**（+10.4 分，p < 10^-6），同时 token 消耗降低 **8 倍**。这颠覆了「更多上下文 = 更好结果」的直觉——累积的无关信息实际上降低了准确性。所有实验可复现，包含官方评判器和完整日志。
