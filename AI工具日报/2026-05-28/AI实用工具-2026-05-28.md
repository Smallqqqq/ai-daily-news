# AI 实用工具 · 2026-05-28

  今天三个，都是直接插进开发流程就能用的。核心方向：前端 UI 感知、自动化测试、代码变更影响分析。

---

## ViewGraph — 给 AI Agent 装上眼睛

  前端开发里有个长期没人认真解决的问题：Claude Code 改了组件代码，但它完全不知道页面现在长什么样——哪个按钮点不了，布局有没有跑偏，CSS 改错了没有。你要么自己截图粘贴进对话，要么靠文字描述，全是低效的人工中继。ViewGraph 就是专门解这个的。

  原理不复杂：装一个 Chrome/Firefox 扩展，点一下工具栏图标，当前页面的完整结构化快照（DOM 树 + CSS 样式 + 元素边界框 + 可交互元素列表 + 无障碍属性 + 网络错误 + 控制台报错）就被捕获成 JSON 文件，通过 MCP 暴露给 Claude Code、Cursor、Windsurf 等任何支持 MCP 的 agent。Agent 可以直接查询「这个页面有哪些按钮」「这个 input 有没有 aria-label」「上次快照和这次有什么结构变化」。

  有几个数据值得关注：ViewGraph v3 格式快照中位数 1930 tokens，直接喂原始 HTML 约需 17000 tokens，节省 88%；selector 准确率 99.7%，`data-testid` 召回率 100%（基于 48 个真实网站测试）。研究数据显示，给 agent 提供结构化布局信息相比给原始无障碍树，web 任务成功率高 14-17 个百分点。另外还有 `@viewgraph/playwright` 包，可以在现有 E2E 测试里直接嵌入 `viewgraph.capture('checkout-page')`，下次测试自动对比两次快照检测 UI 回归。

```json
// 加到 Claude Code 的 MCP 配置（~/.claude.json）
{
  "mcpServers": {
    "viewgraph": { "command": "npx", "args": ["-y", "@viewgraph/core"] }
  }
}
```

  浏览器扩展建议从 GitHub Releases 下载，比 Chrome 商店版本更新更快：https://github.com/sourjya/viewgraph/releases/latest

### 同类工具对比

| 工具 | 安装方式 | 核心能力 | 与 ViewGraph 的区别 | 热度 |
|------|---------|---------|-------------------|------|
| **ViewGraph**（推荐）| npx + 浏览器扩展 | DOM 结构化捕获 → MCP，97% token 节省 | — | ⭐ 6 |
| Playwright MCP | npm | 浏览器自动化（点击/截图/导航）| 主动操控浏览器；ViewGraph 是被动感知，两者互补 | ⭐ 3.2k |
| Chrome DevTools MCP | npm | DevTools 协议全功能暴露 | 功能更全但更复杂；ViewGraph 专注 UI 结构感知 | ⭐ 1.8k |
| Stagehand | npm | AI 驱动的浏览器自动化 | 面向测试自动化执行；ViewGraph 面向开发时 UI 理解 | ⭐ 9.7k |

  热度最高的 Stagehand 和 Playwright MCP 都是主动执行工具（agent 驱动浏览器做事），ViewGraph 是感知工具（让 agent 能看到当前 UI 状态）——定位不同，实际上可以组合用：ViewGraph 告诉 agent 页面现状，Playwright MCP 负责操作。

⭐ 6 · v0.9.9 · AGPL-3.0 · [GitHub](https://github.com/sourjya/viewgraph) · [文档](https://chaoslabz.gitbook.io/viewgraph)

## tailtest — Claude 写完代码自动跑测试

  使用 Claude Code 的一个常见状态：让它写完函数或 API，然后要手动提示「帮我写测试」，或者等 CI 跑完才知道有没有问题。tailtest 把这个循环变成全自动。

  它是一个 Claude Code 插件，作为 PostToolUse hook 运行。每当 Claude 写完或修改一个文件，tailtest 立即基于文件内容生成「业务场景描述」，用项目里已有的测试框架（pytest / vitest / jest / go test / rspec / phpunit / cargo test 等）执行。全部通过静默，有失败弹一行报错并询问是否修复，不产生额外噪音。

  测试生成方式是这个工具比较有意思的地方。它不是照抄函数签名生成「测试 add(1,2) 返回 3」这种没用的用例，而是生成真实业务行为场景。比如 Claude 写了一个计费服务，tailtest 生成的是：「创建金额 800 美元的发票，当前信用额度 1000 美元——验证成功；创建金额 1200 美元的发票——验证被拒并返回正确错误码」。这类边界场景恰恰是手工最容易漏的，也是线上 bug 的高发区。

  支持 Python / TS / JS / Go / Ruby / PHP / Java / Rust，单体仓库（Turborepo、Nx、pnpm workspaces）自动识别。第一次装上时自动扫代码库，按 git commit 频率和路径权重排序，挑出最重要的 7 个文件先补一轮测试。

```bash
claude plugin marketplace add avansaber/tailtest
claude plugin install tailtest@avansaber-tailtest
# 重启 Claude Code 即生效，无需其他配置
```

### 同类工具对比

| 工具 | 触发方式 | 生成场景质量 | 支持语言 | 热度 |
|------|---------|------------|---------|------|
| **tailtest**（推荐）| Claude 写完文件自动触发 | 业务行为场景，非函数签名 | 8种 + 单仓 | ⭐ 5 |
| nit | 手动 `nit pick` | 较全面，支持覆盖率目标 | 8种，多框架 | ⭐ 0（2月停更）|
| testpilot-ai | 手动 CLI 执行 | 自愈循环（生成→运行→修复）| JS/TS 4框架 | ⭐ 1 |
| GitHub Copilot 测试生成 | 手动右键菜单 | 函数级，质量一般 | 多语言 | 闭源付费 |

  tailtest 最大的差异点是**零干预自动化**：装上就忘掉，它在后台默默跑，只在出问题时打扰你。nit 功能更全但需要手动跑且 2 月已停更；testpilot-ai 的自愈循环值得关注但 stars 极低。tailtest 星数也少，但它的定位很精准，且与 Claude Code 工作流结合最紧密。

⭐ 5 · v3.0.0 · Apache 2.0 · [GitHub](https://github.com/avansaber/tailtest)

## LAIN — 改代码之前先搞清楚影响范围

  一个用 Rust 写的持久化代码智能 MCP 服务器。核心是回答一个大多数 AI agent 回答不好的问题：**「如果我改这个函数，会影响到哪些地方？」**

  LAIN 启动后在项目里建一张知识图谱：通过 LSP 和 tree-sitter 提取所有符号及调用关系，叠加 git 协变更历史（哪些文件总是在同一次 commit 里一起改），加上可选的本地语义嵌入（ONNX 模型，不联网）。图谱持久化在 SQLite 里，跨 session 保留，文件变更时通过 watcher 增量更新。

  最实用的几个工具：`get_blast_radius`（给定符号，返回所有受影响调用链）、`get_coupling_radar`（历史上总是一起改的文件，重构时需同步考虑）、`find_anchors`（最被调用的核心符号，快速理解项目骨架）、`run_build`（构建失败时附上调用者上下文，而不只是报错行）。

  有人跑过小规模 A/B 对比（同一个真实 bug，各 20 次）：有 LAIN 中位耗时 39.3 秒，没有 LAIN 是 54.1 秒，节省约 27%；每次请求少用约 6000 input tokens。

```bash
# 一键安装，自动检测 Claude Code / Cursor / Windsurf 并写入 MCP 配置
curl -fsSL https://raw.githubusercontent.com/spuentesp/lain/main/install.sh | bash
source ~/.zshrc

# macOS 也可以用 Homebrew
brew tap spuentesp/lain https://github.com/spuentesp/lain
brew install lain && lain init
```

### 同类工具对比

| 工具 | 实现方式 | 核心能力 | 持久化 | 热度 |
|------|---------|---------|--------|------|
| **LAIN**（推荐）| Rust + LSP + tree-sitter + git 历史 | 爆炸半径分析、协变更相关性、语义搜索 | SQLite 跨 session | ⭐ 7 |
| Continuum MCP | Rust + LSP | 共享代码图谱，多 agent 协作 | SQLite | ⭐ 0（新项目）|
| CodeGraph | 各语言 LSP 桥接 | 符号定位、引用查找 | 内存，不持久 | ⭐ 400+ |
| Sourcegraph Cody | 云服务 + IDE 插件 | 全库语义搜索，代码导航 | 云端 | 闭源，有免费层 |

  同类工具里 LAIN 最独特的是 **git 协变更历史分析**（哪些文件总是一起改）——这是纯 LSP 工具没有的，能发现逻辑上耦合但代码上没有直接调用关系的文件。Sourcegraph Cody 更成熟但要联网且闭源；LAIN 完全本地，代码不出机器。

⭐ 7 · v0.3.0 · MIT · [GitHub](https://github.com/spuentesp/lain)
