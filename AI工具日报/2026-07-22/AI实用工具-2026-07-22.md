## Forge — 给本地 8B 模型套上五层铠甲，工具调用成功率从 53% 飙到 99.3%，反超裸奔的 Claude Sonnet

  用本地模型跑 Agent 的开发者都有一个共同的痛：模型不是不会调工具，是调不对。函数名拼错一个字母、把 JSON 写在 code fence 里而不是 `tool_calls` 字段里、在多步工作流的第三步忽然忘了自己该干什么、或者在应该调工具的时候输出了一段纯文本——任何一个都能让整个 Agent 管线当场崩溃。这不是一个模型能力问题，而是一个基础设施问题。

  Forge 是德州仪器 AI 总监 Antoine Zambelli 用 Python 写的一套可靠性层，专门解决这个问题。核心思想很直接：模型已经知道怎么调工具了，它只是缺少一张安全网。Forge 给了它五张：

  1. **Rescue parsing（救援解析）**：模型输出了格式错误的工具调用（Mistral 的 `[TOOL_CALLS]` 格式、Qwen 的 `<tool_call>` XML、裸 JSON 包在 markdown fence 里）？Forge 不报错，直接从文本中提取出结构化的调用然后重写成标准 OpenAI `tool_calls` 格式。
  2. **Retry nudges（重试推动）**：校验没通过？Forge 不在第一时间就放弃，而是把错误信息用标准的 "tool failed" 通道喂回模型让它重试——用模型自己理解的语言告诉它哪里写错了，而不是扔一个 HTTP 500。
  3. **Step enforcement（步序执行）**：你可以在 workflow 定义里声明 `required_steps`、`prerequisites` 和 `terminal_tool`。Forge 确保模型不会跳过前置步骤、不会在"准备写入"之前忘记"先读取"、不会在应该停止的时候继续瞎编。
  4. **Context compaction（上下文整理）**：长对话里工具调用的返回结果越积越多，VRAM 是有限资源。Forge 用分层策略在上下文字数即将溢出时做一次智能压缩——等价摘要替换旧内容，保留最近两轮的原始细节。
  5. **Synthetic respond tool（合成响应工具）**：这是 Forge 最巧妙的设计。8B 模型有一个结构性问题：它们在"输出文本"和"调用工具"之间做选择时经常选错。Forge 注入一个假的 `respond` 工具——模型永远在"工具调用模式"里，即使只是想回复一句话也通过调用 `respond(message="...")` 来实现。代理层拦截这个调用，把它转换成普通文本返回给客户端。客户端完全不知道这个工具存在。

  在 Forge 的 26 场景 eval 套件上，这套组合拳的效果是：裸的 Ministral-3 8B 本地模型跑多步工具调用，成功率约 53%。挂上 Forge 之后，同一份权重达到 99.3%。更有意思的是对照组——Claude Sonnet 裸跑得分 87.2%，Forge 把它推到约 100%。这张对比表足够直观：

  | 配置 | 综合得分 |
  |------|---------|
  | Ministral-3 8B（裸） | ~53% |
  | Ministral-3 8B + Forge | 99.3% |
  | Claude Sonnet（裸） | 87.2% |
  | Claude Sonnet + Forge | ~100% |

  一台 $600 的 GPU，一个 8B 的本地模型，挂上 Forge 之后在工具调用可靠性上超过了裸奔的云端前沿模型。这个结论背后有正式的学术支撑——Zambelli 把 Forge 写成了一篇 IEEE 论文（DOI: 10.1145/3786335.3813193），框架和消融实验方法论都经过了同行评审。

  Forge 提供了三种使用模式对应不同的集成深度。**Proxy 模式**最简单：一行命令启动代理服务器 (`python -m forge.proxy`)，同时提供 OpenAI 和 Anthropic Messages 两个 API——把你的 Claude Code 或任何 OpenAI 兼容客户端指向它，Forge 透明地应用全套防护。**WorkflowRunner 模式**完全控制 Agent 循环：你来定义工具，Forge 管理 prompt、执行、压缩和防护。**Middleware 模式**适合已有自己循环的项目：直接引入 Forge 的 `ResponseValidator`、`StepEnforcer` 和 `ContextManager`，嵌入你的管线。

```bash
# 安装核心
pip install forge-guardrails

# 启动代理模式（连接本地 Ollama）
python -m forge.proxy --backend ollama --model ministral-3:8b --port 8081

# 然后在 Claude Code 里设置：
# ANTHROPIC_BASE_URL=http://localhost:8081
# ANTHROPIC_AUTH_TOKEN=anything
```

## DebugMCP — AI Agent 的代码出了 bug？让它自己设断点、单步执行、检查变量，不需要你帮它读日志

  用 AI 写代码的人早晚会遇到一种挫败：Agent 写了一段逻辑，跑了，报错了，然后把错误信息喂给 Agent。Agent 读了一遍错误栈，把代码改了，再跑——又报错了。如此反复三四轮，你发现 Agent 一直在错误的假设上打转，因为错误栈里没有足够的上下文告诉它"问题出在进入函数之前变量就已经是 `None` 了"。

  这个困境的根源是：Agent 目前唯一的调试手段是读日志。但真正有效的调试工具是——断点、单步执行、变量检查、表达式求值。DebugMCP 把这一整套 VS Code 调试器能力以 MCP 工具的形式暴露给了 AI Agent。

  安装之后，你的 AI 编程助手多出了 14 个调试工具。它能自己给任意文件的任意一行下断点（包括条件断点和 logpoint），启动调试会话，单步/跳入/跳出执行，在任意暂停点查看所有局部和全局变量的值，甚至能在调试上下文中计算任意表达式——就和人类开发者的调试流程一样，只是不再需要人类去操作。

  一个典型的 Agent 自主调试流程：写完代码 → 运行测试 → 某个测试挂了 → Agent 调用 `start_debugging` 启动调试会话 → `add_breakpoint` 在嫌疑代码行下断点 → `step_into` 进入函数内部 → `get_variables_values` 看了所有局部变量 → 发现某个值在调用前就错了 → `evaluate_expression` 验证了修复假设 → 改代码 → `continue_execution` 跑完验证 → 完工。全程零人工介入。

  技术实现上几个细节。**纯本地运行**——MCP 服务器跑在 `localhost:3001`，代码和数据不离开本机。**零配置**——不需要写 `launch.json`，根据文件语言自动生成调试配置。**自动注册**——安装后在 GitHub Copilot、Cursor、Windsurf、Codex、Cline、Roo Code、Antigravity 等 8 种主流 AI 助手中自动注册，其他 MCP 兼容客户端手动配置也很简单。**测试调试**——能直接调试单个测试用例而不是整个测试文件，减少无关上下文。

  v2.2 版本新增了两个重要能力：**跨代理技能安装**——附带一个 `/debug` Agent Skill，在支持的助手中自动安装到个人技能目录，调用它就能加载完整的系统化调试工作流。**暂停运行中程序**——不需要事先设断点，Agent 可以在程序卡住或运行过久时直接暂停并检查当前状态。

## GitHub MCP Server v1.6.0 — 官方 GitHub MCP 重大更新：150 位贡献者、71 个 Release、32k Stars，这次更新了什么

  6 天前（7 月 15 日），GitHub 官方发布了 github-mcp-server v1.6.0——这个把 GitHub REST 和 GraphQL API 包装成 MCP 工具的服务，在过去四个月里从 0 到 32k Stars、150 位贡献者、71 个 Release，已经成为 MCP 生态里使用最广泛的服务器之一。v1.6.0 不是小修小补，而是上线了一个让 Agent 工作流完整性大幅提升的新能力。

  新版本的亮点是 **Agent workflow tools**——首次让 AI Agent 能够通过 MCP 执行 GitHub Actions 工作流、查看日志、触发发布。在此之前，Agent 能帮你管理 Issues 和 PRs，但不能触发 CI/CD——那个最后的"跑完了通知我"环节，还得你自己切到浏览器里点。现在 Agent 可以直接调用 `actions_workflow_dispatch` 触发任何一个 workflow，`actions_list_workflow_runs` 查看运行状态和结果，`actions_download_workflow_run_logs` 下载日志做 AI 辅助的失败原因分析。如果你的 Agent 工作流是"开发 → 提交 → 创建 PR → CI 通过 → 合并"，现在 CI 的触发和监控可以在 MCP 里闭环了，不用再人工检查那个绿勾。

  新增的第二个能力是 **Copilot Spaces tools**——如果你的团队在用 GitHub Copilot Spaces 做项目级知识管理，Agent 现在能直接搜索空间内容、读取文档。这对需要跨项目共享上下文的大型团队有意义，但对个体开发者来说更像是锦上添花。

  v1.6.0 还改进了认证体验。**OAuth 设备码流程**（device code flow）现在可以在没有浏览器的环境下完成认证——服务器终端、远程 SSH 会话、CI runner 都支持。这对在远程开发环境或 CI 中用 GitHub MCP 的团队是个实用的改善。另外，使用 Personal Access Token 时如果 token 即将过期，Agent 会收到一个明确的 `token_expiry_warning` 工具结果而不是模糊的 `permission denied`。

  一个容易被忽视但影响实际使用的改进：**工具集（toolset）粒度更细了**。现在可以选择只启用特定类别的工具，减少 Agent 面对的工具列表长度。如果你只做 Issue 和 PR 管理，把 Actions、Dependabot、Secret Scanning 等工具关掉——Agent 不会被无关工具干扰，token 开销也更小。可用的工具集从 v1.5 的 20 个增加到 22 个，新增了 `copilot_spaces` 和 `stargazers`。

  回顾一下 GitHub MCP Server 的安装方式——因为它不是旧的 npm 包。官方提供了三条路径：**远程托管**（推荐，`https://api.githubcopilot.com/mcp/`，OAuth 认证，零本地运行时）、**Docker 本地**（`ghcr.io/github/github-mcp-server`，PAT 认证，流量不出机器）、**Go 二进制编译**（`go build`，完全控制）。对个体开发者来说，远程 OAuth 是最省心的方案——不需要维护 Docker，不需要管理 PAT。对团队来说，Docker + fine-grained PAT 是更安全的选择——token 只对特定 repo 的特定操作授权，万一泄露也炸不到其他项目。

```bash
# 远程托管（推荐个体开发者）
# VS Code 1.101+ → 设置 → MCP → 添加：
{
  "servers": {
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/"
    }
  }
}

# 本地 Docker（推荐团队）
docker run -i --rm -e GITHUB_PERSONAL_ACCESS_TOKEN=<fine-grained-PAT> ghcr.io/github/github-mcp-server

# Claude Code
claude mcp add --transport http github https://api.githubcopilot.com/mcp/
```
