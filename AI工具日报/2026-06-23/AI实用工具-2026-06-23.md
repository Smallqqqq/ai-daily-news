## TMax — Allen AI 开源终端 Agent RL 训练套件，27B 模型登顶 Terminal-Bench

  让 AI 在终端里自主完成任务——编译代码、配置服务器、调试环境——是编码 Agent 的终极场景之一。但此前开源模型在这类任务上的表现远落后于闭源方案。6 月 22 日，Allen AI（Ai2）发布了 TMax，一套完整的终端 Agent 强化学习训练工具链，包含从 2B 到 27B 四个尺寸的模型权重、15,000 个训练任务的完整语料、以及可复现的训练和评估代码。

  核心原理是 DPPO（Direct Preference Policy Optimization）——在 Qwen 基座模型上做强化学习，奖励信号来自终端任务是否执行成功。训练数据并非人工标注，而是通过一套「结构化轴独立采样」的合成管线生成：任务按难度、领域、工具链等维度自动组合，每个任务打包成自包含的 Docker/Apptainer 环境，附带程序化验证器（不是 LLM 打分，是真实执行结果）。这意味着任何人在任何机器上都能复现评估，不存在「换个环境分数就变了」的问题。

  实际效果：TMax 9B 在 Terminal-Bench 2.0 上达到 27.2%，比同尺寸 Qwen 3.5 9B 基座（21.1%）提升 6.1 个百分点。在更轻量的 TB Lite 基准上，TMax 9B 达到 57.2%，而基座仅 41.9%——RL 训练带来的提升在更短上下文的场景下更为显著。最大的 TMax 27B 在 Terminal-Bench 2.0 上达到 42.7%，在 2.1 版本上达到 44.9%。

  值得注意的是 65K token 预算下的表现——这恰是多数实际使用场景的约束条件。在此预算下，TMax 全系列均优于同尺寸基座模型和此前所有开源终端 Agent。

## GitHub Agentic Workflows — 把 AI Agent 嵌入 Actions，用 Markdown 写 CI 自动化

  6 月 11 日进入 Public Preview 的 GitHub Agentic Workflows 解决了一个实际问题：CI 流水线里有很多「需要判断但不够格写代码」的任务——issue 自动分类、CI 失败原因分析、文档同步、每日仓库状态报告。以前要么用传统 YAML 写复杂的条件分支（维护成本高），要么靠人肉处理（响应慢）。GitHub 的方案是把 AI Agent 直接嵌入 Actions 运行时，用自然语言 Markdown 定义任务，编译成标准 Actions YAML 执行。

  工作方式：写一个 `.github/workflows/xxx.md` 文件，YAML 前置元数据配置触发条件和 AI 引擎，正文用自然语言描述任务目标。`gh aw compile` 将其编译为 `.lock.yml`，跑在已有的 runner 和权限策略之上。支持四种 AI 引擎——Copilot、Claude、Codex、Gemini——选哪个取决于团队的已有订阅。

  安全模型值得单独提：Agent 默认只读权限，运行在沙箱容器内，前面有 Agent Workflow Firewall，输出经过 safe-outputs 校验，所有拟变更要经过专门的威胁检测 job 扫描后才应用。不再需要手动创建 PAT——直接用内置 `GITHUB_TOKEN`。

```bash
# 安装 CLI 扩展
gh extension install github/gh-aw

# 添加预置工作流模板（交互式向导）
gh aw add-wizard githubnext/agentics/daily-repo-status

# 手动触发运行
gh aw run daily-repo-status

# 编辑 Markdown 后重新编译
gh aw compile
```

| 工具 | 定义方式 | AI 引擎 | 自托管 | 安全模型 | 热度 |
|------|---------|---------|--------|---------|------|

  GitHub Agentic Workflows 的核心优势是零迁移成本——不需要新平台、新账号、新权限体系。它跑在你已有的 Actions runner 上，遵守已有的策略约束。对于已经在 GitHub 上的团队，这是把 AI 引入 CI 的最短路径。

## clihub — 一个 CLI 管理七个 AI 编码工具，配置、技能、MCP 全同步

  2026 年的 AI 编码工具格局已经碎片化到令人头疼：Claude Code 管终端、Codex 管 JetBrains、Gemini CLI 做快速实验、Qwen Code 跑本地模型——每个 CLI 都有自己的一套技能目录、MCP 配置、记忆文件和系统提示词。换台机器或者加入新团队，光是配环境就要半天。

  clihub（v1.50.0，npm 周下载量稳定）用一个 `clihub.yaml` 声明文件终结了这场配置战争。`clihub tool install claude-code` 一条命令装好 CLI，`clihub skill install superpowers` 自动把技能分发到你装的每一个 CLI 里，`clihub preset apply fullstack` 一键套用预设的工具+技能+MCP 组合包。所有配置被锁定在 `clihub.lock.json` 里，团队成员 clone 后 `clihub install --frozen` 即可复现完全一致的开发环境。

  覆盖的 CLI 已扩展到 7 个：Claude Code、Codex、Gemini CLI、Qwen Code、Kiro、Cursor、Goose。6/7 支持技能同步，MCP 配置在所有 JSON/TOML 格式的 CLI 之间自动转换。v1.50 新增了 Cursor 命令文件（`.cursor/commands/*.md`）和 Goose 配方文件（`.goose/recipes/*.yaml`）的技能同步。

```bash
# 一键安装
npm install -g @wikieden/clihub

# 引导式初始化
clihub wizard

# 安装 CLI 工具
clihub tool install claude-code
clihub tool install codex

# 安装技能（自动分发到所有已装 CLI）
clihub skill install superpowers

# 健康检查
clihub doctor

# 升级前自动备份
clihub backup
```
