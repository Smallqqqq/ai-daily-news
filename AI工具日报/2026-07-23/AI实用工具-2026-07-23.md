## Context7 MCP — 一句话让 AI 编程助手读到最新、版本正确的库文档，不再幻觉出不存在的 API

  用 Claude Code 或 Cursor 写代码的人迟早会撞上同一个坑：模型信心十足地调用了一个 API，跑起来才发现这个函数在三个版本前就被改名了，或者参数签名根本不是那样。这不是模型笨——它训练数据截止到几个月前，而你在用的库上周刚发了一个 breaking change。

  Context7 的解法很直接：在 AI 提示词里注入**最新、版本匹配的**库文档。它由 Upstash 团队维护，索引了主流开源库的官方文档，通过 MCP 协议在你问"帮我写一个 Next.js 中间件做 JWT 校验"时，自动把当前版本 Next.js 的真实 API 文档塞进 prompt——模型看到的不再是记忆中模糊的旧 API，而是此时此刻正确的函数签名和参数。

  使用体验已经被压缩到几乎零摩擦。安装后，在 prompt 末尾加一句 `use context7`，或者配一条规则让 Agent 在涉及库文档时自动调用——之后你只管正常提问，文档检索全程在后台静默完成。如果你明确知道要查哪个库，用 `use library /vercel/next.js` 可以跳过库名匹配步骤，直接拿文档。指定版本号也一样自然："Next.js 14 的中间件怎么写？use context7"——Context7 会自动匹配对应版本的文档。

  数据层面，Context7 的 GitHub 仓库 59.6K stars，npm 周下载量 110 万，已经是 MCP 生态里装机量最大的文档类工具之一。支持 15+ 种语言文档、三种传输协议（stdio/http/sse），还有一个可选的 Context7 API Key 提升速率上限（免费额度每月 1000 次请求，付费 $10/月）。

```bash
# 一键安装（推荐）
npx ctx7 setup

# 或手动注册 MCP 服务器
claude mcp add context7 -- npx -y @upstash/context7-mcp@latest

# 在 CLAUDE.md 或 Cursor Rules 中加一条规则，自动触发：
# "Always use context7 when I need library/API documentation without me explicitly asking."
```

## FlowGent — 用一句话描述自动化需求，Claude 生成可直接导入 n8n/Make.com 的工作流 JSON

  用 n8n 搭自动化工作流的人都知道，拖节点不难，难的是每个节点的参数配置、数据映射、错误处理、条件分支——一个中等复杂度的流程从零搭起往往要 30-60 分钟，而且大部分时间花在"试一下这个字段对不对"上。

  FlowGent 把这个过程中的机械部分交给了 Claude Opus。你描述一句需求——"每天早上 9 点抓取 Hacker News 前五条，用 AI 摘要后发到 Slack"——它分析意图、选择节点、配置参数、连接数据流，输出一份完整的 n8n 或 Make.com 工作流 JSON。下载、导入、填上凭据、激活，完事。

  设计上保持轻量。一个 CLI 命令搞定：`flowgent gen "你的需求"`，可选 `--platform make` 切到 Make.com，`--stdout` 直接打印 JSON 方便管道处理。交互模式 `flowgent interactive` 用引导式问答补全你遗漏的参数。内置示例库覆盖了常见模式（RSS→摘要→发布、GitHub Issue→分类→Slack、Airtable→AI 生成→Shopify），可以直接复制 prompt 开跑。Docker 一键部署，Node.js SDK 可以嵌入你自己的工具链。

  176 stars 的仓库不算大，但定位很清晰：它不是要替代 n8n 的 AI Assistant（那个在 n8n 云版里，自托管用户用不了），而是在编辑器外面提供一个独立的 CLI——你写好 prompt，拿到 JSON，导入任何 n8n 实例。对自托管 n8n 用户来说，这刚好填上了官方 AI 功能云独占的缺口。

```bash
# 安装
git clone https://github.com/rnarturovich/flowgent.git
cd flowgent && npm install && cp .env.example .env
# 在 .env 中填入 ANTHROPIC_API_KEY
npm run build && npm link

# 使用
flowgent gen "Shopify 新订单 → 发邮件通知客户 → 记录到 Google Sheets"
flowgent gen --platform make "Stripe 支付失败 → Gmail 发送恢复邮件"
flowgent interactive  # 交互引导模式
```

## Ormayundo — 给 AI Agent 装上持久化长期记忆，知识图谱代替"把所有历史塞回上下文"

  AI Agent 有一个结构性的失忆问题：每次新会话从零开始。你今天告诉它"用 PostgreSQL 14，端口 5433，不要动 users 表"，明天开新会话，它全忘了。目前的补救手段是把历史对话摘要塞回 system prompt——但摘要会丢失细节，上下文窗口有上限，而且 Agent 自己也不知道"哪些信息以后还会用到"。

  Ormayundo（马拉雅拉姆语"你还记得吗？"）做了一件事：给 Agent 一个可以**写入和查询**的外部记忆体。底层是一张自托管的知识图谱——Agent 调用 `remember("Ada Lovelace 为 Charles Babbage 的分析机写了第一个算法")` 写入一条记忆，LLM 在后台自动提取实体（Ada Lovelace、Charles Babbage、分析机）和关系（写了、设计了）存入图谱，将来 Agent 调用 `recall("谁参与了分析机的工作？")` 就能拿到结构化的事实而非模糊的向量相似度结果。

  为什么是知识图谱而不是向量数据库？向量检索找到的是"相似"的文本块，图谱找到的是"关联"的事实——当问题是"Alice 在上线 Y 项目时一起合作的同事是谁"这种多跳推理时，图谱遍历直接沿边走到答案，向量检索只能给你一堆包含"Alice""Y 项目""同事"这些词的临近片段。

  项目还很早期（昨天刚发 v0.1.1），但架构方向明确：pip install 一行安装，LLM_PROVIDER 切换 OpenAI/Claude 做实体抽取，本地 embedding 模型做向量化（不消耗 API 额度），SQLite 持久化到 `~/.ormayundo/memory.db`。MCP 模式只暴露两个工具——`remember` 和 `recall`——设计上的克制值得注意：不需要 Agent 理解图谱结构，它只需要像使用便签本一样"记下这个""查一下那个"。

```bash
pip install ormayundo
export OPENAI_API_KEY=sk-...  # 或用 LLM_PROVIDER=anthropic

# Python 库
from ormayundo import remember, recall
remember("项目数据库端口 5433，只读用户 readonly")
print(recall("数据库端口是多少？"))

# MCP 模式（Claude Code 中使用）
# .mcp.json:
# { "mcpServers": { "ormayundo": { "command": "ormayundo-mcp" } } }
```
