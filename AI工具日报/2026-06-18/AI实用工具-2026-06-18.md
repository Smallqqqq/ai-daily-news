## Vercel eve — 把 Agent 从 demo 跑进生产的开源框架

  昨天（6 月 17 日），Vercel 发布了 eve——一个文件系统优先的 TypeScript Agent 框架，开源、自带生产环境所需的全部基础设施。这不是又一个"把 LLM 包一层"的玩具框架，它解决的是 Agent 开发中最实际的问题：你写完了一个能跑通的 Agent demo，但离上线还缺持久化执行、沙箱隔离、审批流、评测体系、多渠道分发——eve 把这些都内置了。

  eve 的核心理念是"Agent 即目录"：一个 Agent 就是一个包含 `agent.ts`、`instructions.md`、`tools/` 子目录的文件夹。框架负责持久化执行（崩溃可恢复）、沙箱（Agent 操作在受限环境中运行）、审批卡（敏感操作弹确认）、以及评测（evals 内置）。它通过 Vercel Connect 安全地接入 Slack、GitHub、Snowflake、Salesforce、Notion、Linear 等外部服务——你只需要声明 Agent 需要什么，OAuth 和凭据管理由框架处理，模型永远看不到连接 URL 和密钥。

  更值得关注的是多渠道统一：同一个 Agent，通过 HTTP API、Slack、Discord、GitHub 等不同渠道接入，每个渠道只需一个轻量适配器文件。这解决了 Agent 开发中"每个新触点都要重写集成层"的顽疾。

```bash
# 脚手架一个新 Agent 项目
npx eve@latest init my-agent

# 在已有项目中添加 eve
npm install eve@latest
```

## VS Code ACP Client — 一个扩展，终结 11 个 AI 编码工具的界面碎片化

  AI 编码工具的爆发带来了一个尴尬的问题：Claude Code 在终端里、Codex 在另一个终端里、Copilot Chat 在侧边栏里、Gemini CLI 又要新开一个窗口——你在不同工具之间切来切去，每个都有不同的认证流程、不同的交互方式、不同的快捷键。这不是工具太多的问题，是界面层没有统一的问题。

  由 Jun Han（formulahendry）开发的 VS Code ACP Client 用 Agent Client Protocol（ACP）解决了这个碎片化。把它想象成 AI 编码工具的"USB-C 接口"——一个协议、一个扩展、一套交互方式，连接 11 个主流 AI 编码 Agent。目前在 VS Code Marketplace 上免费提供，MIT 开源。

  预配置的 Agent 列表覆盖了几乎所有主流工具：GitHub Copilot、Claude Code、Gemini CLI、Qwen Code、Codex CLI、OpenCode、OpenClaw、Kiro CLI、Hermes Agent、Auggie CLI、Qoder CLI。你在侧边栏的下拉菜单里切换 Agent，聊天面板统一渲染 Markdown、代码块、工具调用卡片和思考过程，不再需要记住每个工具的命令行参数或 API 格式。

  一个容易被忽视但很重要的设计：一次只激活一个 Agent。这不是限制，而是防止多个 Agent 互相干扰——你不会想让 Claude 和 Gemini 同时编辑同一个文件。ACP Client 的定位是"统一遥控器"，不是"多 Agent 混战指挥中心"。

```bash
# 在 VS Code 扩展市场搜索 "ACP Client" 安装
# 或通过协议链接一键安装：
# vscode://extension/formulahendry.acp-client
```

## Vercel json-render — AI 生成 UI，但必须用你自己的组件

  AI 生成 UI 是 2024-2026 年最热的方向之一，但大多数方案有一个根深蒂固的问题：v0、Bolt、Lovable 生成的是"别人的组件"——你拿到一段 React 代码，要么全盘接受它的设计，要么手动改到符合自己的设计系统。json-render 换了一个思路：AI 不生成组件代码，而是输出一个 JSON spec，由你预先定义好的组件目录来渲染。AI 只能在你的组件库里挑选，不能发明新组件。这从根本上解决了 AI 生成 UI 的"幻觉组件"和"风格跑偏"问题。

  json-render 由 Vercel Labs 开发，核心是三个概念：Schema（定义 UI 结构语法）、Catalog（定义可用的组件和 action 词汇表）、Registry（将组件映射到具体平台的渲染代码）。AI 在 Catalog 约束下生成 JSON，Registry 负责渲染——React、Vue、Svelte、Solid、React Native、甚至 3D 场景（Three.js）和 PDF 文档，都用同一套 JSON spec。

  36 个预置的 shadcn/ui 组件可以直接用，你也可以定义自己的组件目录。JSON 输出是流式的——模型边生成、界面边渲染，用户体验和 ChatGPT 打字效果类似。最关键的是类型安全：JSON spec 在构建时通过 Schema 校验，不符合定义的输出在进入渲染层之前就被拦截。

```bash
# React
npm install @json-render/core @json-render/react @json-render/shadcn

# Vue
npm install @json-render/core @json-render/vue
```

| 工具 | 输出形式 | 跨平台 | 组件约束 | 后端 | 价格 | 热度 |
|------|---------|:---:|:---:|:---:|------|------|
| **json-render**（今日推荐）| JSON spec | ✅ 6+ 平台 | ✅ 目录约束 | ❌ | 开源免费 | ⭐ Vercel Labs |
| v0 (Vercel) | React 组件代码 | ❌ React only | ❌ | ❌ | $20/月 | ⭐ 极高 |
| Bolt (StackBlitz) | 全栈应用代码 | ⚠️ JS 生态 | ❌ | ✅ | $20/月 | ⭐ 极高 |

  json-render 不是 v0 或 Bolt 的替代品——它的定位完全不同。v0/Bolt/Lovable 是"给我一个完整应用"的工具，json-render 是"在我的设计系统里让 AI 动态生成 UI"的基础设施。如果你需要一个 AI 聊天界面、一个动态表单生成器、或者一个能根据用户输入实时改变布局的面板，json-render 是目前最优雅的方案。你已经有了组件库和设计规范，缺的是让 AI 在规范内排列组合的能力——这就是 json-render 做的事。
