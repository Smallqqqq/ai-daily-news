## ProofShot — 给 AI 编码 Agent 一双眼睛，录下它亲手验证自己写的 UI

  用 AI 写前端的人迟早撞上一个尴尬时刻：Agent 说"做好了"，你 `npm run dev` 打开浏览器一看——按钮错位、表单不回显、响应式布局炸了。问题不在于 Agent 写不出对的代码，在于它写完以后**自己看不见效果**。它交给你的是文本文件，不是跑起来的页面。

  ProofShot 的解法直接到没有设计：一个 CLI，三个命令——`proofshot start` 启动录屏和服务器日志捕获，Agent 用 `proofshot exec` 在真实浏览器里操作页面，`proofshot stop` 打包成一份包含视频、截图、错误报告的自包含 HTML 验证档案。Agent 自己就是验证者——它导航、点击、填表、截图，每一步都被记录下来。人不需要盯着 Agent 干活，事后打开 `viewer.html`，视频、操作时间线、控制台错误、服务器日志全在同一页面对齐。

  底层浏览器原语来自 Vercel Labs 的 agent-browser（不是 Playwright MCP），ProofShot 做的是验证层的编排：会话管理、自动录屏、服务器错误捕获（支持 JS/Python/Go/Rust 等 10+ 语言的错误模式匹配）、死时间自动裁剪、时间戳同步。`proofshot pr` 直接把整份验证档案上传到 GitHub PR 评论区，reviewer 不用拉代码就能看到 Agent 做的东西长什么样、有没有报错。

  跟所有需要手动操作的验证方案相比，ProofShot 的核心差异是零配置。`proofshot start --run "npm run dev"` 自动检测并启动开发服务器，Agent 拿到浏览器控制权后自己完成验证。支持 Claude Code、Codex、Cursor、Gemini CLI、Windsurf、Copilot、OpenCode——任何能跑 shell 命令的 Agent 都能用。首次安装后 `proofshot install` 自动检测 AI 工具并在技能目录写入 workflow 文件，Agent 自动学会这套验证流程——用户不需要写任何 prompt。

```bash
# 一键安装（含 agent-browser + Chromium）
npm install -g proofshot && proofshot install

# 启动验证会话
proofshot start --run "npm run dev" --description "验证登录表单提交流程"

# Agent 自己操作浏览器
proofshot exec navigate "http://localhost:3000/login"
proofshot exec fill @e2 "admin@example.com"
proofshot exec click @e5
proofshot exec screenshot "after-login.png"

# 停止录制，打包验证档案
proofshot stop

# 上传到 PR（自动找当前分支对应的 PR）
proofshot pr
```

## terminalcp — 终端的 Playwright：让 AI Agent 像人一样操作命令行

  几乎所有 AI 编码工具的终端能力都停留在一个水平：执行命令，等结果，拿文本。对于一次性任务（`npm test`、`git status`）够用，但对于交互式程序——调试器（LLDB/GDB/pdb）、REPL（Python/Node/Ruby）、数据库 shell（psql/mongo）、TUI 应用（vim/htop）——就完全无能为力。这些工具的设计前提是**有人在键盘前**，它们期望逐键输入、实时响应、视觉反馈。terminalcp 用伪终端（PTY）+ 全终端模拟把这个问题彻底解决了：Agent 对终端进程的操控粒度从"一次命令执行"降到了"单个按键"。

  它怎么做到的：启动一个持久的后台守护进程（TerminalServer，Unix domain socket 通信），所有终端会话都在这个守护进程中运行——Agent 断开连接后会话继续存活。Agent 通过 MCP 工具发送 JSON 指令：`start` 启动进程、`stdin` 发送按键（包括方向键、Ctrl+C 等控制序列）、`stdout` 读取渲染后的终端画面（保留颜色和光标位置）、`stream` 增量读取原始输出。两种输出模式对应不同场景：终端模式适合 TUI（vim、htop 的视觉布局需要完整渲染），流模式适合构建日志、服务器输出（只读增量，省 token）。

  一个容易被忽视但非常务实的特性是**人类可随时接管**。每个终端会话自动暴露一个 socket，人用 `terminalcp attach <session>` 进入同一个会话——看到 Agent 正在做什么、随时上手帮忙、完成后 detach 退回旁观。Agent 和人共享同一个终端状态，不是各自看各自的副本。这对调试环节尤其有用：Agent 在 LLDB 里单步执行、检查变量，人觉得方向不对随时切进去调整。

  跟 screen/tmux 的对比能看清 terminalcp 的设计意图。tmux 也能让 Agent 操控终端——先 `tmux new-session` 创建会话，再用 `tmux send-keys` 发指令，`tmux capture-pane` 读输出。但实现同样功能需要处理四个 tmux 特有的问题：会话创建和 pipe 激活之间的输出丢失、无法增量读取（每次 dump 全量 buffer）、exit code 追踪需要额外逻辑、不同版本行为不一致。terminalcp 把这些都做成了内置能力——结构化 JSON 响应、增量 stream、自动退出码检测、跨平台一致的 PTY 行为。

```bash
# Claude Code 一键注册
claude mcp add -s user terminalcp npx @mariozechner/terminalcp@latest --mcp

# Cursor：设置 → MCP → 新增，命令：npx @mariozechner/terminalcp@latest --mcp
# VS Code：code --add-mcp '{"name":"terminalcp","command":"npx","args":["@mariozechner/terminalcp@latest","--mcp"]}'
```

  terminalcp 和 Desktop Commander 解决的是不同层级的问题。Desktop Commander 是"给 Agent 一个终端"——文件操作 + 命令执行 + diff 编辑——覆盖面广但交互深度浅，适合替代 IDE 的文件操作和一次性命令。terminalcp 是"给 Agent 一只会打字的手"——它假定 Agent 已经有了基本的终端能力，缺的是对交互式程序的精细控制。两者互补：Desktop Commander 管广度，terminalcp 管深度。如果你需要 Agent 帮你定位一个 release 模式的 segfault（gdb 逐帧回溯、检查寄存器、dump 内存），terminalcp 是目前唯一能做到的 MCP 方案。
