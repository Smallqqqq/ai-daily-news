# AI 实用工具 · 2026-07-15

![[summary.png]]

  今天两个工具解决的是 AI 编码工作流里两个"看不见"的问题：Agent 调了哪个 MCP 工具、传了什么参数、为什么没反应——这些在 MCP Inspector 里全看不到；以及那些年你背了又忘、忘了又查的 CLI 参数，能不能让 AI 帮你读一次 --help 然后永远别再手动敲。

## mcpsnoop — MCP 界的 Wireshark，坐在 Agent 和 Server 中间看每一帧 JSON-RPC

![[mcpsnoop-demo.png]]

  用 MCP 接外部工具的开发者早晚会遇到同一个场景：Agent 说"我调了那个工具"，但实际结果不对。你打开 MCP Inspector，手动发一个请求——能通。代码没问题，Server 没挂，Agent 却说没调用。问题在哪？

  MCP Inspector 有一个结构性盲区：它以独立客户端身份连接 Server，走的是自己的握手、自己的会话、自己的工具调用。你的 Claude Code 或 Cursor 实际发出的每一个 JSON-RPC 帧，Inspector 一概看不到。当工具被静默跳过、参数被悄悄改掉、或者一次调用卡住不动，Inspector 告诉你一切正常——因为它看的是另一条线路。

  mcpsnoop 的解法简单而暴力：坐进真实的数据通道里。它是一个 Go 写的透明代理，单二进制、零依赖。把你的 Server 启动命令包在 `mcpsnoop --` 后面，它就夹在 Agent 和 Server 之间，原封不动转发每一字节，同时把每一帧 JSON-RPC 复刻一份送到一个带 TUI 的实时终端。你能看到每一次 tool call 的完整请求和响应、Server 的 stderr 输出、每次调用的延迟（挂起的调用有实时计时器）、客户端和服务端的完整 capability 握手过程。

  几个设计让它在调试场景中远胜 Inspector。**实时过滤**——`tool:search_files` 只看一个工具，`status:error` 只看失败调用，`dir:request` 只看发出的请求。**回放功能**——抓到触发 bug 的那个调用，按一下 replay，修完 Server 代码再 replay，看修没修好。不需要重新让 LLM 组织 prompt 碰运气发出同样的调用。**导出**——支持 JSON、HTML、纯文本三种格式，带完整的关联调用、耗时、状态、错误标记。**CI 集成**——`mcpsnoop check` 可以在 CI 里对错误帧、无效帧、慢调用、挂起调用直接返回非零退出码。

  安全模型很清晰：它执行你传给它的 Server 命令，权限和原 Server 一样。只 wrap 你信任的 Server，不信任的放容器里跑。抓到的所有帧存在本地 Unix socket 和磁盘日志里，不往外发。

```bash
# 安装
go install github.com/kerlenton/mcpsnoop/cmd/mcpsnoop@latest
# 或
brew install kerlenton/mcpsnoop/mcpsnoop

# 看一遍 demo（零配置，播放预设会话）
mcpsnoop demo

# 包装你的 stdio MCP Server
# 在 MCP 配置里把 command 改成 mcpsnoop，原命令放 -- 后面
{
  "mcpServers": {
    "my-server": {
      "command": "mcpsnoop",
      "args": ["--", "node", "build/index.js"]
    }
  }
}

# 代理 streamable-HTTP Server
mcpsnoop http --target http://localhost:3000/mcp --listen :7000

# CI 检查：有错误帧/慢调用/挂起就失败
mcpsnoop check --session latest
```

### 同类工具对比

| 工具 | 安装方式 | 核心特点 | 价格 | 热度 |
|------|---------|---------|------|------|
| **mcpsnoop**（今日推荐）| Go install / Homebrew | 透明代理坐入真实数据通道，实时 TUI + 回放 + CI 集成 | 开源免费 | ⭐ 254 |
| MCP Inspector | npx | 独立客户端连接，手动触发工具，看原始 JSON | 开源免费 | 官方维护 |
| Raindrop Workshop | 本地 daemon | 全链路 agent trace + MCP 暴露给 Claude Code 自愈 | 开源免费 | ⭐ 858 |
| WireMCP | GitHub | Wireshark 的 MCP 插件，真正的抓包级分析 | 开源免费 | 🆕 |

  mcpsnoop 和 MCP Inspector 的关系就像 Wireshark 和 ping——一个看真实流量，一个单独发测试包。Inspector 适合开发阶段验证 Server 基本功能；一旦 Agent 和 Server 联调，Inspector 就瞎了。Raindrop Workshop 是更全面的 agent 可观测性方案，但面向的是整个 agent 链路（token、tool call、timing），mcpsnoop 只聚焦 MCP 层的 JSON-RPC 通信——更轻、更专注、更适合排查"为什么这个 tool 没被调用"这类具体问题。

## instagui — 任何一个 CLI，一行命令变 GUI，从此不用背参数

![[instagui-demo.gif]]

  命令行工具很强，但参数难记。ffmpeg 的转码参数、pandoc 的格式转换、yt-dlp 的下载选项——每次用都要 Google 一遍同样的命令。这不是技术能力问题，是认知负荷问题。人类的短期记忆不适合存大量 flag 组合，但 CLI 的设计假设就是让你记住它们。

  instagui 做的事出奇地简单：`npx instagui ffmpeg`，它读 ffmpeg 的 `--help` 输出，用 AI 解析成结构化的 JSON schema，在浏览器里渲染一个干净的表单——复选框、下拉菜单、输入框、数字选择器，对号入座。你在表单里勾选参数，页面实时显示对应的完整命令，点击 Run 执行，stdout/stderr 流式返回。整个过程不隐藏任何东西——你始终能看到它要执行的确切命令，边用边学会 CLI 语法。

  核心巧思在于缓存策略。ffmpeg、yt-dlp、pandoc 三个高频工具内置了预制 schema，**完全不需要 API key**，`npx instagui ffmpeg` 直接能用。对于其他工具，schema 只在第一次使用时通过 AI 提取一次，之后缓存到 `~/.instagui/`，永不再付费。AI 引擎自动检测——你装了 Claude Code 就用 `claude` CLI，设了 `OPENAI_API_KEY` 就用 OpenAI，有 Ollama 就用本地模型。一条命令都不需要额外配置。

  安全性设计得很认真。命令通过参数数组（`spawn`）执行，不走 shell，所以表单输入无法注入额外命令。服务器只绑 `127.0.0.1`，外部网络不可达。`POST /run` 做 Origin 头校验防 CSRF。同一时间只跑一个命令，关掉标签页自动杀进程不残留。API key 从不写入日志、不回显到页面、不嵌入任何缓存文件。代码里没有遥测、没有外部统计、没有 phone-home。

  目前 v0.1 阶段，只支持平铺参数的工具（不支持 `git commit` vs `git push` 这种子命令树，作者说 v0.2 做）。交互式 TUI 程序（vim、top、REPL）也不在支持范围内——这不是 bug，是刻意的范围约束，保持内核小而锋利。

```bash
# 零配置，直接打开 GUI（内置 schema，无 API key）
npx instagui ffmpeg      # 音视频转码
npx instagui yt-dlp      # 媒体下载
npx instagui pandoc      # 文档格式转换

# 其他任意 CLI 工具（首次需 AI 提取 schema）
export ANTHROPIC_API_KEY=sk-ant-...
npx instagui curl
npx instagui imagemagick

# 查看所有可用 AI 引擎及其状态
npx instagui --engines

# 使用指定引擎
npx instagui curl --engine claude     # 用 Claude Code CLI（订阅用户无需额外 key）
npx instagui curl --engine ollama     # 用本地模型，完全免费
```

### 同类工具对比

| 工具 | 安装方式 | 核心特点 | 价格 | 热度 |
|------|---------|---------|------|------|
| **instagui**（今日推荐）| `npx instagui` | AI 读 --help 自动生成 GUI，一次提取永久缓存，多引擎 | 开源免费 | ⭐ 94 |
| Gooey | `pip install Gooey` | Python argparse → 桌面 GUI，纯规则无 AI | 开源免费 | ⭐ 21k |
| CLI-Chart | npm | CLI 输出转图表 | 开源免费 | ⭐ 小 |
| Ask-UI | npm | 自然语言操控任意 UI 界面，非 CLI 工具 | 开源免费 | ⭐ 小 |

  instagui 和 Gooey 的区别是设计哲学的分岔。Gooey 要求工具的**作者**在代码里加一个 `@Gooey` 装饰器——它是给"愿意为 GUI 改代码的开发者"用的。instagui 从外部读取 `--help` 文本，不需要工具本身做任何改变——它是给"不想背参数的终端用户"用的。Gooey 产出的是桌面原生窗口，instagui 产出的是浏览器表单。Gooey 不依赖 AI，纯规则解析 argparse；instagui 用 AI 解析任意格式的 --help 文本（Python、Go、Rust、C 写的都可以）。被 The Register 专题报道、发布一周即 94 stars，社区反馈集中在"终于不用再 Google ffmpeg 命令了"这个具体痛点上。
