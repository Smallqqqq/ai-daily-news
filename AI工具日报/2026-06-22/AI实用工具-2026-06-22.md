## Assay — 用对抗性验证 + 确定性规则双重检查 AI 代码，在 LiteLLM 里找到了 185 个 bug

  用 AI 写代码的人迟早撞上同一个问题：Agent 写的代码通过了所有测试、lint 也没报错、review 看起来也合理，但上线就炸。问题不在语法层，而在语义层——代码做出的隐含假设（"这个函数会处理 null 输入""这个查询是注入安全的"）从未被检查过。Assay 做的事很直接：把代码里每一条隐含声明提取出来，先用对抗性 LLM 验证一遍，再用纯正则的确定性验证器 override 任何 LLM 级别的误判。

  原理分两层。第一层是**声明提取**：Assay 扫描代码，把"这段代码声称自己能做什么"拆成结构化的、可测试的 claim——功能正确性、安全性、数据隐私、性能边界，每一条都有类别和严重等级。第二层是**双重验证**：先让一个对抗性 LLM 逐条判断 PASS/PARTIAL/FAIL，然后用纯正则/模式匹配的 formal verifier（零 LLM，零幻觉风险）覆盖 LLM 的判决。在第一次生产调用时，LLM 对一段含 SQL 注入的代码判了 PASS，formal verifier 直接 override 成 FAIL。

  实际效果：在 4 个流行开源项目上的扫描结果——LiteLLM（18K stars）: 1,381 条声明，185 个 bug，30 个致命；Chatbot UI（28K stars）: 476 条声明，41 个 bug，12 个致命；LobeChat（50K stars）: 205 条声明，14 个 bug，1 个致命；Open Interpreter（55K stars）: 12 条声明，4 个 bug，2 个致命。不是 AI 随便说的，每条报告都绑定到具体文件和行号。

  基准测试同样硬核。HumanEval pass@1 从 86.6% 提到 98.8%，pass@5 直接 100%（164/164 题全过）。SWE-bench resolve@1 从 18.3% 提到 25.0%，best-of-5 提到 30.3%。关键是这些分数不是 LLM 自己评的——是用真实测试套件跑出来的。

  此外 Assay 还提供了一个更宏大的用法：LUCID 方法论——故意让 AI "幻觉"出一份法律级服务条款（ToS），从中提取 80-150 条可测试需求，然后迭代验证代码是否满足这些需求。6 轮迭代后合规率从 ~35% 收敛到 90.8%，总花费约 $17。有正式论文（DOI: 10.5281/zenodo.18522644），理论基础引用了 Xu et al. 2024、Banerjee et al. 2024、Karpowicz 2025 三篇关于 LLM 幻觉不可消除性的独立证明。

```bash
# 快速扫描一个项目
npx tryassay assess /path/to/your/project

# 扫描 GitHub 仓库
npx tryassay assess https://github.com/org/repo --publish

# 全局安装
npm install -g tryassay

# MCP Server 模式（Claude Code / Cursor / Windsurf）
npm install -g assay-mcp
# 然后在 AI 助手里说 "Verify this file with Assay"
```

  Assay 的不可替代之处在于双层验证架构。传统 SAST 工具（ESLint、SonarQube）只能检查已知规则模式，对语义级别的隐含假设无能为力。CodeRabbit 和 Copilot Code Review 能做语义判断，但没有确定性验证器兜底——LLM 自己也可能判错。Assay 是唯一一个把"让 LLM 大胆猜"和"用正则铁腕纠错"放在同一条 pipeline 里的工具。

## Clihub — 一个控制平面管 8 个 AI CLI：统一安装、同步 skill、备份回滚

  2026 年的 AI CLI 生态已经碎片化到令人抓狂。你用 Claude Code 做复杂重构、Codex 跑批量操作、Gemini CLI 处理 Google Cloud、Qwen Code 省钱——每个 CLI 有自己的 skill 目录布局（`.claude/skills/` vs `.codex/skills/` vs `.gemini/commands/*.toml` vs `.qwen/commands/*.toml` vs `.kiro/steering/` vs `.cursor/commands/*.md` vs `.goose/recipes/*.yaml`），有自己的 MCP 配置格式（JSON vs TOML），有自己的记忆文件位置。同一个 superpowers skill 要装七次，每次路径不同。哪天升级炸了配置，没有回滚手段。

  Clihub 是这个问题目前最完整的答案。它是一个 TypeScript 写的终端控制平面——一键安装 8 个主流 AI CLI（Claude Code、Codex、Gemini CLI、Qwen Code、Kiro、Cursor、Goose、OpenCode），然后让 skill、MCP server、memory、system prompt 在所有 CLI 之间自动同步。`clihub skill install superpowers` 一条命令，skill 自动扇出到所有已安装 CLI 的对应目录。

  几个值得展开的能力。**Preset 系统**：`starter`（Claude Code + 5 核心 skill）、`fullstack`（前后端全套）、`python/go/rust`（语言专项包），一条命令配完整个 AI 工具链。**备份与回滚**：`clihub backup` 快照所有 CLI 的配置目录，`clihub rollback` 一键恢复。**版本锁定**：`clihub lock` 生成 `clihub.lock.json`，`clihub status --strict` 在 CI 里检查团队成员的 CLI 版本是否一致。**ed25519 签名目录**：防止供应链投毒。**跨机器加密同步**：scrypt + AES-256-GCM，纯口令，零后端。**多账户 Profile**：工作和个人项目的 API key 和代理配置一键切换。

  截至 v1.65.0（6 月 17 日发布），Clihub 内置了 30 个 skill、14 个 MCP server、8 个 preset。支持中英日韩西五种语言。零运行时依赖。

```bash
# 一键安装
curl -fsSL https://raw.githubusercontent.com/wikieden/clihub/main/scripts/install.sh | sh

# 或 npm
npm install -g @wikieden/clihub

# 新手引导
clihub wizard

# 安装 CLI + 同步 skill
clihub tool install claude-code
clihub tool install codex
clihub skill install superpowers       # 自动扇出到所有已安装 CLI
clihub preset apply fullstack          # 全套工具 + skill + MCP

# 备份与回滚
clihub backup
clihub rollback
```

## Macrokit — 让 7B 本地模型跑出 94.5% 工作流准确率：强模型设计一次，弱模型无限执行

  用本地小模型做实际工作有个根深蒂固的困境：7B/8B 的模型单步推理还行，三步以上就丢上下文，五步以上开始编造工具调用。大多数 Agent 框架对此的解法是"让模型更强"——换更大的模型、加更多的 planning prompt、套更重的 orchestration。Macrokit 走了完全相反的方向：**消除运行时的推理**。

  核心思路简洁而彻底。把工作流拆成两步：**设计时**（design-time）让强模型（Claude、GPT-4o）在开发者监督下把一个多步工作流编码成确定性的、参数化的 macro——本质上是"这个工作流已经被想清楚了，编译成了代码"。**运行时**（runtime）弱模型只需要做一件事——意图分类："用户想要 X → 调 macro Y"。多步推理变成了单次路由，1.5B 模型就能跑。

  几个关键数字。在 100 个任务的维护者 Agent 基准测试上：Qwen 2.5 1.5B 得分 74.0%，Qwen 2.5 7B 得分 82.3%，而生产部署中使用的 Qwen 2.5 7B Q4_K_M（量化版）得分 **94.5%**，零次救场（bail-out），平均延迟 5.85 秒。全部在本地 16GB MacBook 上跑，不调任何云 API。

  这套 SDK 不仅仅是 runtime。它附带的 **distillation gate**（`macrokit gate`）是这个项目的真正创新点：它读取 Agent 会话日志，标记出"Agent 连续调了 3 个以上未被编码的工具"的轮次，自动建议 macro 名称、参数 schema 和 stub handler。**规则很简单：每次会话中出现了没有 macro 的工作流，就必须在会话结束前把它编码成 macro。**这个强制蒸馏循环让 macro 库随着使用自动增长——不是靠人想起来要保存，而是靠工具强制执行。

  Macrokit 还有一个公共 MCP server（`@macrokit/mcp`），可以接入 Claude Code 或 Cursor。Agent 在工作时会自动调用你项目里的 macro 作为工具；会话结束后 `macrokit gate` 扫描日志、标记未编码的工作流。

```bash
# 克隆并构建
git clone https://github.com/macrokit/core.git macrokit && cd macrokit
pnpm install && pnpm -r build

# 脚手架新项目
macrokit init my-app --vertical github

# 接入 Claude Code
claude mcp add macrokit -- macrokit mcp ./my-app

# 会话结束后，检查未编码的工作流
macrokit gate ./.macrokit/sessions --macros ./macros
```

  Macrokit 不是又一个 Agent 框架——它不参与"让模型更会推理"的军备竞赛。它的赌注是：大部分实际工作流是重复的，而重复就应该被编译。这个思路的理论渊源是认知神经科学里的双过程理论（System 1/System 2）——慢速昂贵的深思熟虑被编译成快速廉价的自动反应。在工程上，这意味着你的 macro 库才是真正的护城河，而不是你用的是哪个模型。该项目在 GitHub 上的关注度还很低（5 stars），但思路本身是这一类问题的唯一解——目前在此方向上无可比工具。
