# AI 实用工具 · 2026-06-09

![[summary.png]]

  今天三个工具分别覆盖 AI 编码的三个关键维度：UI 生成的可靠性、编码 Agent 的编程化调用、以及让 AI 先想清楚再动手的规约驱动工作流。每一个都能在今天装上进你的开发环境。

---

## json-render — Vercel 出品的生成式 UI 框架，让 AI 在你这套组件库里画 UI

  AI 生成 UI 的最大痛点不是"画不出来"，而是"画出来的东西没法用"——组件名是编的、颜色值是乱的、布局逻辑跟你项目里的设计系统完全对不上。json-render 的解法很直接：你定义一份组件目录（catalog），AI 只能从这个目录里选组件，输出的是一个 JSON 树，框架把它渲染成真实 UI。AI 的想象力被你圈在一个可控的沙盘里，既保留了灵活性，又不会胡来。

  它的核心思路是"生成式 UI"（Generative UI）——AI 根据自然语言描述生成 JSON 规格，渲染器把 JSON 变成真实的 React/Vue/Svelte/Solid 组件。36 个预置的 shadcn/ui 组件开箱即用，支持流式渲染（边生成边显示），跨框架渲染器覆盖 React、Vue 3、Svelte 5、SolidJS、React Native、甚至 React Three Fiber（3D 场景）和 React Email（邮件模板）。2026 年 1 月发布至今已累计 231 个版本、30 位贡献者，迭代密度极高。

  实际场景：聊天应用里根据用户输入动态生成表单、仪表盘、数据表格；或者做一个"描述你想要的页面，AI 帮你搭出来"的原型工具。因为组件是你定义的，它生成的东西直接就能用在你现有的项目里，不需要二次翻译。

```bash
# React 项目
npm install @json-render/core @json-render/react @json-render/shadcn

# Vue 项目
npm install @json-render/core @json-render/vue
```

## Cursor SDK — 把 Cursor 的编码 Agent 从 IDE 里拆出来，放进 CI/CD 管道

  Cursor SDK 是今年 4 月底进入公测的 TypeScript SDK（`npm install @cursor/sdk`），它把 Cursor 桌面端那个能规划、写代码、跑命令、自我纠错的 Agent 运行时，变成了一个可编程调用的服务。你在代码里 `Agent.create()` 创建一个 agent、`agent.send()` 发一个任务、`run.stream()` 接收流式结果——跟调一个 async 函数没有区别。

  这改变了编码 Agent 的使用边界。以前的 Agent 是"人在编辑器里跟 AI 对话"，Cursor SDK 让 Agent 变成了"脚本里的一个步骤"。CI 跑完测试发现失败 → 启动一个 Cursor Agent 分析根因并提出修复 → 把 patch 贴到 PR 评论里。凌晨三点自动触发 Agent 做大规模重命名迁移 → 早上来审查结果。发布前 Agent 自动生成 release notes → 直接贴到 GitHub Release。这些场景以前需要人守在编辑器前，现在可以写成 Pipeline。

  SDK 继承了 Cursor 桌面端的完整能力：代码索引、MCP 服务器、Skills、Hooks、子 Agent 嵌套。6 月 4 日刚更新的版本加了自定义工具（`local.customTools`）、自动审查（`local.autoReview`）、JSONL 持久化存储、以及 Python SDK 支持。本地 Agent 免费，云端 Agent 按用量计费。

```bash
npm install @cursor/sdk
```

```typescript
import { Agent } from "@cursor/sdk";

const agent = await Agent.create({
  goal: "Review the diff in this repo and summarize the 3 most impactful changes.",
  local: { workDir: "./my-project" }
});

for await (const event of agent.run.stream()) {
  console.log(event);
}
```

## GitHub Spec Kit — 让 AI 编码 Agent 先写规约再动手

  "Vibe coding"（感觉编码）的问题是 AI 会跳过思考直接写代码，结果就是结构漂移、逻辑矛盾、修修补补越改越乱。GitHub 6 月初开源的 Spec Kit 用一个简单但强硬的方式解决这个问题：它不让你和 AI 直接写代码，必须先写完三样东西——规约（Spec）、技术方案（Plan）、任务分解（Tasks）——然后 AI Agent 才能开始写第一行代码。

  Spec Kit 不是另一个 AI 编码工具，它是一套 CLI + 斜杠命令的工作流框架，兼容 30+ 个 AI 编码 Agent（GitHub Copilot、Claude Code、Gemini CLI 等）。核心命令链：`specify init` 初始化规约框架 → 写需求文档 → `specify plan` 生成技术方案 → `specify tasks` 分解为可执行任务 → Agent 按任务清单逐项实现。每一步产出都是 Markdown 文件，可以 commit 到 Git 里做版本管理。

  规约驱动开发（SDD，Spec-Driven Development）的核心价值不在于"文档"，而在于"让 AI 的每一步都有可验证的边界"。任务清单里的每一项都可以独立验证——测试通过、类型检查通过、lint 通过——Agent 完成一项再进入下一项，不会出现"修了 A 坏了 B"的漂移问题。项目方称 v0.9.5 版本已经过大量社区反馈迭代，MIT 协议开源。

```bash
# 安装
npm install -g @github/spec-kit

# 在项目中初始化
cd your-project
specify init

# 按流程推进
specify plan    # 生成技术方案
specify tasks   # 分解任务清单
# → Agent 按任务清单逐项实现
```

### 同类工具对比

| 工具 | 核心思路 | 兼容 Agent | 产物 | 团队协作 |
|------|---------|-----------|------|---------|
| **Spec Kit**（今日推荐）| 规约 → 方案 → 任务三阶段门控 | 30+ | Markdown 文档 | ✅ Git 管理 |
| CLAUDE.md | 自然语言规则文件 | Claude Code | 单一文件 | ✅ |
| OpenSpec | 结构化开发规范 | 不限 | YAML/JSON | ✅ |
| PR-Flow | PR 模板 + CI 检查 | GitHub | GitHub Actions | ✅ |

  Spec Kit 比 CLAUDE.md 更结构化（三阶段门控 vs 一段自由文本），比 OpenSpec 更轻量（Markdown 而非 YAML schema），比 PR-Flow 更上游（从需求阶段就开始约束，而不是等到 PR 阶段）。它是目前唯一一个让"先想再写"这件事变成硬约束而非软建议的工具。

⭐ 社区项目 · v0.9.5 · MIT · [GitHub](https://github.com/github/spec-kit)
