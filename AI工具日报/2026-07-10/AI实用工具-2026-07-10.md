# AI 实用工具 · 2026-07-10

![[summary.png]]

  今天两个工具踩中了"让 AI 替你干杂活"这个永恒主题——但踩的不是写代码本身，而是写代码前后的两件所有开发者都在做、但没人喜欢做的事：记住命令行参数，和给组件写文档。

---

## instagui — 一句话把任意命令行工具变成网页表单，零配置零代码改动

![[instagui-demo.gif]]

  每个开发者心里都有一份"我知道它很强大但我永远记不住参数"的 CLI 工具清单。ffmpeg 的参数组合足够写一本书，pandoc 的 flag 比 markdown 语法还长，yt-dlp 每次用都要重新查文档。这些工具的威力没变过，但访问这些威力的门槛——记住几百个 flag——几十年没降过。

  instagui 的解法直接到让人拍大腿：读工具的 `--help` 输出，用 Claude Haiku 把它转成 JSON schema，然后渲染成一个本地网页表单。你不再需要记 `ffmpeg -i input.mp4 -c:v libx264 -crf 23 -preset medium -c:a aac -b:a 128k output.mp4`，而是在浏览器里勾几个复选框、填几个输入框，点 Run，命令自动拼接、自动执行、输出实时回流。

  核心流程四步。**Capture**：跑 `<tool> --help`，带超时和容量上限，同时读 stdout 和 stderr，防止异常工具挂住启动。**Extract**：把 help 文本发给 Claude Haiku，返回一个经过严格校验的 JSON schema——每个 option 的名字、flag、类型、描述、枚举值、是否必填、所属分组，以及位置参数。校验不过就重试一次，两次都不过就报错，不给半成品。**Serve**：把 schema 渲染成单页表单，按分组排列，bool 用 checkbox、enum 用 dropdown、数字用 number input。**Preview + Run**：表单编辑时实时预览拼接出的完整命令，一键复制；点 Run 用 `spawn`（参数数组，不走 shell，防注入）执行，stdout/stderr 实时流回页面。

  和传统方案的关键差异在于零侵入。Python 生态里的 Gooey 能把 argparse 程序变 GUI，但需要开发者在代码里加 decorator。instagui 完全从外部读——只要这个程序有 `--help`，不管它用什么语言写的、怎么打包的，都能生成表单。而且生成一次就缓存到 `~/.instagui/`，下次再用同一个工具完全免费、秒开。三个 demo 工具（ffmpeg、yt-dlp、pandoc）的 schema 预置在包里，不花一分钱 API 费。

  安全设计值得单独提一句。命令以参数数组形式传给 `spawn`，永远不进 shell——表单里的输入不可能注入额外命令。服务器 bind 在 `127.0.0.1`，只接受本机请求。状态变更请求校验 Origin header，防 CSRF。关了浏览器标签页，子进程跟着杀掉，不留孤儿进程。API Key 不记录、不回显、不嵌入任何页面。

  7 月 4 日发布，三天后被 The Register 报道，作者 Omar Soutari 说这是他的第一个 FOSS 项目。目前是 v0.1，刻意不碰的边界很明确：不支持交互式/TUI 程序（vim、top、REPL），不支持子命令树（`git commit` vs `git push` 得等 v0.2），不做远程托管、不做插件系统。MIT 协议，64 stars，1 位贡献者。

```bash
# 三个 demo 工具零 API Key 直接用
npx instagui ffmpeg      # 视频/音频转码
npx instagui yt-dlp      # 下载媒体
npx instagui pandoc      # 文档格式转换

# 其他工具首次需 Anthropic API Key（仅一次，之后走缓存）
export ANTHROPIC_API_KEY=sk-ant-...
npx instagui curl

# 刷新缓存（重新提取 schema）
npx instagui curl --refresh

# 手写 schema（跳过 AI，零费用）
npx instagui curl --schema ./my-curl-schema.json
```

  instagui 和 Gooey 走的是两条完全不同的路。Gooey 要求工具的作者在代码层面配合——加一个 decorator，程序运行时自动弹出 GUI——所以它是一个"框架"，不是"适配器"。instagui 则是一个完全外挂式的工具——它对目标 CLI 的唯一要求是"有 --help 输出"，不碰源码、不要求语言、不要求框架。这个差异的实际后果是：Gooey 只能管你自己写的那几个 Python 脚本，instagui 能管你系统里所有有 help 文本的二进制。在这个"适配器 vs 框架"的维度上，instagui 目前几乎没有直接竞品。

## Story UI — 用自然语言给组件自动生成 Storybook 故事，支持 React/Vue/Angular/Svelte/Web Components 五大框架

  写 Storybook 故事像写单元测试——所有人都同意它有价值，但没人喜欢写。一个带 5 个 variant 的 Button 组件，对应的 story 文件轻松破 100 行：每个 variant 要 import 正确的组件、传正确的 props、设正确的 argTypes、写正确的 TypeScript 类型。手里的组件库越大，欠下的 story 债越多。

  Story UI 做的事就是让 AI 替你写这些 boilerplate。启动后选框架（React/Vue/Angular/Svelte/Web Components 五选一）、选设计系统（Mantine、Chakra UI、Material UI、Vuetify、Angular Material、shadcn/ui，或者完全自定义）、选 AI provider（Claude/OpenAI/Gemini），然后在聊天框里描述你想要的故事——"生成一个带 loading、error、empty、正常数据四种状态的 DataTable story"——Story UI 自动生成完整的 `.stories.tsx` 文件，写入你的 stories 目录，Storybook 通过文件监听自动加载，不需要刷新页面。

  自修复循环是 Story UI 最被低估的特性。AI 生成的代码有时会有语法错误、非法 import、或不符合你设计系统的写法。Story UI 不直接吐出代码就完事——它先用 TypeScript AST 解析生成结果，发现有语法错误就把错误上下文塞回 LLM 重试，最多三轮。如果三轮都没修好，选最优版本。还有一个 import 白名单机制：AI 只能使用你配置的组件库、框架运行时和 Storybook 包的导出，不能偷偷引入 Tailwind、shadcn/ui 之类你项目里没装的东西。除非你在 `story-ui-considerations.md` 里显式声明"允许额外导入 @tabler/icons-react"，否则违反白名单的 import 在写入文件前就被拦截。

  和 Storybook MCP 的集成让它不止于"从零生成"。连接 Storybook MCP addon 后，Story UI 能读到已有组件的文档、story 模式、以及项目的 UI 构建指南，生成的代码会模仿你已有的风格——不会出现"AI 生成的 story 和手写的 story 看起来像两个项目的代码"这种尴尬。

  Voice Canvas 是一个有趣的实验性功能。打开后直接用语音描述你想看的组件（浏览器语音识别），AI 实时流式生成代码，在 `react-live` iframe 里即时渲染。对于"我想看看如果把 Button 的 variant 改成 ghost 会怎样"这种即时探索，比打字快得多。

  5 个框架、3 个 LLM provider、90+ 个 release、MCP Server 集成（7 个工具：generate-story、list-components、get-component-props 等，Claude Desktop/Claude Code 直接可用）、自带 Railway 一键部署配置。190 stars，2 位贡献者，MIT 协议。

```bash
# 安装
npm install -D @tpitre/story-ui

# 初始化（交互式选择框架、设计系统、AI provider）
npx story-ui init

# 启动面板
npm run story-ui

# MCP Server 模式（供 Claude Code/Cursor 调用）
npx story-ui-mcp
```

  Story UI 和 StorieAI 解决同一个问题但走了两条路。StorieAI 是 VS Code 扩展，强在组件发现和覆盖统计——一眼看出哪些组件缺故事。Story UI 是独立面板，强在生成质量和多框架覆盖——五个框架加三个 AI provider，外加 MCP 集成让任何 AI 客户端都能调用。对于已经在用 Storybook 的团队，两者不互斥；如果只选一个，Story UI 的自修复循环和 import 白名单提供了更高的生成可靠性，而多框架支持对跨栈团队更友好。
