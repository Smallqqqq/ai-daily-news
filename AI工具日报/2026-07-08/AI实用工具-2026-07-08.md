# AI 实用工具 · 2026-07-08

![[summary.png]]

## caveman — 一个 Claude Code 技能，让 AI 用原始人腔调说话，省掉 65% 输出 token，同时保住代码和命令的字节级精度

![[caveman-demo.png]]

  用 Claude Code 或任何 AI 编码 Agent 的人，每天都会接收大量「礼貌性废话」——"Sure! I'd be happy to help you with that. The issue you're experiencing is most likely caused by..."，这些填充词不提供任何技术信息，但每次都消耗输出 token。对于一天几十轮对话的开发者来说，累积的浪费不小。

  caveman 的解法直接到不需要设计：它把 Agent 的输出风格压缩成原始人腔调。不是降智——大脑没变，嘴变小了。同一段代码修复建议，正常 Agent 输出 69 个 token（"The reason your React component is re-rendering is likely because you're creating a new object reference on each render cycle..."），caveman 输出 19 个 token（"New object ref each render. Inline object prop = new ref = re-render. Wrap in `useMemo`."）。代码块、命令、错误信息保持字节级精确，只压缩解释性文字。

  六个压缩档位可以随时切换。`lite` 删掉礼貌用语，`full`（默认）砍掉所有非必要词汇，`ultra` 进一步压到电报体，`wenyan` 用文言文输出——单位 token 承载信息密度最高。所有档位保持你的输入语言不变，写中文就压缩中文，写葡萄牙语就压缩葡萄牙语，只压缩风格，不翻译内容。

  除了日常对话压缩，caveman 还附带一套完整的 token 优化工具链。`/caveman-commit` 生成 ≤50 字符的 Conventional Commit 消息。`/caveman-review` 输出一行式 PR 评论。`/caveman-stats` 显示实时 token 用量和累计节省。最实用的是 `caveman-compress`——把你的 CLAUDE.md 等内存文件也压成原始人语，每次会话加载时都省输入 token。实测一个 706 token 的 claude-md-preferences.md 压缩到 285 token，省 59.6%。

  有一篇 2026 年 3 月的论文值得注意：[*Brevity Constraints Reverse Performance Hierarchies in Language Models*](https://arxiv.org/abs/2604.00025) 测试了 31 个模型，发现对大模型施加简洁性约束后，某些 benchmark 上的准确率反而提升了约 26 个百分点。有时候少说 = 更对。

  诚实地说：caveman 只压缩输出 token，不压缩输入和推理 token，且技能本身每轮增加约 1-1.5k 输入 token。所以整会话节省比例小于纯输出数字，在本身就很简洁的工作负载上甚至可能净亏损。真正的收益是阅读速度和可扫描性——成本节省是附带的。

```bash
# macOS / Linux / WSL / Git Bash
curl -fsSL https://raw.githubusercontent.com/JuliusBrussee/caveman/main/install.sh | bash

# Windows PowerShell
irm https://raw.githubusercontent.com/JuliusBrussee/caveman/main/install.ps1 | iex

# Claude Code 插件安装
claude plugin marketplace add JuliusBrussee/caveman && claude plugin install caveman@caveman
```

  caveman 和 TokenSlayer 解决的是 token 优化这个问题的两面。TokenSlayer 从输入侧入手——把 1200 行代码压成 8 行 AST 骨架，省输入 token。caveman 从输出侧入手——让 Agent 少说废话，省输出 token。两者互补而非互斥。prompt caching 是 Claude API 的内置机制，自动缓存重复前缀，不需要用户干预，但覆盖范围有限——对话一旦偏离缓存前缀就失效了。caveman 的优势在于零配置、跨 Agent、而且附带 commit/review/stats 等额外工具。

## Chrome DevTools MCP — Google 官方出品，让 AI Agent 拿到全套 Chrome DevTools 调试能力：性能追踪、网络抓包、控制台诊断、Lighthouse 审计，26 个工具一把梭

![[chrome-devtools-mcp-demo.png]]

  让 AI Agent 操作浏览器这件事，目前有三条主流路线：Playwright MCP 擅长跨浏览器自动化测试，Puppeteer MCP 提供最小化的浏览器操控面，而 Chrome DevTools MCP 走的是第三条路——它不跟 Playwright 抢「开车」的地盘，它专做「修车」的事。性能瓶颈排查、网络请求失败原因分析、Core Web Vitals 分数诊断、V8 堆内存快照——这些都是只有 CDP（Chrome DevTools Protocol）原生暴露的能力。

  Chrome DevTools MCP 由 Google Chrome DevTools 团队直接维护，底层用 Puppeteer 驱动浏览器，上层把 CDP 的性能、网络、控制台、无障碍域包装成 26 个 MCP 工具。Agent 可以录制性能 trace 并提取 Core Web Vitals（LCP/INP/CLS）的 actionable insights，可以检查每条网络请求的完整 header 和 body，可以读取带 source map 映射的控制台报错，甚至可以直接跑 Lighthouse 审计拿 SEO 和可访问性分数。这些能力在 Playwright MCP 里要么需要手动 `evaluate` JS 来模拟，要么根本不存在。

  一个经常被忽略但非常实用的设计是**实时浏览器会话附着**。从 Chrome M144 开始，Agent 可以请求远程调试连接，附着到你正在看的浏览器页面上。你在 Chrome 里打开一个出 bug 的页面，Agent 直接介入，看网络请求、读控制台、录 trace——不需要你描述问题，它自己看。这个能力在 Playwright MCP 的干净隔离会话模型里完全做不到。

  **Slim 模式**是另一个务实的选择。完整模式有 26 个工具，工具定义本身消耗约 18K token。如果你的 Agent 只需要基础浏览器操作（导航、截图、点击、填表），加 `--slim` 参数后只暴露 3 个核心工具，大幅降低 token 开销。Google 还提供了独立的 `chrome-devtools` CLI（实验性），进一步用 shell 命令替代 MCP 协议来提升 token 效率。

  一份 2026 年 4 月的独立基准测试（4 工具 × 13 场景 × 104 次运行）给出了量化参考：Chrome DevTools MCP 的 scenario mean 为 9.77/10，在所有被测工具中排名第一；Playwright MCP 为 9.38/10。在调试能力维度上，Chrome DevTools MCP 和 Playwright MCP 并列 10/10 满分。

```bash
# 一行接入 Claude Code
claude mcp add chrome-devtools --scope user npx chrome-devtools-mcp@latest

# 或以插件方式安装（含技能）
# 在 Claude Code 中先添加 marketplace：
/plugin marketplace add ChromeDevTools/chrome-devtools-mcp
# 再安装插件：
/plugin install chrome-devtools-mcp@chrome-devtools-plugins

# Slim 模式（仅 3 个核心工具，省 token）
npx chrome-devtools-mcp@latest --slim --headless
```

## Codex Plugin for Claude Code — OpenAI 官方出品，在 Claude Code 里直接调用 Codex 做代码审查、委派任务、跨 Agent 上下文传递

![[codex-plugin-cc-demo.png]]

  Claude Code 和 Codex 是目前最主流的两个 AI 编码 Agent。Claude Code 在复杂推理和自主编程方面更强，Codex 在速度和代码审查方面有独特优势。但在此之前，如果你想用 Codex 审查 Claude Code 写的代码，流程是：Claude Code 写完 → 你人工切到 Codex → 手动描述上下文 → 跑审查 → 把结果贴回 Claude Code。中间每一步都是上下文丢失和手动摩擦。

  OpenAI 的 Codex Plugin for Claude Code 把这个流程变成了几条斜杠命令。`/codex:review` 直接把当前未提交改动或分支 diff 发给 Codex 做只读审查，结果直接显示在 Claude Code 会话中。`/codex:adversarial-review` 更进一步——不是普通的「这里有个 typo」式审查，而是挑战你的设计决策、质疑你的缓存策略、测试你的错误处理假设。适合发布前的最后一道防线。`/codex:rescue` 则反过来——当你需要把一个问题丢给 Codex 去调查或修复时，直接把任务委派过去，Codex 在后台跑，完成后结果返回 Claude Code。

  最值得关注的设计是**跨 Agent 上下文传递**。`/codex:transfer` 把当前 Claude Code 会话的完整上下文（包括你问了什么、Agent 做了什么、中间踩了什么坑）导出为 Codex 可恢复的线程。你在 Claude Code 里调了一小时的 bug，想把后续工作交给 Codex——一条命令，Codex 拿到的是完整上下文，不是你在聊天框里手动写的三行描述。

  后台任务管理做得也干净。`/codex:review --background` 把审查丢到后台，`/codex:status` 随时查进度，`/codex:result` 取回结果，`/codex:cancel` 取消。不需要开着终端等 Codex 跑完。还有一个可选的审查闸门——启用后，Claude Code 的每次回复都会经过 Codex 审查，发现问题会阻止输出。注意这个功能会形成 Claude ↔ Codex 循环，消耗限额较快，适合关键发布节点而不是日常使用。

```bash
# 在 Claude Code 中添加 marketplace
/plugin marketplace add openai/codex-plugin-cc

# 安装插件
/plugin install codex@openai-codex

# 重载插件
/reload-plugins

# 初始化（自动检测 Codex 是否已安装并登录）
/codex:setup

# 快速体验
/codex:review --background
/codex:status
/codex:result
```
