# AI 实用工具 · 2026-07-09

![[summary.png]]

  今天三个工具分别来自三个不同的"权力中心"——Meta、开源社区和微软生态——但踩中了同一个趋势：AI Agent 不再是代码的"消费者"，而是代码的"共同生产者"。一个让设计系统对 Agent 可读可操作，一个把 API 调用成本压缩到接近零，一个让 .NET 开发者终于有了和 Claude Code 平起平坐的 IDE 武器。

## Astryx — Meta 开源的 150+ 组件设计系统，同时为人类和 AI Agent 设计，8 年内部实战，13,000+ 应用验证

  用 AI 写前端的人迟早撞上一个尴尬时刻：Agent 写出了能跑的组件，但每个组件都长得不一样——间距不一致、颜色硬编码、无障碍属性缺失。这不是 Agent 的错，是它没有设计系统的约束。你可以在 prompt 里写"用 shadcn 的 Button 组件"，但 shadcn 的文档是给人读的，Agent 从这里学到的只是 API 表面，学不到设计意图、组合模式和边界条件。

  Astryx 的解法是把设计系统本身做成 Agent 的一等公民。150+ 个组件不只是暴露 props 类型——每个组件带 JSDoc 注解，标注了组合提示（"这个 Dialog 的 trigger 应该用 DialogTrigger 包裹"）、边界约束（"IconButton 不接受 children，用 icon prop"）和主题变量（"此组件响应 `--color-primary` 和 `--radius-md`"）。Agent 读的是结构化知识，不是从文档里猜。

  这套设计系统在 Meta 内部跑了 8 年，支撑了 13,000+ 个内部应用。它不是实验室项目——是经历过 React 版本升级、设计语言迭代、多团队并行开发的实战产物。开源出来的四个核心包各司其职：`@astryxdesign/core` 提供组件和主题系统，`@astryxdesign/cli` 是命令行入口（component 文档、模板注入、主题编译、版本迁移 codemod），`@astryxdesign/build` 负责 StyleX 构建管线，`@astryxdesign/theme-*` 提供 10 套开箱即用的主题（neutral、butter、chocolate、matcha、stone、gothic、brutalist、y2k 等）。

  CLI 是理解 Astryx 设计理念的关键入口。它不是给人看的 `--help` 文本，而是一个**自描述的 Agent 清单**。Agent 一次调用就能拿到每个命令的参数类型、可选值、默认值和响应格式——相当于 CLI 自己提供了一份 OpenAPI 规范。`astryx build "用户登录页面"` 这个命令尤其值得展开：它返回一个按角色分组的"组合套件"——最接近的 PAGE 模板、始终包含的 FRAME（页面外壳）和 FOUNDATION（布局/排版/动作）原语、针对具体场景的 BLOCKS 和 DOMAIN COMPONENTS，以及 SETUP 提醒。实测这套结构把 Agent 的组件召回率从 15% 提升到 71%。

  10 套主题全都基于 CSS 变量级联，改一套变量（颜色、圆角、字体、动效），整个系统跟着变。你不会遇到"用了某个组件库之后整个应用看起来像那个公司的产品"的问题——主题层和组件层是完全解耦的。基于 React 和 StyleX（Meta 自家的 CSS-in-JS 方案），MIT 协议，40 位贡献者。

```bash
# 安装核心 + 主题
pnpm add @astryxdesign/core @astryxdesign/theme-neutral
pnpm add -D @astryxdesign/cli

# 初始化项目（安装依赖、配置主题、生成 Agent 文档）
npx astryx init

# 浏览所有组件
npx astryx component --list

# Agent 组合式构建（返回结构化组合方案）
npx astryx build "数据仪表盘主页"

# 生成自定义主题
npx astryx theme build --config my-theme.config.ts
```

## OmniRoute — 免费 AI 网关，一个端点接入 237 个 LLM 提供商，自动故障转移，token 压缩最高省 95%

  用 Claude Code 或 Codex 写代码的人，几乎都经历过这个循环：Claude Code 的订阅配额用完了 → 切到 Codex → Codex 限速了 → 换 Cursor → Cursor 的免费额度也见底了。每个工具用自己的 API Key、自己的额度、自己的限速策略。你想省钱用免费模型（Kiro、Qoder、Pollinations），但每个都要单独注册、单独配 SDK、单独记额度。管理成本比省下来的钱还高。

  OmniRoute 用一个本地网关解决了这个问题。它在 `localhost:20128` 起一个 OpenAI 兼容的 `/v1` 端点，后面接了 237 个 LLM 提供商，其中 90+ 有免费额度，11 个永久免费。你把 Claude Code、Codex、Cursor、Cline、Copilot 全部指向这一个端点，OmniRoute 负责背后的路由、fallback、格式转换和 token 压缩。

  核心路由引擎有 17 种策略，但日常最有价值的就三个场景。**Cost-optimized 路由**：简单问题自动走免费或廉价模型（GLM-4 Flash $0/百万 token、MiniMax M2.5 $0.3/百万 token），复杂问题才动你的订阅额度。**Priority fallback**：配额用完自动切下一个，毫秒级切换，你永远看不到"rate limit exceeded"。**Context-relay**：同一个会话跨模型切换时，自动把上下文从 Claude 格式转成 Gemini 格式再转成 OpenAI 格式，Agent 不知道背后换了模型。

  RTK + Caveman 双层 token 压缩是 OmniRoute 最被低估的功能。输入侧 RTK 做结构化压缩（去冗余空格、合并重复模板、压缩 JSON），输出侧 Caveman 去掉填充词和礼貌用语。代码、URL、JSON 字段受保护，不会被误压缩。官方数据显示工具密集型会话平均省 89% 的 token，轻度会话省 15-20%。如果你一个月花 $100 在 API token 上，压缩层大概能帮你降到 $11-40。

  内置的 MCP 服务器（95 个工具、3 种传输方式、30 个作用域）让你可以直接用 MCP 协议操作网关——查询额度、切换路由策略、查看压缩统计——不需要开浏览器。A2A（Agent-to-Agent）协议支持 6 个 skill，让 Claude Code 和 Codex 在同一个网关上协作。42 种语言的 i18n、JA3/JA4 TLS 指纹伪装（防止被提供商识别为自动化流量）、PII/注入/视觉三道护栏——这些属于"平时用不到、出问题时会救你一命"的能力。

  诚实地说注意事项：要充分利用 237 个提供商，你需要注册 15-20 个账号并填入 API Key。免费永久提供商（Qoder、Pollinations、LongCat）的服务质量波动较大——不适合关键任务，但做代码补全和简单问答足够。本项目开源 MIT，本地运行，你的 Key 不经过任何云端。

```bash
# 全局安装
npm install -g omniroute

# 启动网关（自动打开 Web 仪表盘）
omniroute

# 一键配置 Claude Code 走网关
omniroute setup-claude-code

# 一键配置 Codex CLI
omniroute setup-codex

# Docker 部署
docker pull diegosouzapw/omniroute
docker run -p 20128:20128 diegosouzapw/omniroute

# 查看当前免费额度
curl http://localhost:20128/dashboard/free-tiers
```

## VS-MCP — Visual Studio 的 Roslyn 驱动 MCP 服务器，41 个工具让 AI Agent 获得编译器级别的代码理解

🏪 Visual Studio Marketplace · ⬇ 4,236 安装 · (6) 评分 · Free · [安装](https://marketplace.visualstudio.com/items?itemName=LadislavSopko.mcpserverforvs)

  .NET 开发者在 AI 编码工具的浪潮中一直站在尴尬的位置。Claude Code、Codex CLI、Cursor 这些工具对 TypeScript/Python/Go 的支持已经很成熟了，但面对一个大型 C# 解决方案——几十个项目、几百个类、复杂的泛型继承链——它们退化成 grep。`FindSymbols` 被降级为文本搜索，`Go to Definition` 变成了文件名猜测。这是因为这些 Agent 工作在文件系统层面，而 .NET 代码的真正结构在编译器的符号表里。

  VS-MCP 的解法是把 Roslyn（C# 编译器）和 Visual Studio 调试器的完整能力暴露为 41 个 MCP 工具。Agent 搜索 `WhisperFactory` 时，grep 可能返回 47 个文本匹配——包括注释、字符串、测试数据里的偶然出现。VS-MCP 的 `FindSymbols` 用 Roslyn 的语义索引，返回的是 `Class, Whisper.net, line 16`——一个精确答案。Agent 想重命名 `ProcessDocument` 时，sed 会把 `ProcessDocumentAsync` 也改了。VS-MCP 的 `RenameSymbol` 用 Roslyn 做编译器验证的重构，23 个调用点全部安全更新。

  41 个工具分四层。**语义导航层**（13 个 Roslyn 工具）：`FindSymbols`、`FindSymbolDefinition`、`FindSymbolUsages`、`GetInheritance`（完整类型层次）、`GetMethodCallers`/`GetMethodCalls`（调用图上下游）。**代码分析层**：`GetDiagnostics` 获取 Roslyn 后台分析结果——不需要 build 就能看到编译错误和警告。**重构层**：`RenameSymbol` 解决方案级安全重命名，`FormatDocument` 调用 VS 原生格式化器。**调试层**（19 个工具，Preview）：完整的断点管理、单步执行、变量检查、表达式求值、Docker/WSL 进程附加——Agent 可以像人一样在调试器里排查运行时 bug。

  一个经常被忽略的细节是"紧凑输出"。VS-MCP 的工具响应被刻意压缩——不返回冗余的元数据、不重复已有的上下文。官方数据是比同类方案省 70-90% 的 token。按 Claude Code 的 token 计费来算，一个 2000 token 的符号查询响应被压缩到 200-600 token。

  兼容所有 MCP 客户端：Claude Code、Codex CLI、Gemini CLI、Cursor、Windsurf、Copilot、Cline 等。配置也简单——装好扩展，启动服务器（默认 `localhost:3010`），在 MCP 客户端里加一行配置即可。

```bash
# 在 Visual Studio 2022/2026 中
# 扩展 → 管理扩展 → 搜索 "MCP AI Server" → 安装

# Claude Code 配置（.mcp.json）
{
  "mcpServers": {
    "vs-mcp": {
      "type": "http",
      "url": "http://localhost:3010/sdk/"
    }
  }
}

# Cursor 配置（.cursor/mcp.json）
{
  "mcpServers": {
    "vs-mcp": {
      "url": "http://localhost:3010/sdk/"
    }
  }
}
```

  VS-MCP 和 VS Code 系的 MCP 扩展走的是两个方向。VS Code 的 MCP 扩展（vscode-mcp-server、VS Code MCP Bridge）做的是"把编辑器功能暴露给 Agent"——文件读写、终端执行、LSP 诊断。这些能力 Claude Code 本身已经具备了，装这些扩展的价值是让 Agent 留在 VS Code 里而不是切到终端。VS-MCP 做的是"把 Agent 不具备的能力给它"——Roslyn 的语义索引精度远超文件系统搜索，VS 调试器的断点/单步/变量检查是 Claude Code 完全做不到的。对于 .NET 开发者，VS-MCP 不是锦上添花，是把 Agent 从一个"文本搜索器"升级成了"有编译器大脑的开发者"。
