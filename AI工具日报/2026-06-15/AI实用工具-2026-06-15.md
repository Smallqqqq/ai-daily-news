# AI 实用工具 · 2026-06-15

![[summary.png]]

  今天三个工具分别解决了 AI 编码 Agent 的三个关键痛点：靠状态机让 Agent 不再瞎跑、靠开源全平台覆盖让 Agent 不挑编辑器、靠 MCP 桥接让 Agent 真正看懂你的 IDE。

---

## Statewright — 用状态机给 AI Agent 装上硬护栏，13B 本地模型也能稳定完成任务

  用 AI Agent 写代码的人迟早会撞上同一个问题：Agent 明明有能力完成任务，但它就是跑偏。读同一个文件五次、在应该修 bug 的时候去重构无关代码、自说自话删掉你没让它碰的东西。这不是模型不够聪明，而是"给一个模型 40 个工具然后说'去干吧'"本身就是有问题的设计。Statewright 的思路很直接：用状态机把 Agent 的工作流切成离散的阶段，每个阶段只开放该阶段需要的工具。

  原理不复杂但实现很扎实。你定义一个 JSON 格式的状态机工作流——比如 `planning → implementing → testing → completed`——每个状态只允许特定工具。在 planning 状态，Agent 只能 Read/Grep/Glob；进入 implementing 后 Edit 和 Write 解锁，但 Bash 只开放重定向写入（`rm`、`shred` 等破坏性命令被硬拦截）；testing 阶段只能跑测试命令。如果你调用当前状态不允许的工具，直接被拒绝，并告知你可以用什么、怎么迁移状态。

  核心引擎是 Rust 写的，确定性执行——状态转换逻辑里没有 LLM。上层通过 MCP 协议和 Claude Code 的 hooks 系统集成，工具拦截发生在协议层，模型根本看不到被禁止的工具。对 Codex、opencode、Pi、Cursor 也都有适配（Cursor 目前是 advisory 模式，因为 MCP 无法在 Cursor 架构中 gate 工具调用）。

  项目方用本地模型做的实验很有说服力。在 5 个任务的 SWE-bench 子集上，两个模型（gpt-oss:20b 和 gemma4:31b）加上 Statewright 约束后，成功率从 2/10 跳到了 10/10。同样的任务、同样的硬件。对于云端大模型，Statewright 的核心价值不再是"防止它删库"这种极端情况（大模型默认行为已经能避免），而是结构性地解决两个更常见的问题：读文件死循环（同一个文件读五次还不改）、以及工具集太大导致推理质量下降。

  工作流定义可以手写 JSON，也可以用 `statewright_create_workflow` 让 Agent 自己生成。视觉编辑器 statewright.ai 提供了可视化编辑界面。免费套餐供个人开发者 3 个工作流、每月 200 次状态转换；Pro $29/月扩展到 10 个工作流和 2500 次转换。引擎核心 Apache 2.0 开源，可嵌入自托管。

```bash
# Claude Code 一键安装
/plugin marketplace add statewright/statewright
/plugin install statewright
/reload-plugins

# 启动 bugfix 工作流
/statewright start bugfix
```

  Statewright 和 prompt 工程不是替代关系——它是 prompt 之上的硬约束层。Hooks 能做类似的拦截，但需要自己写状态逻辑；Statewright 把状态机变成了声明式配置。Guardrails-AI 做的是输出验证（"这个回复合不合规"），Statewright 做的是过程控制（"你现在只能做这些事"），两者维度不同。该工具在此细分场景下几乎没有直接竞品——用状态机约束 AI Agent 工具调用是一个尚未被充分探索的方向。

## Kilo Code — 开源 AI 编码 Agent 的完全重构版，VS Code + JetBrains + CLI 一个引擎全通

  如果你同时用 VS Code 写前端、用 JetBrains 写后端、偶尔在终端里用 Claude Code 做重重构，那你一定有过这种体验：每个编辑器的 AI 能力是割裂的。VS Code 里装的 Copilot 不会出现在 JetBrains 里，终端里的 Claude Code 不知道你 IDE 里正开着什么文件。Kilo Code 用一套思路解决了这个问题——它基于 OpenCode server 重建了一个可移植核心引擎，同一套 Agent 逻辑在 VS Code、JetBrains、CLI 和云端四个表面共用。

  六月这次重构是 2026 年以来最大的一次更新。核心变化是把 VS Code 扩展从依赖 VS Code 内部 API 的架构，迁移到了一个与编辑器无关的 OpenCode server 引擎上。这意味着新功能只要在引擎层实现一次，所有平台同步获得——子 Agent 委派、并行工具调用、Agent Manager、Git Worktree 隔离、行级代码审查，都不再是某个平台的独占能力。

  几个实用细节值得展开。**Agent Manager**：在 IDE 里同时跑多个 Agent——一个在写代码、一个在审查、一个在调试——每个 Agent 可以跑在独立的 git worktree 里。**500+ 模型自由切换**：前端任务用 Claude Sonnet 4.6、重型重构用 Opus 4.7、省钱场景用 DeepSeek，不需要换工具。**MCP Server Marketplace**：直接在 IDE 里浏览和安装 MCP 服务器，一键接入数据库、API、文件系统。**代码审查**：`/local-review` 审查当前分支 vs 基准分支的全部改动，审查结果直接以行级评论贴在编辑器里。6 月 12 日刚更新的版本让审查规则能自动适配项目的 conventions。

  Kilo Code 是 Apache 2.0 开源的，你可以 fork、修改、自托管。没有静默模型切换——你始终能看到当前在用什么模型、上下文窗口还剩多少、完整 prompt 长什么样。这个透明性在商业产品里很少见。

```bash
# VS Code 扩展安装
code --install-extension kilocode.Kilo-Code

# 或通过 CLI（与扩展共用同一引擎）
# 参考 kilo.ai 文档配置
```

## Pylon MCP — 把 VS Code 的终端、诊断、符号导航完整暴露给 AI Agent

  AI Agent 在终端里很能干，但它完全看不见你的 IDE 里正在发生什么——TypeScript 的类型错误、ESLint 的警告、终端的运行状态、文件符号结构，这些 VS Code 实时掌握的信息 Agent 一概不知，只能靠 grep 和猜。Pylon MCP 是一个零依赖的 VS Code 扩展，在本地启动一个 HTTP MCP 服务器，把 VS Code 的终端管理、诊断信息、LSP 符号导航、调试器控制完整暴露给任何 MCP 兼容的 AI Agent。

  架构设计很巧妙。每个 VS Code 窗口跑一个内置的 AgentServer（端口 27691+），第一个启动的窗口同时跑 Hub（端口 27681），作为唯一的 MCP 入口。后续窗口自动注册为卫星节点。Agent 调用 `terminal_list_sessions` 发现所有连接的窗口，然后用 `session_id` 指定操作目标。Hub 如果关闭（比如你关掉了第一个窗口），卫星窗口自动检测心跳丢失并在 5 秒内升级为新的 Hub。

  工具覆盖四个层面。**终端**：列出所有终端、执行命令并捕获输出（支持 shell integration 的命令边界检测）、发送按键（Ctrl+C/D）、读取缓冲输出。**IDE**：获取 VS Code 诊断信息（TypeScript 错误、ESLint 警告等）、LSP 文档符号大纲（不读文件就能了解结构）、查找引用、LSP 重命名。**编辑器**：打开文件并跳转到指定行、格式化文档、整理 imports、一键修复所有自动修复项。**调试器**：设置断点、启动/停止调试会话、获取完整调试状态快照（线程、调用栈、作用域、变量）。

  实际场景：Agent 改完代码后调用 `get_diagnostics` 检查是否引入了新错误；重构前用 `get_references` 确认影响范围；用 `terminal_run` 启动 dev server 然后 `terminal_read_output` 看启动日志。关键是这一切不需要 Agent 在终端里跑 `tsc` 或 `eslint`——诊断信息直接来自 VS Code 的实时语言服务。

```bash
# 构建安装
npm run compile
npm install -g @vscode/vsce
vsce package
code --install-extension pylon-mcp-0.5.0.vsix

# Claude Code 配置
claude mcp add --transport http pylon http://127.0.0.1:27681/mcp
```
