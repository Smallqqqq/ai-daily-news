# AI 实用工具 · 2026-06-29

![[summary.png]]

  今天两个工具，一个让 VS Code 的代码补全不再依赖云端 API，一个把浏览器变成了 AI Agent 的"可读写文件系统"——分别踩中了代码补全的本地化需求和浏览器自动化的确定性交互两个方向的最新落点。

---

## Leyline — VS Code 代码补全，自带 API Key，支持 Ollama 本地模型，零遥测

🏪 VS Code Marketplace · v0.4.1 · MIT License · [GitHub](https://github.com/pasqualesalza/vscode-leyline)

  大多数 VS Code 的 AI 代码补全工具都有一个共同的隐形成本：你敲的每一行代码都会被发送到某个云服务。GitHub Copilot 发给微软，Tabnine 发给他们的服务器。如果你在做涉密项目、离线开发、或者单纯不想让自己的代码离开机器，这些工具都不适用。Leyline 的定位就是填补这个空白——一个完全本地可控的代码补全扩展。

  Leyline 本身不内置任何模型，它只是一个"插座"。你把 API 端点插上去，它在后台帮你做代码补全。支持两条路径：一是 Codestral（Mistral 的 FIM 模型），填个 API Key 就能用；二是 Ollama 本地模型，装好 `qwen2.5-coder`、`codellama` 或 `deepseek-coder`，补全请求全部走本机推理，不经过任何外部网络。按 `Alt+\` 手动触发补全，或在设置中开启自动触发——像一个完全在你掌控之下的 Copilot。

  一个容易被忽略但很实在的设计：跨文件上下文。补全不只是看当前文件，还会把其他已打开文件中相关的类型定义和函数签名拉进来。这意味着你在写 `processOrder` 的时候，Leyline 知道 `Order` 类型长什么样、`validateOrder` 接受什么参数——跟你自己切窗口去看是一样的，但不需要你切。

  没有遥测。只有你配置的那个端点会收到代码片段。没有第三方收集使用数据。如果你用 Ollama，那就连端点都不涉及——代码完全不出机器。对于合规环境、金融系统、涉密项目，这个"不收集、不上传"的设计比任何功能都更有分量。

```bash
# 在 VS Code 扩展面板搜索 "Leyline" 安装
# 或命令行安装：
code --install-extension pasqualesalza.vscode-leyline

# 使用 Codestral：命令面板 → Leyline: Set API Key → 填入 Codestral API Key
# 使用 Ollama：先装好模型（ollama pull qwen2.5-coder），
#              命令面板 → Leyline: Select Provider → 选 Ollama
# 编辑器中 Alt+\ 手动触发补全
```

## DOMShell — 把浏览器变成文件系统，AI Agent 用 ls、cd、cat 操控网页

🏪 Chrome Web Store · ⬇ 503 用户 · v2.0.1 · MIT License · [GitHub](https://github.com/apireno/DOMShell) · [Chrome 安装](https://pireno.com/domshell)

  AI Agent 操控浏览器的主流方案有两个流派：截图流（给模型看页面截图，让它预测点击坐标）和选择器流（让模型写 CSS 选择器或 Playwright 脚本）。两种方案各有一个致命弱点——截图流消耗大量 token 且坐标预测不准，选择器流需要针对每个网站写适配逻辑且 DOM 结构一改就崩。DOMShell 走了一条完全不同的路：把浏览器的无障碍树映射成一个虚拟文件系统，Agent 用 `ls`、`cd`、`cat`、`grep`、`click` 这些每个模型都会用的 Unix 命令来操控网页。

  原理不复杂但很巧妙。Chrome 内置的无障碍树（Accessibility Tree）本来就是浏览器对页面结构的语义化描述——哪些元素是导航栏、哪些是按钮、哪些是输入框、它们之间的层级关系是什么。DOMShell 用 Chrome DevTools Protocol 读取这棵树，把容器元素映射为目录，把交互元素映射为文件。浏览器标签页变成 `~/tabs/` 下的子目录，页面内的导航栏变成 `navigation/` 目录，按钮变成 `submit_btn` 这样的"文件"。

  这套抽象带来的核心收益是确定性和低 token 消耗。Agent 不需要解析截图、不需要猜坐标、不需要写选择器。`cd main/article` 进入文章区域，`cat` 读取内容，`find --type button` 找所有按钮，`click submit_btn` 点击提交——每一步都是精确操作，不存在"点偏了"或"选择器过期了"的问题。在实际测试中，DOMShell 完成相同任务所需的 API 调用次数约是 Computer-in-the-Cloud 方案的一半（平均 4.3 vs 8.6 次）。

  DOMShell 内置了 39 个 MCP 工具，分四级权限：Read（`ls`、`cat`、`find`、`grep`、`text` 等）、Navigate（`navigate`、`open`、`back`）、Write（`click`、`type`、`js`）、Sensitive（`whoami`）。安装 Chrome 扩展后，Agent 通过 HTTP（localhost:3001）或 WebSocket（localhost:9876）连接，支持 Claude Code、Cursor、Windsurf、Gemini CLI 等任何 MCP 兼容客户端。

  另一个实用特性是脚本批处理。`script save` 把一组命令保存为脚本，`for` + `each` 在多标签页上并行执行——O(2n) 的工具调用变成 O(2)。对于需要批量抓取、多页监控、重复性 QA 测试的场景，这个效率增益是量级上的。

```bash
# 1. 从 Chrome Web Store 安装 DOMShell 扩展
#    https://pireno.com/domshell

# 2. 安装 MCP 服务器
npm install -g @apireno/domshell

# 3. 注册到 Claude Code
claude mcp add domshell -- npx -y @apireno/domshell@latest --allow-write

# 4. Agent 即可使用 ls/cd/cat/click 等命令操控浏览器
#    例如：在 Claude Code 中直接说"帮我在 github.com 上搜索 mcp server"
```

| Browser MCP            | 工具调用 + 截图                         |      中      |    部分     |       npm       | ⭐ 较新         |

  DOMShell 的最大差异化优势是"用 Agent 已有的知识解决新问题"。每个 LLM 都知道 `ls`、`cd`、`grep` 怎么用——这些命令在训练数据中出现了无数次。你不需要让模型学一套新的浏览器操作范式，它只需要把"浏览网页"理解为"浏览文件系统"。这个思维模型转换带来的零样本可用性是 Playwright MCP 和 agent-browser 都不具备的。

  目前 DOMShell 的 GitHub 关注度还不高（44 stars），Chrome Web Store 用户 500 出头，属于典型的"思路独到但还不出圈"的阶段。如果你需要一个让 AI Agent 稳定、低消耗地操控浏览器的方案，且不介意做一点初始配置，这个工具值得花一个下午试一试。
