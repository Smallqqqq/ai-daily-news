## Weave Router — 一个代理端点，自动把每次 AI 请求路由到最便宜但足够强的模型

  你用 Claude Code、Codex 或 Cursor 写代码，Agent 后台发的每一次 API 请求——规划架构、实现功能、生成测试、写文档——默认都走同一个模型。规划时确实需要 Opus 的推理能力，但写测试、生成文档这些活，DeepSeek 或 GLM 完全够用，价格却差一个数量级。你不可能手动给每次请求切换模型。

  Weave Router 的解法是把路由决策交给一个不到 50ms 延迟的本地代理。它在本地 8080 端口启动一个兼容 Anthropic Messages API、OpenAI Chat Completions API 和 Gemini 原生协议的代理服务。每次请求到达时，内嵌的 cluster scorer（基于 Avengers-Pro 论文，RouterArena 排行榜第一）分析请求的复杂度和语义，自动选择最便宜但能胜任的模型。

  核心技术路径：路由器不是靠 prompt 去"猜"该用哪个模型——那样每次路由决策本身就是一次 LLM 调用，得不偿失。它用一个轻量级的 on-box embedder 做语义分类，开销不到 50ms。路由决策完成后，请求被转发到对应的 provider（Anthropic、OpenAI、OpenRouter 等），响应原样返回。工具端感知不到中间多了一层代理，只是账单变少了。

  对日常开发最有价值的两个功能：一是**按 phase 路由**。Claude Code 在规划阶段发复杂推理请求→ Opus/Sonnet；实现阶段发中等复杂度请求→ Sonnet/DeepSeek V4；测试和文档阶段发简单请求→ GLM/Kimi。二是在 Weave Dashboard（localhost:8080/ui/dashboard）看到完整的 OTLP trace，每笔请求走了哪个模型、花了多少钱、路由理由是什么，全透明。

```bash
# 一行安装，自动检测 Claude Code/Codex/OpenCode 并配置
npx @workweave/router

# 指定工具（跳过交互选择）
npx @workweave/router --claude       # Claude Code
npx @workweave/router --codex        # OpenAI Codex CLI
npx @workweave/router --opencode     # OpenCode

# 自托管（需要 Docker + Postgres）
git clone https://github.com/workweave/router.git
cd router
echo "OPENROUTER_API_KEY=sk-or-v1-..." >> .env.local
make full-setup
```

### 同类工具对比

| 工具 | 路由方式 | 编码感知 | 延迟 | 自托管 | 典型节省 | 价格 |
|------|---------|:---:|------|:---:|------|------|
| **Weave Router**（今日推荐）| embedder 语义分类 | ✅ 按 phase 路由 | <50ms | ✅ ELv2 | 40-70% | 免费自托管 / 托管版免费 |
| CodeRouter | phase 感知路由 | ✅ | ~30ms | ❌ 仅云 | 70-90% | $19-299/月 |
| OpenRouter | 手动选模型 | ❌ | ~40ms | ❌ | 0%（仅省中间费） | 5.5% 加价 |
| LiteLLM | 规则路由 | ❌ | 50ms+ | ✅ OSS | 平台费节省 | 免费（OSS） |
| Portkey | 规则+缓存路由 | 部分 | ~40ms | 部分 | 中等 | $49/月起 |

  Weave Router 的定位很明确：不是模型市场（OpenRouter 有 600+ 模型，它只用 15+ 编码验证过的），不是企业网关（Portkey 有合规和策略引擎），而是专为 AI 编码 Agent 做的省钱代理。提供 BYOK（自带 Key），数据不离开你的机器。相比 CodeRouter 云服务省钱更多但需月费，Weave Router 自托管零成本、开源可审计、RouterArena 排行榜第一——这三条加在一起，目前没有直接竞品能做到。

## DebugMCP — VS Code 扩展，让 AI Agent 直接控制调试器

🏪 VS Code Marketplace · ⬇ 3,983 次安装 · v2.0.0 · Microsoft DevLabs 维护 · [安装](https://marketplace.visualstudio.com/items?itemName=ozzafar.debugmcpextension)

  AI 编码 Agent 最让人抓狂的时刻：它改了代码，测试挂了，然后开始猜——"可能是这里的问题，让我改一下这个条件判断"。你盯着屏幕心里清楚它在瞎蒙，因为它没有运行时的信息。它看不到变量值、不知道哪行代码实际被执行了、不知道异常发生时的调用栈长什么样。

  DebugMCP 把这些信息全部开放给 AI Agent。装了这个 VS Code 扩展后，你的 Agent 获得了完整的调试器控制权：设断点、启动调试会话、单步执行（step over/into/out）、查看所有变量和调用栈、评估任意表达式。22 个 MCP 工具覆盖了人类开发者使用 VS Code 调试器的全部操作。

  2.0 版本加入了一个关键功能：`start_debugging` 现在通过 VS Code Testing API 发现和启动测试，而不是靠字符串匹配去猜测试名。这对 pytest、Jest/Vitest、Java JUnit、.NET xUnit、Go testing 等所有对接了 Testing API 的测试框架都生效，断点能稳定命中单个测试用例内部。之前的最佳尝试（best-effort）路径在复杂测试文件里经常定位不到正确的测试函数。

  另一个值得注意的设计：DebugMCP 安装后会自动向所有支持的 AI 助手注册自己——GitHub Copilot、Copilot CLI、Cline、Cursor、Codex、Windsurf、Roo Code、Antigravity——不需要手动配置 MCP JSON。同时附带一个 `/debug` Agent Skill，自动安装到各助手的 skills 目录，调用时加载系统化的调试工作流，确保 Agent 按正确的顺序使用调试工具。

```bash
# 从 VS Code Marketplace 安装
# 搜索 "DebugMCP" 或使用直接链接：
# vscode:extension/ozzafar.debugmcpextension

# 安装后自动注册到所有 MCP 兼容的 AI 助手
# 在支持 /debug skill 的助手中，直接调用：
/debug
```

  同类工具中，JetBrains Debugger MCP 和 VS MCP Server 分别在 IntelliJ 和 Visual Studio 上提供了类似能力，但 DebugMCP 有两个独特优势：一是 VS Code 用户基数最大，大部分 AI 编码 Agent 的主要载体就是 VS Code；二是自动注册机制——其他工具需要手动复制 JSON 配置，DebugMCP 弹出提示框一键完成。对于 "让 AI Agent 帮我 debug" 这个需求，DebugMCP 是目前门槛最低的选择。Microsoft DevLabs 官方维护也意味着它能跟进 VS Code 的调试 API 变更。
