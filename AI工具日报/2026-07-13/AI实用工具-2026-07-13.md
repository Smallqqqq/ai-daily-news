# AI 实用工具 · 2026-07-13

![[summary.png]]

  今天两个工具都是"给 AI Agent 套上缰绳"——但套的不是同一个部位。一个在你和 Agent 的 shell 之间加了道 <1ms 的闸门，Agent 跑任何命令之前先过一遍 50+ 条安全规则。另一个在 Agent 生成 UI 之后再做一次"去 AI 味"审计——不是检查代码能不能跑，是检查代码有没有审美。

## destructive_command_guard — 给 AI Agent 的终端命令装上 <1ms 的安全闸门

  用 Claude Code 或 Codex CLI 写过代码的人早晚会经历那个瞬间：Agent 跑了一个 `git reset --hard HEAD~5`，把一上午没 commit 的改动全丢了。或者 `rm -rf ./src` 真的把 src 删了。Agent 不会犹豫——你让它修 bug，它判断最快路径就是重置到干净状态重来。它不在乎你 stash 里有什么。

  dcg（Destructive Command Guard）做的事很简单：在 AI Agent 执行任何 shell 命令之前拦截它。它是一个 Rust 编写的 PreToolUse hook，作为 JSON stdin 接收命令，四阶段流水线处理：先拆复合命令（`&&`、`||`、`;`、管道），再 O(n) 快速过滤没有危险关键词的命令（覆盖 99%+ 的正常命令），然后是安全模式白名单先行、危险模式黑名单拦截，最后输出——被拦截时 stdout 给机器可读 JSON、stderr 给人可读彩色面板。整个链路亚毫秒级延迟，你根本感觉不到它的存在。

  50+ 安全规则包覆盖了 Agent 最可能闯祸的所有领域：Git 操作（`reset --hard`、`push --force`、`branch -D`）、文件系统（`rm -rf`、`chmod 777`、`find -delete`）、数据库（`DROP TABLE`、`TRUNCATE`）、Docker（`system prune`、`volume rm`）、Kubernetes（`delete namespace`）、AWS/GCP/Azure CLI 的危险操作、Terraform 的 `destroy -auto-approve`。默认只开启基础包，需要更多保护时在 `~/.config/dcg/config.toml` 里加一行 `enabled = ["database.postgresql", "kubernetes.kubectl", "cloud.aws"]`。

  两个关键设计让它区别于简单的正则匹配。第一，**上下文感知**——`grep "rm -rf" log.txt` 不会被拦，因为 `rm -rf` 在字符串参数里，不是被执行的命令。`dcg scan` 模式能读懂文件结构（shell 脚本、Dockerfile、CI workflow、Makefile、package.json scripts），只在真正会执行的命令上做检查，注释和字符串里的危险关键词一律放行。第二，**Agent 专属配置文件**——你可以对不同 Agent 设置不同信任级别。Claude Code 设 `trust_level = "high"`，加白名单放行 `npm run build`；未知 Agent 设 `trust_level = "low"`，多加载偏执规则包。

  支持 12 种 AI 编码工具：Claude Code、Codex CLI、Gemini CLI、GitHub Copilot CLI、VS Code Copilot Chat、Cursor、Hermes Agent、Grok (xAI)、Antigravity CLI、OpenCode、Pi、Aider。安装一条 curl 命令，自动检测平台、下载正确二进制、配置所有已安装 Agent 的 hook。Windows 原生支持 PowerShell 安装器。

```bash
# macOS / Linux / WSL（一键安装）
curl -fsSL "https://raw.githubusercontent.com/Dicklesworthstone/destructive_command_guard/main/install.sh" | bash -s -- --easy-mode

# Windows（PowerShell 原生）
& ([scriptblock]::Create((irm "https://raw.githubusercontent.com/Dicklesworthstone/destructive_command_guard/main/install.ps1"))) -EasyMode -Verify

# 安装后验证
dcg explain "git push --force"    # 查看为什么被拦截
dcg explain "rm -rf /tmp/cache"   # 有时 /tmp 路径在白名单中

# 启用更多规则包
mkdir -p ~/.config/dcg
cat > ~/.config/dcg/config.toml << 'EOF'
[packs]
enabled = ["database.postgresql", "kubernetes.kubectl", "cloud.aws", "containers.docker"]
EOF

# CI 扫描模式（检查代码中是否有危险命令）
dcg scan ./scripts/
```

## Hallmark — 给 AI 生成的 UI 装上审美闸门，57 道检查拒绝"一看就是 AI 做的"

  用 v0、Bolt.new、Lovable 这些 AI 设计工具的人都在经历同一件事：生成出来的页面"能用"，但所有人都能一眼认出这是 AI 做的。同样的留白比例、同样的配色套路、同样的卡片圆角、同样的渐变按钮。丑倒不丑，就是千篇一律。模型在训练数据里看到最多的就是这些"安全""不出错"的设计模式，它学会了走最宽的马路。

  Hallmark 的解法是：在 AI 输出 UI 代码之后、呈现给用户之前，跑 57 道"slop test"闸门。它不是检查代码能不能运行——它检查的是设计有没有个性。先选一个宏观布局结构，再从 20 套主题中挑一套穿上，跑完 57 道反模板化检查，最后做一次 pre-emit 自我批判。同一个 prompt 用 Hallmark 跑两次，会得到两个感觉完全不同的页面——不是同一套模板换了颜色。

  四个核心命令（Hallmark 称为"verbs"）对应四种使用场景。默认模式是**从零生成**——给一句描述，Hallmark 选布局、选主题、跑检查、输出 HTML+CSS。`hallmark audit` 是**审计现有代码**——把已经生成的页面丢进去，按 57 条反模式逐项打分，输出 punch list 而不是直接改。`hallmark redesign` 是**翻新**——保留原文案、信息架构和品牌色，扔掉布局结构，用不同的风格指纹重建。最独特的是 `hallmark study`——给它一个你喜欢的网站 URL，它提取"设计 DNA"：宏观布局、字体配对、色彩锚点，输出一份可携带的 `design.md`，可以交给其他 AI 工具作为设计约束。明确拒绝像素级复制和有版权的付费模板。

  20 套主题覆盖了从极简到戏剧化的光谱：Sourdough（暖色调面包店）、Cobalt（冷峻科技蓝）、Carnival（大胆撞色）、Garden（自然绿植）、Lumen（亮白干净）、Riso（粗粝印刷风）……每套主题不仅仅是颜色不同——字体、间距、动效节奏、信息密度都不一样。当 prompt 中蕴含的创意意图在现有主题中找不到匹配时，Hallmark 自动切换到 Custom 模式，从零设计配色、字体和布局。

  Nutlope（Together AI 的知名开发者）出品，MIT 协议。作为一个 Claude Code/Cursor/Codex 的 skill 分发，安装一行命令。输出是自包含的 HTML+CSS——没有任何框架依赖，可以直接部署。

```bash
# 安装 skill（支持 Claude Code / Cursor / Codex）
npx skills add nutlope/hallmark

# 在 Claude Code 中使用
# 默认生成：
#   "Build a landing page for a sourdough baking app"
# 审计现有代码：
#   "hallmark audit index.html"
# 翻新（保留内容，换风格）：
#   "hallmark redesign index.html --theme carnival"
# 提取设计 DNA：
#   "hallmark study https://stripe.com"  → 输出 design.md

# 手动安装（Cursor）
# 复制 skills/hallmark/SKILL.md 的内容到 .cursor/rules/hallmark.mdc
```

  Hallmark 和 v0/Bolt/Lovable 解决的不是同一个问题——后者负责"把需求变成能跑的代码"，Hallmark 负责"让这些代码看起来不像模板"。真正独特的是 `study` 和 `audit` 两个命令：`study` 让你把喜欢的设计变成可复用的约束文件，`audit` 让你在任何 AI 生成的页面上跑质量检查。这两件事目前的 AI 设计工具都做不了。57 道 slop test 的规则集也是目前开源的独一份——没有其他工具在"防 AI 味"这个具体问题上投入了同等工程资源。
