# AI 实用工具 · 2026-06-04

![[summary.png]]

  今天两个工具解决 AI 辅助开发中两个被忽视的基础设施问题：前端组件文档写完就过时、agent 每次新会话都从零开始。Story UI 把 Storybook 故事生成自动化，pmb-ai 给 agent 装上不丢失的本地记忆。

---

## Story UI — 用自然语言生成 Storybook 故事，支持任意框架和设计系统

  写组件文档是前端开发中最容易被跳过的一步——功能做完了，Storybook 故事还没写。等到三个月后有人问"这个 Select 组件有多少种状态组合"，只能去翻源码。Story UI 用 AI 把这件事自动化了：描述你想要的，它生成完整可运行的 Storybook 故事文件，使用你项目里已有的组件库、正确的 import 路径和 TypeScript 类型。

  工作流程极其直接。在项目里 `npm install -D @tpitre/story-ui` 然后 `npx story-ui init`，会引导你选择框架（React / Vue / Angular / Svelte / Web Components）、设计系统（Mantine / Vuetify / Angular Material / Shoelace / shadcn/ui 等任意组件库）和 AI 供应商（Claude / OpenAI / Gemini）。然后你就可以用自然语言描述需要什么样的 story——"生成一个 DataTable 组件的所有变体故事，包括空状态、加载中、排序、筛选、分页、行选择"——Story UI 生成 `.stories.tsx` 文件，带正确的组件 import、props 和 TypeScript 类型，直接写到你的 stories 目录，Storybook 的文件监听器自动捡起来。

  它的自我修复机制值得专门提一下：生成的代码先过 TypeScript AST 解析，如果编译失败——比如 import 路径不对、props 类型不匹配——错误信息会作为纠错 prompt 反馈给 LLM，自动重试。这个 retry loop 对实际体验影响很大：手写 story 时你也会 typo，AI 生成时同样会，但有自动修复就意味着你不会被这些低级错误打断。

  Voice Canvas 模式是个有趣的实验特性：说话描述组件，story 实时渲染在 Storybook 里。MCP 集成让 agent 可以直接读取组件文档和已有的 story 模式来提升生成质量。项目自 2025 年 6 月发布以来已经迭代了 90 个版本，MIT 协议，两个维护者。

```bash
npm install -D @tpitre/story-ui
npx story-ui init
npm run story-ui
```

### 同类工具对比

| 工具 | 定位 | 框架支持 | AI 供应商 | 自我修复 | 热度 |
|------|------|---------|----------|---------|------|
| **Story UI**（今日推荐）| AI 生成 Storybook 故事 | React/Vue/Angular/Svelte/Web Components | Claude/OpenAI/Gemini | AST 解析 + retry | ⭐ 185 |
| Chromatic（Storybook 官方）| 视觉回归测试 + CI | 与 Storybook 相同 | 无 AI 生成 | N/A | Storybook 生态核心 |
| 手写 stories | 人工编写 | 任意 | 无 | 人工修复 | — |

  Story UI 在这个细分场景几乎没有直接竞品——Chromatic 做的是"故事写好之后的视觉测试"，Story UI 做的是"帮你把故事写出来"。两者互补：Story UI 生成初始故事和变体覆盖，Chromatic 在 CI 里跑视觉回归。目前没有其他工具专攻"AI 生成 Storybook 故事"这个具体问题。

## pmb-ai — 给编码 agent 装上不丢失的本地记忆

  每次打开 Claude Code 或 Cursor 开始新会话，agent 对你项目的理解都从零开始——它不知道你昨天做了什么决策、上周踩过什么坑、上个月定了什么规范。CLAUDE.md 能存一部分，但它是静态文件，不会从对话中学习。pmb-ai 是一个本地优先的持久化记忆 MCP 服务器，专门解决这个问题：agent 在对话中产生的知识自动存入本地数据库，下次会话自动召回，不需要 API key，不需要云服务。

  技术上它不是一个简单的"把对话存到向量库然后相似度搜索"的方案。它的核心是一个多层检索管道：关键词匹配 → BM25 文本检索 → 模糊搜索 → 语义向量搜索。在 LoCoMo（Long Conversation Memory）基准测试上达到 94.5% recall@10，中位延迟 70ms。支持多语言（中文、英文、日文等），所有数据存在本地 SQLite + 文件系统中，代码和数据都不出机器。

  使用方式简单：`pip install pmb-ai`，然后加到 MCP 配置里。之后 agent 在对话中产生的"记忆"——项目约定、踩坑经验、架构决策、偏好设置——就自动持久化了。下次新会话开始时，相关记忆会自动注入上下文。它还带一个 PreCompact hook：当对话上下文即将被压缩时，自动把当前会话的关键信息存入记忆，避免压缩导致的遗忘。

  这在多项目场景尤其有用：你在项目 A 里学到"这个库的 v3 版本改变了 authentication API 的调用方式"，切换到项目 B 时如果也用到同一个库，记忆会自动召回这条经验。相比 CLAUDE.md 的静态手写，pmb-ai 是活的——随着你用的越多，agent 对你的项目和偏好的理解越深。

```bash
pip install pmb-ai

# 加到 Claude Code 的 MCP 配置
claude mcp add pmb -- python -m pmb_ai
```

### 同类工具对比

| 工具 | 存储方式 | 检索方式 | 多语言 | 本地优先 | 热度 |
|------|---------|---------|--------|---------|------|
| **pmb-ai**（今日推荐）| SQLite + 文件系统 | 关键词 → BM25 → 模糊 → 语义（四层管道）| ✅ | ✅ 零 API | ⭐ 73 |
| MemPalace | ChromaDB 向量库 | 语义 + L0/L1 分层 | 部分 | ✅ | ⭐ 较新 |
| Hindsight | 向量库 | BEAM 基准验证（10M token 级）| 部分 | ✅ 需 Docker | ⭐ 较新 |
| Mem0 | Qdrant + SQLite | 纯语义向量 | 部分 | ❌ 云优先 | ⭐ 41K |
| agentmemory | 自定义 | 53 个 MCP 工具 | 部分 | ✅ | ⭐ 1.5K |

  pmb-ai 在这个赛道里最突出的两点：**零依赖启动**（不需要 Docker、不需要 Neo4j、不需要 ChromaDB）和**四层检索管道**（不是纯语义搜索，而是从精确到模糊逐层递进）。MemPalace 的 170 token 启动开销最小但只有两层分层；Hindsight 在超大规模（10M token）下表现最好但需要 Docker；Mem0 生态最大但是云优先；agentmemory 工具最多（53 个）但 token 开销也最大。如果你的需求是"装完就能用，本地跑，agent 自动记住该记住的东西"，pmb-ai 是路径最短的选项。94.5% 的 recall 意味着 agent 几乎不会漏掉你真正需要它回忆起来的内容。
