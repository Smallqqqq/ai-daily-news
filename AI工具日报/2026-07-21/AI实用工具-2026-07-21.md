# AI 实用工具 · 2026-07-21

![[summary.png]]

  今天两个工具分别解决 AI Agent 实践中两个被忽视的基础问题：循环什么时候该停（不是靠猜一个数字），以及本地小模型做工具调用时谁来兜底（不是靠祈祷）。

---

## LoopGain — 用 1921 年的控制论公式替代 `max_iterations=20`，让 Agent 循环在收敛时自动停止，不再烧钱跑空转

  几乎所有 AI Agent 的 verify-revise 循环都靠一个魔法数字终止：`max_iterations=20`。没人能解释为什么是 20——太早截断还在改进的循环，太晚则为已经发散的回答继续买单。LoopGain 的解决方案来自一个你意想不到的方向：1921 年 Barkhausen 提出的反馈系统稳定性判据——环路增益 Aβ < 1 则收敛，≥ 1 则振荡或发散。Agent 循环本质就是一个反馈系统：输出进去，打分出来，再喂回去。

  LoopGain 在每个迭代计算 Aβ = E(n)/E(n-1)（当前误差/上一轮误差），用一个四特征轨迹分类器把循环标为五种状态之一：FAST_CONVERGE（累计误差降到初始 10% 以下，继续）、CONVERGING（负斜率且统计显著，继续但要盯住）、STALLING（无显著变化，连续两次就停）、OSCILLATING（高残差方差+平坦趋势，停）、DIVERGING（正斜率且累计 > 110%，立即回滚到最佳版本）。它不只是"停"——还维护一个滚动缓冲区，返回 argmin(error) 对应的输出，而不是最后一次迭代可能已经劣化的版本。

  在 2000 次配对的真实 API 跑分中（五种循环模式 × 六种框架适配器 × 三种模型提供商，预注册协议+六项 kill 标准），LoopGain 相比 `max_iterations=20` 将 API 总支出砍掉 92.8%（$27.05 → $1.94），中位端到端延迟压缩约 15 倍（30.9s → 2.1s）。跨供应商评判者在 1800 次成对比较中加权偏好 LoopGain 的输出（0.678）——主要因为它返回的是整个循环中误差最低的那版输出，而固定上限法交出来的是最后一版的，有时候已经比中间某版退化了。

```bash
pip install loopgain
```

```python
from loopgain import LoopGain

lg = LoopGain(target_error=0.0)

while lg.should_continue():
    errors = verify(output)        # 你的验证器返回的错误计数
    lg.observe(errors, output=output)
    output = revise(output, errors)

result = lg.result                 # .best_output, .outcome, .savings_vs_fixed_cap
```

### 同类工具对比

| 工具 | 安装方式 | 核心特点 | 价格 | 热度 |
|------|---------|---------|------|------|
| **LoopGain**（今日推荐）| `pip install loopgain` | 控制论驱动的实时收敛检测+最佳版本回滚，六种框架适配器 | 开源 Apache 2.0 | ⭐ 73 · HN 首页 |
| agent-loop-guard | `pip install agent-loop-guard` | 滑动窗口检测重复/模糊重复/循环模式 | 开源 | ⭐ 极少 |
| ai-loopguard | `pip install ai-loopguard` | 检测卡死模式后升级到更强模型 | 开源 | ⭐ 极少 |
| max_iterations=N | 内置 | 固定迭代上限——所有人都用，没人能解释为什么是这个数 | 免费 | 通用 |

  agent-loop-guard 和 ai-loopguard 也能检测循环，但 LoopGain 把停止决策建立在控制论的数学基础上，并且区分了五种不同的循环状态。更关键的区别：它不是观察工具（LangSmith、Langfuse 只能事后看），而是坐在循环里面实时做决策的。唯一需要警惕的是：LoopGain 检测的是收敛，不是正确性——如果验证器本身有盲区（比如跑过了所有内环检查但通不过完整的 held-out 测试），LoopGain 会自信地停在一个错误的答案上。作者的基准测试中这个比例为 4.5%，且明确说"这是下限不是上限"。

## Forge — 把自托管 LLM 的工具调用成功率从个位数拉到 84%，还能当 Claude Code 的透明代理用

  本地跑个 8B 模型做工具调用，十个请求里九个返回的 JSON 格式不对——缺括号的、函数名拼错的、`[TOOL_CALLS]` 标签没闭合的。Forge 做的事很直接：在客户端和本地模型服务器之间插一层代理，对模型的输出做救援解析、格式修正和重试，让你的 OpenAI 兼容客户端（或 Claude Code）以为自己在跟一个更聪明的模型对话。上 Forge 之后，同一个 8B 模型在 26 个场景的 eval 套件上从个位数冲到 84%，甚至能把 Sonnet 4.6 从 85% 抬到 98%。

  三种用法逐级递进：（1）Proxy 模式——`python -m forge.proxy` 启一个同时讲 OpenAI 和 Anthropic Messages API 的代理，把 Claude Code 的 `ANTHROPIC_BASE_URL` 指向它，Forge 透明地对本地模型输出做救援解析和验证重试；（2）WorkflowRunner——自己写工具定义，Forge 管整个生命周期的 system prompt、工具执行、上下文压缩和护栏；（3）中间件模式——把 Forge 的护栏栈嵌入你自己的编排循环。支持的推理后端覆盖 Ollama、llama-server、Llamafile、vLLM 和 Anthropic API。

  代理模式下每次请求经过四道工序：响应校验（检查工具名和参数是否符合 tools 定义）→ 救援解析（Mistral 的 `[TOOL_CALLS]` 格式、Qwen 的 `<tool_call>` XML、code fence 里的 JSON，全部提取成标准 OpenAI `tool_calls`）→ 重试循环（默认最多 3 次，每次把错误信息以 tool-result 消息喂回去）→ 合成 respond 工具注入（给模型一个"说话"的工具，让它不会在工具调用和纯文本之间乱选——对 8B 小模型尤其关键）。从客户端视角看，就是一个稍慢但从不返回畸形工具调用的模型。

```bash
pip install forge-guardrails

# 代理模式——假设 Ollama 在 localhost:11434 跑着 ministral-3:8b
python -m forge.proxy --backend ollama --model ministral-3:8b-instruct-2512-q4_K_M --port 8081

# Claude Code 指向它
export ANTHROPIC_BASE_URL=http://localhost:8081
export ANTHROPIC_AUTH_TOKEN=anything
claude
```

|------|---------|---------|------|------|

| Janus | Docker / Rust 二进制 | 企业级 AI 网关+语义缓存+智能路由，0.84ms p99 | 开源 BUSL-1.1 | ⭐ 较少 |
| LiteLLM | `pip install litellm` | 100+ 模型提供商统一代理，社区最大 | 开源 MIT | ⭐ 16K+ |

  Forge 和 LiteLLM/Olla/Janus 不是竞品关系——后者是 API 网关（统一多提供商的入口），Forge 是可靠性层（让单模型的工具调用输出格式正确）。两者的代理都是"坐在中间"，但做的事完全不同：网关做路由和负载均衡，Forge 做输出修复。如果你的场景是"想让本地小模型可靠地调用工具"，Forge 是目前最专注这个问题的方案。8B 模型配合 Ministral 3 的 Q8 量化版跑在 llama-server 上，是作者验证过的最佳性价比组合。
