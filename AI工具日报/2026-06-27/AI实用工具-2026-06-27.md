## FreePeak db-mcp-server — 一个 MCP Server 同时连接 MySQL、PostgreSQL、SQLite、Oracle

  AI 编码助手查数据库通常靠两种方式：要么你手动复制 schema 和连接串贴进 prompt，要么装一个只支持单数据库的 MCP server。当你的项目同时用了 MySQL 主库和一个 SQLite 缓存，再加一个 PostgreSQL 的分析库时，你就得配三个 MCP server，每个吐一堆 tool 到 Agent 的上下文窗口里，token 消耗翻倍甚至三倍。FreePeak 的 db-mcp-server 用一个 Go 编译的单二进制文件解决了这个问题：一份配置，同时连接多个异构数据库，自动为每个库生成专用工具。

  核心设计是"按数据库 ID 动态生成工具"。你在 `config.json` 里给每个数据库连接分配一个 ID（比如 `mysql1`、`postgres1`、`sqlite_cache`），服务器启动后自动为每个 ID 生成五类工具：`query_<id>`（只读查询）、`execute_<id>`（写操作）、`transaction_<id>`（事务管理）、`schema_<id>`（结构查看）、`performance_<id>`（性能分析）。Agent 在对话中调用 `query_mysql1` 查用户表，调用 `query_sqlite_cache` 查缓存，完全不需要切换 server。

  对日常开发来说，最有价值的两个细节：一是懒加载模式（`--lazy-loading`），配了 10 个数据库但只在 Agent 实际调用时才建立连接，启动快，不浪费连接池；二是支持 SQLite 的 SQLCipher 加密和 WAL 模式，本地开发时也能安全地读写加密的 SQLite 文件。TimescaleDB 的时序表、连续聚合、压缩策略这些高级特性也都被正确暴露成 MCP 工具。

```bash
# 下载预编译二进制（macOS/Linux/Windows 均有发布）
# 或使用 Docker
docker pull freepeak/db-mcp-server:latest
docker run -p 9092:9092 \
  -v $(pwd)/config.json:/app/my-config.json \
  -e TRANSPORT_MODE=sse \
  -e CONFIG_PATH=/app/my-config.json \
  freepeak/db-mcp-server

# 或从源码构建
git clone https://github.com/FreePeak/db-mcp-server.git
cd db-mcp-server && make build
./bin/server -t stdio -c config.json
```

### 同类工具对比

| 工具 | 语言 | 同时多库 | 特有功能 | 热度 |
|------|------|---------|---------|------|
| **FreePeak db-mcp-server**（今日推荐）| Go 单二进制 | MySQL + PG + SQLite + Oracle + TimescaleDB | 按库 ID 动态生成工具、懒加载、SQLCipher | ⭐ 387 |
| iPraBhu/mcp-database-server | Node.js | PG + MySQL + SQLite + MSSQL + Oracle | Schema 缓存 TTL、Join 路径推荐、查询重写 | ⭐ 较低 |
| haymon-ai/dbmcp | Rust 单二进制 | MySQL + PG + SQLite | 内置 PII 脱敏（46 种实体类型）| ⭐ 27 |
| roshanponnappa/multidb-mcp-server | Python + Docker | PG + MySQL + MSSQL + SQLite + ClickHouse | 同一条 SQL 跨多个数据库并行执行 | ⭐ 较低 |

  FreePeak 的核心优势是"一个二进制搞定一切"——Go 编译的约 20MB 单文件，不依赖 Node/Python/Docker 运行时，在开发机和 CI 里都能直接跑。iPraBhu 的版本在查询智能（Join 推荐、查询重写）上更深入，但安装链路更长（npm + Node）；haymon-ai 的 dbmcp 的 PII 脱敏是一个独特加分项，但 stars 和社区活跃度尚不及 FreePeak。需要"开箱即连、不用想太多"的话，FreePeak 是当前最省力的选择。

## Multi-CLI — 让 Claude、Codex、Gemini、OpenCode 在同一会话里互相调用

  现在几乎每个大模型厂商都推出了自己的 AI 编码 CLI：Anthropic 有 Claude Code，OpenAI 有 Codex，Google 有 Gemini CLI，开源社区有 OpenCode。它们各有各的优势——Claude Code 的代码理解和重构能力强，Codex 在处理复杂多步任务时稳定，Gemini 对多模态文件的理解好。实际工作中你只能选一个，遇到它的短板场景时，你得手动切到另一个 CLI、重新描述上下文、重新开始对话。Multi-CLI 用一个 MCP server 打破了这堵墙：你在 Claude Code 的对话里直接 `Ask-Codex` 让它审一段代码，或者 `Ask-Gemini` 让它分析一张截图。

  它的工作方式很直接：安装 Multi-CLI 后，当前 AI 客户端的工具列表里会自动出现其他已安装 CLI 的调用工具。如果你在用 Claude Code，会看到 `List-Antigravity-Models`、`Ask-Antigravity`、`List-Codex-Models`、`Ask-Codex`、`List-OpenCode-Models`、`Ask-OpenCode` 等工具。调用 `Ask-Codex` 时，Multi-CLI 在新进程中启动 Codex CLI，传入你的 prompt，收集输出，返回给 Claude。你自己的 CLI 的工具不会出现在列表里（自己调自己是死循环）。

  一个比较特别的细节：这个工具的代码和维护本身也是这些 AI CLI 协作完成的。Claude Code、Codex、Gemini CLI、OpenCode 各写了一部分代码。每天晚上 CI 自动抓取各个 CLI 的最新模型列表，diff 对比后自动发版。大多数 MCP 工具一两周就会过时，这个工具是"自维护"的：模型列表由 CI 自动同步。

```bash
# 一键安装（自动检测你装了哪些 CLI）
curl -fsSL https://raw.githubusercontent.com/osanoai/multicli/main/install.sh | bash

# 或分步手动安装
npm install -g @osanoai/multicli
multicli service install --configure-claude

# 其他 CLI 的安装参考
codex mcp add Multi-CLI -- npx -y @osanoai/multicli@latest
```

### 同类工具对比

| 工具 | 支持的 CLI 互相调用 | 安装方式 | 状态 |
|------|-------------------|---------|------|
| **Multi-CLI**（今日推荐）| Claude ↔ Codex ↔ Gemini ↔ OpenCode | npm 全局安装 + 一键脚本 | 活跃维护，自更新 |
| terminalcp | 通用终端进程控制（可用于 CLI 互调）| npm (npx) | 稳定，通用性强 |
| n8n MCP Server | n8n 工作流构建与执行 | npm / self-host | 偏向工作流自动化 |
| Oh-My-ClaudeCode (OMCC) | Claude 调度 Codex + OpenCode | Python + uv | 仅 Claude 作为主控 |

  Multi-CLI 做到了真正的平等互调：任何一方都可以调用其他方，没有单向的调度关系。terminalcp 是一个更通用的方案（可以操控任意终端进程），但在 AI CLI 互调这个具体场景下需要自己写胶水逻辑。Oh-My-ClaudeCode 只支持 Claude 作为主调度者。我试用下来，如果你已经装了两个以上的 AI CLI，Multi-CLI 是实现多模型协作当前最短的路径。
