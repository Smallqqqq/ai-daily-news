## Wayfinder Router — 不用模型调用决定调哪个模型，离线、确定性、微秒级路由

  每个用 AI API 的开发者迟早会撞上一个荒诞的成本现实：你为了省 token 给一个模型路由器付费，但这个路由器自己先调一次模型来判断"这个 prompt 该不该用贵的模型"——用推理成本去解决推理成本，逻辑上自相矛盾。Wayfinder Router 的解法直接到几乎没有设计：读 prompt 的结构特征（长度、标题、列表、代码块、表格），算出一个 0-1 的复杂度分数，低于阈值走本地便宜模型，高于阈值走云端强模型。整个过程不需要 API Key、不联网、不用 GPU，微秒级出结果，同一个 prompt 永远给出同一个答案。

  背后的直觉很简单但被行业长期忽略：prompt 的"难度"在很大比例上是被其结构形式暴露的。一个 200 字的纯文本提问和一个 3000 字的带代码块的复杂指令，人一眼就能分辨，不需要让 LLM 来判断。Wayfinder 默认仅用结构特征评分（长度、标题层级、列表项、链接数、代码块和表格），词汇线索（"证明""数学""约束"等硬词）默认关闭——作者自己做了双盲测试，词汇线索在未见过的 prompt 上提升不显著（只多抓到约 20% 的难 prompt，且输给纯词数基线），所以设为 opt-in。

  更务实的细节在网关模式。`pip install "wayfinder-router[gateway]"` 后启动一个 OpenAI 兼容的 `/chat/completions` 端点，你的应用改一行 `base_url` 就切过去，零代码改动。配置一个 TOML 文件，左边填 Ollama 的 `base_url` 和模型名，右边填 Anthropic 或 OpenAI 的端点，阈值你定。API Key 永远从环境变量读取，不进配置文件，不进日志。带预算上限（`[gateway.budget]`），超支自动降级到便宜 tier，防止账单爆炸。

  安装后的终端聊天（`wayfinder-router chat`）也做得干净——每轮显示路由去向（`● LOCAL` / `◆ CLOUD`）、复杂度分数、为什么这么判断（`/why` 展开逐项评分），以及相比全走云端省了多少钱。还有 Web 聊天 UI（`wayfinder-router webchat`）带实时阈值滑块，拖一下就能看不同阈值下多少请求被路由到本地。

```bash
# 安装网关模式
pip install "wayfinder-router[gateway]"

# 交互式配置（选 Ollama → Anthropic / 两个 OpenAI tier / 两个 Gemini tier）
wayfinder-router init --interactive

# 或直接写 wayfinder-router.toml，启动网关
wayfinder-router serve

# 试一下终端聊天（零配置，仅展示路由决策）
uvx wayfinder-router chat --dry-run
```

  Wayfinder 和 LiteLLM/OpenRouter 解决的是不同层级的问题：后者回答"选哪个 provider 来服务这个调用"（价格、可用性、故障转移），Wayfinder 回答"这个 prompt 值不值得用贵模型"（难度、离线决策）。两者互补——Wayfinder 做 cheap-vs-expensive 的路由，LiteLLM 或 OpenRouter 在下面做 provider 网关。其他路由器（RouteLLM、NotDiamond）都需要调模型来决策，这就回到了"花 token 省 token"的矛盾。

## OpenSRE — 自托管 AI SRE Agent，60+ 集成，记住每一次故障调查

  生产环境出问题时，证据散落在五个地方：Grafana 面板里一条异常曲线、Loki 里一串报错日志、Datadog APM 里一个延迟尖峰、Kubernetes 里一个重启计数、以及三天前的一次部署。值班工程师凌晨三点被叫起来，做的就是把这些分散的信号手动拼成一条因果链。OpenSRE 的设计目标就是把这个过程变成 Agent 的自动化调查循环——它不只是一个"帮我查一下 CPU 为什么高"的诊断工具，而是从告警触发到根因报告的全流程 AI Agent，跑在你自己的基础设施上。

  核心工作流分六步。告警触发后，OpenSRE 自动拉取告警上下文，从已连接的观测工具中收集关联的日志、指标、链路追踪和最近部署记录。然后在一个工具调用循环中跨系统推理验证假设——查 Datadog 看指标异常的时间点，翻 Loki 找对应时间段的错误，调 Kubernetes API 确认 Pod 状态，跑 Git diff 检查最近的部署变更。每轮推理产出一个结构化调查报吿，带根因判定、证据链接和建议的下一步操作，直接发到 Slack、PagerDuty 或 Telegram。

  两个架构设计让它与"调个 LLM 问一下"的浅层方案拉开距离。一是**情景记忆**（episodic memory）：每次调查的结果和修复手段被持久化，下次类似告警触发时 Agent 直接调出历史方案，不需要从零推理。二是 **Neo4j 知识图谱**：服务之间的依赖关系、爆炸半径、上下游调用链被建模成图，Agent 在调查开始前就知道"这个服务挂了会影响哪些下游"，而不是盲目扫描。

  OpenSRE 连接 60+ 工具，涵盖 LLM 提供商（Anthropic、OpenAI、Ollama、Gemini、Bedrock）、观测平台（Grafana Loki/Mimir/Tempo、Datadog、Honeycomb、CloudWatch、Sentry、Elasticsearch）、基础设施（Kubernetes、AWS S3/Lambda/EC2、GCP、Azure）、数据库（PostgreSQL、MySQL、MongoDB、ClickHouse、Redis、Snowflake）、数据平台（Airflow、Kafka、Spark）、事件管理（PagerDuty、Opsgenie、Jira、incident.io）和开发工具（GitHub、GitLab、Bitbucket）。

  模型选择完全由你控制。通过 LiteLLM 路由，可以用 Anthropic Claude 追求质量，切到开源模型省成本，或者在隔离环境里跑完全本地模型。你的事故数据、情景记忆、知识图谱永远不离开你的环境——这对合规要求严格的团队是硬需求。目前还在 Public Alpha 阶段（API 和集成仍在演进），但 210 个贡献者、每周多次发布的节奏说明社区活跃度很高。

```bash
# 一键安装
curl -fsSL https://install.opensre.com | bash

# 或 Homebrew
brew tap tracer-cloud/tap && brew install tracer-cloud/tap/opensre

# 初始化并启动
opensre onboard
opensre

# 从告警 JSON 启动调查
opensre investigate -i tests/e2e/kubernetes/fixtures/datadog_k8s_alert.json

# 按服务名调查
opensre investigate --service api-backend
```
