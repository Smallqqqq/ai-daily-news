# AI 实用工具 · 2026-05-30

![[summary.png]]

  今天三个工具都围绕一个共同主题：AI 辅助编程的质量防线。当 AI 写的代码能通过测试但暗藏腐化模式时，当本地 8B 小模型做不了多步工具调用时，当你需要一个真正开源的全能 VS Code 编码 agent 时——这三个工具正好各解决一个。

## aislop — 给 AI 写的代码加上质量闸门

  使用 Claude Code、Cursor、Codex 写过代码的人大概率都见过这些模式：自解释代码上面加一行叙述性注释、被吞掉的异常、`as any` 类型强转、幻觉出来的 import、重复的辅助函数、写了一半的 TODO 桩。测试通过，lint 通过，但代码在悄悄腐化——这就是所谓"AI slop"。

  aislop 专门检测这些模式。40+ 条规则覆盖 TS/JS、Python、Go、Rust、Ruby、PHP、Java 七种语言。给每次变更打一个 0-100 的分数，亚秒级执行，完全确定性——没有 LLM 在运行时路径里，同样的代码进来，同样的分数出去。MIT 协议，免费 CLI。

  核心引擎有六个，并行运行：Formatting 调用 Biome/ruff/gofmt 等原生格式化工具；Linting 跑 oxlint/golangci-lint/clippy；Code Quality 检测函数/文件体积超标、深度嵌套、死代码；AI Slop 专门抓叙述性注释、无意义注释、死模式、未用 import、`console.log` 残留、TODO 桩、泛型命名；Security 查 eval/innerHTML/SQL 注入/依赖审计；Architecture 支持自定义 import 禁止和分层规则。

  最实用的三个使用姿势：第一，`npx aislop scan --changes` 只看变更文件的分数，适合 PR 审查前自检；第二，`npx aislop fix --claude` 把机械性能自动修的修掉，修不掉的连完整诊断信息一起交给 Claude Code；第三，`npx aislop hook install --claude` 在每个 agent 编辑完成后自动跑一轮，质量反馈即时回到 agent 上下文里。还支持 MCP 模式，暴露 `aislop_scan`、`aislop_fix`、`aislop_why`、`aislop_baseline` 四个工具给任何 MCP 客户端。

```bash
# 零配置即用
npx aislop scan

# 只看变更
npx aislop scan --changes

# 自动修复 + 交给 Claude Code 处理剩余的
npx aislop fix --claude

# 安装 per-edit hook（Agent 每次编辑后自动扫描）
npx aislop hook install --claude

# CI 质量闸门
npx aislop ci
```

## Forge — 让自托管 8B 模型的多步工具调用可靠性追上 GPT-4

  自托管 LLM 有一个尴尬的现实：8B 参数的小模型跑得快、成本零、数据不出机器，但做多步 agentic 任务时工具调用格式经常出错——JSON 套在 markdown 代码块里、用 Mistral 的 `[TOOL_CALLS]` 格式而不是 OpenAI schema、该调工具时直接返回文本。一个 8B 模型裸跑多步工具调用，完成率往往只有个位数。

  Forge 是一个 Python 可靠性层，专门解决这个问题。它在模型和你的 agent 之间加了一层 guardrail：校验每个工具调用的格式和参数、从错误格式中 rescue 出结构化调用、校验失败自动 retry 并附带纠错 nudge、给 8B 小模型注入一个合成 `respond` 工具防止它"忘记"自己在工具调用模式里。结果：Ministral 8B 从裸跑的 ~53% 飙升到 99.3%（ACM CAIS '26 正式论文数据，97 组配置 × 18 场景 × 50 次运行）。甚至 Claude Sonnet 加 Forge 后也能从 85% 提升到 98%。

  使用方式有三种。最易上手的是 Proxy 模式——`python -m forge.proxy` 起一个 OpenAI 兼容代理，你的 opencode / Continue / aider / Cline 把 API base URL 指向它，guardrail 透明生效，客户端以为自己连了一个更强的模型。WorkflowRunner 模式适合基于 Forge 构建新项目，它管理完整的 agent 循环生命周期。Middleware 模式让你在自己的循环里按需引入 Forge 的 guardrail 组件。

```bash
# 安装
pip install forge-guardrails

# Proxy 模式（最易上手）
python -m forge.proxy --backend-url http://localhost:8080 --port 8081

# 然后把你现有工具的 API base URL 改为 http://localhost:8081/v1
```

## Kilo Code — 113 万安装量的开源全能 VS Code 编码 Agent

🏪 VS Code Marketplace · ⬇ 1,138,397 次安装 · 评分 4.5/5 (186) · [安装](https://marketplace.visualstudio.com/items?itemName=kilocode.Kilo-Code)

  2026 年 5 月 15 日，装机量 300 万的 Roo Code 官方宣布归档停更。对几十万开发者来说，Kil Code 几乎是从 Roo Code fork 出来的唯一仍在积极维护的替代品——Roo Code 官方团队甚至公开推荐用户迁移到 Kilo Code。

  Kilo Code 是一个全能的 agentic 编码平台。和 Claude Code 的纯终端模式不同，它在 VS Code 里有完整的面板 UI，支持 500+ AI 模型（OpenAI、Anthropic、Gemini、DeepSeek、Ollama 本地模型等），BYOK 零加价——你只付模型供应商的 API 费。四种结构化模式切换不同工作状态：Architect 做规划不碰代码、Code 专注实现、Debug 查问题修 bug、Orchestrator 把复杂任务拆成子任务分发给专项 agent。

  和 Cline（59K stars）相比：Kilo Code 从 Cline fork 出来后迁移到了 OpenCode 的服务端架构，这让它支持并行 subagent——可以同时跑代码审查和功能开发两个 agent。JetBrains 全家桶（IntelliJ、PyCharm、WebStorm 等）原生支持，这是 Cline 和已停更的 Roo Code 都没有的。CLI 工具可以嵌入 CI/CD pipeline。KiloClaw 云服务 $9/月，含有一周免费试用。

```bash
# VS Code 扩展商店搜索 "Kilo Code" 安装
# 或者命令行安装
code --install-extension kilocode.Kilo-Code
```
