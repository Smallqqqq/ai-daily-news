## RunPod MCP Server — 在 Claude Code 里说一句话，开 GPU、跑训练、查账单，不用切到控制台

  用 RunPod 跑 GPU 训练的开发者每天的 workflow 是这样的：在编辑器里让 AI 写完训练脚本 → 切到浏览器打开 RunPod 控制台 → 选 GPU 类型 → 配置网络卷 → 等 Pod 启动 → SSH 进去 → 把脚本传上去 → 跑训练 → 回控制台看日志 → 训练完别忘了关 Pod（否则一直计费）。这个 workflow 的每一步都在不同工具之间切换，每一步都在丢失上下文。

  RunPod 昨天（7 月 17 日）发布的 MCP Server 把这个流程压缩成了一句话。在 Claude Code 里说"开一台带 48GB 显存的 A100，挂载我的网络卷，跑我刚写的微调脚本，跑完把结果下载到本地然后关掉 Pod"——Agent 调 `list-gpu-types` 看库存和价格，调 `create-pod` 开机，调 `execute-ssh-command` 跑脚本，调 `download-files` 拉结果，调 `stop-pod` 关机。这些操作没有一个需要你打开浏览器。

  约 50 个 MCP 工具覆盖了 RunPod 的整个资源面。Pod 全生命周期（创建/启动/停止/重启/删除/更新配置），Serverless 端点（部署/扩缩容/提交任务/轮询结果/流式输出），模板管理（把常用配置做成可复用模板），网络卷 CRUD（持久化存储不随 Pod 销毁），容器镜像仓库凭证管理，标签系统，账单查询（按时段/按资源类型拆分）。每个工具都标注了 `read-only`、`write` 或 `destructive`——客户端可以自动批准只读操作，涉及花钱或破坏性操作则要求人工确认。

  几个设计细节体现了对实际使用场景的理解。GPU 发现工具 `list-gpu-types` 支持按显存大小过滤——你说"至少 48GB VRAM"，它返回符合条件的 GPU 型号、实时库存和单价，Agent 不会幻觉出不存在的 GPU 型号。Pod 创建时如果不指定 GPU 类型，Server 不会默默 fallback 到 CPU Pod，而是返回明确错误提示你先查 GPU 列表。Serverless 端点支持 scale-to-zero——最小 worker 数设 0，空闲时不计费。API 版本兼容也做了处理：同时支持即将发布的 v2 REST API 和当前 v1。

  安全模型要说清楚。MCP 意味着你的 AI 助理正在用你的 API Key 操作你的云账户——创建 Pod 会立刻产生费用，删除 Pod 不可逆。RunPod 的建议很务实：用权限受限的专用 API Key、保留客户端的破坏性操作确认提示、不要把 MCP 直接接入生产自动化流水线不加人工审查。

```bash
# Claude Code 全局安装（一条命令，所有项目可用）
claude mcp add runpod -s user \
  -e RUNPOD_API_KEY=你的API_KEY \
  -- npx -y @runpod/mcp-server@latest

# 项目级安装（生成 .mcp.json 可提交 Git，不含 Key）
claude mcp add runpod -s project \
  -e RUNPOD_API_KEY=你的API_KEY \
  -- npx -y @runpod/mcp-server@latest

# 验证连接
claude mcp list
```

## Grok Build 开源 — SpaceXAI 把 Grok 编码 Agent 的 Rust 核心全部开源，Apache 2.0，随便改

  7 月 15 日，SpaceXAI 把 Grok Build 的源代码推到了 GitHub 上——`xai-org/grok-build`，Apache 2.0 协议。这不是"开放 API"或"开源 SDK"，而是 Grok 编码 CLI 的完整 Rust 源码：agent 循环、工具实现、TUI 渲染引擎、工作区管理层、扩展系统——全部可读、可编译、可 fork。

  Grok Build 是 xAI 在 5 月 25 日以早期 beta 发布的终端 AI 编码 Agent。它和其他终端 Agent（Claude Code、Codex CLI）的核心区别在于：默认走 Plan 模式——先出方案让你审批，通过后才动手改代码；16 个并行子 Agent 可以把大任务拆开同时跑；256K token 上下文窗口；全屏鼠标可交互的 TUI。但之前它是闭源的——你能用，不能看里面怎么写的。

  开源出来的代码分四层，每一层都可以独立阅读和复用。`xai-grok-shell` 是 agent 运行时：上下文组装、响应解析、工具调用分发——这是 agent 的大脑中控。`xai-grok-tools` 是工具实现层：文件编辑、终端命令执行、代码搜索——这是 agent 的手脚。`xai-grok-pager` 是 TUI 层：滚动回溯、输入处理、方案审核界面、内联 diff 查看器——这是 agent 的脸。`xai-grok-workspace` 是宿主层：文件系统、版本控制、执行环境、检查点——这是 agent 脚下的地板。

  开源的实际价值分三个层次。第一层是**审计**——在让一个 Agent 跑 shell 命令之前，你可以读它的工具实现确认它不会做什么奇怪的事。这在合规要求严格的团队里不是 nice-to-have，是 blocker。第二层是**自建**——编译自己的 Grok Build，把 `base_url` 指向内网的推理端点，整套 Agent 管线完全离线运行，数据不出机房。第三层是**学习**——读一个 Rust 写的生产级 Agent 循环是怎么处理上下文组装、工具调用分发和并行子 Agent 调度的，这是目前公开可读的最完整的 Agent 架构参考实现之一。

  开源不代表接受外部 PR。README 明确说了：外部贡献不被接受，仓库从 SpaceXAI 内部 monorepo 定期同步。这意味着它不是一个社区项目——它是一份公开的设计文档加一个可自由使用的代码库。Apache 2.0 意味着你可以 fork 它、改它、把它嵌进自己的产品，不需要开源你的改动。

```bash
# 从源码编译（需要 Rust toolchain + protoc）
git clone https://github.com/xai-org/grok-build.git
cd grok-build
cargo build -p xai-grok-pager-bin --release
# 二进制在 target/release/xai-grok-pager

# 快速验证（不开完整构建）
cargo check -p xai-grok-tools    # 检查工具层代码
cargo test -p xai-grok-config    # 跑配置层测试

# 指向自己的推理端点（完全离线运行）
grok -p "Explain the architecture" -m my-local-model

# 无头模式（CI/脚本）
grok -p "Run the test suite and report failures" --output-format streaming-json
```

  Grok Build 开源的意义不在"又多了一个开源 Agent"——Codex CLI 和 OpenCode 早就是开源的了。意义在于：它是目前唯一一个用 Rust 写的、Apache 2.0 协议的、来自顶级 AI 实验室的生产级 Agent 代码库。Rust 意味着你可以把它编译成单个静态二进制，塞进 Docker 镜像、嵌入其他 Rust 项目、或者在资源受限的环境里跑——没有 Node.js 运行时依赖，没有 Python 虚拟环境。Apache 2.0 意味着商业使用零障碍。而"来自 SpaceXAI 生产环境"意味着你读的不是一个教程项目的 demo 代码，是真正在成千上万用户终端里跑过的 Agent 循环。对于正在自己造 Agent 工具的团队来说，这份代码是一座金矿。
