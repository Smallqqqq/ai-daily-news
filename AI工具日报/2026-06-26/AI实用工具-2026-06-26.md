# AI 实用工具 · 2026-06-26

![[summary.png]]

  今天两个工具，一个把 VS Code 调试器变成 AI Agent 的"眼睛和手"，一个让 React 组件直接接入大模型对话——都是装完当天就能用。

---

## DebugMCP — 让 AI Agent 自己设断点、单步调试、看变量

![[debugmcp-demo.gif]]

  AI 编码助手能读代码、写代码，但一直缺一项关键能力：调试。当代码跑不通时，Agent 只能猜：读日志、看报错、推测原因。DebugMCP 填上了这个缺口：它把 VS Code 的调试器完整暴露给 MCP 兼容的 AI Agent，让 Agent 能像人类开发者一样设断点、单步执行、检查变量、求值表达式。

  核心原理不复杂：DebugMCP 是一个 VS Code 扩展，内嵌了 MCP（Model Context Protocol）服务器。Agent 通过 MCP 协议调用 DebugMCP 的工具：`start_debugging` 启动调试、`add_breakpoint` 设断点、`step_over` 单步跳过、`get_variables_values` 查看变量值、`evaluate_expression` 求值表达式，总共 13 个工具，覆盖了完整的手动调试流程。底层走的是 VS Code 的 Debug Adapter Protocol（DAP），因此支持 VS Code 能调试的所有语言（Python、JS/TS、Go、Java、C#、Rust 等九种以上）。

  典型场景：你在 Cursor 或 Copilot Chat 里写了一段代码，跑起来行为不对。以前你得自己切到调试面板、设断点、F5、F10 一步步看。现在直接告诉 Agent"调试一下 `src/api/handler.ts` 的 `processRequest` 函数，变量 `payload` 在第 45 行之后为什么是 null"，Agent 自己调 DebugMCP 完成所有步骤，返回诊断结果。

  v2.0.0（2026 年 6 月发布）新增了两项关键能力：借助 VS Code Testing API 精确调试单个测试用例（`start_debugging` 新增 `testName` 参数），以及配套的 `/debug` Agent Skill。安装扩展后自动注入 Agent 的技能目录，Agent 知道怎么用调试工具。

```bash
# 从 VS Code 扩展商店安装（搜索 DebugMCP）
# 安装后自动启动 MCP 服务器，Agent 自动发现工具
# 无需任何配置
```

### 同类工具对比

| 工具 | 运行方式 | 语言覆盖 | 无头模式 | 核心差异 |
|------|---------|---------|---------|---------|
| **DebugMCP**（今日推荐）| VS Code 扩展内嵌 MCP | 9+ 语言（VS Code 支持的全部）| 需要 VS Code 运行 | 人类可实时看到调试过程，可随时接管 |
| mcp-debugger | 独立 MCP 服务器 | 7 语言（Python/Ruby/JS/Go/Rust/Java/.NET）| 支持 | 纯无头，适合 CI/远程容器内调试 |
| ClrVoyant | dotnet 工具 | .NET 专属 | 支持 | 独有 async/Task 堆内省，DAP 做不到的深度 |
| hwanyong/mcp-debug-tools | VS Code 扩展 + CLI | VS Code 支持的全部 | 部分支持 | 29 个工具，支持一次性 CLI 命令调用 |

  DebugMCP 的最大优势是"有人在环"：人类和 Agent 共享同一个调试会话。Agent 在操作时你可以在 VS Code 里看到断点亮起、变量面板更新，发现不对随时接管。这个设计对于日常开发调试场景比纯无头方案更友好。如果你需要 CI 环境完全无人值守，mcp-debugger 更合适；如果你做 .NET 且需要分析异步 Task 状态，ClrVoyant 是唯一选择。

## Tambo — React 组件直接接入大模型对话，AI 自己决定渲染什么

（注：Tambo 官方演示均为视频，暂无静态截图）

  "生成式 UI"这个词听起来玄，落地其实就是一件事：大模型不只是回文字，而是直接渲染你已有的 React 组件。你告诉它"帮我生成一个上周销售数据的柱状图"，它调出你的 Chart 组件，填上正确的 props，呈现在界面上。不是给你一段代码让你自己粘。Tambo 做的是这件事的完整基建：React SDK + 对话状态管理 + 流式渲染 + MCP 集成，开源 MIT 协议。

  它的工作方式分三步：① 在 `TamboProvider` 里注册你的组件（带 Zod schema 定义 props），② 用户发消息，③ Tambo 的 Agent 决定渲染哪个组件、给什么 props，像 AI 版的 React Router。组件分为两类：生成式组件（generative）渲染一次（图表、摘要、数据卡片），交互式组件（interactable）持久存在、随对话更新（购物车、任务看板、表单）。props 支持流式更新，大模型生成到一半，组件就能先渲染出骨架。

  一个典型例子：你的 SaaS 后台有个"数据看板"页面，原来用户得手动点筛选器、选日期范围。接入 Tambo 后，用户直接说"把华南区 Q2 的客户留存率做成折线图，和全国均值放在一起对比"，Tambo 调出你的 LineChart 组件，填好数据查询参数，渲染结果。用户继续说"把华东区也加上"，图表更新，对话历史保持上下文。我试用下来，从注册组件到第一个对话驱动的图表渲染出来，前后不到 15 分钟。

```bash
# 脚手架一键创建
npm create tambo-app my-app
cd my-app
npx tambo init      # 选择云端或自托管
npm run dev

# 或者在已有项目里
npm install @tambo-ai/react
npx tambo init
```

### 同类工具对比

| 工具 | 组件选择方式 | MCP 集成 | 自托管 | 最适合 |
|------|------------|---------|--------|--------|
| **Tambo**（今日推荐）| AI 自动决定渲染哪个组件 | 内置 | MIT 全栈可自托管 | 给现有 React 应用加 AI 驱动 UI |
| Vercel AI SDK | 手动 tool→component 映射 | v4.2+ 实验性 | Apache 2.0（仅 SDK）| Next.js 全栈流式 AI 应用 |
| CopilotKit | 通过 LangGraph Agent 框架 | 最近加入 | MIT | 多 Agent 工作流 + 聊天 UI |
| Assistant UI | 聊天聚焦的工具 UI | 需要 AI SDK v5 | MIT | 聊天界面为主 |

  Tambo 的杀手锏是"电池全装"——不只是 SDK，还带了一个开箱即用的 Agent 后端（对话状态管理、组件选择、流式传输），自托管只需 Docker 跑起来。Vercel AI SDK 在流式基础设施上更强（20M+ 月下载量），但只给工具抽象，组件选择逻辑你得自己写；CopilotKit 在聊天 UI 组件上更丰富，但组件注册和流式 props 不如 Tambo 自然。如果你的需求是"给现有 React 应用加一个 AI 能直接操控 UI 的能力"，Tambo 是当前最短路径。
