# AI 实用工具 · 2026-07-01

![[summary.png]]

  今天聚焦一个被长期忽视的问题：现有应用的每一层（后端路由、前端组件、数据库 schema、OpenAPI 文档）对 AI Agent 来说都是一堆需要手动翻译的"外语"，而 MCPify 充当了编译器——一次扫描，把整站编译成 Agent 可直接调用的 MCP 工具。这填补了"API 开发与调试"领域 AI 工具链的关键缺口。

---

## MCPify — 把现有应用一键"编译"成 AI Agent 可操作的系统

  大多数开发团队想让 AI Agent 操作自己的应用时，只有两条路：要么手写 MCP 工具包装器（一个接口一个 tool，维护噩梦），要么让 Agent 走浏览器自动化去点按钮（脆弱、慢、token 消耗巨大）。两条路的共同问题是——应用的能力已经存在了，但 Agent 看不见它们。

  MCPify 的解法很直接：它不让你写任何胶水代码，而是像一个编译器一样扫描你的项目——AST 分析后端路由和控制器、提取前端组件的可交互动作、解析 OpenAPI/Swagger 规范、读取 Prisma/Drizzle/Mongoose 的数据库 schema——然后自动生成一个完整的、可运行的 MCP 服务器。Agent 连接这个服务器后，就能像超级用户一样操作你的应用。

  核心流程分五层。第一层是**后端分析器**：对 Express、FastAPI、Spring Boot 等框架做 AST 深解析，把每个路由、控制器、服务方法映射为可调用的 MCP 工具。第二层是**前端动作提取**：React、Vue、Svelte 组件中的按钮、表单、导航被识别为 Agent 可控的操作。第三层是**数据库智能**：Prisma、Drizzle、Mongoose 的 schema 被转换为安全的只读查询面——Agent 能查但不能删。第四层是**工作流引擎**：自动发现跨多个接口的用户操作链（如"下单→支付→发货"），暴露为原子化的 Agent 能力。第五层是**权限层**：每个生成的工具都带 scope、角色和审计日志，调用在到达你的系统之前先过权限校验。

  一个容易被忽视但非常务实的特性是**自更新同步**。每次代码提交后 MCP 定义自动重新生成，不会出现"接口改了但 MCP 工具还是旧的"这种漂移问题。另外 `--ai-enhance` 开关会用 Claude 自动改善工具描述，让 Agent 更容易理解每个工具的用途——这比你手动写描述准确得多。

  还有一个值得单独提的特性：**AI 仿真**。设置 `ANTHROPIC_API_KEY` 后，`mcpify simulate` 会在沙箱中跑 Agent 对你的编译结果做一轮测试，确保生成的工具确实能被 Agent 正确使用，而不是"编译通过了但 Agent 调不动"。

```bash
# 一键分析（无需安装）
npx mcpify-cli analyze ./my-app

# 完整示例：带 OpenAPI + Prisma + Swagger 多数据源
npx mcpify-cli analyze ./examples/ecommerce-saas \
  --output ./examples/ecommerce-saas/.mcpify \
  --prisma ./examples/ecommerce-saas/prisma/schema.prisma \
  --swagger ./examples/ecommerce-saas/openapi.json

# 只看前端动作提取（输出 JSON）
npx mcpify-cli frontend ./src --json

# 把 OpenAPI 规范直接转 MCP 工具
npx mcpify-cli swagger ./openapi.json

# 安全审计（不写文件）
npx mcpify-cli audit ./my-app

# AI 仿真测试（需要 ANTHROPIC_API_KEY）
npx mcpify-cli simulate ./my-app

# 监听模式：代码变更时自动重新编译
npx mcpify-cli analyze . --watch
```

### 同类工具对比

| 工具 | 输入方式 | 覆盖层 | AI 增强 | 自更新 | 热度 |
|------|---------|:---:|:---:|:---:|------|
| **MCPify**（今日推荐）| AST 扫描代码 + OpenAPI + DB Schema | 后端 + 前端 + 数据库 + 工作流 | ✅ `--ai-enhance` | ✅ 每次提交自动同步 | ⭐ 73 |
| mcp-anything | 自然语言 brief + 代码扫描 | 后端 + API Spec（27 种框架） | ✅ LLM 驱动工具设计 | ❌ | ⭐ 37 |
| ivo-toby/openapi-mcp-server | OpenAPI 规范 URL | API 层 | ❌ | ❌ | ⭐ 275 |
| MCP Generator (ChristopherDond) | OpenAPI 规范文件 | API 层 | ❌ | ❌ | ⭐ 17 |
| 手写 MCP Server | 人工逐接口编写 | 取决于投入 | 取决于投入 | ❌ 手动维护 | N/A |

  MCPify 和竞品的核心差异不在"能不能把 API 转成 MCP"——openapi-mcp-server（275 stars）在这方面更成熟。差异在于**覆盖广度**：MCPify 不只读 OpenAPI，它直接读源码——后端路由、前端组件、数据库 schema、事件系统——是多层并行分析而非单层转换。mcp-anything 在框架覆盖上更广（27 种 Python/Java/Go/Rust 框架），但需人工写 brief 描述文件，且 star 数更低、没有自更新机制。如果你的应用已经有 OpenAPI 文档，openapi-mcp-server 是更轻量的选择；如果你想让 Agent 理解整个应用的全部可操作面——从按钮到数据库到工作流——MCPify 是目前唯一一个把这件事做成编译器的工具。
