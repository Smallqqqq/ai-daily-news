## Emdash — YC 背书的开源 Agentic Development Environment，31 个 Agent 并行跑，互不打架

  用 AI 写代码的人迟早会撞上一个效率天花板：你手里有三个 bug 要修、一个 feature 要写，但你一次只能跑一个 Agent。等 Claude Code 修完 bug1 再修 bug2，时间线性叠加。你可以在多个终端里手动开多个 Agent，但它们共用同一个工作目录。两个 Agent 同时编辑同一个文件，冲突立刻炸开。

  Emdash 的解法是用 Git worktree 做隔离层。每个任务启动时，Emdash 从你的基础分支创建一个独立的 worktree（完整的工作副本），Agent 在里面随便改，不会碰到其他 Agent 的文件。任务完成后，你可以在内置的 diff 视图里逐文件对比改动，满意就合并，不满意就丢弃。这套隔离机制的核心价值是确定性：不是更快，而是不会乱。

  底层工程做得相当扎实。`WorktreePoolService` 在后台预创建 reserve worktree，当你点「New Task」时不需要等 3-7 秒的 `git worktree add`。它直接从池子里改名分配，一个哈希校验确认远端没变，瞬间就位。`TaskLifecycleService` 通过 PTY 进程管理 Agent 的完整生命周期，内置操作去重映射防止重复启动，stop intent 确保 teardown 等 run 进程优雅退出后再清理。状态全部存本地 SQLite（Drizzle ORM），不经过任何云端。

  Emdash 不是 Agent 本身，而是 Agent 的调度层。它支持 31 个 CLI Agent：Claude Code、Codex、Gemini、Cursor、Qwen Code、OpenCode、Amp、Devin、GitHub Copilot 等。你可以在一个任务上跑 Claude Code，另一个任务上跑 Codex，第三个任务上跑 Gemini。各自用各自的 API Key，Emdash 不抽成。支持从 Linear、Jira、GitHub Issues、GitLab、Asana 等 issue tracker 直接把工单丢给 Agent，Agent 拿到完整上下文开始干活。

  远程开发支持是一个容易被低估但非常实用的特性。通过 SSH/SFTP 把 Agent 跑到远程服务器上。你在笔记本上操作 Emdash 界面，Agent 在 128 核的云主机上编译。对于需要跑长时间 Agent 任务的场景（比如 overnight 让 Agent 重构一个模块），不用把笔记本开着不关机。

  YC W26 批次，Apache 2.0 协议，110 位贡献者，130 个 release，最新提交 4 天前。Telemetry 可选关闭，代码和聊天记录不经过 Emdash 服务器。

```bash

brew install --cask emdash

# Windows（MSI 安装包或便携版）
# 下载：https://github.com/generalaction/emdash/releases/latest

# Linux
# AppImage 或 deb 包，同上 releases 页面下载
```

## Task Master — 把 PRD 拆成带依赖图的任务树，让 AI Agent 不再边写边忘自己要干嘛

  用 AI Agent 写代码最常见的翻车场景：你给 Agent 扔了一个需求文档，它开始写，写到第三步的时候已经忘了第一步做了什么假设、第五步依赖第二步的哪个输出。这不是模型能力问题，是工作记忆问题。Agent 的上下文窗口是有限的，需求越长，越容易在中间某一步丢失关键上下文。

  Task Master 做的事就是把人类 PM 的活自动化：你给它一个 PRD 文件（`prd.txt`），它调 LLM 分析需求，拆成带依赖关系的结构化任务树。每个任务有 ID、标题、描述、验收标准、依赖列表，存为项目里的 JSON 文件。Agent 每次只需要看当前任务和它的依赖，不需要把整个 PRD 塞进上下文窗口。

  36 个 MCP 工具分三档按需加载。Core 模式（7 个工具）只做最基本的任务 CRUD，省 70% token；Standard 模式（15 个工具）加上项目初始化和复杂度分析；Full 模式（36 个工具）包含研究、标签、依赖图等全部能力。你不需要一次性吃掉 21000 token 的工具描述，在简单项目里开 Core，复杂项目里开 Full。

  多模型角色分工是一个被低估的设计。你配三个模型角色：主模型做核心的任务拆解和代码生成（比如 Claude Opus），研究模型做 web 搜索和信息收集（比如 DeepSeek 或 GLM），fallback 模型在主模型挂了或超预算时顶上。不同任务类型自动路由到不同模型，贵模型只用在刀刃上。

  autopilot 模式值得单独说。`tm autopilot` 启动一个自主循环：生成测试→实现代码→跑测试验证→提交→取下一个任务。七个专门的 MCP 工具驱动这个循环。适用场景很明确：需求已经被正确拆解、每个任务边界清晰、你知道自己要什么只是不想手动一步步调 Agent。

  Task Master 做的是任务拆解和依赖管理，它不替代你用的编码 Agent，它和 Agent 是协作关系。Agent 负责执行，Task Master 负责告诉 Agent「现在该做什么、做完这个下一个是什么」。支持 Cursor、Windsurf、VS Code、Claude Code、Roo Code 等编辑器，一键 MCP 注册。27K+ stars，60 位贡献者，92 个 release。

```bash
# 全局安装
npm install -g task-master-ai

# 注册到 Claude Code
claude mcp add taskmaster-ai -- npx -y task-master-ai

# 解析 PRD，生成任务
task-master parse-prd your-prd.txt

# 查看下一个待办任务
task-master next

# 扩展某个任务（自动拆子任务）
task-master expand-task --id=3

# 启动 autopilot 自主循环
tm autopilot
```

## mcpm — MCP 生态的 npm，带运行时安全护栏的包管理器

  MCP 服务器现在遍地都是，GitHub 上随便一搜几百个。你让 Agent 推荐一个文件系统 MCP，它给你一个名字，你 `claude mcp add` 装上。这个过程缺少一个关键环节：你知不知道这个 MCP 服务器里有什么？

  这不是危言耸听。AgentSeal 去年扫了一遍 MCP 生态，66% 的 MCP 服务器有安全发现：硬编码的 API Key、工具描述里的 prompt injection 模板、拼写相似的 typosquatting 包名。更隐蔽的风险在运行时才暴露：服务器安装后偷偷改了工具定义（rug-pull）、工具响应里嵌了操纵 Agent 下一步行为的指令（prompt injection）。这些问题 `npm audit` 查不出来，因为 MCP 服务器不是 npm 包。它们是通过 `npx` 或 `uvx` 直接跑的进程。

  mcpm 的解决思路很直接：做 MCP 生态的 npm。提供 `search`（搜官方 MCP Registry）、`install`（带信任评分）、`audit`（扫所有已装服务器）、`guard`（运行时拦截）四层防线。

  信任评分（0-100）由四个维度加权：健康检查（30 分，服务器能否正常启动并响应 `list_tools`）、静态扫描（40 分，正则检测硬编码密钥、工具描述中的 prompt injection 模式、typosquatting 包名）、外部扫描器（20 分，可选接入 MCP-Scan 做深度分析）、Registry 元数据（10 分，已验证发布者、发布日期、下载量）。评分是元数据级别的，不是源码审计。它不能替代安全审查，但能帮你快速过滤掉最危险的 80%。

  `mcpm guard` 是 mcpm 最独特的功能。它在 AI 客户端和每个 MCP 服务器之间插入一个 stdio 中继层，实时扫描工具描述、响应和参数中的 OWASP MCP Top 10 攻击模式。安装时快照每个工具的 schema，运行时如果服务器返回的响应和快照不一致（schema drift）就阻断调用。这个防御针对的是 MCP 特有的攻击面。传统包管理器不需要管「工具描述里有没有藏 prompt injection」，因为传统包没有工具描述这个概念。

  `mcpm.yaml` stack file 设计让团队共享 MCP 配置变得可行。`mcpm export` 导出当前配置，`mcpm lock` 锁定版本和信任快照，`mcpm up` 一键复现整队人的 MCP 环境。stack file 带信任策略（`minTrustScore: 60`），如果某个服务器的信任分跌到阈值以下，`mcpm up` 直接 block。

  目前项目还很早期（GitHub 2 stars，554 周下载），但方向踩得很准。MCP 生态正在经历和 npm 早期一模一样的阶段：人人都在装，没人审查。npm 花了四年才等来 `npm audit`（2018 年），MCP 不该再花四年。

```bash
# 全局安装
npm install -g @getmcpm/cli

# 搜索 MCP 服务器
mcpm search filesystem

# 查看某个服务器的信任评分和详情
mcpm info io.github.domdomegg/filesystem-mcp

# 安装（自动过信任评估）
mcpm install io.github.domdomegg/filesystem-mcp

# 审计所有已装服务器
mcpm audit

# 诊断 MCP 配置健康状态
mcpm doctor

# 导出 stack file 供团队共享
mcpm export > mcpm.yaml
mcpm lock
mcpm up
```

  mcpm 和 mception 解决的是同一枚硬币的两面。mception 做深度静态审计：SAST + SCA + 依赖溯源，覆盖 Python、TypeScript、JavaScript、Go、Rust、Ruby 六种语言，60+ 检测规则，适合在 CI 里做 MCP 服务器的安全门禁。mcpm 做的是日常使用的全流程：搜、装、审、护。两者互补。mception 管「装之前深度检查」，mcpm 管「装之后持续防护」。如果你只需要在 CI 里跑一次审计，mception 更专业；如果你每天都和 MCP 服务器打交道、需要管好自己机器上的 MCP 环境，mcpm 是目前唯一覆盖全流程的工具。
