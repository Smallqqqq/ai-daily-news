# AI 实用工具 · 2026-06-03

![[summary.png]]

  今天三个工具解决 AI 辅助开发中三个层级的可靠性问题：本地小模型做工具调用时频繁出错怎么办（Forge 用五层护栏把成功率从 53% 拉到 99%）、MCP 工具集写死以后想加新能力必须重启部署（Genesis MCP 让 AI 自己给自己写工具）、以及 AI agent 无法可靠地读懂 API 测试结果（API Workbench 把响应转成结构化机器可读格式）。

---

## Forge — 给本地小模型的工具调用套上可靠性护栏

  用 Ollama 跑本地模型做 agent 工具调用的人大概率都经历过这种挫败：模型输出格式不对、调了不存在的工具名、JSON 参数丢了一半、text 和 tool_call 之间反复横跳。单次裸调 8B 模型的工具调用成功率大概只有 53%——也就是说近一半的请求在到达"干活"这一步之前就已经失败了。

  Forge 是一个专门解决这个问题的 Python 可靠性层，由 Texas Instruments 的 AI Director Antoine Zambelli 独力开发。它的论文已被 ACM CAIS 2026 接收，HN 首页 686 分，GitHub 1735 stars。核心论点很简单：问题不在模型本身，而在调用框架缺少护栏。同一个 Ministral-3 8B 模型，裸跑 53%，加上 Forge 的护栏后达到 99.3%。更惊人的是：这个 600 美元 GPU 上跑的 8B 本地模型 + Forge 的组合，超过了不使用任何护栏的 Claude Sonnet（87.2%）。

  Forge 的护栏栈由五层组成，每一层解决一类具体的失败模式：第一层 **rescue parsing**（救援解析）处理格式错误——当模型把 tool call 包在 markdown 代码块里、用 Mistral 的 `[TOOL_CALLS]` 格式、或者 Qwen 的 XML 格式输出时，Forge 自动提取并重写为标准 OpenAI tool_calls 格式。第二层 **response validation**（响应校验）检查每次 tool call 的工具名是否合法、参数 schema 是否匹配。第三层 **retry loop with error tracking**（带错误追踪的重试循环）在校验失败时自动重试（默认最多 3 次），把修正提示注入到下一轮消息中。第四层 **step enforcement**（步骤强制）确保 agent 按预定顺序走完多步工作流。第五层 **synthetic respond tool**（合成响应工具）是最巧妙的设计——当请求中包含工具时，Forge 注入一个虚拟的 `respond` 工具，强制模型始终处于工具调用模式，避免 8B 模型在"发文本"和"发 tool call"之间错误切换。

  三种使用方式覆盖了不同的集成深度。**Proxy 模式**最轻量：`python -m forge.proxy --backend-url http://localhost:8080 --port 8081`，启动一个 OpenAI 兼容代理，把任何现有客户端（opencode、Continue、aider、Cline）指向它即可，护栏透明生效。**WorkflowRunner** 提供完整生命周期管理：定义工具集、选后端、跑工作流，Forge 管理系统提示、工具执行、上下文压缩和护栏。**Middleware 模式**最灵活：直接在你自己编排的 agent 循环里调用 Forge 的 `ResponseValidator`、nudge 系统和 `ContextManager`。

```bash
# 核心安装
pip install forge-guardrails

# Proxy 模式（最简单的接入方式）
python -m forge.proxy --backend-url http://localhost:8080 --port 8081

# 带 Anthropic 客户端支持
pip install "forge-guardrails[anthropic]"
```

### 同类工具对比

| 工具                    | 核心能力                    | 优势                | 与 Forge 的区别                |
| --------------------- | ----------------------- | ----------------- | -------------------------- |
| **Forge**（今日推荐）       | 自托管 LLM 工具调用可靠性 + 上下文管理 | 扫除本地 vs 云端差距，论文验证 | Python only，尚无公开延迟数据       |
| Outlines / Instructor | 结构化输出 / JSON Schema     | 成熟、多语言            | 不管理多步重试和步骤强制               |
| LangGraph             | Agent 编排                | 生态巨大              | 无自托管可靠性护栏，假设模型调用正确         |
| vLLM guided decoding  | token 级受限生成             | 速度最快              | 仅服务端，绑定 vLLM               |
| AgentGuard            | 启发式护栏 FastAPI 服务        | 透明可审计             | 侧重安全（注入检测/PII），不解决工具调用格式问题 |

  Forge 和这些工具是互补的：Outlines 约束 schema、vLLM 约束 token、LangGraph 编排流程、Forge 约束整个工作流的可靠性。论文的消融实验表明，五个护栏叠加使用效果最好，去掉任意一层都显著降分。

## Genesis MCP — 一个只有一把工具的工具箱：造工具的工具

  MCP 生态有一个结构性问题：每个 MCP 服务器的工具集是写死的。想加一个"把 Markdown 转 PDF"的工具？改代码、重启服务、重新连接客户端。GitHub 上每天新出的 MCP 服务器越来越多，但每个都只解决一个具体场景，想组合使用得配十几个 server。Genesis MCP 的思路完全不同：它只提供 6 个元工具，其中核心是 `create_tool` 和 `register_tool`——你用自然语言描述需要什么工具，host LLM 生成 Python 代码，Genesis 做 AST 校验后热加载注册，全过程不重启、不改配置文件、不写一行代码。

  工作时序是：先说"帮我创建一个能把摄氏转华氏的工具"，Claude（或其他 MCP 客户端）调用 `create_tool`，Genesis 返回一个代码生成 prompt；host LLM 生成 Python 工具代码；Claude 调用 `register_tool(code=...)`；Genesis 的 AST 校验器扫描代码——检查 import 白名单（requests、json、pathlib 等）、拦截 eval/exec/os.system/subprocess 等危险调用、禁止 dunder 反射绕过——校验通过后写入 `generated_tools/` 目录并热加载到运行中的 server。接下来在当前对话里就能直接调用新工具。

  内置示例覆盖了三个典型场景：天气查询（调 Open-Meteo API，免费无需 key）、笔记保存（写入 `~/genesis_notes/` 目录）、汇率转换（调 open.er-api.com，免费无需 key）。每个示例都是一份 walkthrough 文档，告诉你具体怎么在 Claude Desktop 中用 `create_tool` 创建这个工具。

  安全方面 v0.1 的态度非常诚实：目前运行在 MCP server 同一进程内，AST 校验器是唯一的安全边界，没有沙箱隔离，没有子进程隔离。`open()` 是内置函数、`pathlib.Path.write_text()` 在白名单中、`requests` 可以发 HTTPS 到任意 URL——理论上生成的工具可以读 `~/.ssh/id_rsa` 然后外传。所以作者明确写了：v0.1 只适合本地单用户使用，把 `generated_tools/` 里的代码当自己写的代码来审查。v0.2 计划加入子进程隔离、外发白名单、文件系统约束。

```bash
# 安装（需要 Python 3.11+）
git clone https://github.com/adeoluwaadesina/genesis-mcp.git
cd genesis-mcp
pip install -e .

# 测试启动
python -m genesis.server

# 配置到 Claude Desktop（%APPDATA%\Claude\claude_desktop_config.json）
{
  "mcpServers": {
    "genesis": {
      "command": "python",
      "args": ["-m", "genesis.server"],
      "cwd": "C:\\path\\to\\genesis-mcp"
    }
  }
}
```

## API Workbench — AI Agent 能读懂的 API 测试结果

  后端和全栈开发者的日常：在 `.http` 文件里写好 API 请求，点 Send，看响应面板。但当 AI coding agent 需要理解测试结果时，这套流程完全失效——agent 无法可靠地读取自定义 webview、无法点击面板切换标签页、无法判断哪个断言失败了。你只能在对话里手动复制粘贴 400 响应和错误信息。

  API Workbench 是一个 VS Code 扩展，专门解决这个 AI agent 和 API 测试之间的信息断层。它的核心创新不是"又一个 HTTP 客户端"，而是输出格式：每个请求的结果被写成结构化的机器可读文本——`REQUEST:`、`STATUS:`、`DURATION:`、`BODY_JSON:` 逐行标注，AI agent 不需要解析 HTML、不需要猜测 UI 布局，直接读行标签就能提取全部关键信息。同时还会生成一份结构化的 `.agent.json` 报告文件，包含完整的 failed assertion、expected vs actual、request body 和 response body。

  对开发者来说，这意味着一个顺滑的工作流：在 `.http` 文件里写好请求 → 一键运行整个集合 → agent 读到结构化报告 → 自动定位失败原因 → 修改后端代码 → 再跑一遍验证。整个过程 agent 不需要人类在中间做信息转述。

  API Workbench 还暴露了 MCP stdio 工具，可以在 CI 流程中被 agent 直接调用——跑完全部测试、解析结构化报告、打开修复 PR，全程不用人类介入。v0.1.0 beta，MIT 协议开源，支持 VS Code、Cursor 和 Windsurf。

```bash
# 从 GitHub Releases 下载 VSIX
# https://github.com/sapph1re/api-workbench/releases/latest

# 安装
code --install-extension api-workbench-0.1.0.vsix
```

### 同类工具对比

| 工具 | Agent 可读输出 | 安装方式 | 费用 | 特点 |
|------|-------------|---------|------|------|
| **API Workbench**（今日推荐）| ✅ 结构化标记行 + .agent.json | VSIX | 免费 MIT | 专为 agent 设计的输出格式，MCP 工具 |
| REST Client (VS Code) | ❌ | Marketplace | 免费 | 最流行的 .http 客户端，但输出仅人类可读 |
| Postman | ❌ | 独立应用 | 部分免费 | 功能最全但云端锁定，agent 无法消费输出 |
| Thunder Client | ❌ | Marketplace | 部分免费 | 轻量但无 agent 输出 |
| Glubean | ✅ MCP + 结构化 trace | Marketplace | 免费 | TypeScript 写测试，AI 闭环更强但语言绑定 |
| API Studio | 部分（MCP 管理）| Marketplace | 免费 | 离线全功能，REST+GraphQL+SSE+WS 全覆盖 |

  API Workbench 目前在"给 agent 提供可读 API 测试结果"这个细分场景中没有直接竞品。REST Client 和 Thunder Client 的响应面板对人类友好但对 agent 封闭；Glubean 的 MCP + trace 方案更强但要求用 TypeScript 写测试；Postman 有 Newman CLI 但输出同样是面向人类的。API Workbench 用最简单的 `.http` 文件 + 结构化文本输出，填了一个具体的坑。
