## testpilot-ai — 生成测试、跑测试、通不过就修，修到全绿为止

  用 AI 生成单元测试不是什么新鲜事——Copilot 能内联补全测试代码，Cursor 的 Composer 能按函数签名生成测试文件。但几乎所有 AI 测试生成器都有一个共同的问题：它们只负责"生成"，不负责"通过"。测试写出来能不能跑、断言合不合理、mock 是不是编的——这些统统不管，留给你自己收拾。

  testpilot-ai 做的事不同。它不是一个"测试补全"工具，而是一个完整的测试自动化管线：先用 TypeScript 编译器 API 解析源码（函数签名、类型、JSDoc、导入依赖），构建一个包含完整类型信息的 prompt 发给 LLM，生成测试文件之后立刻跑一遍——如果没通过，把失败信息喂回 LLM 修，修完再跑，直到全绿或达到最大迭代次数。整个过程对你来说就是一行命令。

  这个"验证-修复循环"（verify & auto-fix loop）是 testpilot-ai 和其他测试生成器最本质的区别。Copilot 和 Cursor 生成测试时只看到当前文件窗口内的上下文，testpilot-ai 会沿着 `tsconfig` 的路径别名（`@/foo`、`~lib/*`）追溯导入依赖，上限 5 个文件 × 3KB，让 LLM 知道被 mock 的依赖到底有哪些方法、什么签名，减少幻觉。生成后跑测试，只把失败 delta 发回 LLM（不是整个文件），叠加"最优版本持久化"——如果新一轮修复反而让更多测试挂了，自动回滚到上一版。

  另一个实用功能是 `analyze` 命令。跑一下 `npx testpilot analyze`，它读 lcov/cobertura 覆盖率数据，列出哪些文件测试覆盖率低于阈值、哪些文件根本没有对应测试。比手动翻 coverage report 快得多。如果没有覆盖率数据，就扫描源文件目录直接列出缺测试的文件。

  支持四种 LLM 提供商（OpenAI、Anthropic、Google、Ollama 本地模型），四种测试框架（Vitest、Jest、Mocha、Node test runner），自动从 `package.json` 检测框架。支持并发处理多文件（`--concurrency`）和成本上限（`--max-cost`），不会跑着跑着吃光 API 额度。

```bash
# 安装
npm install -D @amit641/testpilot-ai

# 生成测试 + 自动验证修复
npx testpilot src/utils.ts --verify

# 使用 Claude 做生成
npx testpilot src/utils.ts --verify --provider anthropic --model claude-sonnet-4-20250514

# 分析项目中哪些文件缺测试
npx testpilot analyze

# 用本地 Ollama 模型（无 API 费用）
npx testpilot src/utils.ts --provider ollama --model llama3
```

  testpilot-ai 相比 Qodo 的优势在于它是一个无 UI 依赖的 CLI——你可以在 CI 里跑、在 pre-commit hook 里跑、在任何一个终端里跑，不绑定特定 IDE。相比 Copilot 和 Cursor，它的"验证-修复循环"让测试不是"建议"而是"交付物"。劣势在于目前只支持 JS/TS，不像 Qodo 那样覆盖 Python/Java/Go。stars 数低是事实——但这个工具的实际能力远超过它的热度数字所暗示的。

## pilotfish — 让 Fable 5 只做判断，Sonnet 干体力活，省下 58% 的配额

  用 Claude Code 的人这一个月都踩到了同一个坑：Fable 5 的配额消耗速度大约是 Opus 的 2 倍（官方 UI 原话），而一次 agentic session 里大部分 token 根本不是花在"判断"上——搜索文件、机械编辑、跑测试、更新文档，这些活 Sonnet 甚至 Haiku 一样能干好。问题是 Claude Code 默认整场会话都用同一个模型，搜索文件花的是 Fable 5 的钱。

  pilotfish 的解法是给 Claude Code 装一个"多模型编排层"：主会话始终用最强模型（Fable 5 / Opus）做规划、判断和审查，具体执行通过六个角色子 Agent 分发到便宜模型上。`scout`（Haiku）负责只读查询——"这个符号在哪定义的""这个配置项的值是什么"；`mech-executor`（Sonnet）负责完全规格化的机械工作——模式重构、约定测试、文档更新、批量编辑；`executor`（Opus）负责需要判断力的实现——功能开发、bug 修复、涉及设计的重构；`verifier`（Opus）做独立上下文对抗性验证，只返回 CONFIRMED/REFUTED，从不自己修；`security-executor`（Opus）处理安全相关工作——故意不用 Fable 5，因为 Fable 5 的安全分类器偶尔会拒绝良性的防御性安全任务。

  这套拆分的效益有硬数据支撑。Anthropic 自己的多 Agent 文档里给出了官方 benchmark：Fable 5 编排 + Sonnet 5 执行，在 BrowseComp 上达到全 Fable 5 的 96% 准确率，只花 46% 的成本（$18.53 vs $40.56/题）。社区实验更激进——一个 12 工人的审计任务，全 Fable 5 花了 $14.50，换成 Fable 5 编排 + Sonnet 工人后降到 $6.10（省 58%），再降到 Haiku 工人后只要 $3.70（省 74%）。

  对 Claude 订阅用户还有一个额外利好：Claude 的订阅配额有两个桶——一个所有模型共享的桶，外加一个 Sonnet 专用桶。把执行路由到 Sonnet 子 Agent，不但单 token 更便宜，还能用那个额外的 Sonnet 专用额度。

  pilotfish 的安装方式很特别——不是 npm install，而是让 Claude Code 自己读一份安装脚本然后帮你配。克隆仓库后启动 Claude Code，粘贴一句 prompt，Claude 会展示完整的配置变更计划（改 `settings.json`、`CLAUDE.md`、六个 agent 文件），你批准后才写入。安装是幂等的——再跑一次就是升级。关闭也很简单：在 `~/.claude/CLAUDE.md` 里注释掉 `pilotfish:begin/end` 之间的策略块，六个 agent 文件不删也不碍事。

  降级策略也是内置的。如果 Fable 5 不可用（比如七月订阅变动），`best` 别名自动回退到 Opus；如果某个模型过载或 API 报错，`fallbackModel` 链自动切换。所有角色 agent 用模型别名（`opus`、`sonnet`、`haiku`）而非硬编码版本号，Anthropic 升级推荐版本时你的配置跟着自动走。

```bash
# 克隆并进入
git clone --branch v1.1.5 --depth 1 https://github.com/Nanako0129/pilotfish.git
cd pilotfish
claude

# 在 Claude Code 会话中粘贴：
# Read the local file install/AGENT-INSTALL.md and follow it to install pilotfish.
# Show me the full plan before writing anything.
```

| AgentOpt | pip install | 框架无关的模型选择/路由，HTTP 拦截 | 开源免费 | ⭐ 中 |

  pilotfish 和 Claude Code 内置的 `opusplan` 模式解决的是同一个问题，但粒度不同。`opusplan` 让你整场会话用 Opus 规划 + Sonnet 执行——够简单，但不够细。pilotfish 让你精确控制"哪种任务用哪个模型"，且内置了验证层（verifier）和降级链路。如果你只需要"便宜点就行"，`/model opusplan` 够用；如果你想知道"省了多少、省在哪、结果有没有变差"，pilotfish 值得一试。
