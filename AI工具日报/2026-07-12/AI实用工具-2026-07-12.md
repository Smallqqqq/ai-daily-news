# AI 实用工具 · 2026-07-12

![[summary.png]]

  今天三个工具踩中了 AI 编码链条中最薄弱的一环——写完代码之后的事。一个在后台持续审查 Agent 的每一次提交，审完还能让 Agent 自己去修。一个让你在自建服务器上跑自己的 Copilot，代码永不离开你的网络。一个生成测试后自己跑、跑不过自己修、修不好就回滚——直到全绿为止。

## Roborev — Agent 每提交一次代码，它就在后台审一次，审出问题让 Agent 自己修

  用 AI Agent 写代码的人迟早会发现一个尴尬的事实：Agent 写代码的速度远超你审查代码的速度。你让 Claude Code 修了三个 bug、加了一个 feature，它提交了七八个 commit，等你回过头来 review 的时候，上下文已经凉了。更糟的是——Agent 自己不会回头看。它在 commit 3 里引入的问题，会被 commit 7 的改动埋得更深。

  Roborev 的解法是把代码审查从"人手动触发"变成"Agent 自动触发"。`roborev init` 在你的仓库里装一个 post-commit hook，之后每次 `git commit`——不管是你写的还是 Agent 写的——都会在后台触发一次自动审查。审查结果进入一个持久化的 TUI 队列，按严重程度排列，支持 vim 风格导航。你不去 dismiss，它就一直在那里。

  两层自动化对应两种使用场景。第一层是 post-commit hook——Agent 提交代码，Roborev 在后台审，审完把结果丢进队列。你在终端里 `roborev tui` 随时查看。第二层是 agent hook——它监听 Claude Code 或 Codex 的会话，当累积的未处理审查结果超过阈值时，主动在会话里提示 Agent："你还有 3 个 review 没过，要不要修一下？"Agent 调用 `/roborev-fix` 技能，读取审查意见，改代码，提交——新 commit 触发新一轮审查，闭环。

  `roborev refine` 把这个循环推到了完全自动。它在隔离的 git worktree 里启动一个 fix→review→fix 循环：Agent 根据审查意见修改代码→提交→自动审查→如果还有问题继续修→直到全部通过或达到最大迭代次数。worktree 隔离保证了修复过程不会弄脏你的主工作区，失败了直接丢弃，没有任何副作用。

  内置分析覆盖六类问题：代码重复、圈复杂度、重构机会、测试缺口、死代码、安全问题。审查结果不只是"这里有问题"——每条 finding 带文件路径、行号和具体的修复建议。`roborev compact` 会在展示前做一次验证——确认 finding 指向的代码在当前版本里还存在，过滤掉已被后续提交修复的误报，把相关问题合并成一条。40 位贡献者、107 个 release、最新 v0.61.3，迭代速度很快。

  支持 10 种 AI 编码工具：Claude Code、Codex、Gemini、GitHub Copilot、OpenCode、Cursor、Kiro、Kilo、Droid、Pi。纯 Go 编写，本地运行，不需要外部服务或额外基础设施。审查数据存本地数据库，你拥有一份完整的"审查账本"——谁在什么时候提交了什么、审出了什么问题、修了没有。

```bash
# macOS / Linux
brew install kenn-io/tap/roborev

# 或直接下载预编译二进制
# https://github.com/kenn-io/roborev/releases

# 在当前仓库初始化
cd your-repo
roborev init

# 之后每次 git commit 自动触发审查
# 查看审查结果
roborev tui

# 安装 Agent 技能和 Hook（让 Agent 自动修）
roborev skills install
roborev agent-hook install

# 全自动 refine 循环（隔离 worktree，修复直到全绿）
roborev refine
```

  Roborev 和其他代码审查工具走的是两条不同的路。Open Code Review、PR-Agent、CodeRabbit 的审查时机在 PR 阶段——代码已经写完了，改动已经合在一起了，审查是合并前的最后一道关卡。Roborev 把审查时机前移到了每一次 commit——Agent 刚写完、上下文还热着、修起来成本最低。这个差异的实际后果是：PR 阶段的审查告诉你"这个 PR 有问题"，commit 阶段的审查告诉你"第三个 commit 引入了这个问题，现在修只需要改 5 行"。对于每天和 Agent 协作、commit 频率远超手动编码时代的开发者，这个时间窗口的差异就是"修一个小问题"和"重构一整块代码"的差异。

## Tabby — 自托管版 GitHub Copilot，代码不离开你的服务器，33K stars

🏪 Visual Studio Marketplace · ⬇ 80,840 安装 · ⭐ 33,680 GitHub stars · (12) 评分 · Free · [安装](https://marketplace.visualstudio.com/items?itemName=TabbyML.vscode-tabby)

  GitHub Copilot 很好用，但有两个问题始终绕不开：你的代码要离开你的机器，以及你要按月付费。对于个人开发者来说，$10/月不算贵；但对于一个 50 人的团队、或者代码合规要求不允许出内网的环境，这两个问题就是 blocker。

  Tabby 是目前开源社区里最成熟的 Copilot 替代方案。它在你的服务器上跑一个自包含的代码补全服务——不需要数据库、不需要云服务、不需要注册任何账号。装好之后，VS Code 扩展连到你自己的服务器，代码补全的整个链路都在你的网络里。支持消费级 GPU（RTX 3060 就能跑 7B 模型），也支持纯 CPU 推理。

  三个核心能力对应三种使用场景。**代码补全**：像 Copilot 一样，在你敲代码时实时给出多行建议，Tab 接受、Esc 拒绝——肌肉记忆完全不用改。**聊天面板**：侧边栏对话，能用 `@` 引用文件作为上下文，选中代码右键直接提问。**Answer Engine**：连接团队内部文档和代码仓库，把分散在各处的知识变成可查询的答案——不需要人去翻 Wiki 或翻 Confluence。

  模型选择很灵活。你可以用 Tabby 自带的模型，也可以接任何 OpenAI 兼容的 API——本机的 Ollama、局域网里的 vLLM、云端的 GPT-4o，全都可以。这种"自带模型"的设计意味着你不需要在"代码不出内网"和"用最好的模型"之间做选择——你可以在内网部署一个强模型，Tabby 只负责补全和聊天的交互层。

  Rust 编写，性能开销极低。120 位贡献者、249 个 release、Docker 一键部署。2023 年至今一直在活跃迭代，不是在"即将支持"的阶段——它已经支持了。

## testpilot-ai — AI 生成测试不稀奇，稀奇的是生成后自己跑、跑不过自己修、修不好就回滚

  大多数 AI 测试生成工具做的是同一件事：读你的源码，吐一堆测试文件，然后说"搞定"。你跑一下——红了 8 个。你把错误信息贴回去让它修——它又生成 3 个新问题。几个来回之后你发现手动写测试可能还快一点。

  testpilot-ai 的策略不同：它从来不把"生成测试"当作终点。加 `--verify` 参数后，它进入一个 verify & auto-fix 循环：生成测试→跑测试→收集失败的用例和错误堆栈→把失败信息喂回 LLM→生成修复→再跑测试。最多 3 轮，每轮只在上一轮的基础上改进，不会推倒重来。最关键的设计是**最佳版本持久化**——如果第 2 轮修复引入了新问题、比第 1 轮还差，testpilot-ai 会自动回滚到第 1 轮的版本。不会出现"越修越烂"的情况。还有一个振荡检测——如果连续两轮出现同样的失败，提前终止循环，不浪费 token。

  TypeScript 编译器 API 做 AST 分析是它和纯 prompt 方案的核心差异。它不需要在 prompt 里写"这是一个 TypeScript 函数，参数类型是 X"——编译器直接告诉它。导入上下文也是精确的：它会沿着相对导入和 tsconfig path alias（`@/foo`、`~lib/*`）追回你的项目里，上限 5 个文件 × 3KB，去重后作为上下文传给 LLM。`node_modules` 自动跳过。这意味着生成的测试里不会出现从 `express` 导入 `Request` 但你的项目用的是 `fastify` 这种低级错误。

  支持四种测试框架（Vitest、Jest、Mocha、Node test runner），自动从 `package.json` 检测该用哪个。支持四种 LLM provider（OpenAI、Anthropic、Google、Ollama），可以用本地模型完全离线跑。覆盖度缺口填充——解析 lcov/cobertura 覆盖率报告，只为没覆盖到的代码生成测试，而不是每次都重来一遍。成本控制——`--max-cost <usd>` 设定美元上限，超出就停。

```bash
# 安装
npm install -D @amit641/testpilot-ai

# 生成测试（不验证）
npx testpilot src/utils.ts

# 生成 + 验证 + 自动修复（推荐）
npx testpilot src/utils.ts --verify

# 使用本地 Ollama 模型（完全离线）
npx testpilot src/utils.ts --verify --provider ollama --model codellama

# 覆盖率缺口填充
npx testpilot src/ --verify --provider openai

# 分析项目中哪些文件需要测试
npx testpilot analyze

# 成本控制 + 并发
npx testpilot src/ --verify --max-cost 0.50 --concurrency 2
```

| Ponicode（已停止维护）| ❌ | ❌ | ❌ | ❌ | Jest/Mocha | 已归档 |

  testpilot-ai 最独特的特性不是"能生成测试"——Copilot 也能，Codium 也能，很多工具都能。独特的是 verify & auto-fix 循环里的两个细节：最佳版本持久化（不让修复把测试改得更烂）和振荡检测（同样的问题反复出现就停，不浪费 token）。这两个设计解决的是同一个根问题——LLM 的输出是不确定的，你对同一个代码跑两次测试生成，可能得到两个质量完全不同的结果。没有回滚保护的工具会把第二次的烂结果覆盖掉第一次的好结果，你甚至不知道发生了这件事。testpilot-ai 把这个不确定性关进了笼子里。
