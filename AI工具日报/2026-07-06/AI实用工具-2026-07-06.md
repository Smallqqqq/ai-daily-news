## mcpsnoop — "Wireshark for MCP"，透明代理让你看到 AI 客户端和 MCP 服务器之间真正发生了什么

![[mcpsnoop-demo.gif]]

  MCP（Model Context Protocol）已经铺得足够广了——截至 2026 年 3 月，月均 SDK 下载量 9700 万次，活跃公共服务端超过 10000 个，Claude Code、Cursor、GitHub Copilot 都在用。但调试工具始终没跟上。官方 MCP Inspector 是当前最主流的调试方案，它的问题在于架构：Inspector 把自己当作一个独立的 MCP 客户端连接服务器，等于是开了一条旁路去做测试。它能告诉你「如果我自己去调这个工具，会返回什么」，但它永远看不到你的真实客户端（Claude Desktop、Cursor、Claude Code）和你的服务器之间实际发生了什么。工具调用静默失败、参数被传错、客户端协商出了和预期不同的能力集——这些故障只发生在真实流量中，Inspector 完全看不见。

  mcpsnoop 的解法是把自己塞进真实数据通路。它不是旁路观察者，而是透明代理——用 `mcpsnoop --` 包裹你原有的服务器启动命令，它就在客户端和服务器之间原封不动地转发每一字节，同时把每个 JSON-RPC 帧拷贝一份送到终端 UI。你看到的是真实客户端和真实服务器之间实际在传的东西，不是模拟测试的结果。

  这个架构差异决定了它能捕捉到日志和 Inspector 都看不到的故障。一个典型场景：MCP 协议中，工具调用可以返回 HTTP 200 的成功响应，但 `result.isError` 字段为 true——这表示「调用完成了，但工具本身报告了失败」。只看 HTTP 状态码的日志监控会完全漏掉这类错误，mcpsnoop 在流中显式标记 `result.isError`，把协议层成功和工具层失败区分开来。

  几个日常高频使用的功能：**Live Stream** 实时展示请求、响应、通知和服务器 stderr，按类型着色，错误和慢调用高亮标注。**Replay** 把捕获到的任意一次工具调用重新发给一份全新的服务器副本，不需要重启客户端、不需要手动复制 curl 命令——修完 bug 按一下就知道好了没有。**Capability Inspector** 按 `c` 键查看客户端和服务器握手时协商的确切能力集，排查「为什么工具列表里少了一个」这类问题不用再对着配置文件猜。**Hung-call Detection** 进行中的请求显示 `PENDING` 加实时计时器，哪个工具卡住了、卡了多久，一眼就能看出来。**Filter Query** 支持 `tool:`、`status:`、`dir:`、`kind:`、`id:` 和纯文本过滤，几十条消息中快速定位目标。

  它的零配置设计值得单独提一句。shim（代理层）和 hub（TUI 界面）通过固定的 Unix socket 和磁盘日志通信，不要求任何启动顺序。你可以先开客户端再开 TUI，也可以反过来；关了 TUI 再开，它会自动回填之前的会话记录。和竞品比，MCP Inspector 需要你打开浏览器、手动填入服务器路径；mcp-trace 虽然也能看到真实流量，但没有交互式 TUI 和零配置启动。mcpsnoop 是目前唯一一个同时做到「看到真实流量」「交互式终端界面」「零配置启动」「能力检查」「调用回放」「单二进制无运行时依赖」六项全中的工具。

  纯 Go 编写，编译为单个二进制文件，无运行时依赖。7 月 4 日刚刚在 Hacker News 上作为 Show HN 发布，当天冲上首页。3 位贡献者，194 stars，MIT 协议。

```bash
# 方式一：Go 安装
go install github.com/kerlenton/mcpsnoop/cmd/mcpsnoop@latest

# 方式二：Homebrew
brew tap kerlenton/mcpsnoop
brew install mcpsnoop

# 方式三：直接下载预编译二进制
# https://github.com/kerlenton/mcpsnoop/releases
```

  配置只需改一行——在 MCP 客户端配置里把原来的服务器命令前面加上 `mcpsnoop --`：

```jsonc
{ "mcpServers": {
    "my-server": { "command": "mcpsnoop", "args": ["--", "node", "build/index.js"] }
}}
```

  对于 HTTP 类型的 MCP 服务器，mcpsnoop 也可以作为反向代理运行：

```bash
mcpsnoop http --target http://localhost:3000/mcp --listen :7000
```

  然后打开 TUI：

```bash
mcpsnoop
```

  先用 `mcpsnoop demo` 体验一下预录的演示会话，不需要配置任何东西。

### 同类工具对比

| 工具 | 看到真实客户端流量 | 交互式终端 UI | 零配置启动 | 能力检查 | 调用回放 | 单二进制 |
|------|:---:|:---:|:---:|:---:|:---:|:---:|
| **mcpsnoop**（今日推荐）| 是 | 是 | 是 | 是 | 是 | 是 |
| MCP Inspector | 否 | 否（Web） | 否 | 部分 | 否 | 否 |
| mcp-trace | 是 | 是 | 否 | 否 | 否 | 视情况 |
| mcptools | 是 | 是（Textual） | 否 | 否 | 是 | 否（需 Python） |

  mcpsnoop 是目前唯一覆盖全部六项的工具。MCP Inspector 适合开发阶段的交互式测试（类似 Postman 之于 REST API），但无法排查真实客户端的问题。mcp-trace 能看到真实流量，但需要手动配置。mcptools 功能最全面（六条命令覆盖 proxy/inspect/call/record/replay/doctor），但用 Python 编写、依赖 Textual 框架，不能像 mcpsnoop 那样一个二进制丢到任何机器上就跑。如果你已经在用其他调试工具，它们各自覆盖了不同的场景；但如果你只装一个，mcpsnoop 的零配置 + 单二进制 + 全功能覆盖是当前最好的组合。
