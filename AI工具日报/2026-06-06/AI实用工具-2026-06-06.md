# AI 实用工具 · 2026-06-06

  今天的三个工具围绕同一个问题：AI 编码 agent 的上下文质量。当 agent 被扔进一个十万行代码库，它只能靠 grep 和逐文件阅读来建立认知——慢、贵、容易跑偏。SocratiCode 给 agent 装上代码库的语义索引，Cost.dev 让写基础设施代码的 agent 不再瞎猜云成本，GitHub Copilot 的新 API 则让 agent 任务从「人盯着跑」升级为「按计划自动跑」。

---

## SocratiCode — 给 AI Agent 装上代码库的语义索引

  用 Claude Code 或 Cursor 问过「这个认证中间件是怎么工作的」的人都知道这个流程：agent 先 `grep -rl "auth"` 找到 200 个文件，然后逐个 `read` 猜测哪个才是核心实现，读到第 15 个文件时上下文窗口已经烧掉一半，最后给了一个看似合理但不完整的答案。SocratiCode 用一个预建的混合语义索引彻底改变了这套流程。

  原理不复杂但实现扎实：先对代码库做 AST 感知的代码分块（不是粗暴按行切），然后建立 dense vector + BM25 双路索引，查询时用 Reciprocal Rank Fusion 融合两路结果。这意味着「找认证中间件的实现」这种概念性查询和「`createSession` 函数定义」这种精确查询都能一次命中。索引由文件监听器自动增量更新，Docker 容器管理 Qdrant + Ollama，零配置启动。额外还有跨 18+ 语言的依赖图谱、符号级影响分析（「改 `processOrder` 会波及哪些调用链？」）、交互式图谱浏览器、以及可搜索的上下文制品（数据库 schema、API 规范、架构文档）。

  数据来自项目方自己在 VS Code 2.45M 行代码库上的 benchmark：同一组五个架构级问题，SocratiCode 相比 grep-based agent 少用了 61.5% 的上下文数据、减少了 84% 的工具调用次数、响应速度快 37 倍。更关键的是，因为影响分析和调用链是预计算的，小模型也能处理原本需要顶级推理能力的架构级任务——进一步降低 token 成本。

```bash
# Claude Code 一键安装（推荐，包含工作流 skills）
claude plugin marketplace add giancarloerra/socraticode
claude plugin install socraticode@socraticode

# 或 VS Code / Cursor 扩展安装
code --install-extension GiancarloErra.socraticode

# 也支持纯 MCP 接入（任何 MCP 客户端）
npx socraticode
```

## Cost.dev — 让 AI Agent 写 Terraform 时不再瞎猜云成本

  用 AI agent 写基础设施代码有一个隐蔽但昂贵的风险：agent 对云资源定价一无所知。它可能会「好心」帮你把一个 NAT Gateway 写进 dev 环境（$32/月），或者选了一个 IOPS 拉满的 EBS 卷（$200+/月），而你直到月底账单出来才发现。Infracost 团队 6 月 4 日在 HN 发布了 Cost.dev——一个专门为 AI coding agent 设计的云成本 CLI。

  它和原版 Infracost 的核心区别在于：整个 CLI 的输出格式和调用方式完全针对 agent 调用者优化。团队在博客中公布的数据是：相比裸 Claude 直接做成本估算，新 CLI 让 agent 的 output token 使用量降低了 79%，API 成本降低了 67%。怎么做到的？用谓词 flag 替代 agent 手写 `jq | python | wc` 管道，输出格式剔除了 JSON 中冗余的字段名，用确定性代码完成价格抓取和校验（而不是让 LLM 凭记忆猜价格）。

  支持 AWS、Azure、GCP 三朵云上超过 1,100 种资源，定价数据持续从云厂商价格源更新。三种使用模式：`iac-generation` skill 让 agent 在生成 Terraform 代码时实时遵守成本策略；`scan` skill 分析已有项目的成本违规；`price-lookup` skill 回答「`us-east-1` 的 `m7i.xlarge` 多少钱」这类独立查询。所有数据存在本地，代码不上传。

  它还集成了各云厂商 Well-Architected Framework 中的成本优化支柱——agent 写代码时自动应用最佳实践。自定义价格书（EDP 折扣、私有定价协议）和 FinOps 策略也可以注入，确保 agent 看到的成本数字和你公司财务实际支付的价格一致。Tagging 合规扫描能一次性修复整个仓库的标签违规。

```bash

brew install infracost/tap/infracost
infracost dev

# 或直接下载二进制
# https://cost.dev/

# 在 Claude Code 中安装 agent skills
claude plugin install infracost-agent-skills
```

## GitHub Copilot 云端自动化 — Agent 任务从手动触发升级为计划调度

  6 月 2 日，GitHub 为 Copilot 用户同时放出了两个关键能力：**Cloud Automations**（云端自动化）和 **Agent Tasks REST API**（公开预览扩大至 Pro/Pro+/Max 用户）。这两个功能合在一起，把 Copilot 从一个「你在编辑器里对话的助手」变成了「可以按计划或事件自动运行的编码 agent」。

  Cloud Automations 的核心概念很简单：你可以为仓库配置自动化规则，让 Copilot cloud agent 按固定时间间隔或响应仓库事件（新 Issue 创建、PR 创建/更新）自动执行任务。三个典型场景直击开发者痛点——**Issue 自动分类**：新 Issue 创建时自动根据内容标记 `bug`/`enhancement`/`other`；**夜间测试修复**：每晚检查 `main` 分支的失败测试，尝试自动修复并开 draft PR；**每周发版准备**：定时生成 release notes 并开 PR。Agent 运行在 GitHub 托管的隔离 Linux 沙箱中，每次写操作默认需要人工批准（可切换为 autopilot 模式）。

  Agent Tasks REST API 则是编程化入口。一个 POST 请求就能在一个仓库启动 cloud agent 任务，agent 自动分析代码库、做出修改、开 PR。请求体只需一个 `prompt` 字段，可选配置 `base_ref`、`model` 和 `create_pull_request`。任务状态可通过 API 轮询——`queued` → `in_progress` → `completed`/`failed`/`waiting_for_user`。`waiting_for_user` 状态值得注意：agent 遇到不确定的情况会暂停并请求澄清，而不是悄悄产出错误代码。

  这意味着 Copilot 可以被集成到任何自动化流程中：从内部开发者门户（Backstage/Port）一键初始化新仓库、跨数十个仓库的批量重构和迁移、CI/CD 流水线中自动生成变更日志。API 支持 PAT（classic 和 fine-grained）和 OAuth token 认证。成本方面，economy 模型（Claude Haiku 4.5、GPT-5.4-mini）按 0.33x 用量乘数计费，对于批量任务可以显著控制成本。

```bash
# REST API 示例：在仓库中启动一个 cloud agent 任务
curl -X POST \
  -H "Accept: application/vnd.github+json" \
  -H "Authorization: Bearer $GH_TOKEN" \
  https://api.github.com/agents/repos/OWNER/REPO/tasks \
  -d '{
    "prompt": "Migrate all fetch() calls to use the new ApiClient wrapper",
    "base_ref": "main",
    "model": "auto",
    "create_pull_request": true
  }'

# 自动化配置在 github.com 仓库的 Agents > Automations 页面完成
```

  Copilot Automations 的压倒性优势是「零基础设施」：不需要 Kubernetes 集群、不需要配置 agent 适配器、不需要维护状态数据库。缺点是灵活性不如 Optio——后者支持多供应商 agent、持久化 agent 进程和自动合并闭环。如果你只需要在 GitHub 仓库上做定时自动化，Copilot Automations 是零摩擦的选择；如果需要跨平台、自托管、深度定制的 agent 编排，Optio 值得评估。
