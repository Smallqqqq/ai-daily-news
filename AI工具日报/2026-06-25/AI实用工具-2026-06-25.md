## Shofer — 用声明式 .slang 文件描述多 Agent 工作流，非 LLM 执行器驱动，可复现、可观测

🏪 VS Code Marketplace · Apache 2.0 · June 2026 首发 · [安装](https://marketplace.visualstudio.com/items?itemName=shofer-dev.shofer)

  当前的 AI 编码 Agent 有个共同的软肋：当你需要多个 Agent 协作完成一个复杂任务时，协调逻辑全在模型自己手里。模型决定什么时候 spawn 子 Agent、怎么路由消息、什么时候收敛。每次运行都是黑箱，上次成功的流程下次未必能复现。问题不在模型能力，运行时推理本身就是不确定性的来源。

  Shofer 的解法是把协调逻辑从运行时剥离到设计时。它引入了一个叫 **Slang** 的声明式 DSL：你在 `.slang` 文件中定义哪些 Agent 参与、各自什么 mode、消息怎么路由（`stake`/`await`）、控制流怎么走（`when`/`repeat`）、收敛条件是什么、预算上限是多少。然后一个**完全不调 LLM 的确定性状态机**来驱动执行。Agent 还是 LLM Agent，但编排逻辑变成了可审查、可复现的代码。

  这跟 Claude Code 的 Dynamic Workflows 走了两条完全不同的路。Dynamic Workflows 是模型在运行时自己决定拆解和调度，灵活但不可复现。Shofer 是开发者在设计时把工作流写成声明式 spec，执行引擎保证每次跑出来的拓扑结构一致。少了一些灵活性，换来了可审计、可调试、可版本控制。

  Shofer 内置了两个开箱即用的工作流。**Collaborative Debug**：三个 Agent 各自独立分析同一个 bug，收敛到一致的根本原因后由一个 Agent 执行修复。**Implement a Feature**：Architect 出设计方案 → 你审批 → Developer 分片实现 → Reviewer 逐片审查 → 循环直到双方都满意。两个工作流都以 `.slang` 文件形式存放在 `.shofer/workflows/` 下，可以直接用、可以改、可以写自己的。

  除了工作流引擎，Shofer 在其他维度上也做得相当完整。**语义代码搜索**基于 tree-sitter AST 分块 + 增量嵌入，不会每次重建索引。**Git 历史搜索**可以查到"这个改动是谁什么时候为什么引入的"。**原生的 Git Worktree 支持**让并行 Session 跑在不同分支上，内核级沙箱（Landlock/bwrap）隔离。**异步 MCP 调用**（`call_mcp_tool_async`）不阻塞 Agent 主循环。**硬性 USD 预算上限**按 Session 强制执行，防止失控循环烧钱。

  从 Copilot、Claude Code、OpenCode 迁移都有一键命令（`/migrate-from-copilot`、`/migrate-from-claude-code`、`/migrate-from-opencode`），配置和规则文件自动转换。

```bash
# 从 VS Code Marketplace 安装
# 搜索 "Shofer" 并安装，或点击：
# https://marketplace.visualstudio.com/items?itemName=shofer-dev.shofer

# 迁移 Claude Code 配置
# 在 Shofer 中运行：
/migrate-from-claude-code
```

| **Shofer**（今日推荐）| 声明式 .slang DSL + 非 LLM 执行器 | ✅ 可复现 | ✅ 拓扑/时序/泳道图 | ✅ Apache 2.0 | 免费 |

## Repobridge — 单文件 MCP Server，让 AI Agent 同时搜索你所有本地 Git 仓库

  「这个配置 key 在哪个服务里设的？」「哪个微服务调了这个接口？」在微服务架构下，这类问题每天发生。每次都要手动 cd 到正确的仓库、grep、切回 AI 会话粘贴结果。一个本该 5 秒的问题要花 5 分钟。

  Repobridge 的解法直接到几乎没有设计：一个单文件 Python MCP Server，配置一组 `repo_roots` 目录，Server 自动扫描一级子目录中的所有 Git 仓库，然后暴露 5 个工具——`list_repos`、`search_code`、`read_file`、`find_repos_with`、`run_build`。AI Agent 不需要知道你在哪个仓库，直接问"哪个服务里有 PaymentService 这个类"就行。

  实现上用的是 Python FastMCP SDK，线程安全缓存（默认 300 秒 TTL），优先调 ripgrep（装了就有，没装降级到 grep），支持 stdio / SSE / Streamable HTTP 三种传输模式，可选的 Bearer Token 鉴权。作者是 Java/Spring 开发者，用了一周学会 Python 和 MCP SDK 写出来的。README 里坦言"最难的是一边学 Python 一边理解 MCP 规范"。

  这个工具的实用之处在于它不改变任何工作流。配好 `~/.repobridge.json`，Claude Code 启动时自动加载，之后你所有的跨仓库问题都变成一句话的事。不需要索引服务、不需要向量数据库、不需要 Docker。一个 venv 下的 Python 脚本，跑就完了。

```bash
# 1. 克隆安装
git clone https://github.com/sundeepsuddala/repobridge.git
cd repobridge
python -m venv .venv && source .venv/bin/activate
pip install -e .

# 2. 配置仓库根目录
cp example-config.json ~/.repobridge.json
# 编辑 ~/.repobridge.json，设置 repo_roots 为你的项目目录

# 3. 注册到 Claude Code（全局生效）
# 在 ~/.claude/settings.json 中添加 mcpServers.repobridge 配置
```

### 同类工具对比

| 工具 | 实现方式 | 搜索能力 | 跨仓库 | 部署复杂度 | 热度 |
|------|---------|---------|:---:|:---:|------|
| **Repobridge**（今日推荐）| 单文件 Python + ripgrep | 正则全文搜索 | ✅ 原生支持 | 极低（venv + pip） | 🆕 MIT |
| RepoRelay | Postgres + ParadeDB + pgvector | BM25 + 向量混合搜索 | ✅ | 高（需 Postgres） | ⭐ 较成熟 |
| code-intel-mcp | Zoekt 三元组索引 | 结构化代码搜索 + 符号引用 | ✅ | 中（需 Zoekt 二进制） | ⭐ 较成熟 |
| ssot-mcp | SQLite FTS5 + LanceDB | 关键词 + 语义搜索 | ✅ | 中（需嵌入模型） | ⭐ 一般 |

  选 Repobridge 的理由是启动成本最低。RepoRelay 的混合搜索更强大，code-intel-mcp 的符号级搜索更精准，但如果你只是想"让 AI 帮我跨仓库 grep"，它们的前置依赖太重了。Repobridge 是 venv + pip install -e . 一条路走到亮，不需要数据库、不需要索引服务、不需要向量嵌入。对于个人开发者和中小团队，这是目前该场景下摩擦最小的方案。
