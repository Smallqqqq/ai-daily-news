# AI 实用工具 · 2026-07-16

![[summary.png]]

  今天三个工具分别踩中了 AI 编码链条上三个截然不同的环节：一个把「AI 在干什么」从终端黑箱变成了可点击、可分支、可回退的视觉工作台；一个把工具调用能力从云端 GPU 压缩到了 26M 参数的模型里，塞进手机就能跑；一个让 JetBrains IDE 不再是 AI Agent 的盲区——Agent 终于能「看见」你的 IDE 而不只是读文件。

---

## Juggler — 对话不是聊天记录，是一棵可以分叉、回退、检查每一步的可编辑树

  Julian Storer 做了 30 年的音频开发工具——JUCE（全球最广泛使用的音频框架之一）、Tracktion DAW、Cmajor DSP 语言。他最近发布的 Juggler，瞄准的是另一个他忍了很久的问题：现有的 AI 编码 Agent 全是终端应用，而终端是世界上最不适合查看大量结构化信息的地方。

  大多数 AI 编码 Agent 给你的是线性聊天记录——你只能上下翻，最多回退到某个节点重新来。Juggler 把整场 Agent 会话变成了一棵可编辑的 **Yjs 文档树**。任何一个节点都可以分叉出子线程（sub-thread），子线程还能继续分。工具调用的「批准」「拒绝」不再是终端里一闪而过的状态——它们是 Miller 列视图里你能点开、检查、编辑的对象。每个工具调用后面都带着完整的上下文快照，你看得到 Agent 在调用 `replace-text` 时到底读了哪些文件、引用了哪些参数。

  核心技术栈的选择体现了 Storer 30 年的经验沉淀。后端用 Go 写，前端是 HTML/JS，通过 Wails 做原生窗口——**不是 Electron**。这意味着内存开销极低，启动速度和原生应用一样快。扩展系统是纯 JavaScript，context items、slash commands、LLM 循环策略全部是可 fork 或替换的插件。`read-file` 怎么和 LLM 交互、怎么在 UI 中渲染，都由插件定义——它不是「支持插件」，而是「本身就是插件拼出来的」。

  部署模式有三种：桌面应用（macOS/Windows/Linux），无头服务器模式（`juggler` 命令启动，浏览器连接），以及混合模式（同时从桌面和远程浏览器连接同一个会话）。服务器默认只绑 `127.0.0.1`，局域网访问需手动开启，WAN 访问仅限官方编译版。无注册、无遥测、无注册墙——下载即用。

```bash
# 从 releases 页面或官网下载
# https://github.com/juggler-ai/juggler/releases
# https://juggler.studio

# macOS: 下载 .dmg → 拖到 Applications → 双击启动
# Windows: 下载 setup.exe → 安装并启动
# Linux: 下载 juggler 二进制 → 终端运行 → 浏览器连接
juggler     # 无头模式，打印 URL + 二维码，浏览器扫码即连

# 支持的后端模型
# Claude Code (CLI/API), OpenAI (Codex plan/API), Gemini,
# Ollama, OpenRouter, Z.AI, Deepseek 等
```

  Juggler 和 Claude Code/Codex 的区别不是「谁更强」——它们调的是同一批 LLM，操作的是同一批文件。区别在于你**看不看得到** Agent 在做什么、**能不能中途介入**、**想不想从某个节点分叉尝试另一条路**。如果你对 Agent 的每一次操作都想保持完全可见性和可控性，Juggler 是目前唯一提供这种粒度图形化控制的工具。终端党可能觉得这是多余的设计，但对于需要审查 Agent 每一步操作——或是单纯对终端里疯狂滚屏感到生理性不适——的开发者，这是一个等了很久的答案。

## Needle — 把 Gemini 的工具调用能力蒸馏进 26M 参数，6000 tok/s，塞进手机就能跑

  工具调用（function calling）是 AI Agent 的核心能力——理解自然语言意图、把它映射到具体函数和参数。但今天主流的工具调用方案全是大模型：GPT-4o、Claude Sonnet、Gemini——动辄几千亿参数，需要 GPU 集群推理，延迟上百毫秒起。如果你的 Agent 需要跑在手机上、手表上、眼镜上，这些方案根本不可行。

  Cactus Compute 开源的 Needle 走了另一条路：把 Gemini 3.1 的工具调用能力蒸馏进一个 **26M 参数**的 Simple Attention Network。是的，两千六百万——不是二十六亿。预训练跑了 200B token（16 个 TPU v6e，27 小时），后训练用了 2B token 的单次工具调用数据（45 分钟）。结果是一个能在消费级设备上跑到 6000 tok/s 预填充、1200 tok/s 解码的工具调用模型。模型权重完全开放在 HuggingFace，数据集生成管线也开源。

  架构设计值得展开。Needle 没有用传统的 FFN（前馈网络）——它是一个 8 层解码器 + 12 层编码器，纯粹靠注意力机制。BPE 词表 8192，维度 512，8 注意力头 / 4 KV 头，用 RoPE 位置编码。Gated Residual 连接替代了传统的残差连接。模型文件只有一个 `.pkl` checkpoint，不到 100MB——可以塞进任何移动应用包里。

  实际能力需要诚实评估。Needle 做的是**单次工具调用**——给定一个自然语言查询和一组工具定义，选择正确的工具并填入参数。它在 FunctionGemma-270M、Qwen-0.6B、Granite-350M、LFM2.5-350M 的对比中胜出，但那些模型有更广的对话能力和常识推理。Needle 是一个专门工具——像一把极其锋利的螺丝刀，不是瑞士军刀。如果你需要的是一个能做多轮对话、写代码、翻译文档的全能模型，Needle 不是答案。如果你需要的是一个能理解「帮我查一下旧金山的天气，如果低于 10 度就打开暖气」这句话、然后输出 `get_weather(location="San Francisco")` 的微型模型——这就是 Needle 的精确靶心。

  微调门槛极低。每个工具给 120 个示例（100 训练 / 10 验证 / 10 测试），一行 `needle finetune data.jsonl`，在你的 Mac/PC 上就能跑。Playground 模式可以直接调用 Gemini 帮你生成训练数据、微调、评估、打包——全流程在浏览器里完成。

```bash
# 克隆并启动 Playground（自动下载权重）
git clone https://github.com/cactus-compute/needle.git
cd needle && source ./setup
needle playground      # 打开 http://127.0.0.1:7860

# Python 调用
from needle import SimpleAttentionNetwork, load_checkpoint, generate, get_tokenizer
params, config = load_checkpoint("checkpoints/needle.pkl")
model = SimpleAttentionNetwork(config)
result = generate(model, params, get_tokenizer(),
    query="What's the weather in San Francisco?",
    tools='[{"name":"get_weather","description":"Get current weather for a city.","parameters":{"location":{"type":"string","description":"City name.","required":true}}}]')
# → [{"name":"get_weather","arguments":{"location":"San Francisco"}}]

# 微调自己的工具
needle finetune data.jsonl    # 每个工具至少 120 条示例
```

## MCP Steroid — JetBrains 官方出品，给 AI Agent 装上 IntelliJ 的大脑和眼睛

🏪 JetBrains Marketplace · 官方孵化项目 · Apache 2.0 · [安装](https://plugins.jetbrains.com/plugin/30019-mcp-steroid)

  .NET 开发者上周有了 VS-MCP（Roslyn 编译器驱动的 Visual Studio MCP），Java/Kotlin 开发者这周有了 MCP Steroid——JetBrains 官方孵化的 IntelliJ 平台 MCP 服务器。但两者的设计哲学完全不同。VS-MCP 把 Roslyn 和调试器的能力包装成了 41 个精确的 MCP 工具——你给 Agent 一个具体任务，Agent 调用具体工具。MCP Steroid 的策略是：只暴露 **8 个 MCP 工具**，然后把整个 IntelliJ 平台的 API 全部打开——Agent 可以在 IDE 的 JVM 里写 Kotlin 脚本，直接调用 PSI（Program Structure Interface）、重构引擎、调试器、测试运行器、VCS 层。

  这个设计选择值得仔细品味。大多数 MCP 服务器的思路是「工具越多越好」——把每个能力包装成一个工具，工具列表越来越长。MCP Steroid 反其道而行：工具数压到最少，然后给你一个 `steroid_execute_code` 工具——它能在 IDE 的 JVM 进程里执行任意 Kotlin 代码。配合 `mcp-steroid://` 协议的资源 URI（技能指南和食谱），Agent 被教会了**直接在 IntelliJ 平台上写代码来操作 IntelliJ**。这是一个「教鱼捕鱼」而非「给鱼吃鱼」的设计。

  8 个工具各司其职：`steroid_execute_code`（在 IDE JVM 里跑 Kotlin 代码，完整 API 访问）、`steroid_fetch_resource`（读取 `mcp-steroid://` 技能指南）、`steroid_take_screenshot`（IDE 截图 + OCR + UI 组件树元数据）、`steroid_input`（键盘/鼠标事件注入，用 DSL 序列字符串描述操作）、`steroid_list_projects`（罗列所有打开的 IntelliJ 项目）、`steroid_list_windows`（枚举 IDE 窗口和组件）、`steroid_open_project`（通过程序打开项目）、`steroid_execute_feedback`（Agent 给自己的执行结果打分）。

  配套的 `devrig` CLI 是一个独立桥接器。注册一次，它能自动发现本机所有正在运行的 IntelliJ IDE（跨项目），把它们全部桥接到你的 AI Agent。如果一个 IDE 都没开——`devrig` 会自动下载、安装、启动一个托管 IDE 后端（IDEA Community/Ultimate、PyCharm、Android Studio 都支持）。这对 headless CI 环境和全新机器极其有用。

  官方 benchmark 给出了量化参考。在 DPAIA Spring Boot 任务上，有 MCP Steroid 的 Agent 比纯文件操作的 Agent 快 20-54%——复杂多文件操作的提升最显著（Rename ROLE_ADMIN 跨 9 个文件：202s vs 440s，-54%）。简单文本替换差异不大（+4%），说明 IDE 的结构化理解主要省的是「搜索-理解-定位-修改」循环中的人工/Agent 探索时间。

```bash
# 在 IntelliJ IDEA 2026.1+ 中
# Settings → Plugins → Marketplace → 搜索 "MCP Steroid" → 安装

# 或添加自定义插件仓库获取更快更新：
# Settings → Plugins → 齿轮 → Manage Plugin Repositories →
# 添加 https://mcp-steroid.jonnyzzz.com/updatePlugins.xml

# 一键注册 devrig 桥接器（需要 JDK 25）
git clone https://github.com/jonnyzzz/mcp-steroid.git
cd mcp-steroid
./gradlew deployNpx
~/.mcp-steroid/devrig/bin/devrig install claude    # 注册到 Claude Code
devrig install codex                                # 注册到 Codex
devrig install gemini                               # 注册到 Gemini

# 验证连接
claude mcp list                    # mcp-steroid: ✓ Connected
claude -p "List all open projects using steroid_list_projects"
```

| 工具 | IDE | 核心技术 | 视觉感知 | UI 自动化 | 代码执行 | 热度 |

| IDE Index MCP | JetBrains | IDE 索引查询 | ❌ | ❌ | ❌ | 🏪 较新 |

  MCP Steroid 和 VS-MCP 代表了两条完全不同的 MCP 设计路线。VS-MCP 把 Roslyn 编译器的语义理解能力包装成 41 个精确工具——确定性高、token 消耗可控，每步操作成本可预测。MCP Steroid 选择了「少工具 + 开放执行环境」——8 个工具 + Kotlin 脚本引擎 = 理论上无限的能力，但每步操作需要 Agent 先生成 Kotlin 代码再执行，token 消耗更高，出错模式也更多样。如果你在用 JetBrains IDE 写 Java/Kotlin/Android 代码，这是目前唯一一个能帮 Agent 真正「看见并操作」你的 IDE 的工具——官方背书保证了它不会因为 IntelliJ 平台 API 变更而突然坏掉。
