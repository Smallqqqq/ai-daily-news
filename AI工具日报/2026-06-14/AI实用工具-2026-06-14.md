## Serge — HuggingFace 开源的 GitHub-Native AI 代码审查，规则由仓库自己定义

  大多数 AI 代码审查工具的问题是：它们在 GitHub 之外运行，通过 OAuth 绑定、独立的 Dashboard、或者 IDE 插件来工作。维护者要在 GitHub 和一个外部工具之间来回切换，审查规则散落在第三方平台的配置里，fork PR 的权限问题更是噩梦。HuggingFace 的 Serge 用一个简单但关键的设计决策解决了这个问题——它完全跑在 GitHub 的 PR 流程内部，通过 GitHub Actions、GitHub App、或 Web App 三种模式部署，触发方式就是在 PR 下评论一句 `@askserge please review`。

  Serge 的核心差异在于"仓库自有的审查规则"。每个仓库在 `.ai/review-rules.md` 里定义自己的审查策略——关注什么、忽略什么、安全边界在哪里。这个文件始终从默认分支加载，意味着 PR 作者无法通过修改规则文件来绕过审查。仓库还可以提供 `.ai/context-script` 给 reviewer 提供额外上下文，以及 `read_file`、`list_dir`、`grep` 三个只读工具让 Serge 在 diff 之外做 bounded inspection。

  Serge 支持任意 OpenAI-compatible 的 chat completion 端点，包括 OpenAI 官方、HuggingFace Router（可访问大量开源和商业模型）、本地 vLLM/TGI/LM Studio 等。在 HuggingFace 的内部部署中，reviewer 可以直接从 HuggingFace Router 支持的模型列表里选择。三个部署模式各有侧重：GitHub Action 适合单仓库快速试用；GitHub App 适合组织和 fork 密集的项目（避免了 Action 拿不到 fork PR 的 secrets 的经典问题）；Staged Web App 则提供了人在回路中的审查——reviewer 可以编辑、丢弃或只发布部分 LLM 生成的评论，且默认禁止 approval review。

  安全设计值得注意。Serge 的系统 prompt 明确告诉模型不要把 diff 里的内容当指令执行（防 prompt injection），辅助工具运行在无 shell 环境中，环境变量剥离了 GitHub token、LLM key、OAuth secret 等敏感信息。HuggingFace 已经在自己的 `diffusers` 和 `transformers` 仓库中使用 Serge，Serge 自身也用 Serge 做自审。

```bash
# 方式一：GitHub Action（最快上手）
# 1. 在仓库 Settings > Secrets 中添加 LLM_API_KEY
# 2. 从 github.com/huggingface/serge 复制 workflow 到 .github/workflows/
# 3. 在任意 PR 下评论：
@askserge please review

# 方式二：克隆自托管
git clone https://github.com/huggingface/serge.git
cd serge
# 参考 docs/ 中的部署文档配置 GitHub App 或 Web App
```

### 同类工具对比

| 工具 | 部署方式 | 模型自由度 | 审查规则位置 | 人在回路 | 热度 |
|------|---------|-----------|------------|---------|------|
| **Serge**（今日推荐）| GitHub Action / App / Web | 任意 OpenAI-compatible | 仓库 `.ai/review-rules.md` | ✅ Web App 模式 | 🆕 HuggingFace 官方 |
| CodeRabbit | SaaS + 自托管 | 固定模型 | 平台配置 | ✅ 有限 | ⭐ 12K |
| GitHub Copilot Code Review | GitHub 内置 | Copilot 模型 | `.github/copilot-instructions.md` | ❌ | 需 Copilot 订阅 |
| Greptile | SaaS | 内置模型 | 平台配置 | ❌ | ⭐ 6K |
| Qodo Merge | SaaS + 自托管 | 固定模型 | 平台配置 | ❌ | ⭐ 1.5K |
| Sourcery | SaaS | 内置模型 | 平台配置 | ❌ | ⭐ 1.2K |

  Serge 的不可替代之处在于它的架构决策：规则由仓库自己定义、审查发生在 GitHub 原生流程中、模型可以自由选择。CodeRabbit 在成熟度和功能深度上领先，但它的审查规则和模型选择都锁在平台上。GitHub Copilot Code Review 最方便（Copilot 用户零额外成本），但模型和规则灵活度最低。Serge 适合那些"不想为了 AI 审查而改变现有 PR 流程"的团队——你不需要新账号、新 Dashboard、新通知渠道，只需要在 PR 下多打一行字。

## Omnigent — Databricks 开源的元编排层，让 Claude Code、Codex、Pi 在同一个会话里协作

  现在的 AI 编码工具体系是割裂的。你有 Claude Code 处理复杂重构、Codex 跑快速批量操作、Pi 做轻量级问答——但它们各说各话。你手动把 Claude 的分析结论复制给 Codex，再把 Codex 的输出喂给 Claude 做 sense check。Databricks 6 月 13 日以 Apache 2.0 开源的 Omnigent 就是来解决这个问题的：它不是你还需要再装一个的"又一个 Agent"，而是坐在你现有 Agent 之上的元编排层。

  Omnigent 的核心洞察很简洁：不管每个 Agent 内部怎么调 LLM，它们对外的接口都是相同的——消息和文件进、文本流和工具调用出。Omnigent 在这个共同接口之上包了一层统一的 API，把 Claude Code、Codex CLI、Pi、以及通过 YAML 定义的自定义 Agent 都变成可互换的 worker。一个 `omnigent` 命令启动后，同一个会话可以从终端、浏览器、手机三个入口同时访问——消息、子 Agent、终端、文件在三个端之间实时同步。

  几个独特的场景值得展开。**Polly 模式**：一个不写代码的编排者 Agent 负责规划和任务分配，把实际编码工作委派给 Claude Code、Codex、Pi 子 Agent，每个子 Agent 跑在独立的 git worktree 里，做完后自动路由到不同厂商的 reviewer（比如 Claude 写的代码由 Codex 审，反之亦然）——跨厂商交叉审查。**Debby 模式**：一个问题同时发给 Claude 和 GPT 两个"头"，回答并排展示，输入 `/debate` 后两个头互相 critique 几轮再收敛。**会话共享**：生成一个链接发给同事，对方可以实时看你 Agent 的工作过程、和 Agent 对话，甚至可以"共驾"（co-drive）——对方的消息在你的机器上执行。

  Omnigent 的策略系统（Policies）是另一个区别于纯 prompt 工程的机制。你可以定义"执行 shell 命令前问我"、"单会话最多 50 次工具调用"、"硬性费用上限 $5、$3 时预警"等规则，策略在三个层级叠加：服务器级（管理员设）、Agent 级（开发者设）、会话级（你自己设），会话级规则最先检查。这些策略是真正的代码检查而非 prompt 建议。

```bash
# 一键安装
curl -fsSL https://raw.githubusercontent.com/omnigent-ai/omnigent/main/scripts/install_oss.sh | sh

# 或 Homebrew
brew install omnigent-ai/tap/omnigent

# 启动（自动检测已登录的 CLI 和环境变量中的 API key）
omnigent

# 启动特定 Agent
omnigent claude    # Claude Code，同一会话可团队加入
omnigent codex     # Codex

# 运行自定义 Agent（YAML 定义）
omnigent run path/to/agent.yaml

# 部署服务器（手机也能接入）
omnigent server start
```

### 同类工具对比

| 工具 | 定位 | 编排层级 | 跨 Agent 协作 | 策略系统 | 会话共享 | 热度 |
|------|------|---------|-------------|---------|---------|------|
| **Omnigent**（今日推荐）| 元编排层 | Agent 之上 | ✅ 多 Agent 混合编排 | ✅ 三层策略叠加 | ✅ 实时共享+共驾 | ⭐ 138（🆕） |
| LangGraph | Agent 框架 | Agent 内部 | ✅ 图状态机 | ❌ 需自行实现 | ❌ | ⭐ 24K |
| CrewAI | Agent 框架 | Agent 内部 | ✅ 角色扮演式 | ❌ 需自行实现 | ❌ | ⭐ 46K |
| Multi-CLI | MCP 桥接 | Agent 之间 | ✅ 双向调用 | ❌ | ❌ | ⭐ 45 |
| Claude Code Workflows | 内置编排 | Agent 内部 | ✅ 子 Agent 并行 | ❌ 仅权限控制 | ❌ | 内置功能 |

  Omnigent 区别于 LangGraph 和 CrewAI 的关键在于：后者是让你**重新构建** Agent 系统的框架，而 Omnigent 是坐在你**已有** Agent 之上的编排层。你不需要学 Python SDK、写 Agent 图、定义 workflow——你已经在用的 Claude Code 和 Codex 直接就是 Omnigent 的 worker。和 Multi-CLI（6 月 11 日推荐过）相比，Multi-CLI 解决的是"让 Agent A 能调 Agent B"，Omnigent 解决的是"让 A 和 B 在同一个会话里被统一编排、从手机也能看进度、同事也能加入"。Omnigent 刚发布两天，138 星，状态 Alpha，但 Databricks 的背书和 Apache 2.0 协议给了它足够的关注理由。
