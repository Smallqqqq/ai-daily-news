## Ponytail — 一个开关，让 AI 写的代码从"能跑"变成"该跑"

  用 AI 写代码的人迟早会撞上一个尴尬的事实：Agent 写的代码能跑、测试也过了、lint 也没报错，但就是**臃肿得让人难受**——日期选择器不直接用 `<input type="date">` 而是引了 400 行第三方组件库，简单工具函数套了三层抽象包装，本来标准库一行搞定的事硬是写成 50 行带注释的企业级实现。这不是模型能力不够，恰恰相反——模型被训练得太"勤奋"了，以至于默认输出就是过度工程的。

  Ponytail 的解法极其简单但极其有效：**在 AI 动笔之前，注入一套强制性的六阶梯决策模型**，让 Agent 在每次生成代码前依次问自己六个问题——第一个成立的阶梯就是答案，后面的阶梯不用再想。这套阶梯是：① 这功能真的需要存在吗？（YAGNI）→ ② 标准库能做吗？→ ③ 原生平台 API 有吗？（比如 `<input type="date">`）→ ④ 已安装的依赖里有吗？→ ⑤ 能一行代码解决吗？→ ⑥ 以上都不行才写最小实现。规则的核心精神是"需要什么写什么，但校验逻辑、异常处理、安全防护一样不少"——代码之所以少，**是因为必要，不是被压缩过**。

  实测数据很能说明问题。在 FastAPI + React 真实开源仓库上，用 Claude Code + Haiku 4.5 完成 12 个功能任务（每组 n=4），有 Ponytail 和没有 Ponytail 的对比：

  | 指标 | 无 Ponytail | 有 Ponytail |
  |------|:---:|:---:|
  | 代码行数 | baseline | **-54%** |
  | Token 消耗 | baseline | **-22%** |
  | API 费用 | baseline | **-20%** |
  | 完成时间 | baseline | **-27%** |
  | 安全评分 | baseline | **100%（无退化）** |

  最极端的案例来自日期选择器场景——无 Ponytail 时 Agent 生成了 404 行代码（引了第三方组件库、写了完整的键盘导航、无障碍适配层），有 Ponytail 时只写了 23 行，因为模型在第三阶梯就意识到 `<input type="date">` 已经解决了所有需求。

  Ponytail 支持四级强度模式：`lite`（轻度约束）、`full`（默认，标准精锐）、`ultra`（极致精简：单行优先、最少变量、无装饰注释）、`off`（关闭）。被 Ponytail 跳过的"更优方案"会自动打上 `<!-- ponytail: browser has one -->` 注释标记，之后可以用 `/ponytail-debt` 命令一键汇集成待办清单——你随时可以审计"我到底为了精简放弃了什么"。

  覆盖 13 种 AI 编码工具：Claude Code、Codex、GitHub Copilot CLI、OpenCode、Gemini CLI、Cursor、Windsurf、Cline、Kiro、Antigravity、Pi、Aider、Qwen Code。Claude Code / Codex / Copilot 等支持生命周期钩子的工具获得完整能力（规则层 + 钩子层），Cursor / Windsurf 等纯编辑器工具通过规则文件静默注入——装完即生效，不需要记快捷键。

```bash
# Claude Code：完整插件模式
/plugin marketplace add DietrichGebert/ponytail
/plugin install ponytail@ponytail

# Codex
codex plugin marketplace add DietrichGebert/ponytail

# GitHub Copilot CLI
copilot plugin marketplace add DietrichGebert/ponytail
copilot plugin install ponytail@ponytail

# Cursor / Windsurf / Cline（轻量版，无需装插件）
# 将仓库中对应目录的规则文件复制到项目或全局配置文件夹即可
```

## codebase-memory-mcp — 158 种语言的代码知识图谱，Linux 内核 3 分钟索引，亚毫秒查询

  AI 编码助手最大的能力盲区之一，是**对大型代码库缺乏真正的"理解"**。当 Agent 需要回答"这个函数被哪些地方调用了"、"改了这个接口会影响哪些模块"、"这两个服务之间是什么调用关系"时，传统做法是逐文件 grep——慢、有假阳性、消耗大量 token。codebase-memory-mcp 的方案是：把你的整个代码仓库预先编译成一个存储在 SQLite 里的知识图谱，AI 查询的不再是文件内容，而是图结构。亚毫秒级响应，120 倍更少 token 消耗。

  核心架构分四层。第一层是 **tree-sitter AST 解析**——158 种语言的语法全部编译进一个约 7MB 的单静态 C 二进制文件，零运行时依赖，不需要安装任何语言工具链。第二层是 **Hybrid LSP 类型解析**——对 Python、TypeScript、PHP、C#、Go、C/C++、Java、Kotlin、Rust 九种语言提供类型感知的调用解析（类层次、重载匹配、泛型处理、跨文件类型推断）。第三层是**知识图谱构建**——节点是函数、类、HTTP 路由、Helm 模板等实体，边是 CALLS（调用链）、INHERITS（继承）、IMPORTS（导入）、HTTP_CALLS（跨服务调用）、CROSS_*（跨仓库引用）等关系，加上 FTS5 全文索引和 Leiden 社区检测算法自动发现模块边界。第四层是 **MCP Server**——暴露 14 个工具给 AI 编码助手，包括结构化图搜索、调用链追踪、Git diff 影响分析、自然语言语义搜索等。

  几个关键的性能数字。在 Apple M3 Pro 上：Django 全量索引约 6 秒，Linux 内核（28M LOC、75K 文件）全量索引约 **3 分钟**。Cypher 图查询 **<1ms**，调用链追踪（深度 5 层）**<10ms**。官方基准测试中，5 次结构化图查询消耗约 3,400 tokens——如果换做逐文件 grep，同样的信息需要扫约 412,000 tokens，差距是 **120 倍**。

  内嵌的语义向量搜索是另一个亮点。nomic-embed-code 模型（40K tokens 上下文、68 维 int8 向量）直接编译进二进制，完全本地运行——不需要 API Key、不需要 Ollama、不需要 Docker。搜 `send` 能命中 `publish`、`emit`、`dispatch`，搜 `auth` 能匹配 `login`、`authenticate`、`verify_credentials`。代码不出机器，这对企业合规场景是硬需求。

  安装覆盖 8 种包管理器（npm、PyPI、Go、Homebrew、Scoop、Winget、Chocolatey、AUR），装完执行 `codebase-memory-mcp install` 自动检测并配置 11 种 AI 编码 Agent（Claude Code、Codex CLI、Gemini CLI、Zed、OpenCode、Antigravity、Aider、KiloCode、VS Code、OpenClaw、Kiro）。团队共享模式：用 zstd 压缩的知识图谱 artifacts 可以提交到仓库，队友 clone 后直接导入，免去重复索引。

```bash
# 8 种包管理器任选其一
npm install -g codebase-memory-mcp     # npm
pip install codebase-memory-mcp        # PyPI
brew install codebase-memory-mcp       # Homebrew (macOS)
scoop install codebase-memory-mcp      # Scoop (Windows)

# 一键自动配置 AI 编码 Agent
codebase-memory-mcp install

# 索引当前仓库
codebase-memory-mcp index
```
