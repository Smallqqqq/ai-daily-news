# AI 实用工具 · 2026-06-11

![[summary.png]]

  今天三个工具分别切入了 AI 编程的三个不同层面：一个从零入局终端 Agent 赛道，一个在运行时自动修复代码错误，一个让不同 AI Agent 之间互相调用——都指向同一个趋势：AI 编程工具正在从"单打独斗"走向"分工协作"。

## Kimi Code CLI — 月之暗面开源的终端 AI 编程 Agent

  终端 AI 编程 Agent 的牌桌上已经有 Claude Code、Codex CLI、Gemini CLI 三位玩家。2026 年 5 月 22 日，月之暗面（Moonshot AI）以 MIT 协议发布了 Kimi Code CLI，成为第四位——也是唯一一个模型和代码都开源的选择。

  Kimi Code CLI 是一个 TypeScript 编写的终端 Agent，默认搭载 Kimi K2.6 模型（万亿参数 MoE 架构，256K 上下文窗口，100 tokens/s 输出速度）。它读代码、写文件、执行 shell 命令、搜索文件和网页，根据反馈自动规划下一步。和 Claude Code 一样，它也支持 MCP 协议、Plan Mode、子 Agent、生命周期 Hook。但区别于前三家的是——K2.6 模型权重开放，CLI 本身 MIT 协议开源，且允许通过 config.toml 切换到 Anthropic、OpenAI、Google 等第三方模型提供商。

  几个独特功能值得一提。**视频输入**：可以把屏幕录制或 demo 视频拖进聊天，Agent 直接"看懂"你难以用文字描述的操作流程。**AI 原生 MCP 配置**：通过对话式 `/mcp-config` 添加和认证 MCP 服务器，不用手写 JSON。**ACP 协议支持**：Kimi Code CLI 支持 Agent Client Protocol，可以在 Zed、JetBrains 等 IDE 中原生驱动——不需要额外登录，编辑器直接通过 stdio 连接 `kimi acp` 子命令。

  子 Agent 体系也值得关注：内置 `coder`（编码）、`explore`（探索代码库）、`plan`（方案设计）三个子 Agent，在隔离上下文中并行工作，主会话保持整洁。这在处理多文件重构或跨模块功能时特别有用——比如让 explorer 先梳理现有代码结构，coder 负责实现，plan 核验方案一致性，三个子 Agent 可以并行启动。

```bash
# 安装（无需 Node.js 预装）
# macOS / Linux
curl -fsSL https://code.kimi.com/kimi-code/install.sh | bash

# Homebrew
brew install kimi-code

# Windows (PowerShell)
irm https://code.kimi.com/kimi-code/install.ps1 | iex

# 首次启动
cd your-project
kimi
# 进入 TUI 后 /login 完成认证
```

## pfix — Python 运行时自我修复，错误即修即跑

  写 Python 时最常见的场景：脚本跑到一半崩了，看 traceback，改几行，再跑，又崩在另一个地方，再看，再改……这个循环的每一次迭代都在消耗注意力。pfix 的思路是把"崩溃→修复→重跑"做成自动化——在运行时捕获异常，拿错误上下文调 LLM 修代码，然后自动重启进程。

  pfix 用起来极其简单。零配置模式下，只需在 `.env` 中设置 `PFIX_AUTO_APPLY=true`，然后 `import pfix`，项目中任何未捕获异常都会自动触发修复流程。如果只想保护特定函数，用 `@pfix` 装饰器即可。对于 `ModuleNotFoundError` 这类依赖缺失，pfix 会跳过 LLM 直接调 `pip`/`uv` 安装——不浪费 token。真正的代码逻辑错误才会发到 LLM（通过 LiteLLM 支持 OpenRouter/Claude/GPT/DeepSeek 等 100+ 模型），附带完整 traceback 和上下文。

  更实用的是它的安全机制：修复前生成 unified diff 让用户确认，所有修改自动备份到 `.pfix_backups/` 目录，修完后通过 `os.execv` 原地重启进程。这意味着在开发阶段，你可以在一个长任务中途出错时让 pfix 自动抢救——比如批量数据处理脚本跑到第 800 条记录时因为一个 edge case 崩了，pfix 修复后自动从断点继续。此外还内置了 MCP server（FastMCP），可以在 Claude Code、Cursor 等 IDE 中直接调用修复工具。

```bash
# 安装
pip install pfix

# 零配置模式（.env 中设置）
echo "PFIX_AUTO_APPLY=true" >> .env

# 在入口文件最顶部
import pfix  # 全局自动修复

# 或装饰器模式
from pfix import pfix

@pfix
def my_fragile_function():
    ...
```

  pfix 的优势在于零配置启动和 MCP 集成——`import pfix` 一行就能覆盖整个项目，且内置的 MCP server 让它能直接嵌入 Claude Code 等 Agent 的工作流。self-heal-llm 的"记忆之前失败尝试"机制在修复杂 bug 时更精准，但需要为每个函数加装饰器。pfix 适合"不想改现有代码、希望全局保护"的场景。

## Multi-CLI — 让 Claude、Gemini、Codex 互相调用的 MCP 桥接器

  你有多个 AI 编程 Agent 装在电脑上——Claude Code 适合复杂重构，Codex CLI 适合快速批量操作，Gemini CLI 有免费的百万级上下文窗口。但它们各说各话，没法协作。你手动把 Claude 的结论复制粘贴给 Codex，再把 Codex 的输出喂给 Gemini 做 sense check——这恰好是多 Agent 工作流中最无聊的部分。

  Multi-CLI 用一个 MCP 服务器打通了这些 Agent。安装后，Claude Code 会多出 `Ask-Gemini`、`Ask-Codex`、`Ask-OpenCode` 等工具；Codex CLI 会多出 `Ask-Claude`、`Ask-Gemini` 等工具。任何一个 Agent 都能把任务委派给其他 Agent，拿到结果后继续自己的工作——不需要你手动搬运中间结果。

  它的设计有几个聪明的细节。每个 Agent 看不到调用自己的工具，避免死循环；自动检测你装了哪些 CLI，只显示可用的；支持 MCP Task 模式，长任务不会阻塞主 Agent 的请求。更值得说的是它的维护方式——这个项目的代码和模型列表由它自己连接的 AI Agent 自动维护，每晚 CI 查询各 CLI 最新模型列表、对比差异、自动发版。一个元到令人舒适的设计：桥接 Agent 的工具本身由 Agent 维护。

```bash
# 一键安装（自动检测已安装的 CLI 并配置）
curl -fsSL https://raw.githubusercontent.com/osanoai/multicli/main/install.sh | bash

# 或各客户端独立安装
# Claude Code
claude mcp add --scope user Multi-CLI -- npx -y @osanoai/multicli@latest

# Gemini CLI
gemini mcp add --scope user Multi-CLI npx -y @osanoai/multicli@latest

# Codex CLI
codex mcp add Multi-CLI -- npx -y @osanoai/multicli@latest
```

| 工具 | 安装方式 | 支持 Agent | 通信方向 | 热度 | 维护状态 |
|------|---------|-----------|---------|------|---------|
| **Multi-CLI**（今日推荐）| `npx -y @osanoai/multicli@latest` | Claude/Gemini/Codex/OpenCode | 双向 | ⭐ 45 / 116 w/wk | 活跃（自动发版） |
| all-agents-mcp | `npx all-agents-mcp` | Claude/Codex/Gemini/Copilot | 双向 | ⭐ 已归档 | 已归档，建议用 Skills 替代 |

  Multi-CLI 是目前唯一在生产维护的多 Agent 互调工具。all-agents-mcp 已归档，作者在 README 中明确指出"直接 CLI 调用 + Skills 集成"是更主流的方向。agent-link-mcp 有双向对话协议（spawn 的 Agent 可以反问主 Agent），适合需要深度协作的场景。但 Multi-CLI 的一键安装脚本和自动维护机制让它最适合"装完就忘"的日常使用——配置一次，每个 Agent 就都能调用其他 Agent 了。
