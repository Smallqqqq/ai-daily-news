## Claude Code Workflow — 用代码取代 Prompt 编排多 Agent

  5 月中旬发布的 Claude Code v2.1.147 中，Anthropic 在发布说明里删掉了对这个功能的描述，但它稳定可用：**Workflow 模式**。核心就是把多 agent 编排从自然语言"临场指挥"升级为 JavaScript 脚本"代码驱动"——用代码处理控制流（循环/条件/并行/重试），LLM 只负责每个步骤内的判断工作。

  为什么这件事重要？当前所有 AI agent 编排方案都有一个根本缺陷：主 agent 需要把每个子 agent 的输出拼进自己的上下文窗口，子 agent 越多、上下文越长，性能越差、token 成本越高（学术界称为"lost in the middle"问题）。Workflow 把这个负担从 LLM 上下文里拆出来，放到 JavaScript 变量里——编排脚本本身不消耗模型 token，只有每个 `agent()` 调用消耗。方案完成后只把最终结果返回给你，中间所有 agent 的对话历史都不进入主会话。

  使用时只需设置一个环境变量启动 Claude Code，然后在 prompt 中包含 `workflow` 关键词，或者使用 `ultracode` 模式（xhigh effort + 自动 workflow 编排）：

```bash
export ANTHROPIC_WORKFLOW=1
claude
# 然后在对话中输入：workflow 分析当前 PR 并生成多 agent review 工作流后运行
```

  Claude Code 不会直接执行任务，而是先编写一个完整的 JavaScript 脚本。脚本包含元数据（名称、描述）、多个执行阶段（Stage）、每个阶段的 Agent 定义（角色、工具、提示词、token 上限）。你可以用 `/workflows` 命令打开可视化监控界面，查看每个阶段的进度、耗时、Token 消耗、工具调用。脚本默认存在临时目录（3 天生命周期），满意的脚本可以复制到 `~/.claude/workflows/` 复用。

  一个完整的工作流脚本示例——deep-research 深度研究（官方内置）：

```javascript
export default {
  metadata: {
    name: "deep-research",
    description: "Multi-source parallel search, fact verification, report synthesis"
  },
  stages: [
    {
      id: "search",
      description: "并行搜索官方文档、学术论文和社区讨论",
      parallel: true,  // 阶段内 agent 并行执行
      agents: [
        {
          id: "official-docs",
          tools: ["web-search"],
          systemPrompt: "你是技术文档研究员，擅长从官方文档、API 参考和发行说明中提取准确信息。",
          prompt: "搜索并整理与主题相关的最新官方文档和最佳实践。",
          maxTokens: 4000, temperature: 0.1
        },
        {
          id: "papers",
          tools: ["arxiv-search"],
          systemPrompt: "你是学术论文研究员，擅长检索论文并提取方法、实验结果和结论。",
          prompt: "搜索相关学术论文（近三年优先），总结核心贡献和技术细节。",
          maxTokens: 4000, temperature: 0.2
        },
        {
          id: "community",
          tools: ["reddit", "github"],
          systemPrompt: "你是开源社区分析师，擅长从讨论中提取实际案例和痛点。",
          prompt: "收集社区讨论中的实际案例、常见问题和推荐实践。",
          maxTokens: 3000, temperature: 0.3
        }
      ]
    },
    {
      id: "verification",
      description: "交叉验证三个来源，标记矛盾",
      dependsOn: ["search"],
      agents: [{
        id: "fact-checker",
        tools: ["web-search"],
        systemPrompt: "你是严格的事实核查员。",
        prompt: "对比官方、学术、社区三路结果，输出一致性摘要、矛盾点（注明哪个来源更可信）、缺失信息、验证建议。",
        maxTokens: 6000, temperature: 0.0
      }]
    },
    {
      id: "synthesis",
      description: "生成最终技术报告",
      dependsOn: ["verification"],
      agents: [{
        id: "report-writer",
        tools: [],
        systemPrompt: "你是高级技术报告撰稿人。",
        prompt: "基于验证阶段输出，生成深度研究报告，含：研究背景、核心发现、一致性结论、风险评估、可执行建议、参考文献。Markdown 格式。",
        maxTokens: 8000, temperature: 0.2
      }]
    }
  ],
  return: async (context) => {
    const report = context.stages.synthesis.agents["report-writer"].output;
    await context.saveFile("deep-research-report.md", report);
    return { summary: "完成", reportPath: "deep-research-report.md" };
  }
};
```

  工作量限制：最多 16 个并发 agent，单次运行最多 1000 个 agent。支持六种编排形态——流水线、并行聚合、对抗验证（多个 agent 互相校验）、Fan-out/Fan-in、递归式、嵌套式。中断后可在同一 session 恢复。

  Workflow 和已有的 Skills / Sub-agent / Agent Teams 的关系值得理清：Skills 解决"能力封装"（告诉模型什么时候用什么工具），Sub-agent 解决"临时委派"（一次性的子任务分发），Agent Teams 解决"多角色并行"（人类主导调度）。Workflow 解决的是"流程固化"——把编排写成代码，让多 agent 协作变得可复现、可追踪、可验证。用一句概括：Skills 封装能力，Workflow 封装流程。

  社区已有响应：`ray-amjad/claude-code-workflow-creator` 是一个专门用来创建 Workflow 脚本的 Claude Code Skill，内含 API 参考手册、编排模式示例（fan-out / pipeline / judge panel 等）、验证 lint 工具。虽然 Workflow 功能尚未正式官宣，但社区已经开始建设生态。

```bash
# 安装 workflow creator skill
claude plugin marketplace add ray-amjad/claude-code-workflow-creator
claude plugin install claude-code-workflow-creator
```

## AgentKit — 像 create-vite 一样脚手架多 Agent 工作流

  大多数开发者使用 Claude Code 的方式是单次长对话：描述需求、Claude 写代码、纠错、再提示。这能工作，但问题也明显——上下文逐渐污染、每次新项目重讲一遍规范、没有审计记录、agent 之间需要手动协调切换。AgentKit 要解决的就是这个：把 Claude Code 从"聊天式一对一代码助手"升级到"可编排的多 agent 工程流水线"。

  核心思路很简洁。AgentKit 本身不含任何 AI——它不做 API 调用，不需要任何 key，零成本。它只是一个结构生成工具，在项目里脚手架出一套编排文件：`CLAUDE.md`（每个 agent 启动前读取的规范手册）、`PLAYBOOK.md`（自主执行引擎，包含阶段划分、重试逻辑、人工升级机制）、`AGENT_WORKFLOW.md`（项目路线图）。你只需要写一份纯语言的项目蓝图，然后运行一条命令，再在 Claude Code 里输入一句「Read PLAYBOOK.md and execute the procedure.」，整个流水线就自动运行。

  工作流分三个阶段：Phase 0 分解——Claude Code 读蓝图（或问你三个问题），提出 agent 拆分方案，你确认；技能注入暂停——Claude Code 创建 agent 文件夹后暂停，你可以往每个 agent 目录丢 API 文档、数据库 schema、算法参考资料，再输入 proceed；Phase 1 执行——每个 agent 按顺序跑，完成后验证成功标准，失败自动重试最多三次，真正阻塞时才升级给人类。

  作者用 AgentKit 自举构建了一个真实项目——DevLog Desktop，一个基于 Tauri v2 + Ollama 本地 RAG 的桌面应用。结果：6 个 agent 自主执行，68 个测试通过，产生了 Windows 安装包（MSI + NSIS），完整中英文国际化、暗色模式、语义搜索。整个过程在一个 Claude Code 会话中完成。

```bash
# 初始化（自动检测栈或交互式选择）
npx @patricksardinha/agentkit-cli init

# 带蓝图初始化
npx @patricksardinha/agentkit-cli init --blueprint PROJECT_BLUEPRINT.md

# 然后在 Claude Code 中输入：
# Read PLAYBOOK.md and execute the procedure.
```

## Mirage — 给 AI Agent 一个统一的虚拟文件系统

  AI agent 面临一个根本性的碎片化问题：要访问 S3 桶里的日志、Google Drive 里的文档、Slack 频道里的消息、GitHub 仓库里的代码——每种数据源有自己的 SDK、自己的认证方式、自己的数据格式。Agent 要在 N 套 API 之间来回切换，每次切换都是上下文消耗和出错机会。Mirage 用一个统一的虚拟文件系统解决这个问题：把所有数据源挂载到同一个文件树下，agent 用 ls、cat、grep、cp 这些最熟悉的 bash 命令就能操作一切。

  原理上，Mirage 是一个可嵌入的虚拟文件系统层，支持 Python（`mirage-ai`）和 TypeScript（`@struktoai/mirage-node` / `@struktoai/mirage-browser`）SDK。你声明式地配置挂载点——把 S3 bucket 挂到 `/s3`、Google Drive 挂到 `/drive`、Slack 挂到 `/slack`、GitHub 挂到 `/github`——然后 agent 就可以用统一语义读写所有后端。LLM 最擅长 bash 和文件系统操作，Mirage 恰好把一切变成了 LLM 最擅长操作的形式。

```bash
# Python
uv add mirage-ai

# TypeScript
npm install @struktoai/mirage-node      # Node.js
npm install @struktoai/mirage-browser   # 浏览器/边缘

# CLI
npm install -g @struktoai/mirage-cli
```

```python
from mirage import Workspace
from mirage.resource.s3 import S3Resource, S3Config
from mirage.resource.ram import RAMResource

ws = Workspace({
    "/data": RAMResource(),
    "/s3":   S3Resource(S3Config(bucket="my-bucket")),
})

await ws.execute("cp /s3/report.csv /data/report.csv")
await ws.execute("grep alert /s3/log.jsonl | wc -l")
```

## Pullfrog — 开源版 CodeRabbit，Zod 作者新作

  AI 代码审查赛道过去一年增长有目共睹：CodeRabbit 是品类定义者，GitHub Copilot Code Review 上线后迅速铺开。但这些工具要么闭源托管 SaaS（数据离开你的仓库），要么绑定特定模型供应商。Pullfrog 走了第三条路：完全开源，运行在你自己仓库的 GitHub Actions 里，模型自选（BYOK），代码和数据都不出你的 GitHub 环境。

  Pullfrog 由 Colin McDonnell 创建——他是 Zod（TypeScript 生态最流行的 schema 验证库，42,000+ stars）的作者。Pullfrog 是一个运行在 CI 中的 agent 编排层：监听 GitHub webhook 事件（PR 创建、issue 创建、review 提交、CI 失败等），按配置触发 agent 执行。Agent 带有一个专用 MCP 服务器，可以创建 PR、提交 review、读取 CI 日志、管理 issue。还内置了 headless 浏览器工具，agent 可以直接跑 E2E 测试、截图、迭代 UI。

  使用方式简单：安装 Pullfrog GitHub App → 在仓库里加一个 `.github/workflows/pullfrog.yml` → 在 PR 或 issue 里 @pullfrog 触发。也支持自动触发器：新 PR 自动审查、CI 失败自动修、新 issue 自动分类。支持 Anthropic、OpenAI、Google、DeepSeek 等所有主流 LLM 供应商。

```yaml
# .github/workflows/pullfrog.yml
jobs:
  pullfrog:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: pullfrog/pullfrog@v0
        with:
          prompt: ${{ inputs.prompt }}
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
```

  Pullfrog 和 Copilot cloud agent 的核心差异在于触发机制：Copilot 以人驱动为主（分配 issue、@ 提及 agent），Pullfrog 支持事件驱动自动触发（新 PR 自动审、CI 失败自动修）。对于想降低 PR 审查响应延迟的团队，事件驱动是实质性的效率提升。Zod 作者的身份也让这个项目在 TypeScript 社区有了天然信任基础。
