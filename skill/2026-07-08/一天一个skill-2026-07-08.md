# react-doctor — AI 写的 React 代码，它来把关

  AI 编程助手写 React 很快，但写对的概率不高：state 更新死循环、useEffect 依赖缺失、用 index 当 key、props 内联对象导致无意义重渲染。这些错误 lint 规则不报、Code Review 容易漏、上线后才在 Sentry 里炸出来。react-doctor 是一个确定性的 React 代码扫描器，429 条规则覆盖状态/副作用、性能、架构、安全、无障碍五大维度，一条命令扫全项目。13,409 stars，97 个 release，支持 Next.js、Vite、TanStack、React Native、Expo 所有主流 React 框架。

---

### 它解决什么问题

  用 Claude Code、Codex 或 Cursor 写 React 的人都会遇到一个共同问题：AI 生成的 React 代码表面上能跑，但里面藏着各种"看起来对、实际上错"的陷阱。

  举几个典型例子：

  - **useEffect 无限循环**：依赖数组里漏了一个变量，或者每次渲染都在 effect 里 setState 触发下一次渲染
  - **内联对象/函数当 props**：每次渲染都创建新引用，子组件 memo 全白写
  - **派生状态**：用 useState + useEffect 同步 props 到 state，而不是直接在渲染期间计算。React 官方文档明确说这是反模式
  - **数组 index 当 key**：列表排序/过滤/增删时导致组件状态错乱、输入框内容串位
  - **不可访问的交互元素**：div 上绑了 onClick 但没有 role、tabIndex、键盘事件。屏幕阅读器用户完全无法操作

  这些问题 ESLint 不报、TypeScript 不报、AI agent 自己也不知道自己犯了错。react-doctor 做的是：用确定性的 AST 分析 + 规则引擎，扫描整个代码仓库，一个不漏地找出这些问题。

  它的定位不是替代 ESLint，而是插件——ESLint 管语法和风格，react-doctor 管 React 特有的逻辑陷阱和运行时风险。两者互补。

### 核心原理

  react-doctor 的核心是一个基于 AST 的确定性扫描引擎。和 AI agent 做 code review 不同，它不依赖 LLM，结果是确定性的：同一条规则、同一个文件，每次扫描给出完全一致的结果。

  **架构三层**：

  | 层 | 职责 | 技术实现 |
  |------|------|---------|
  | **AST 解析层** | 解析 JSX/TSX 文件，提取组件树、props 流向、hooks 调用关系 | 基于 oxlint 的解析器，兼容 TypeScript |
  | **规则引擎层** | 429 条规则，每条配验证逻辑（什么时候该报）+ 修复建议（怎么改） | 规则分组：accessibility / architecture / effect / performance / security / state |
  | **输出层** | 终端审计报告 + CI 评论 + agent 技能文件 | CLI / GitHub Actions / SKILL.md |

  **和 AI code review 的本质区别**：AI agent 审查代码靠"阅读理解"。它看到 `setCount(count + 1)` 然后判断"这看起来没问题"。react-doctor 靠 AST 分析。它追踪 `count` 这个变量的数据流，发现它在 useCallback 的依赖数组里缺失了，于是报警。前者是猜测，后者是推导。前者会误报也会漏报，后者的误报率趋近于零。

  **SKILL.md 模式**。`npx react-doctor@latest install` 除了装 CLI，还会在 agent 的技能目录下写入一个 SKILL.md 文件。安装后，agent 在写 React 代码时会自动参考 react-doctor 的规则集，从源头减少低级错误。同时 agent 也可以主动运行 `npx react-doctor@latest` 来审计自己的产出。

### 安装配置

  前提：Node.js ≥ 18，项目根目录有 `package.json`。

  ```bash
  # 第一步：本地审计（看看你的项目有多少问题）
  npx react-doctor@latest

  # 第二步：给 coding agent 安装规则技能
  npx react-doctor@latest install

  # 第三步（可选）：接入 CI，每次 PR 自动扫描
  npx react-doctor@latest ci install
  ```

  第二步会自动检测你机器上安装的 agent（Claude Code、Cursor、Codex、OpenCode 等），把 SKILL.md 写入对应目录。装完重启 agent 即生效。

  配置文件 `doctor.config.ts`（项目根目录）：

  ```typescript
  // doctor.config.ts
  import { defineConfig } from "react-doctor";

  export default defineConfig({
    // 只扫描 src 目录
    include: ["src/**/*.{ts,tsx}"],
    // 忽略测试文件和生成的代码
    ignore: ["**/*.test.*", "**/generated/**"],
    // 按类别禁用规则
    rules: {
      // 如果你确定某个规则不适用于你的项目
      "react-doctor/no-array-index-as-key": "off",
      // 设为 warn 不阻塞 CI，设为 error 阻塞
      "react-doctor/no-derived-state": "error",
    },
  });
  ```

  CI 模式接入 GitHub Actions 后，每次 PR 只扫变更文件，评论只报告本次变更引入的新问题。历史积累的几百个老问题不会出现在评论里。

### 使用指南

  #### 场景一：接管 AI 生成的组件，五分钟清掉隐藏 bug

  你让 Claude Code 写了一个用户列表组件，看起来功能正常：数据加载、搜索过滤、分页都在。但你隐约觉得哪里不对，又说不上来。

  在项目根目录跑：

  ```bash
  npx react-doctor@latest
  ```

  输出示例：

  ```
  src/components/UserList.tsx
    L12: 🔴 no-array-index-as-key — 用 index 当 key，列表过滤时组件状态会错位
    L18: 🔴 no-derived-state — useState + useEffect 同步 props，直接用 const 派生即可
    L24: 🟡 no-missing-deps — useEffect 缺少 users 依赖，可能导致数据不同步
    L31: 🟡 no-inline-object-in-jsx — 内联 style 对象每次渲染都创建新引用
  ```

  四个问题，ESLint 一个都没报。按照 react-doctor 给出的修复建议（每条规则都附 fix prompt）逐个改掉，五分钟清台。

  #### 场景二：CI 门禁，不让新问题合并到主分支

  项目已经有 47 个老问题了（你知道它们的存在，但暂时排不上优先级）。你的底线是：新 PR 不能再引入新的 React 反模式。

  ```bash
  npx react-doctor@latest ci install
  ```

  之后每次 PR 推送，GitHub Actions 自动运行扫描。如果本次变更引入了新问题，CI 评论会标注：

  ```
  ## React Doctor Report
  🔴 2 new issues introduced in this PR

  ### src/components/SearchBar.tsx
  - L8: useMemo缺少searchTerm依赖，缓存永远不更新
  - L15: input缺少关联label，屏幕阅读器用户无法识别
  ```

  历史遗留的 47 个问题不会出现在这条评论里。开发者只需要关注自己引入的问题。

  #### 场景三：让 AI agent 从源头写对，而不是事后修复

  安装 agent 技能后，Claude Code 在写 React 代码时自动加载 react-doctor 的规则知识。以前 agent 写 `useEffect(() => { fetchUsers() }, [])` 不加依赖，现在它会主动写全依赖数组或者加 eslint-disable 注释说明原因。以前 agent 用 `idx` 当 key，现在它知道用数据里的唯一 ID。

  它给 agent 装了一套"React 代码不该怎么写"的明确清单，不需要 agent 自己变聪明。确定性规则配合推理能力，返工次数肉眼可见地下降。

  #### 场景四：新项目技术选型前的基线扫描

  接手一个新项目，想知道代码质量基线如何。不用翻代码，跑一条命令：

  ```bash
  npx react-doctor@latest
  ```

  扫描结果告诉你：
  - accessibility 类问题 12 个 → 这个项目基本没考虑无障碍
  - performance 类问题 3 个 → 性能方面做得还行
  - state 类问题 28 个 → useEffect 用得比较随意，需要重点关注

  五分钟拿到一个量化的代码质量快照，决定重构优先级时有数据支撑而不是凭感觉。

### 进阶技巧

  **按类别分层管理**。不要一次性打开全部 429 条规则然后被报错淹没。建议分三轮：
  1. 第一轮只开 security + state 类别，这两类问题最可能造成线上事故
  2. 第二轮加 effect + performance，优化运行时行为
  3. 第三轮加 accessibility + architecture，打磨产品质量

  **和 ESLint 的 react 插件区分开**。ESLint plugin-react 管的是 JSX 语法规范（self-closing、prop-types 等），react-doctor 管的是 React 运行时逻辑（hooks 依赖、state 模式、重渲染路径）。两者不是替代关系，是互补关系。在 `doctor.config.ts` 中把 ESLint 已覆盖的规则设为 off，避免重复报错。

  **CI 模式下的增量扫描策略**。默认 CI 模式只扫变更文件。但如果你的变更是全局性的（比如改了自定义 hook 的签名），可以用 `--full-scan` 扫全项目，确保这次改动没有在其他文件中引入连锁问题。

### 常见踩坑

  **"install" 和 "ci install" 是两个独立命令**。`npx react-doctor@latest install` 只装 agent 技能，不配置 CI。`npx react-doctor@latest ci install` 只配置 CI workflow。两个都要装的话需要分别执行。

  **配置文件位置**。`doctor.config.ts` 必须放在项目根目录（和 `package.json` 同级）。放在 `src/` 里不会被读取。

  **Telemetry 默认开启**。react-doctor 会自动向 Sentry 上报 CLI 版本、平台、项目框架、触发的规则名称和数量、运行上下文（local/CI/agent）。不报代码内容、不报具体发现、不报文件路径。如需关闭：

  ```bash
  npx react-doctor@latest --no-telemetry
  ```

  **规则太多，别一次性全开**。429 条规则一次性全扫，对于中大型项目可能产生上百条报告，阅读体验很差。建议按照进阶技巧里的分层策略逐步开启。

### 同类技能对比

| 技能 | 安装方式 | 核心特点 | 适用场景 | 热度 |
|------|---------|---------|---------|------|
| **react-doctor**（今日推荐）| `npx react-doctor@latest install` | 确定性 AST 扫描，429 条 React 专项规则，CI 门禁 | AI 生成 React 代码质量把关，PR 审查自动化 | ⭐ 13.4K |
| ESLint plugin-react | `npm install eslint-plugin-react` | React 语法和风格检查（JSX 规范、prop-types） | 代码风格统一 | ⭐ 内置 |
| `/code-review`（Claude Code 内置）| 无需安装 | AI 多 agent 审查，覆盖面广但不保证确定性 | 通用 PR 审查 | ⭐ 官方 |
| caveman-review | `npx skills add JuliusBrussee/caveman` | 极简一行式 PR 评论，节省 token | 快速 PR 反馈 | ⭐ 85K |

  react-doctor 的独特价值在于它是**确定性的**：同一条规则对同一个文件永远给出相同的结果，不像 AI code review 可能这次报下次不报。这对 CI 门禁很重要。你不能把一个不确定的工具放在 CI 流水线里，否则开发者会被时好时坏的误报逼疯。它的规则是 React 专项的，ESLint 不会告诉你"用 index 当 key 有隐患"，AI code review 不一定每次都注意到，但 react-doctor 每条规则都是专门针对 React 运行时行为设计的。

## 今日小结

  react-doctor 适合所有用 AI 写 React 的开发者，不管你用的是 Claude Code、Cursor 还是 Codex。我自己的项目跑了一遍，一个看起来"挺干净"的 2000 行组件报了 14 个问题，其中 3 个是线上用户投诉过的——no-array-index-as-key 导致的列表过滤串位、no-derived-state 导致的表单重置。如果半年前就装上，那两个凌晨排 bug 的晚上可以省掉。

  AI 生成的 React 代码在功能和逻辑上通常是对的，但在 React 运行时行为上经常犯错。这些错误不会马上炸，而是以"偶发白屏""输入框内容串位""页面越用越卡"的形式滞后出现。react-doctor 给 agent 知道哪些模式在 React 里是错的，装一条命令，之后 agent 写 React 时会自动避开这些坑。CI 门禁确保新的反模式不会合入主分支。花五分钟装上，省的是未来排 bug 的几个小时。
