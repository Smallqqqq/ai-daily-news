# Playwright Skill — AI 帮你操控浏览器，截图测试一条指令搞定

  让 Claude Code 写代码你已习以为常，但让它帮你打开浏览器、填表单、截图、验证页面是否正常——这才是日常开发中被忽略的高频痛点。lackeyjb/playwright-skill 把 Playwright 的完整能力封装为一个 Claude Code Skill，装完以后对着终端说"测试一下登录流程"就行。2.8K stars，MIT 协议，是浏览器自动化领域最成熟的 Agent Skill。

### 它解决什么问题

  用 Claude Code 开发 Web 应用时，有一个尴尬的断层：AI 能帮你写前端代码，但写完之后你得自己打开浏览器看效果、手动点来点去验证功能是否正常。这个"写了代码 → 切到浏览器 → 手动测试 → 切回终端描述 bug"的循环，每天至少重复十几次。

  Playwright MCP 是微软官方的浏览器自动化方案，但它有一个致命问题——工具列表就吃掉约 10000 tokens 的上下文窗口。在一个已经跑了半小时的会话里，这 10000 tokens 意味着你要么精简对话历史，要么接受更高的费用。

  playwright-skill 走的是另一条路：Skill 的渐进式披露机制让它只在 metadata 层面占约 100 tokens，Claude 决定使用它时才加载完整指令。实际测试中，完成同一个"截取首页全屏截图"的任务，Skill 方案比 MCP 方案快 14%、便宜 39%、对话轮次少 43%。

### 核心原理

  playwright-skill 的设计很务实——它不是给 Claude 一本 Playwright 百科全书让它自己翻，而是给了一套"写脚本 → 执行 → 返回结果"的标准化流程：

  **自动检测开发服务器**。你本地跑了 `npm run dev`，Skill 会自动扫描 `localhost` 上的端口，找到你的应用。只有一个服务器就直接用，多个就问你选哪个，没有就让你提供 URL。

  **脚本写到 /tmp，不污染项目**。Claude 根据你的需求现场写一段 Playwright 脚本，写到 `/tmp/playwright-test-*.js`。执行完就交给操作系统的临时文件清理机制，不会在你的项目里留下一堆测试文件。

  **可见浏览器优先**。默认 `headless: false`，你能实时看到浏览器窗口里发生的一切——不像 MCP 方案每一步都要等快照，脚本一次性跑完，结果和截图一起返回。

  **通用执行器（run.js）**。这是 Skill 最巧妙的设计。Playwright 的模块加载经常因为路径问题报错，run.js 作为统一入口解决了所有模块解析问题，Claude 不需要关心 `node_modules` 在哪、`require` 路径怎么写。

  **渐进式披露**。SKILL.md 只有核心工作流指令（约 300 行），完整的 Playwright API 参考（630 行）放在 `API_REFERENCE.md` 里，只在 Claude 需要查 API 细节时才加载。这意味着日常的简单任务——截个图、填个表单——不会因为加载了完整的 Playwright 文档而浪费 token。

### 安装配置

  推荐插件方式安装（支持自动更新）：

```bash
# 添加插件市场
/plugin marketplace add lackeyjb/playwright-skill

# 安装插件
/plugin install playwright-skill@playwright-skill

# 进入技能目录安装依赖
cd ~/.claude/plugins/marketplaces/playwright-skill/skills/playwright-skill
npm run setup
```

  也可以作为独立技能手动安装：

```bash
git clone https://github.com/lackeyjb/playwright-skill.git /tmp/playwright-skill-temp
mkdir -p ~/.claude/skills
cp -r /tmp/playwright-skill-temp/skills/playwright-skill ~/.claude/skills/
cd ~/.claude/skills/playwright-skill
npm run setup
rm -rf /tmp/playwright-skill-temp
```

  `npm run setup` 会安装 Playwright 和 Chromium 浏览器。验证安装：重启 Claude Code 后输入 `/help`，确认 Playwright Browser Automation 出现在技能列表中。

### 使用指南

#### 场景一：开发中快速验证——"这个页面现在长什么样？"

  你刚改完首页的 CSS，想知道在 375px（iPhone）和 1440px（桌面）下分别是什么效果。不用切浏览器、不用打开 DevTools、不用手动调窗口大小。

  直接在 Claude Code 里说：

  ```
  用 Playwright 截一下首页，分别截 375px 和 1440px 宽度的
  ```

  Claude 会自动检测本地开发服务器，写两段脚本，分别设置 viewport 为 375×812 和 1440×900，截两张全页截图，返回给你对比。整个过程你不需要离开终端。

#### 场景二：表单交互测试——"登录流程还能走通吗？"

  你重构了认证中间件，想确认登录流程没有断。但手动测试需要：打开浏览器 → 输入 URL → 填邮箱 → 填密码 → 点登录 → 检查是否跳转到 dashboard → 检查 cookie 是否设置。

  在 Claude Code 里说：

  ```
  测试一下登录流程：打开 /login，输入 test@example.com / password123，
  提交，确认跳转到 /dashboard 并且页面包含 "Welcome"
  ```

  Claude 写的脚本会精确执行每一步，每一步都有 `waitForSelector` 确保页面加载完成，最后断言 URL 和页面内容。失败了会保留浏览器窗口和截图让你排查。

#### 场景三：截取功能截图用于文档或 PR

  你写完了新功能的代码，想在 PR 描述里附一张效果截图。以前你得手动截、裁剪、拖到 PR 里。

  在 Claude Code 里说：

  ```
  截取用户设置页面的截图，保存到 /tmp/settings-page.png
  ```

  Claude 截图后你得到完整路径。如果要加标注或对比，继续描述即可——"把这张截图和上周的版本对比一下，看看布局有什么变化"。

#### 场景四：检查响应式布局

  你说：

  ```
  分别用 iPhone 12、iPad、桌面分辨率打开首页，截三张图，
  看看有没有元素溢出或者错位
  ```

  Claude 会用三种 viewport 各截一张，然后分析是否有水平滚动条、文字截断、按钮重叠等问题。

### 进阶技巧

  **和 `/code-review` 组合使用**。写完前端代码后，先让 Playwright Skill 截图验证 UI 正确，再让 `/code-review` 检查代码质量。两个 Skill 互不冲突，分别覆盖"看起来对不对"和"写得对不对"两个维度。

  **把常用测试流程写成自然语言模板**。如果你每次发版前都要跑同一组冒烟测试（首页加载 → 登录 → 核心页面 → 表单提交），可以写一段固定描述存到项目文档里，每次发版前复制粘贴给 Claude。不需要维护脆弱的 E2E 测试脚本——每次 Claude 现场生成的脚本会自动适配你最新的页面结构。

### 常见踩坑

  **Playwright MCP 和 Skill 不要同时启用**。如果你之前装了 `playwright-mcp`，同时启用两个浏览器自动化工具会导致 Claude 不知道该用哪个，或者两个都加载浪费大量 token。建议只保留 Skill（上下文占用小得多）。

  **开发服务器检测可能漏掉非标准端口**。自动检测覆盖 3000、3001、5173、8080、8000 等常见端口。如果你用的是 4200 或其他非标准端口，直接在指令里带上 URL 即可——"打开 http://localhost:4200 测试首页"。

  **首次安装后 Chromium 下载可能较慢**。`npm run setup` 里的 `playwright install chromium` 会下载约 150MB 的浏览器二进制文件。网络不好的话可能超时，重试即可。

### 同类技能对比

| 技能 | 安装方式 | 核心特点 | 适用场景 | 热度 |
|------|---------|---------|---------|------|
| **playwright-skill**（今日推荐）| `/plugin install` | 脚本式执行、100 tokens 上下文、可见浏览器 | 日常开发快速验证、截图、冒烟测试 | ⭐ 2.8K |
| playwright-mcp | `claude mcp add` | 步骤式交互、实时快照反馈 | 需要逐步探索未知页面的场景 | ⭐ 微软官方 |
| dev-browser | 手动安装 | 有状态服务器、页面跨脚本保持 | 复杂多步骤调试 | ⭐ 新项目 |
| agent-browser（内置） | 无需安装 | Chrome DevTools MCP 封装 | 轻量网页操作 | ⭐ 内置 |

  playwright-skill 的差异化在于"脚本式执行"。不是每一步都问 Claude 下一步做什么（MCP 模式），而是把整个测试写成一段脚本一次性跑完。对于确定性的验证任务——截个图、填个表单、检查页面元素——这种方式又快又省 token。MCP 更适合"探索性"操作——你不知道页面长什么样，需要一步步看快照决定下一步。

## 今日小结

  playwright-skill 适合所有做 Web 开发的工程师。安装耗时 3 分钟（主要是下载 Chromium），之后每次想验证页面效果时，不用再切浏览器手动操作了。它不是替代你的 E2E 测试套件——那些应该在 CI 里跑——而是填补"写了代码想看效果"这个日常高频瞬间的空白。如果你已经装了 playwright-mcp 但被上下文占用困扰，换成这个 Skill 能立刻感受到 token 消耗的下降。
