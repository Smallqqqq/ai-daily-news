# Agent Skills — 像高级工程师一样写代码

  AI 编程 agent 写代码快，但写出来的代码往往"能用但不够好"——跳过 spec、省掉测试、安全审查缺失、变更粒度太大难以 review。Google 工程副总裁 Addy Osmani 把高级工程师的判断力、流程纪律和质量闸门编码成了 24 个 SKILL.md 文件，安装后你的 Claude Code（或 Codex / Cursor / Gemini CLI）就能像一位 Google 高级工程师那样工作——每一步都有规范、每一个决策都有验证、每一段代码都经得起推敲。60K+ GitHub stars，这是目前最受欢迎的 Agent Skill 包，没有之一。

---

### 它解决什么问题

  所有 AI 编程 agent 都有一个共同的毛病：**走捷径**。它们默认选择最直接的路径——直接写代码，跳过需求澄清、架构设计、测试驱动、安全审查、性能验证这些"慢但对"的步骤。

  具体来说：

  - **需求不清就开始写**：你说"加个登录"，它就开始写代码。没问你："用什么认证方式？JWT 还是 session？要不要支持 OAuth？"
  - **没有 spec 就动工**：写了一半发现方向错了，再来回改。大量 token 浪费在返工上。
  - **不写测试**：口头承诺"我会补测试的"——但它永远不会主动补。
  - **变更粒度太大**：一个 commit 改了 500 行、涉及 8 个文件，根本没法 review。
  - **安全是事后考虑**：用户输入没验证、SQL 注入没防、密钥硬编码——agent 默认不关心这些。
  - **发布前没有 checklist**：功能写完了就 push，没人检查是否要加 feature flag、是否要做灰度、是否配了监控告警。

  Agent Skills 用 **强制工作流** 替代了 agent 的自由发挥。每个 skill 不是"建议"，而是"流程"——有步骤、有检查点、有防狡辩表、有验证闸门。

### 核心原理

  这个 skill 包的哲学可以用一句话概括：**过程正确，结果才正确**。

  它不是给 agent 塞一堆"最佳实践"文档让它自己理解，而是把每个实践变成 agent 必须执行的步骤序列。每个 SKILL.md 都遵循同一套模板：

  ```
  1. Overview（这个 skill 干什么）
  2. When to Use（什么条件下触发）
  3. Process（分步骤工作流，每步有明确产出）
  4. Anti-Rationalization Table（常见借口 + 反驳论据）
  5. Red Flags（哪些信号说明出问题了）
  6. Verification（必须产出的证据——测试结果、构建输出、运行时数据）
  ```

  最犀利的设计是 **Anti-Rationalization Table**。agent 会因为"这个改动很小"、"我稍后补测试"、"这看起来是对的"而跳过步骤。这个表格预判了 agent 所有可能的偷懒借口，并给出了不可辩驳的反驳。当你看到 agent 准备跳过某步时，skill 已经在那里等着它了。

  **7 个入口命令，24 个底层 skill**：

  ```
    DEFINE          PLAN           BUILD          VERIFY         REVIEW          SHIP
   ┌──────┐      ┌──────┐      ┌──────┐      ┌──────┐      ┌──────┐      ┌──────┐
   │ Idea │ ───▶ │ Spec │ ───▶ │ Code │ ───▶ │ Test │ ───▶ │  QA  │ ───▶ │  Go  │
   │Refine│      │  PRD │      │ Impl │      │Debug │      │ Gate │      │ Live │
   └──────┘      └──────┘      └──────┘      └──────┘      └──────┘      └──────┘
    /spec          /plan          /build        /test         /review       /ship
  ```

  每个命令都会自动激活对应的 skill 组。比如 `/spec` 会串起 interview-me → idea-refine → spec-driven-development，一步一步把模糊想法变成可执行的 spec。

  **渐进式加载**：24 个 skill 的文件名和描述加起来只有约 2400 tokens（每个约 100 tokens），agent 启动时只加载这些元数据。只有当你真的在做 review 时，`code-review-and-quality` 的完整内容才会被加载到上下文。平时不占 token，用时才拉取。

### 安装配置

  **Claude Code（推荐，一键安装）：**

  ```
  /plugin marketplace add addyosmani/agent-skills
  /plugin install agent-skills@addy-agent-skills
  ```

  > 如果遇到 SSH 报错，改用完整 HTTPS URL：
  > ```
  > /plugin marketplace add https://github.com/addyosmani/agent-skills.git
  > /plugin install agent-skills@addy-agent-skills
  > ```

  **Cursor：** 复制任意 `SKILL.md` 到 `.cursor/rules/`，或引用整个 `skills/` 目录。

  **Gemini CLI：**

  ```bash
  gemini skills install https://github.com/addyosmani/agent-skills.git --path skills
  ```

  **通用安装（支持 40+ agent）：**

  ```bash
  npx skills add addyosmani/agent-skills --skill '*' --global -y
  ```

  安装完成重启 agent 即可。不需要额外配置、不需要 API key、不需要依赖。

### 使用指南

  #### 场景一：从一句话需求到可发布功能

  你对 Claude Code 说："我想加一个暗色模式。"

  **没有 Agent Skills 时**：Claude 可能直接开始改 CSS、加几个 CSS 变量、写个 toggle 按钮，然后告诉你"搞定了"。代码可能能用，但：
  - 没有考虑系统主题检测（`prefers-color-scheme`）
  - 没有考虑无障碍（高对比度模式）
  - CSS 变量命名可能不一致
  - 没有测试

  **装了 Agent Skills 后**，你键入 `/spec`：

  1. `interview-me` skill 激活——Claude 不再直接写代码，而是开始问你：
     > "暗色模式的触发方式是什么？手动 toggle 还是跟随系统？两个都要？"
     > "需要用户偏好持久化吗？存 localStorage 还是后端？"
     > "现有的组件库支持主题切换吗？"
  2. `spec-driven-development` 生成一份 spec 文档，覆盖：目标、命令、项目结构、代码风格、测试策略、边界条件。
  3. 你审阅并批准 spec。
  4. `/build auto` 自动生成任务列表并逐个实现，每个任务都有对应的测试。

  整个过程 agent 不会跳过任何一步，因为每个 skill 的流程都是锁定的。

  #### 场景二：代码审查——不只是"看起来没问题"

  你写了一个新功能，想在做 PR 之前自查一下。

  键入 `/review`：

  1. `code-review-and-quality` skill 激活，启动五轴审查：
     - **正确性**：逻辑对吗？边界条件覆盖了吗？
     - **安全性**：有注入风险吗？认证/授权正确吗？
     - **性能**：有 N+1 查询吗？有不必要的重渲染吗？
     - **可维护性**：命名清晰吗？职责单一吗？
     - **一致性**：是否符合项目现有模式？

  2. 每个发现标注严重级别：Nit / Optional / FYI / Important / Critical。
  3. 如果一次变更超过 100 行，skill 会建议拆分为多个小 commit。
  4. 检查完毕后给出 verdict：Ship it / Needs fixes / Needs rework。

  最妙的是 `code-simplification` skill——它不只提问题，还会应用 Chesterton's Fence 原则（不理解为什么存在的代码不要删）和 Rule of 500（一个文件超过 500 行就该拆），帮你把代码变简洁但不改变行为。

  #### 场景三：安全审计——别等到上线再想安全

  你的功能涉及用户输入和数据库操作。键入 `/review` 后，`security-and-hardening` skill 会在审查中自动激活：

  1. **OWASP Top 10 逐项检查**：SQL 注入、XSS、CSRF、认证绕过、敏感数据泄露……
  2. **依赖审计**：检查 `package.json` / `requirements.txt` 中是否有已知漏洞。
  3. **密钥管理**：扫描代码中是否有硬编码的 API key、密码、token。
  4. **三层安全边界**：
     - L1（自动通过）：代码行数 < 20，不涉及用户输入、auth、外部 API
     - L2（标准审查）：涉及用户输入或外部调用，需要完整安全审查
     - L3（深度审查）：涉及 auth、加密、支付——需要威胁建模

  从 L1 到 L3 逐步升级，不会对一行文案修改做全套安全审查（浪费 token），也不会放过真正危险的代码。

  #### 场景四：发布上线——有 checklist 不慌张

  功能写完了、测试过了、review 批了。现在要上线。键入 `/ship`：

  1. `shipping-and-launch` skill 启动，逐项检查：
     - Feature flag 配了吗？
     - 回滚方案是什么？
     - 监控告警设了吗？
     - 数据库 migration 是可逆的吗？
     - 文档更新了吗？
  2. 如果你在部署 CI/CD pipeline，`ci-cd-and-automation` skill 会确保你遵循 Shift Left 原则——越早发现问题，修复成本越低。
  3. 如果你在替换旧系统，`deprecation-and-migration` skill 会引导你完成代码即负债的迁移流程。

### 进阶技巧

  **组合技：`/build auto`——批准一次计划，自动执行全程**

  这是这个 pack 里效率最高的功能。你只需要：

  1. `/spec` 产出并批准 spec
  2. `/build auto` 自动生成 plan → 逐个实现 task → 每个 task 都走 TDD → 所有测试通过 → 所有 commit 原子化
  3. 最后 `/review` 做最终审查

  中间不需要你手动确认每一步。但这不是盲目的自动化——每个 task 都独立验证，任何一步失败都会暂停并报告。

  **使用 Agent Personas 做专项审查**

  这个 pack 还附带了 4 个预设的 agent 角色，你可以用它们做针对性审查：

  ```bash
  # 用安全工程师视角审查代码
  /security-auditor

  # 用性能工程师视角做 Core Web Vitals 审计
  /webperf
  ```

  `webperf` agent 有一个"度量诚实规则"——它必须从 Chrome DevTools 拿到真实的 Core Web Vitals 数据才能下结论，不能靠猜测。

### 常见踩坑

  **1. 安装了 skill 但 agent 没变聪明？**

  确保重启了 agent。Claude Code 启动时扫描 skill 目录，运行中新增的 skill 需要 `/reload-skills` 或重启才能被识别。Gemini CLI 同理。

  **2. `/build auto` 跑了一半停了？**

  `/build auto` 在遇到测试失败、lint 错误、或"高风险步骤"时会暂停。这不是 bug，是设计——它不会强制通过失败的验证。你需要告诉它下一步怎么做："继续"、"跳过这个 task"、或"换个方案"。

  **3. Agent 试图跳过某个 skill 的步骤？**

  这就是 Anti-Rationalization Table 发挥作用的时候。如果你发现 agent 说"这个改动太小，不需要测试"，直接在对话里说"检查 test-driven-development skill 的 anti-rationalization 表"。skill 里的反驳论据会让 agent 乖乖回去写测试。

  **4. SSH 克隆失败？**

  如果你没有配置 GitHub SSH key，插件市场克隆会失败。解决方案：用完整 HTTPS URL 添加市场：
  ```
  /plugin marketplace add https://github.com/addyosmani/agent-skills.git
  ```

| **Agent Skills**（今日推荐）| `/plugin install` | 24 个 skill 覆盖完整开发生命周期，强制工作流 + 防狡辩设计 | 全栈开发、团队协作、需要工程纪律的项目 | ⭐ 60K+ |
| superpowers (obra) | `/plugin marketplace add obra/superpowers` | 20+ skill，侧重 brainstorming → plan → execute 三阶段，强调 TDD | 需要强流程约束的 solo 开发 | ⭐ 2K+ |

  Agent Skills 的独特之处在于它的**覆盖面**和**防狡辩机制**。其他 skill 包可能在某一个环节做得更深，但没有一个像它这样把 Define → Plan → Build → Verify → Review → Ship 六个阶段全部覆盖，且每个阶段都有不可跳过的验证闸门。特别是 Anti-Rationalization Table——这是从 Google 工程文化中提炼出来的实战经验，知道 agent 会在哪里偷懒，提前把反驳写好。

## 今日小结

  今天推荐的 Agent Skills 适合**每一个用 AI 编程 agent 写生产代码的开发者**。它不挑语言、不挑框架、不挑 agent——Claude Code、Codex、Cursor、Gemini CLI 都能用。安装只需要一条命令，24 个 skill 的元数据只占约 2400 tokens 上下文，日常使用几乎无感。但它能给你的 agent 带来的改变是根本性的：从"能写代码"变成"能写好代码"。5 分钟装上，一辈子的工程纪律免费拿走。
