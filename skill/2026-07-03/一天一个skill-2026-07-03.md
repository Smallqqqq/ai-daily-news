# Supabase Agent Skills — 把你用的 AI 编程助手，变成懂 Postgres 的 DBA

  AI 编程助手写 SQL 的水平，大致相当于一个看过 Stack Overflow 但没上过生产的实习生——查询能跑通，索引想不起来加，RLS 策略经常忘，连接池耗尽之后只会说"再试一次"。Supabase 把自家团队十年踩过的 Postgres 坑，打包成了 30 条机器可读的铁律，一条命令装进 Claude Code、Cursor、Copilot、Codex 等 18 个 AI Agent。装上之后，AI 写 SQL 不再靠"训练数据里好像见过"，而是靠一份你随时能打开审查的硬规则。

### 它解决什么问题

  Supabase 跑着几十万个项目的 Postgres 数据库。他们的支持团队、数据库顾问、内部文档里沉淀了大量排障经验——但这些都是给人看的，AI 读不到。结果就是：

  - Agent 生成的 SQL **能跑但会全表扫描**，因为它根本没想过要建索引
  - Agent 建的表**默认没有 RLS 策略**，因为它不知道 Supabase 的安全模型
  - Agent 写 migration **不知道哪些 DDL 会锁表**，凌晨上线锁了五分钟
  - Agent 用 ORM 生成了 **N+1 查询**，因为它没有"每一条 ORM 调用背后都对应一条 SQL"的直觉

  Supabase Agent Skills 把这些经验从人类文档翻译成了 Agent Skills 格式——不是"参考文档"让 AI 自己去翻，而是**硬编码在 SKILL.md 里的铁律**，Agent 加载技能的那一刻就拿到了。

  实际的评估数据验证了效果：在 6 个 Supabase 典型场景下，Claude Code（Sonnet 4.6）装上 skill 后，任务完成准确率从 46% 提升到 71%。Codex（GPT-5.4）从 71% 提升到 88%。光靠 MCP 给 Agent 开数据库权限是不够的——MCP 只给了方向盘，skill 才是驾驶课。

### 核心原理

  Supabase Agent Skills 包含两个技能，解决不同层次的问题：

  **`supabase-postgres-best-practices`**（30 条规则，8 个分类）：这是给所有 Postgres 用户用的通用规则集，不管你是否用 Supabase。8 个分类按影响程度排列：

  | 分类 | 优先级 | 典型规则 |
  |------|--------|---------|
  | 查询性能 | CRITICAL | 缺索引检测、覆盖索引、复合索引设计、部分索引 |
  | 连接管理 | CRITICAL | 连接池配置、客户端生命周期、资源限制 |
  | 安全与 RLS | CRITICAL | 行级安全策略、访问控制模式、强制 RLS |
  | Schema 设计 | HIGH | 表结构、数据类型选择、范式化决策 |
  | 并发与锁 | MEDIUM-HIGH | 事务隔离级别、死锁预防、锁管理 |
  | 数据访问模式 | MEDIUM | 分页策略、批量操作、查询设计 |
  | 监控与诊断 | LOW-MEDIUM | 查询分析、性能追踪、调优 |
  | 高级特性 | LOW | CTE、窗口函数、扩展 |

  每条规则都有固定的格式：标题、影响级别、为什么重要、**错误代码示例**和**正确代码示例**。Agent 看到的不只是"你应该建索引"，而是一段真实的反面 SQL 和对应的修复版本。例如 RLS 规则直接展示了"只在应用层过滤"的漏洞代码和"数据库层 RLS 策略"的正确实现。

  **`supabase`**（Supabase 全产品技能）：这个技能覆盖 Supabase 的所有产品——Auth、Edge Functions、Realtime、Storage、Vectors 等。它的核心设计原则是"授人以渔"：不教 Agent 当前 API 长什么样，而是教它**怎么找到最新文档**。因为 Supabase 更新太快，写在 skill 里的 API 细节三个月就过期了。技能告诉 Agent：先用 MCP `search_docs` 查文档，查不到就用 `curl` 拉取 docs 页面的 markdown 版本（URL 后面加 `.md` 即可），再不行才用 web search。

  安全规则是个例外。Supabase 团队发现 Agent 在查文档这件事上天然懒惰——即使有 `search_docs` 工具可用，Agent 经常选择跳过，直接用训练数据里的记忆。所以 **6 条硬性安全铁律直接写进了 SKILL.md**：

  - `user_metadata` 不可用于鉴权（用户可编辑），必须用 `app_metadata`
  - `service_role` key 不能出现在前端（Next.js 中 `NEXT_PUBLIC_` 开头的变量会被打包进浏览器 JS）
  - View 默认绕过 RLS，必须加 `security_invoker = true`
  - UPDATE 需要配套的 SELECT 策略（否则静默返回 0 行，不报错）
  - Storage upsert 需要 INSERT + SELECT + UPDATE 三权限（只给 INSERT 会让文件覆盖静默失败）
  - 删除用户不会自动让 JWT 失效，必须先 revoke session

### 安装配置

  推荐用 `npx skills` 一键安装（自动检测已安装的 Agent，兼容 18+ 工具）：

```bash
# 安装所有技能
npx skills add supabase/agent-skills

# 只装 Postgres 最佳实践
npx skills add supabase/agent-skills --skill supabase-postgres-best-practices

# 只装 Supabase 全产品技能
npx skills add supabase/agent-skills --skill supabase
```

  Claude Code 用户也可以用插件市场安装：

```bash
# 添加市场
claude plugin marketplace add supabase/agent-skills

# 安装 Postgres 最佳实践插件
claude plugin install postgres-best-practices@supabase-agent-skills

# 安装 Supabase 全产品插件
claude plugin install supabase@supabase-agent-skills
```

  零依赖，不需要额外配置文件。安装后重启 Claude Code 或新开会话即可。技能会在相关任务触发时自动加载——当你写 SQL、设计 Schema、审查数据库变更时，Agent 自动参考这 30 条规则。

### 使用指南

#### 场景一：AI 帮你设计表结构，它不会再漏索引了

  你说"帮我设计一个电商订单表"，没装 skill 之前，Agent 大概率给你一个能跑的 `CREATE TABLE`——字段齐全、类型合理，但缺少外键索引、没有 RLS 策略、可能用了 `text` 类型存状态枚举。

  装了 `supabase-postgres-best-practices` 之后，同样的请求，Agent 会：

  - 给 `user_id` 外键自动加索引（规则 `query-missing-indexes`）
  - 建议用 `enum` 或查找表替代 `text` 存状态（规则 `schema-data-types`）
  - 自动加上 `created_at` 和 `updated_at` 时间戳（规则 `schema-timestamps`）
  - 如果是 Supabase 项目，默认 `enable row level security` 并提示你创建策略（规则 `security-rls-basics`）

  输出不再是"能用就行"，而是"可以直接上生产"。这不依赖 Agent 变聪明了——它只是在生成代码时对照了那 30 条规则。

#### 场景二：慢查询排查，AI 告诉你问题在索引还是查询

  你把一个慢查询丢给 Agent：

```sql
SELECT o.*, u.email FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 'pending'
AND o.created_at > now() - interval '30 days'
ORDER BY o.created_at DESC
LIMIT 20;
```

  没装 skill 时，Agent 可能建议加一个 `status` 的单列索引就完事了。装了 skill 后，`query-composite-indexes` 规则会让它分析整个查询模式：

  - 这个查询同时过滤 `status` 和 `created_at`，排序也用 `created_at`——应该建复合索引 `(status, created_at DESC)`
  - `SELECT o.*` 会导致回表，考虑建覆盖索引把常用列 `INCLUDE` 进去
  - 检查 `orders` 表上是否有未使用的冗余索引（`idx_orders_status` 如果存在且为单列索引，现在应该被复合索引替代）

  输出从"给你一条 SQL 建议"变成了"给你一套索引策略"。

#### 场景三：代码审查时，AI 帮你抓 RLS 漏配

  你提交了一个 PR，在 Supabase 项目里加了一张新表 `projects` 和对应的 API 路由。没装 skill 时，AI code review 看的是 TypeScript 代码——参数校验对不对、错误处理全不全。RLS 策略完全在盲区。

  装了 skill 后，AI 审查 PR 时自动加载 `security-rls-basics` 规则，检查点多了：

  - `projects` 表有没有 `enable row level security`？
  - 有没有配套的 SELECT/INSERT/UPDATE/DELETE 策略？
  - 如果用了 view，view 定义里有没有 `security_invoker = true`？
  - API 路由里有没有把 `service_role` key 暴露给客户端？

  这 4 条检查，人类 reviewer 也经常忘掉第二和第三条。AI 不会忘——因为它是 checklist，不是判断力。

#### 场景四：新人上手 Supabase，AI 带着走

  团队新成员第一次用 Supabase，只听说过 Postgres 但没摸过 Auth、RLS、Storage 这些模块。装了 `supabase` 全产品技能后，Agent 变成随身导师：

  - 新人说"帮我加用户登录"，Agent 会先去 docs 拉取最新的 `@supabase/ssr` 文档，确认 API 没变，再生成代码
  - 新人说"上传头像怎么不行"，Agent 知道 Storage upsert 需要三权限（INSERT + SELECT + UPDATE），直接定位到 RLS 策略只配了 INSERT
  - 新人不知道 `supabase db query` 这个命令需要 CLI ≥ 2.79.0，Agent 会先检查版本再提示升级

  效果等同于每个新人入职时配了一个熟悉 Supabase 所有坑的老同事。实际上 Supabase 团队内部就是这么用的。

### 进阶技巧

  **和 Supabase MCP Server 配合使用，效果翻倍**。MCP Server 给 Agent 开了一扇门——它可以直连数据库，建表、跑查询、管理 Schema。但只有门是不够的。把 MCP Server 当成方向盘，Agent Skills 当成驾驶课——装了两个之后，Agent 不仅能把车开到目的地，还能做到不闯红灯、不压线、不超速。实测数据也支持这一点：纯 MCP（无 skill）在某些场景下准确率反而下降（Claude Opus 从 58% 降到 50%），因为 Agent 有工具但不知道怎么用。加上 skill 后跳到 67%。

  **贡献你自己的 Postgres 踩坑经验**。30 条规则是 Supabase 内部经验的起点，不是终点。如果你碰到一个 Postgres 坑觉得 AI 也应该知道，可以直接给 `supabase/agent-skills` 仓库提 PR。每条规则需要：标题、优先级（critical/high/medium/low）、为什么重要、错误和正确的代码示例。`CONTRIBUTING.md` 里有完整模板。

### 常见踩坑

  **Agent 可能不主动加载 reference 文件**。`supabase-postgres-best-practices` 的 30 条规则存在 `references/` 目录下的独立文件中。早期版本中 Agent 经常只加载 SKILL.md 本体的概述，不想麻烦去翻 reference。Supabase 团队后来在 SKILL.md 里加了一个"Reference Loading Guide"表格，明确告诉 Agent 什么场景该读哪个文件。但如果你发现 Agent 给出的建议不像参考了规则，可以直接说"先读一下 supabase-postgres-best-practices 的 references"。

  **不要把 MCP 的生产环境连接和 skill 的工作流搞混**。skill 里推荐的开发工作流是：在本地/ staging 数据库上用 MCP 的 `execute_sql` 快速迭代 → 跑 Supabase database advisor 检查 → 确认无误后 `supabase db diff` 生成 migration → commit。这个流程**绝对不能对着生产数据库跑**。在 production 上，老老实实写 migration、review、apply。

  **如果你是纯 Postgres 用户（不用 Supabase）**，`supabase-postgres-best-practices` 对你也完全适用。30 条规则中绝大多数是通用 Postgres 最佳实践——查询优化、索引设计、连接池——跟 Supabase 特有的 Auth/RLS 无关。单独安装这个技能即可。

  Supabase Agent Skills 的差异化在于三点：(1) **来自生产环境**——不是"理论上的最佳实践"，是 Supabase 几十万项目踩出来的坑；(2) **有量化评估**——6 个场景 × 4 个模型/条件的 benchmark 数据，不是"应该能提升"；(3) **Agent Skills 格式的原生示范**——Supabase 通过大量实验（agent 偷懒不读 reference、安全检查容易被跳过等）打磨出的 skill 结构设计，对任何想写 skill 的人都是教科书级参考。

## 今日小结

  Supabase Agent Skills 适合所有让 AI 写 SQL 或操作 Postgres 数据库的开发者——不管你是 Supabase 用户还是裸 Postgres 用户。安装一条命令，之后 Agent 生成的每条 SQL、每个 CREATE TABLE、每个索引建议都会对照那 30 条规则。它不是让 AI 变聪明了，而是给了 AI 一份你不会逐条口述的 Postgres 生产经验。花 2 分钟安装，换来的是：AI 写的 SQL 从"能跑就行"变成了"可以直接 code review 通过"。如果你们团队已经在用 AI 编程助手操作数据库，这个 skill 可能是你今天 ROI 最高的两分钟投入。
