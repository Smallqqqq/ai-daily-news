# Debug Mode — 让 AI Agent 用科学方法调试，而不是瞎猜乱改

  让 Claude Code 修 bug 最常见的翻车场景是什么？你说"登录接口返回 500"，它二话不说开始改代码。改路由、改中间件、改数据库查询。三趟下来 bug 没修好，反而多了一堆你不想要的"优化"。Debug Mode 这个技能把 Cursor 的假设驱动调试方法论搬到了 Claude Code：先提假设、再插日志、复现收集证据、确认根因、精准修复、最后清理现场。100 stars，MIT 协议，最近更新在三天前。

### 它解决什么问题

  AI 编程助手在处理 bug 时天然倾向于"先改再说"。不是因为懒，而是因为它的优化目标就是尽快给你一个看起来能用的结果。你描述一个现象，它基于静态代码分析猜一个原因，改一行代码，然后说"修好了"。等你验证发现没修好，它又猜另一个原因，再改另一行代码。

  这个循环每多一轮，代码库就多一层"尝试性修改"的痕迹。改过的函数签名可能已经不对了，中间件的调用顺序可能已经乱了。但你注意不到，因为每次 Agent 只告诉你"这次应该行了"。

  Debug Mode 把这个过程反转了。它不玩"猜一个修复然后改代码"，而是走另一条路：猜一组根因，然后收集证据。在证据确凿之前，一行代码都不改。这个方法论来自 Cursor 的 Debug Mode 功能，Cursor 把它做成了 IDE 内置功能，而这个技能把它移植到了 Claude Code、Codex 和 Gemini CLI 中。

### 核心原理

  Debug Mode 是一个约 200 行的 SKILL.md 文件，定义了 8 个阶段的调试流水线：

  ```
  理解 Bug → 生成假设 → 插入日志探针 → 用户复现 → 分析日志 → 精准修复 → 用户验证 → 清理探针
  ```

  每一阶段都有硬性规则，Agent 不能跳步：

  **阶段 1-2：理解 + 假设**。Agent 先收集上下文——预期行为、实际行为、复现步骤、报错信息——然后阅读相关源码，理解调用链和数据流。之后生成 3-5 个可证伪的假设（不是"可能是什么问题"，而是"如果 H1 成立，日志应该显示 X；如果 H1 不成立，日志应该显示 Y"）。

  **阶段 3：插探针**。这是最关键的阶段。Agent 在代码中插入带标记的调试日志，所有日志写入 `.claude/debug.log`（绝对路径），不用 `console.log` 或 `print`。这样既避免污染终端输出，也避免日志撑爆上下文窗口。每条日志带假设编号：`[DEBUG H1]`、`[DEBUG H2]`。所有探针代码用 `#region DEBUG` 块包裹（兼容 JS/TS/Python/Go/Rust/Java/C# 等 12 种语言），确保修复后能一键清除。

  **阶段 4-5：分析 + 修复**。你复现 bug 后，Agent 读取日志文件，用 `grep "[DEBUG H"` 提取关键行（避免全量日志撑爆上下文），逐一对照假设：H1 被排除（日志显示变量值正常）、H2 被确认（日志显示这个分支根本没走到）。确认根因后，Agent 写最小修复。只改导致 bug 的那几行，不顺便重构。

  **阶段 6-8：验证 + 清理**。修复后探针保持不动，清空日志，你再次复现确认修好。如果没修好，带着新日志回到阶段 2 重新假设。确认修好后，Agent 用 Grep 搜索所有 `#region DEBUG` 块，一键删除。工作区恢复干净，不留任何调试痕迹。

  这套流水线的核心哲学是：在证据确凿之前不写修复，在用户确认之前不清理探针。Agent 不能自己说"修好了"就完事，必须你确认。

### 安装配置

  Claude Code 安装（注意嵌套目录问题）：

```bash
# 克隆仓库
mkdir -p ~/.claude/skills && git clone https://github.com/doraemonkeys/claude-code-debug-mode.git ~/.claude/skills/claude-code-debug-mode

# Claude Code 只扫描 ~/.claude/skills/ 的直接子目录，不递归
# 必须把内层 debug-mode/ 提上来，否则技能不会加载
mv ~/.claude/skills/claude-code-debug-mode/debug-mode ~/.claude/skills/debug-mode
rm -rf ~/.claude/skills/claude-code-debug-mode
```

  Codex CLI 安装（支持递归扫描，无需额外步骤）：

```bash
mkdir -p ~/.codex/skills && git clone https://github.com/doraemonkeys/claude-code-debug-mode.git ~/.codex/skills/claude-code-debug-mode
```

  Gemini CLI 安装：

```bash
gemini skills install https://github.com/doraemonkeys/claude-code-debug-mode.git
```

  验证安装：

```bash
ls ~/.claude/skills/debug-mode/SKILL.md
```

  文件存在即安装成功。重启 Claude Code 后，描述一个 bug 即可自动激活，也可以用 `/debug-mode` 显式调用。

### 使用指南

#### 场景一：API 返回 200 但数据为空——用假设排除法定位根因

  你有一个用户列表接口，传了正确的分页参数，API 返回 `200 OK`，但 `data` 字段是空数组。数据库里明明有数据，SQL 日志也显示查询执行了。

  直接对 Claude Code 说：

  ```
  /debug-mode GET /api/users?page=1&limit=20 返回 200 但 data 是空数组，数据库里有 50 条记录
  ```

  Agent 读完代码后生成假设：

  ```
  1. [H1] 分页 offset 计算错误——SQL 查询跳过了所有记录
  2. [H2] 中间件修改了 query 参数——传到 controller 的 page/limit 不对
  3. [H3] 序列化层过滤掉了结果——ORM 返回了数据但 JSON 序列化时丢了
  ```

  然后在关键位置插入探针：

  ```javascript
  // #region DEBUG
  const fs = require('fs');
  const log = (msg) => fs.appendFileSync('/project/.claude/debug.log', msg + '\n');
  log(`[DEBUG H1] page=${page}, limit=${limit}, offset=${(page-1)*limit}, SQL: ${query}`);
  log(`[DEBUG H2] req.query=${JSON.stringify(req.query)}, after middleware=${JSON.stringify(locals.pagination)}`);
  log(`[DEBUG H3] ormResult.length=${results.length}, serialized.length=${JSON.stringify(serialized).length}`);
  // #endregion DEBUG
  ```

  你复现 bug 后，Agent 分析日志：

  ```
  [DEBUG H1] page=1, limit=20, offset=0, SQL: SELECT * FROM users LIMIT 20 OFFSET 0
  [DEBUG H2] req.query={"page":"1","limit":"20"}, after middleware={"page":"1","limit":"20"}
  [DEBUG H3] ormResult.length=20, serialized.length=4
  ```

  一目了然：H3 被确认。ORM 查到了 20 条，但序列化后只剩 4 个字符（`[]` 的 JSON 长度）。问题在序列化层的字段映射，不在 SQL 也不在中间件。

#### 场景二：偶发 bug 等不到复现窗口——预埋探针、随时收集

  用户反馈"偶尔提交订单后页面空白"，但你复现了十次都没触发。传统的"Agent 等你复现"模式在这个场景下完全失效。你不能让 Agent 干等着。

  Debug Mode 支持"预埋探针"模式：

  ```
  /debug-mode 订单提交后偶发白屏，我复现不出来，先帮我埋好探针，触发后我把日志给你
  ```

  Agent 会在关键路径上插入探针。订单创建、支付回调、页面渲染，每个关键节点一个探针。插完后告诉你探针已就位。你继续正常使用系统，什么时候 bug 触发了，`.claude/debug.log` 里就有那次触发时所有探针的完整记录。把日志丢给 Agent，它从"准备就绪"状态直接跳到"分析诊断"阶段，不需要重现整个调试流程。

#### 场景三：重构后行为不一致——用探针做前后对比

  你重构了认证中间件，把 session 校验改成了 JWT 校验。单元测试全绿，但总觉得哪里不对——登录后的重定向行为好像变了。

  ```
  /debug-mode 重构认证中间件后，登录成功后的重定向 URL 可能不对，帮我验证
  ```

  Agent 在旧中间件和新中间件的出口各插一个探针，记录 `res.redirect` 的目标 URL。你分别用旧版本和新版本各登录一次，对比两份日志：

  ```
  [DEBUG H1] OLD: redirect=/dashboard
  [DEBUG H1] NEW: redirect=/
  ```

  新中间件在某个分支下没有正确设置 redirect URL，fallback 到了根路径。如果不用探针，这个差异几乎不可能通过手动测试发现。浏览器地址栏一闪而过，你只会觉得"好像哪里不对"。

### 进阶技巧

  **结合 Git 做"调试快照"**。在 Agent 插入探针之前，先手动 `git stash` 保存当前工作。探针插入后，你的工作区是脏的。修复完成、探针清理后，`git diff` 显示的只有修复本身——干净的 diff，干净的 commit。如果修复过程中你想放弃回到原点，`git stash pop` 即可。

  **日志不用非得写本地文件**。SKILL.md 默认把日志写到 `.claude/debug.log`（绝对路径），但这个默认值只为最常见的场景设计。如果你的代码跑在手机上、远程设备上、或者 Docker 容器里，让 Agent 把日志写到那个环境能访问的位置，复现后把日志内容复制回来即可。Skill 的设计文档明确写了："传输方式变了，但工作流没变——只要 Agent 拿到复现日志，结果是一样的。"

### 常见踩坑

  **Claude Code 安装后技能不生效**。这是最高频的问题，根因是 Claude Code 只扫描 `~/.claude/skills/` 的**直接子目录**，不递归。如果你 `git clone` 后没有把内层 `debug-mode/` 目录提上来，`SKILL.md` 实际在 `~/.claude/skills/claude-code-debug-mode/debug-mode/SKILL.md`，Claude Code 看不到它。症状是静默的——不报错，但技能列表里没有它，`/debug-mode` 命令无效。执行 `mv ~/.claude/skills/claude-code-debug-mode/debug-mode ~/.claude/skills/debug-mode` 后重启即可。Codex 和 Gemini CLI 没有这个问题，它们递归扫描。

  **探针没有清理干净**。修复完成后，Agent 用 Grep 搜索 `#region DEBUG` 来删除探针。如果你的代码中本身就用了 `#region` 注释（比如 C# 项目），Grep 可能会匹配到非探针代码。解决方案：修复完成后手动 `git diff` 确认所有改动都符合预期，再 commit。

  **Agent 在应该等待时继续执行**。Skill 要求 Agent 在插入探针后"停止并等待用户复现"，但在某些模型上 Agent 可能继续往下执行。如果遇到这种情况，显式说"等我复现完再继续"即可打断。

### 同类技能对比

| 技能 | 安装方式 | 核心特点 | 适用场景 | 热度 |
|------|---------|---------|---------|------|
| **Debug Mode**（今日推荐）| `git clone` + 提目录 | 假设驱动、探针标记、自动清理、人机协同 | 难复现/难定位的逻辑 bug | ⭐ 100 |
| Cursor Debug Mode | IDE 内置 | 同方法论的原生实现，更深度 IDE 集成 | Cursor 用户 | ⭐ 内置 |
| ferrucc-io/clog | `clog init` | Rust CLI + 本地日志服务器 + Skill | 需要实时日志收集的结构化调试 | ⭐ 3 |
| abekdwight/code-debug-skills | `/plugin install` | Log 服务器 + 标准化 Schema + Claude Code 插件 | 需要跨会话标准化日志格式的团队 | ⭐ 新 |

  Debug Mode 的差异化在于**极简 + 通用**。它不依赖外部服务器（不像 clog 需要起一个 Rust 进程）、不需要 npm 包（不像 debugsk 需要 `npx`）、不绑定任何语言或框架。一个 200 行的 SKILL.md 文件加上你项目里已有的文件系统，就能跑完整的假设驱动调试流程。如果你的团队用多种语言（JS + Python + Go），其他调试技能往往只支持一种语言，而 Debug Mode 的 `#region DEBUG` 标记兼容 12 种语言的注释语法。

## 今日小结

  Debug Mode 适合每天和 bug 打交道的开发者——尤其是那些"看起来不应该出问题但就是出问题"的场景。安装一分钟（注意那个嵌套目录的坑），之后每次遇到难啃的 bug，`/debug-mode` 启动的就是一套完整的科学调试流程：假设 → 探针 → 复现 → 证据 → 修复 → 清理。它不会让你的 bug 自动消失，但它会让你的 AI Agent 从一个"乱猜乱改的实习生"变成一个"先收集证据再下刀的外科医生"。
