# debug-skill — 给你的 AI 编程助手装一个真正的调试器

  Claude Code 改 bug 的时候只会加 print 然后猜。这不是你的问题，是它手里没有调试器。debug-skill 把微软的 Debug Adapter Protocol 包装成一套 CLI 命令，让 AI agent 能像人类开发者一样设断点、单步执行、检查调用栈、在运行时求值表达式。287 stars，MIT 协议，19 个 release，支持 Python / Go / Node.js / Rust / C / C++。

### 它解决什么问题

  用 AI 编程助手改 bug 的人都会遇到同一个瓶颈：agent 没有真正的调试器。

  你给它一个崩溃堆栈，它读一遍源码，然后开始猜："可能是这里返回了 null"、"试试加个空值检查"。猜错了就再猜一次，每次猜错都浪费一轮 token 和几分钟等待。agent 够聪明，少的是工具箱里的一件东西：调试器。

  debug-skill 给了 agent 两样东西：

  1. **一套调试方法论**（SKILL.md），教 Claude 什么时候该设断点、怎么单步跟踪、如何从调用栈倒推根因
  2. **一个真实的调试器接口**（`dap` CLI），底层对接 debugpy、Delve、js-debug、lldb-dap，agent 在 Bash 里敲命令就能操控真正的调试器

  两者组合起来的效果是：agent 先重现 bug，设断点，单步执行，检查变量，定位根因，修一行代码。和人类开发者的调试流程完全一致。

### 核心原理

  debug-skill 的架构分两层：

  **上层：SKILL.md 技能文件**。安装后在 Claude Code 中以 `/debugging-code` 命令的形式存在。技能描述约 200 词，覆盖了完整的调试决策树——什么时候用断点而不是 print、怎么缩小搜索范围、如何从调用栈定位根因。Claude 遇到 bug 相关描述时自动加载。

  **下层：`dap` CLI**。一个用 Go 写的无状态命令行工具，通过 Unix socket 与后台守护进程通信，守护进程再通过 DAP 协议对接真正的调试器后端。架构链路：

  ```
  dap <命令> → Unix socket → 守护进程 → DAP 协议 → debugpy/dlv/js-debug/lldb-dap → 你的程序
  ```

  守护进程在首次 `dap debug` 时自动启动，在 `dap stop` 或空闲 10 分钟后自动关闭。所有执行类命令（debug、continue、step）返回完整上下文：当前位置、周围源码、局部变量、调用栈、程序输出，不需要后续查询。

### 安装配置

  **第一步：给 Claude Code 安装技能**

  ```
  /plugin marketplace add AlmogBaku/debug-skill
  /plugin install debugging-code@debug-skill-marketplace
  ```

  **第二步：安装 `dap` CLI**

  ```bash
  # macOS / Linux 一键安装
  bash <(curl -fsSL https://raw.githubusercontent.com/AlmogBaku/debug-skill/master/skills/debugging-code/scripts/install-dap.sh)

  # macOS 也可以用 Homebrew
  brew install AlmogBaku/tap/dap

  # 从源码安装（需要 Go）
  go install github.com/AlmogBaku/debug-skill/cmd/dap@latest
  ```

  **第三步：安装对应语言的调试器后端**

  ```bash
  pip install debugpy        # Python
  go install github.com/go-delve/delve/cmd/dlv@latest  # Go
  npm install -g js-debug    # Node.js / TypeScript
  # Rust/C/C++ 需要系统自带 lldb，macOS 已内置，Linux 需 apt install lldb
  ```

  给其他 agent（Cursor、Codex、Copilot）安装：

  ```bash
  npx skills add AlmogBaku/debug-skill
  ```

### 使用指南

  #### 场景一：Python 程序崩溃，定位空指针

  你的 Flask 应用在处理某个请求时抛了 `AttributeError: 'NoneType' object has no attribute 'id'`，但你不确定是哪个中间件把 user 设成了 None。

  在 Claude Code 中描述 bug，Claude 会自动启用 debug-skill，执行：

  ```bash
  dap debug app.py --break app.py:156
  dap continue
  # 请求到达断点，程序暂停
  dap eval "user"
  # → null
  dap eval "request.headers.get('Authorization')"
  # → "Bearer invalid_token"
  dap step
  # 进入 authenticate() 函数，发现异常路径返回了 None 而不是抛异常
  ```

  Claude 看到 `user=null` 且 token 无效后，直接定位到认证中间件的异常返回路径，修一行代码解决。

  #### 场景二：Go 服务的并发竞态

  你的 Go 服务在压测时偶发 panic，日志只留下一个堆栈，无法稳定复现。

  ```bash
  dap debug main.go --break main.go:42 -- --config prod.yaml
  dap continue
  # 程序在断点处暂停
  dap inspect sharedMap --depth 2
  # 显示 map 内部状态，发现两个 goroutine 同时写入
  dap threads
  # 列出所有 goroutine
  dap thread 7
  # 切换到 goroutine 7，检查它的调用栈
  ```

  Claude 通过切线程、检查共享状态，确认了竞态位置，加上 `sync.Mutex` 解决。

  #### 场景三：远程容器内调试

  你的服务在 Docker 容器里跑，本地改代码不方便，想在容器内设断点调试。

  ```bash
  # 先在容器内启动 debugpy 并暴露端口
  # docker exec -it myapp python -m debugpy --listen 0.0.0.0:5678 --wait-for-client app.py

  # 本地用 dap attach 上去
  dap debug --attach container:5678 --backend debugpy --break handler.py:20
  dap continue
  dap eval "request.body"
  # 直接在容器进程内求值，不需要加日志重新构建镜像
  ```

  #### 场景四：调试前端 TypeScript 逻辑

  ```bash
  dap debug server.ts --break server.ts:88
  dap continue
  dap eval "JSON.stringify(responseBody)"
  # 直接检查运行时数据结构
  dap context --frame 2
  # 上移两层调用栈，看是谁传入了这个参数
  ```

### 进阶技巧

  **多会话并行调试**：不同的 agent 或终端窗口可以同时调试不同的程序，互不干扰。

  ```bash
  dap debug app.py --session agent1 --break app.py:10
  dap debug main.go --session agent2 --break main.go:8
  dap stop --session agent1  # 只停 agent1 的会话
  ```

  **条件断点（通过 continue --to 模拟）**：虽然没有原生的条件断点，但可以先设一个较早的断点，然后用 `dap continue --to file:line` 跑到精确位置，中间用 `dap eval` 检查条件。

  **与 git bisect 组合**：当不确定 bug 是哪个 commit 引入的，让 Claude 配合 `git bisect` 在每个可疑 commit 上跑 `dap debug` + 断点验证，全自动定位引入 bug 的 commit。

### 常见踩坑

  **后端未安装**。`dap debug app.py` 报 `debugpy not found` 时，确认 `pip install debugpy` 且 debugpy 在 PATH 中。Go 需要 `dlv`，Node 需要 `js-debug`。`dap` 不会自动安装后端，只负责对接。

  **断点不触发**。Python 的 debugpy 在某些异步框架（如 gevent、eventlet）下断点可能不触发。解决方案：在启动参数中加 `--backend debugpy` 并确认程序入口文件的路径是绝对路径。

  **远程 attach 超时**。`dap debug --attach host:port` 超时通常是容器网络不通。确认容器端口已映射（`-p 5678:5678`），且 debugpy 启动时用了 `--listen 0.0.0.0:5678`（不是 127.0.0.1）。
