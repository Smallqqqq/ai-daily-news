# skill-propose — 打破第一个想法的陷阱，一次给你三个方案

  做技术决策时，我们往往被"第一个想到的方案"绑架——不是因为它最好，只是因为大脑的锚定效应让你很难跳出那个框架。skill-propose 用三条不同的"设计人格"并行探索方案空间，然后用排名告诉你：哪个方案最适合当下的你。

---

### 它解决什么问题

  和 AI 结对编程时，你问"这个功能怎么做好"，AI 通常给你一个方案——而且往往是它最先"想到"的、最常规的方案。这和人脑的锚定效应一样危险：你看到第一个方案后，后续思考就不自觉地围着它打转。

  skill-propose 的做法是：同时启动三个 AI Agent，每个被注入完全不同的设计哲学——实用主义者追求最小改动、架构师着眼长期可维护性、野路子故意挑战问题本身的前提假设。三个方案并行生成，彼此互不知晓，保证了方案的独立性和多样性。最后，Claude 把三个方案排好名，标注优劣，给出推荐。

  你不是从零开始想方案，你是从三个有完整论证的方案中做选择。这比"再帮我想想还有没有别的办法"高效得多。

### 核心原理

  这个 Skill 的本质是一个"并行多角色提案系统"。它利用了 Claude Code 的原生子 Agent（`Agent` 工具）能力，在一次调用中同时启动三个独立子 Agent，每个被赋予不同的角色 prompt：

  - **实用主义者（Pragmatist）**：最小改动、最低风险、复用现有模式。核心问题："什么是最小的变化就能达到目标？"
  - **架构师（Architect）**：健壮、可扩展、考虑边界情况。核心问题："正确的做法是什么？"
  - **野路子（Wildcard）**：挑战前提、突破常规、横向思考。核心问题："如果我们完全换个思路呢？"

  每个 Agent 被要求先阅读实际代码库，确保方案接地气；然后按四段式输出（方案概述、实现路径、取舍分析、工作量估算）；最后 Claude 做综合排名和推荐。整个过程 Agent 只做研究，不修改任何文件。

  SKILL.md 只有约 80 行 Markdown，没有依赖、没有脚本、没有 MCP 服务器——纯粹是用 prompt 工程构建的多角色决策系统。

### 安装配置

```bash
git clone https://github.com/brianharms/skill-propose.git
mkdir -p ~/.claude/skills/propose
cp skill-propose/SKILL.md ~/.claude/skills/propose/SKILL.md
```

  重启 Claude Code（或新开会话），输入 `/propose` 即可触发。

  就这么简单。不需要 Node.js、不需要 pip、不需要任何外部依赖。SKILL.md 就是整个产品——YAML 头部的 `name` 和 `description` 让 Claude 自动识别和触发，文件体就是 Claude 执行的指令集。

### 使用指南

#### 场景一：功能设计——"怎么加离线同步"

  你正在开发一个笔记应用，需要加离线同步能力。你不确定哪种方案最合适——是本地优先+后台同步，还是乐观锁+冲突合并，还是干脆用 CRDT？

  直接在对话中输入：

```
/propose 怎么给笔记应用加离线同步功能？
```

  如果之前的对话已经讨论过这个问题，甚至不需要重新描述——Skill 会从上下文自动提取当前讨论的设计问题。

  三个 Agent 并行启动：
  - 实用主义者可能提议：用 LocalStorage + 定时 sync，最小改动
  - 架构师可能提议：IndexedDB + Service Worker + 冲突合并策略
  - 野路子可能提议：直接用 PouchDB/CouchDB 生态，放弃自建

  每个方案附带：具体怎么做、改了哪些文件、取舍分析、工作量（S/M/L）。Claude 排名后告诉你哪个最适合你的团队规模和技术栈。

#### 场景二：架构决策——"选什么状态管理方案"

  项目中要引入状态管理，团队在 Zustand、Jotai、Redux Toolkit 之间犹豫。你不知道选哪个最合适。

```
/propose 我们的 React 项目该用什么状态管理方案？
```

  三个 Agent 会先读你的 package.json、现有组件结构、数据流复杂度，然后各自提案：
  - 实用主义者：如果项目不大，可能推荐 Zustand——API 最简、bundle 最小
  - 架构师：如果检测到复杂的数据依赖，可能推荐 Redux Toolkit——生态完整、中间件丰富
  - 野路子：可能提议用 React Query + Context 替代全局状态，或者用 XState 做状态机

  排名输出后，你可能会发现：原来 90% 的状态管理需求其实可以用 URL search params + React Query 解决，根本不需要引入新库。

#### 场景三：重构策略——"怎么拆分这个 2000 行的大组件"

  一个组件已经膨胀到 2000 行，你知道需要拆分，但不知道从哪里下手。

```
/propose 怎么拆分这个巨型订单详情组件？
```

  三个 Agent 各自读取实际代码后：
  - 实用主义者：找到 3-4 个最明显的可抽取子组件，按优先级排
  - 架构师：设计完整的分层结构——容器/展示分离、业务逻辑 Hook 化、子组件树
  - 野路子：可能提议用 Compound Component 模式 + Render Props，甚至考虑用 Slot 模式彻底改变组件消费方式

  Claude 的排名会考虑你的代码库现状——如果项目已经有很多类似的拆分模式，实用主义者的方案会因为"复用现有模式"而获得更高排名。

### 进阶技巧

  **组合技：propose → plan → build**。先用 `/propose` 拿到排名方案，选定方向后，让 Claude 直接按选定方案生成实施计划，再用子 Agent 执行。整个流程从"想不清楚"到"写完代码"一条龙。

  **自定义角色**：SKILL.md 约 80 行，读完就能改。如果你觉得"安全审计师"比"野路子"更适合你的场景，直接在 SKILL.md 里替换 Agent C 的角色定义即可。三种角色的设计哲学是开放的——Pragmatist / Architect / Wildcard 只是作者的默认推荐，你可以改成 Security-Auditor、Performance-Nerd、UX-Purist 等任何对你有价值的视角。

### 常见踩坑

  **空仓库无法测试**：Agent 被要求"阅读实际代码"来接地气地提案。如果你在空仓库或刚初始化的项目中测试，Agent 只能给出泛泛的通用建议，无法体现这个 Skill 真正的价值。建议在至少有几百行代码的真实项目中首次使用。

  **非设计类问题不要用**：这个 Skill 是为"怎么做"这种设计决策设计的，不是为"这段代码有什么 bug"这种诊断类任务设计的。问诊断类问题会让三个 Agent 输出相似的 bug 分析结果，失去"多视角"的意义。

|------|---------|---------|---------|------|
| **skill-propose**（今日推荐）| 复制 SKILL.md | 3 角色并行提案 + 排名 | 设计决策、技术选型、架构讨论 | ⭐ 新发布 |

  skill-propose 的独特之处在于：它不是让 AI 帮你做事，而是让 AI 帮你**想清楚再做事**。三个不同人格的并行探索，本质上是在最短时间内模拟了"和三个不同风格的同事各聊了半小时"的效果。

## 今日小结

  今天的 skill-propose 适合每一个在技术决策面前犹豫过的开发者。安装成本几乎是零（复制一个 Markdown 文件），换来的是每次设计决策时三个独立视角的并行探索。它不是帮你写代码的工具，它是帮你做决策的工具——从这个意义上说，它的价值不在"执行层"而在"判断层"。配置花费 1 分钟，每次使用时节省的"纠结时间"可能是 30 分钟到一个下午。如果你经常和 AI 结对编程并且经常面临"怎么做更好"的决策，今天就装上它。
