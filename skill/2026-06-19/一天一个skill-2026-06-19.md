# Waza（技）— 把你已有的工程习惯，变成 AI 代理可执行的技能

  写了十年代码的老工程师，遇到新项目时脑子里会自动跑一套流程——先想清楚再动手、写完自查一遍、出 bug 先定位根因再修。但 AI 编程代理没有这种肌肉记忆，它只会对着你的指令埋头就干。Waza 把这八个"本能级"的工程习惯，封装成了 Claude Code、Codex、Cursor 等代理可以直接调用的技能。

  tw93 用 300+ 次真实会话、7 个项目、500 小时的实战经验，提炼出这八个技能。每个技能只做一件事，有清晰的触发时机，用完就退出，不污染上下文。

---

### 它解决什么问题

  AI 编程代理最让人头疼的不是"不够聪明"，而是**缺少工程纪律**。你让它加个功能，它二话不说就开始写代码——不先确认需求边界、不考虑边界情况、写完不自查、出 bug 就瞎改。这些问题不是因为模型能力不够，而是因为没人告诉它"先做什么、后做什么"。

  Waza 的本质是一套**工程流程的外骨骼**。它不改变模型的底层能力，而是在模型外面套了一层工作流约束——/think 强制你先想清楚、/check 强制你写完后自查、/hunt 强制你先定位根因再动手修。就像老工程师带新人：不是教你怎么写代码，而是教你在什么时候该做什么事。

### 核心原理

  Waza 的每个技能就是一个 `SKILL.md` 文件，遵循 Agent Skills 开放标准。当你在 Claude Code 中输入 `/think`，Claude 会加载对应的 SKILL.md，按照其中定义的工作流执行——比如 /think 会先挑战问题本身（"这个功能真的需要吗？"），再压力测试设计方案，最后产出一份其他 agent 可以直接实施的完备计划。

  与 Superpowers 等重型技能包不同，Waza 刻意保持极简——八个技能，每个只做一件事。tw93 认为："工具太重的技能包，配置多、学习曲线陡，最终工程师不会用。而每一条作者写的规则，也会成为模型的天花板。"所以 Waza 的设计哲学是：设定清晰的目标和关键约束，然后退后一步，让模型发挥它最擅长的事。

  技能之间可以链式组合，但不会自动触发彼此：
  - 设计功能：`/think` → 人工批准 → 实现 → `/check` → 合并
  - 修复发布：`/hunt` → 修复 → `/check` → 发布后续
  - 研究写作：`/read`（获取素材）→ `/learn`（综合理解）→ `/write`（润色输出）
  - 调试验证：`/hunt`（找根因）→ 修复 → `/check`（审查变更）

### 安装配置

  **一键安装全部八个技能（推荐）：**

```bash
npx skills add tw93/Waza -a claude-code -g -y
```

  **只安装某一个技能：**

```bash
npx skills add tw93/Waza --skill think -a claude-code -g -y
```

  **Claude Code 插件市场安装：**

```bash
/plugin marketplace add tw93/Waza
/plugin install waza-think@waza    # 按需安装单个
```

  **更新：**

```bash
npx skills update -g -y
```

  支持六种 AI 编程工具：Claude Code、Codex、Antigravity、OpenCode、Claude Desktop、Pi。安装后无需重启，Claude Code 会自动发现新技能。

### 使用指南

#### 场景一：新功能开发前——用 /think 避免"做错东西"

  产品经理说"加个暗色模式"，AI 代理的直觉反应是直接开始写 CSS 变量。但暗色模式有很多设计决策：是跟随系统还是手动切换？覆盖哪些组件？已有设计系统怎么兼容？

  输入 `/think 给这个后台管理系统加暗色模式`，Waza 会让 Claude：
  1. 先挑战问题本身——"暗色模式对这个后台系统的用户来说，使用频率有多高？"
  2. 列出所有设计决策点——跟随系统 vs 手动切换、覆盖范围、设计 token 改造量
  3. 产出一份决策完整的实施计划，另一个 agent 可以直接照着执行

  这一步花 3 分钟，省掉后面因为方向错误浪费的 3 小时。

#### 场景二：任务完成后——用 /check 做发布前自查

  写完代码准备合并 PR，但你不知道有没有遗漏——日志打了吗？错误处理写了吗？有没有不小心改了不该改的文件？

  输入 `/check`，Waza 会：
  1. 审查当前 diff，提取项目专属约束（比如你的项目要求所有 API 调用必须有超时处理）
  2. 自动修复安全的小问题（格式化、import 排序）
  3. 标记需要人工确认的危险操作（删除文件、修改数据库 schema）
  4. 用实际证据验证——不是"看起来没问题"，而是"测试通过了、lint 过了、类型检查过了"

  最关键的细节：/check 对破坏性命令（`git push --force`、`rm -rf`、数据库变更）有专门的拦截逻辑，会要求你明确确认后才放行。

#### 场景三：遇到诡异的 bug——用 /hunt 系统化定位根因

  测试报了一个"之前还能用的功能现在挂了"的回归 bug，AI 代理最常见的错误是：看到报错信息就猜测原因，然后开始改代码。改了三次还没好，才发现根因在别的地方。

  输入 `/hunt 用户登录后页面白屏，昨天还正常`，Waza 会强制 Claude 走系统化调试流程：
  1. 确认现象——白屏发生在哪个页面、什么条件触发
  2. 二分法缩小范围——最近的哪次提交引入的？哪个组件渲染失败？
  3. 确认根因后才允许动手修——用 git bisect、console 日志、网络请求等多种手段交叉验证
  4. 修完后用 /check 验证——防止修复引入新问题

  "确认根因前不许修"这个约束看似简单，但实际使用中能避免大量无效修改。

#### 场景四：写前端界面——用 /design 摆脱 AI 模板味

  AI 生成的 UI 有一个通病：千篇一律的"现代化简约风"——蓝白配色、圆角卡片、Inter 字体。放在一起看根本分不出哪个是哪个产品。

  输入 `/design 做一个 SaaS 数据仪表盘首页`，Waza 会：
  1. 先让你回答五个方向性问题——目标用户是谁、竞品有哪些、品牌调性是什么
  2. 调研 2-3 个同类产品是怎么解决同一个设计问题的
  3. 产出有辨识度的设计方案，包含截图驱动的美学迭代
  4. 对仪表盘、数据表格等特定场景有专门的设计规则

#### 场景五：深入不熟悉的领域——用 /learn 系统化学习

  需要快速了解一个新技术领域（比如 WebAssembly、CRDT、RAG 优化），但不知道从哪看起。

  输入 `/learn CRDT 冲突解决算法`，Waza 会走六阶段研究流程：
  1. 收集——多源搜索相关资料
  2. 消化——提取核心概念和关键论文
  3. 大纲——结构化组织知识点
  4. 填充——逐节展开详细内容
  5. 精炼——删冗余、补遗漏
  6. 自审——检查准确性和完整性后输出

  产出不是简单的信息堆砌，而是结构化的学习笔记。

### 进阶技巧

  **技能链式组合**：Waza 的技能可以像 Unix 管道一样串联。做完一个功能后：`/think` → 实现 → `/check`，保证每个环节都有质量闸门。遇到 bug：`/hunt` → 修复 → `/check`，先定位再修复再验证。

  **按需安装**：不需要全部八个技能。如果你主要做后端开发，可以只装 `/think`、`/check`、`/hunt`。如果你经常写技术文章，加装 `/write` 和 `/learn`。每个技能独立运行，互不依赖。

  **和 CLAUDE.md 配合**：在 CLAUDE.md 中写明项目特定的约束（如"所有 API 必须带超时"），/check 会在审查 diff 时自动应用这些约束。Waza 不替代 CLAUDE.md，而是让它发挥更大作用。

### 常见踩坑

  **技能不会自动触发**：和很多人的直觉相反，Waza 的技能需要你手动输入 `/think`、`/check` 等命令。它不会因为你说了"我要写一个新功能"就自动跑 /think。这是刻意设计的——tw93 认为自动触发会增加不必要的上下文开销，而且在错误时机触发比不触发更糟。

  **/health 目前仅支持 Claude Code**：如果你想审计 Codex 或 Pi 的健康状态，需要等后续版本。其余七个技能在 Claude Code 和 Codex 上都可以正常使用。

  **不要和重型技能包同时装**：如果你已经装了 Superpowers 或 gstack，可能会和 Waza 的部分技能（如调试流程）产生冲突。建议先试用 Waza 的轻量模式，如果觉得不够再上重型包。
