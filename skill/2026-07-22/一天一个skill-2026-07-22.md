# agent-browser — AI 终于有了自己的浏览器

  Vercel Labs 出品，38,879 stars，110 位贡献者，97 个 release。一条命令装好，agent 就能像人一样浏览网页——打开页面、点按钮、填表单、截图、抓数据，全部通过 CLI 完成。100% 原生 Rust，不依赖 Playwright/Puppeteer，专为 AI 编程 agent 设计。

### 它解决什么问题

  AI 编程 agent 最大的短板不是推理能力，是没有浏览器。你说"帮我看看这个页面的表单怎么填"，它得靠读 HTML 源码猜布局。你说"部署完帮我截图验证一下"，它束手无策。你说"帮我测一下登录流程能不能走通"，它只能写测试代码让你自己跑。

  现有的浏览器自动化方案（Playwright MCP、Puppeteer）有两个致命问题：第一，太重了。加载一个页面 dump 出来的 DOM 动辄 3000-5000 tokens，一个表单页面就塞满上下文窗口。第二，版本漂移。你把 Playwright 的使用方法写进 SKILL.md，三个月后 Playwright 升级了 API，skill 里的指令全过期。

  agent-browser 用两个核心设计解决了这两个问题：

  **Snapshot + Ref 系统取代 DOM dump。** 不把整个 HTML 丢给 agent，而是生成一个压缩版的 accessibility tree，每个可交互元素分配一个 `@eN` 引用号。Agent 看到的是"heading 'Login' [ref=e1], textbox 'Email' [ref=e2], button 'Submit' [ref=e3]"，然后说 `click @e3`。实测 token 使用量是传统 DOM 方式的 2.2%-21.6%——页面越复杂，省得越多。

  **Skill 内容由 CLI 动态提供，不写死在文件里。** 你安装的 SKILL.md 只是一个轻薄的路由器——它告诉 agent"用 `agent-browser skills get core` 获取当前版本的使用指南"。CLI 永远提供与已安装版本匹配的内容，不存在 skill 内容过期的版本漂移问题。

### 核心原理

  agent-browser 的架构分两层：

  **Rust CLI 层。** 你（或你的 agent）输入 `agent-browser open example.com`，CLI 解析命令，通过 IPC 发给后台守护进程。CLI 本身是纯 Rust 编译的二进制文件，启动几乎无延迟。

  **Rust Daemon 层。** 守护进程通过 Chrome DevTools Protocol (CDP) 直接操控 Chrome/Chromium。不经过 Playwright，不经过 Puppeteer，不经过 Node.js。守护进程在第一次命令时自动启动，之后常驻内存，后续命令零冷启动成本。

  关键的数据流：`snapshot` 命令拿到 Chrome 的 accessibility tree → Daemon 将其渲染为精简文本 → 每个元素打上 `@eN` ref → 返回给 agent。Agent 选择 ref 发出操作指令 → Daemon 将 ref 翻译为 CDP 命令 → Chrome 执行。

  Session 隔离：`--session` 参数创建独立的浏览器上下文，每个 session 有自己的 Cookie、Storage、代理配置。多个 agent 可以并行使用同一个 agent-browser 而不互相干扰。

### 安装配置

```bash
# 全局安装（推荐）
npm install -g agent-browser
agent-browser install   # 下载 Chrome for Testing（仅首次需要）

# macOS 也可以走 Homebrew
brew install agent-browser
agent-browser install
```

安装 agent-browser 的 Claude Code Skill：

```bash
npx skills add vercel-labs/agent-browser
```

重启 Claude Code，之后凡是涉及浏览器的操作——打开网页、填表单、截图验证——agent 都会自动调用 agent-browser。

### 使用指南

#### 场景一：部署后的页面验证

  你让 Claude Code 改完前端代码，部署到 staging。正常流程是你手动打开浏览器、输入 URL、肉眼检查。有了 agent-browser，一句话完成：

```
帮我在 staging 环境验证一下登录页面是否正常渲染，截图给我看
```

Agent 的执行流程：
1. `agent-browser open https://staging.example.com/login`
2. `agent-browser snapshot` → 检查关键元素（用户名输入框、密码框、登录按钮）是否存在
3. `agent-browser screenshot login-check.png` → 截图返回给你

  不需要写一行 Playwright 脚本，不需要配 MCP server。

#### 场景二：自动填表测试

  开发完注册流程，需要验证全链路——输入邮箱、密码、确认密码、勾选协议、点击注册。用 agent-browser 就是六条命令：

```bash
agent-browser open https://example.com/register
agent-browser snapshot                         # 获取 ref 映射
agent-browser fill @e2 "test@example.com"      # 邮箱输入框
agent-browser fill @e3 "MyPassword123"         # 密码输入框
agent-browser fill @e4 "MyPassword123"         # 确认密码
agent-browser check @e5                        # 勾选用户协议
agent-browser click @e6                        # 点击注册按钮
agent-browser snapshot                         # 验证跳转后的页面
```

  对 AI agent 来说，这比写 Selenium 脚本自然得多——它操作浏览器就像你在操作一样，只是用命令行代替了鼠标键盘。

#### 场景三：数据抓取与信息提取

  需要从某个页面提取特定信息？不用写爬虫，不用处理 HTML 解析：

```bash
agent-browser open https://news.ycombinator.com
agent-browser snapshot -i                       # 获取页面结构
agent-browser get text @e45                     # 提取某条新闻标题
agent-browser read                              # 获取 agent 友好的纯文本内容
```

  `read` 命令更有意思——它直接请求 URL 的 markdown 版本（`Accept: text/markdown`），不启动浏览器。对于文档站、博客、新闻页面，一条命令就能拿到干净的 markdown 内容，token 消耗比 `snapshot` 还要低一个数量级。

#### 场景四：Electron 桌面应用自动化

  agent-browser 不只是操作网页，它还能操控任何 Electron 应用——VS Code、Slack、Discord、Figma、Notion。只要应用支持 Chrome DevTools Protocol：

```bash
agent-browser skills get electron               # 加载 Electron 专用 skill
agent-browser connect <debug-port>              # 连接到 Electron 的 CDP 端口
agent-browser snapshot                           # 获取应用界面结构
```

  这让 agent 可以操作那些没有 Web 版也没有 API 的桌面应用。

### 进阶技巧

  **Annotated Screenshot——截图带上 ref 编号。** `screenshot --annotate` 会在截图上给每个可交互元素画上编号框，把截图发给多模态模型时，模型可以直接指着编号说"点 12 号按钮"。这解决了一个经典问题：截图和 ref 树是两套坐标体系，annotate 把两套体系统一到一张图上。

  **HAR 录制 + API 客户端生成。** `derive-client` 这个 skill 可以录制页面的网络请求，从 HAR 文件中提取 API 端点，自动生成一个独立的 API 客户端。不需要读源码、不需要翻文档，直接把网站的"后台接口"扒出来。对逆向工程和 API 调研场景特别实用。

### 常见踩坑

  **Windows 上 `open` 报告成功但 `get url` 返回 `about:blank`。** 这是已知 bug（#1262），Windows 上 session 状态同步存在竞态条件。临时解决：`open` 后手动 `get url` 验证是否真正导航成功，如果返回 `about:blank`，`close --all` 后重试。官方正在修复中。

  **`close` 失败后继续 `open` 会读到旧页面内容。** 当 `close` 返回 "Failed to connect" 时，守护进程可能处于半死状态。后续 `open` 报告成功但实际内容来自之前的 URL——AI agent 会被误导，以为自己在读维基百科实际在读诗歌网站。临时解决：`close` 失败后务必 `close --all` 或手动杀掉残留进程再继续。

  **Cloudflare 等反爬系统无法绕过。** agent-browser 和 Playwright 一样使用 Chrome CDP，在 Cloudflare、Akamai 等专业反爬系统面前是"裸奔"的。如果你的场景涉及对抗性爬取，需要 Browserbase 这类改过内核的专业方案。

### 同类技能对比

| 技能 | 安装方式 | 核心特点 | 适用场景 | 热度 |
|------|---------|---------|---------|------|
| **agent-browser**（今日推荐）| `npm i -g agent-browser` | Rust 原生，Snapshot+Ref 系统，token 节省 80%+，内置专用 skill 体系 | AI agent 驱动的浏览器操作 | ⭐ 38.9k |
| Playwright MCP | npm + MCP 配置 | 完整的浏览器自动化框架，三引擎支持 | 人类开发者写测试脚本 | ⭐ 70k+ |
| Puppeteer | npm 包 | Google 维护的 Chrome 控制库 | Node.js 爬虫和自动化 | ⭐ 88k+ |
| Browserbase | SaaS 服务 | 改过内核的隐形浏览器，反检测 | 对抗性爬取、大规模数据采集 | ⭐ 10k+ |

  推荐 agent-browser 而不是 Playwright MCP 的理由：Playwright MCP 会把完整的 DOM accessibility tree dump 到上下文里，一个复杂页面就吃掉几千 tokens。agent-browser 的 snapshot 输出只有 200-400 tokens——agent 长期运行的 token 账单差距非常大。如果你确实需要 Playwright 的复杂交互逻辑（拖拽组合、多 iframe 操作、网络拦截），Playwright 更灵活；如果只是让 agent 看看页面、点点按钮、填填表单，agent-browser 是最佳选择。

## 今日小结

  agent-browser 解决的是一个基础架构问题：AI agent 需要一个浏览器的"眼睛"和"手"。它用 Rust 原生 CLI + accessibility tree snapshot + ref 系统，把 token 消耗降到传统方案的零头，同时内置了一套版本不漂移的 skill 体系。一条 npm 命令装好，Claude Code、Cursor、Codex、Gemini CLI 全兼容。如果你经常让 agent 帮你验证页面、测试流程、抓取信息，花两分钟装上 agent-browser，代理就从此不再"失明"。
