# guard-skills — AI 写的代码，它来把关

  AI 编码 agent 写代码很快，但系统性缺陷也很一致：吞掉异常、幻觉出 API、过度抽象、文档与代码脱节。guard-skills 是五个专注的"质量闸门"skill，在 agent 产出代码之后、你合入之前，做第二遍审查，专门抓这些 AI 生成代码的典型失败模式。

---

### 它解决什么问题

  用 AI agent 写代码的人迟早会撞上同一类问题：agent 信心十足地交了一个 PR，但测试用例全是 `MagicMock()`，README 引用了不存在的函数，WordPress 插件忘了做转义和 nonce 校验。这些问题不是 agent 不够聪明——它们是当前 AI 编码的系统性缺陷，单一 agent 很难自我审查出来。

  guard-skills 的思路是"写归写，审归审"：让 agent 把活干完，然后在 diff 上跑对应的 guard 做第二遍审查。每个 guard 专注一个领域，规则来自真实踩坑经验加上已发表的研究数据（如幻觉包比例、agent 假性成功的统计等）。

### 核心原理

  五个 guard 各自独立，按需触发。每个 guard 是一个标准 SKILL.md，加载成本很低；深度规则放在 `references/` 目录中按需加载。

  - **clean-code-guard**：检查 SOLID、DRY/KISS/YAGNI 违规，外加 AI 专属层——吞异常返回 ok、硬编码"success"、幻觉 API、过早抽象、注释污染、复制粘贴 bug
  - **test-guard**：九条通用规则——只在系统边界 mock、不 mock 自己的状态对象、参数化替代复制粘贴、删除没抓到任何东西的测试、回归测试不可动
  - **docs-guard**：把文档当成声明列表，逐条与代码库对账。README 里引用的函数是否存在、`@param` 类型是否对得上、代码示例是否真能跑通
  - **wp-guard**：WordPress 专属——转义、消毒、nonce、权限检查、预处理 SQL、i18n、查询/缓存规范
  - **woo-guard**：WooCommerce 专属——HPOS 安全的订单访问、CRUD 替代直接读写 meta、服务端校验结账逻辑、货币处理规范

### 安装配置

```bash
# 先浏览有哪些 guard
npx skills add amElnagdy/guard-skills --list

# 一键安装全部五个
npx skills add amElnagdy/guard-skills

# 或按需安装
npx skills add amElnagdy/guard-skills --skill clean-code-guard
npx skills add amElnagdy/guard-skills --skill test-guard
npx skills add amElnagdy/guard-skills --skill docs-guard

# 安装到指定 agent
npx skills add amElnagdy/guard-skills --skill clean-code-guard --agent claude-code
```

安装完成后重启你的 agent 即可使用。零依赖、无脚本执行、无外部网络调用，纯 Markdown 指令。

### 使用指南

#### 场景一：agent 刚写完一堆代码，提交前跑一遍清洁审查

  最常用的姿势：agent 噼里啪啦改完几个文件后，说一句：

  > Use $clean-code-guard on the diff you just produced.

  guard 会逐文件检查：有没有默默改变了对外接口（而 agent 没告诉你）、有没有 `try/catch` 吞掉所有异常返回 ok、有没有引用根本不存在的库。如果发现问题，它会逐条列出，并给出具体的修复方向——不是"建议优化错误处理"这种废话，而是"第 47 行的 `catch(Exception e) { return true; }` 会掩盖所有下游故障，至少加上日志，最好重抛"。

#### 场景二：agent 写的测试看起来覆盖率高，但实际上测了个寂寞

  AI 写测试有一个经典套路：把所有依赖 mock 掉，断言日志输出，复制粘贴同一个测试改个标题。跑一下 test-guard：

  > Use $test-guard on the tests you just wrote.

  test-guard 会标记：用 `MagicMock()` mock 了自身状态对象的测试、内容完全相同只是描述不同的测试、只断言日志消息不检验行为的测试。一个典型的审查结果可能是"这个测试文件里 12 个用例中有 7 个存在 mock 滥用，3 个是复制粘贴的重复体，2 个不会在任何断言失败时失败——请重写"。

#### 场景三：agent 帮你写了 README 和 API 文档，上线前做一次事实核查

  agent 特别擅长写看起来专业、实际上充满幻觉的文档。函数名是编的，参数类型是想的，示例代码跑不通。

  > Use $docs-guard on this README update before we ship it.

  docs-guard 的核心逻辑是"声明即待验证"：把文档里每个函数引用、每个参数类型、每个代码示例都当成需要证伪的声明，逐条与代码库实际内容对比。README 里写了 `await createUser(email, password)`，但实际签名是 `createUser(email, password, role)`——它会指出来，并给出修正建议。

#### 场景四：WordPress/WooCommerce 插件开发的安全闸门

  做 WP 插件开发的人都知道，WordPress 有自己的一套安全范式：`esc_html()`、`wp_nonce_field()`、`current_user_can()`、`$wpdb->prepare()`……AI agent 不了解这些就会写出看起来正常但实际有安全漏洞的代码。

  > Use $wp-guard on this WordPress plugin change.
  > Use $woo-guard on this WooCommerce checkout change.

  wp-guard 检查转义、消毒、nonce、权限、预处理 SQL、i18n、查询分页。woo-guard 在 wp-guard 基础上追加 HPOS 兼容、订单 meta 直接读写、结账逻辑仅前端校验、货币字符串拼接等问题。

### 进阶技巧

  **组合技：写完即审**。在 CLAUDE.md 里加一条规则，让 agent 在每次提交代码后自动触发对应的 guard：

  ```markdown
  After writing production code, always self-check with $clean-code-guard on the diff.
  After writing tests, always self-check with $test-guard on the test file.
  Before writing documentation, note that all doc claims will be verified by $docs-guard.
  ```

  这样 guard 成为你的 agent 工作流的一部分，而不是事后补救。

  **按需组合**：clean-code-guard + wp-guard 覆盖 WordPress 开发的完整质量面；clean-code-guard + test-guard 覆盖任何语言的代码+测试双审。不需要一次装五个，按你的实际技术栈选。

### 常见踩坑

  **guard 不是 process 框架**：它不会告诉 agent 怎么规划、怎么拆分任务、怎么管理 git 分支。它只做一件事——审查已产出的代码。如果你的 agent 流程本身有结构性问题（比如没有规划阶段、直接上手写代码），先解决流程问题，再用 guard 做质量闸门。

  **wp-guard 和 woo-guard 有依赖关系**：如果你的代码同时涉及 WordPress 和 WooCommerce，两个 guard 都要跑。woo-guard 只检查 WooCommerce 专有层（HPOS、结账、支付网关），不重复检查 WordPress 基础安全层。正确的顺序是：先跑 wp-guard 过基础安全，再跑 woo-guard 过电商专有规则。

## 今日小结

  guard-skills 适合每一个正在用 AI agent 写生产代码的开发者。安装只需一条 `npx skills add` 命令，之后在你的 agent 产出代码后加一句话就能触发审查。它最核心的价值不是"又一个审查工具"，而是"精准打击 AI 代码的系统性缺陷"——吞异常、幻觉 API、文档与代码脱节、mock 滥用、WordPress 安全漏掉——这些是人类 review 容易疲劳漏掉、AI 自己也意识不到的典型问题。花 5 分钟装上，每次 agent 交活后跑一下，能挡住一大类"看着没问题，上线就炸"的 bug。
