# Prompt Refine — 用模型的母语跟模型说话

  你随口说一句"帮我分析这个市场"，它自动按当前模型最擅长的格式重写成结构化的指令，然后静默输出结果——整个过程你只看到最终答案，感觉什么都没发生，但答案质量却显著提升。这就是 Prompt Refine，一个 2026 年 6 月刚发布就自带双盲 A/B 测试数据的 Agent Skill。

---

### 它解决什么问题

  不同模型有不同的"母语"。Claude 在 XML 结构标签下表现最优，GPT 更适应 Markdown 风格的任务描述，Gemini、DeepSeek、Llama 也各有偏好。你用同一种方式跟所有模型说话，就像对法国人、日本人、阿拉伯人全部用英语——能沟通，但不精准。

  Prompt Refine 做的事情很简单：它检测当前会话跑的是哪个模型，然后用那个模型家族的官方提示词策略，在内部把你的请求重写一遍。Claude 收到的是 Anthropic 风格的 XML 结构，GPT 收到的是 OpenAI 风格的指令块，DeepSeek 收到的是 DeepSeek 风格的格式。**你不变，它替你变。**

  更重要的是，它很小——只有一个 `SKILL.md` 加一份策略文件进入上下文，没有额外 API 调用，没有外部服务，token 开销极低。

### 核心原理

  Prompt Refine 的设计架构非常简洁：

  ```
  用户输入（可能模糊、简短、缺少约束）
       ↓
  检测当前模型家族（Claude / GPT / Gemini / DeepSeek / ...）
       ↓
  加载对应策略文件（strategies/anthropic.md / openai.md / ...）
       ↓
  在内部重写 prompt（添加角色、约束、输出格式、成功标准）
       ↓
  模型用优化后的 prompt 生成答案（用户只看到最终答案）
  ```

  关键设计决策：**按宿主模型路由，不按任务路由**。写代码不会切到 GPT 风格，写文档不会切到 Claude 风格。模型策略是固定的，任务不改变策略选择。

  上下文感知：多轮对话中的后续请求会继承前文的目标、约束和术语，但最新的用户指令始终优先。

  三个对话内控制命令：
  - `/prompt-refine` — 激活 skill
  - `/refine verbose` — 显示改写前后的对比（调试用）
  - `/refine off` — 关闭优化

### 安装配置

  ```bash
  # Claude Code（项目级安装）
  git clone https://github.com/Li-Bailiang/prompt-refine-skill.git .claude/skills/prompt-refine

  # 或用 degit 避免 .git 目录
  npx degit Li-Bailiang/prompt-refine-skill .claude/skills/prompt-refine

  # 或通过 npm 安装
  npm pack prompt-refine-skill
  tar -xzf prompt-refine-skill-*.tgz --strip-components=1 -C .claude/skills/prompt-refine
  ```

  安装后重启 Claude Code，执行 `/prompt-refine` 即可激活。

### 使用指南

  #### 场景一：模糊需求一键结构化

  你提了一个很宽泛的问题，模型经常答不到点子上或者答得太泛。

  激活 Prompt Refine 后提问：

  > 帮我分析一下这个市场

  内部改写（Claude 策略）会自动补全：角色定义（资深市场分析师）、上下文（未提供市场名称、地域等，保留不确定性）、硬约束（不编造当前市场数据，最多追问 1-2 个关键问题）、输出格式（假设 → 竞争地图 → 壁垒 → 战略含义 → 未知项 → 下一步问题）和成功标准。

  你看到的只是一个非常有条理的市场分析，而不是泛泛而谈。

  #### 场景二：约束性任务保持不变

  你已经给了很精确的约束，不希望模型"过度发挥"。

  > 写一个 5 条 npm 发布 checklist，每条不超过 8 个字

  Prompt Refine 识别到用户已经提供了硬约束，干预降到最低：只添加成功标准和输出格式校验，不扩展任务范围。最终输出严格遵守 5 条、每条不超过 8 字的限制。

  #### 场景三：跨模型协作保持一致性

  你在 Claude Code 中写完一个 feature，然后用 Codex CLI 做 code review，再用 Gemini CLI 生成文档。三个模型默认的输出风格可能完全不同。

  在三个工具中都安装 Prompt Refine 后，同样的请求在每个模型中会被改写为各自最优的格式，产出质量更加稳定，不再有"这个模型好用、那个不好用"的波动。

  #### 场景四：中英文混合场景

  你用中文提问，模型有时会莫名其妙切到英文回答，或者中英混杂。

  在双盲测试中，Prompt Refine 在 60 个中文 prompt 上的语言保真度 δ = +0.03（与 raw 几乎持平），且零例输出语言切换。这意味着它不会因为内部改写就把中文 prompt 变成英文回答。

### 评测数据

  这是 Prompt Refine 最令人信服的部分——它不是拍脑袋说"我觉得有用"，而是跑了严格的双盲 A/B 测试：

  - **120 个模糊 prompt**（60 英文 + 60 中文，横跨 32 个领域）
  - 同一个模型回答两次——一次 raw、一次带 Prompt Refine
  - 独立裁判不知道哪份是哪个条件，且每对交换位置评两次消除顺序偏见
  - 生成模型：Claude Sonnet 4.6，裁判模型：Claude Opus 4.8

  **核心数据**：

  | 指标 | 结果 |
  |------|------|
  | Refine vs Raw 胜率 | **74.0%**（167 胜 / 52 负 / 21 平） |
  | 95% 置信区间 | [66.9%, 80.6%] |
  | 显著性 | **p < 0.0001** |
  | 中文胜率 | 72.9% |
  | 长度匹配胜率 | **64.7%**（排除"写得更长=更好"的偏差） |

  **分维度提升**（1-5 分量表，refine − raw）：

  | 维度 | 提升 |
  |------|------|
  | 可操作性 (actionability) | **+0.96** |
  | 完整性 (completeness) | **+0.81** |
  | 结构 (structure) | +0.49 |
  | 澄清度 (clarification) | +0.35 |
  | 语言保真度 | +0.03 |

  安全性检查：0/120 例 scaffold 泄漏（输出中没有出现 XML 标签或改写后的 prompt），0/60 例中文 prompt 语言切换。

### 常见踩坑

  **长对话后 skill 失效**：上下文压缩（compaction）会清掉 skill 激活状态。解决方法：压缩后重新执行 `/prompt-refine`。

  **对已经写得很好的 prompt 提升有限**：这是设计如此。Prompt Refine 对模糊、缺少约束的 prompt 效果最显著，对于已经结构化的清晰 prompt，内部改写可能几乎为零。不要期待它把好 prompt 变得"更好"——它解决的是"说不清楚"的问题。

  **只有 Anthropic 策略经过大规模评测**：其他 11 个策略文件结构相同但未经过同等级别的双盲测试。如果你主要用 Claude，数据是可靠的；用其他模型时视为"合理但未经严格验证"。

  **需要模型能遵循元指令**：Prompt Refine 依赖模型理解并执行"静默改写，然后回答"这个元指令。大多数前沿模型可以做到，但小型或旧版模型可能效果打折。

## 今日小结

  Prompt Refine 适合所有日常重度使用 AI Agent 的开发者。安装只需要一条 git clone 命令，之后每次启动会话执行一次 `/prompt-refine`，就能在几乎不增加 token 开销的情况下，让模糊的问题得到更结构化的答案。它不改变你的工作流，不要求你学习新的提示词技巧——它只在你和模型之间加了一层极薄的翻译层，用模型的母语帮你说出你想说的话。如果你经常觉得"模型没理解我的意思"，今天花 2 分钟装上，明天开始你的每一次提问都自带一位隐形的 prompt 工程师。
