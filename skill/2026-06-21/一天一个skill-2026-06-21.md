# claude-orchestration — 像 N8N 一样编排你的 AI Agent 工作流

  Claude Code 单 agent 做事没问题，但复杂任务需要多个 agent 并行排查、分阶段执行、条件分支路由。每次手写 Agent 调用、手动传递上下文、人工判断下一步走向，既低效又容易出错。claude-orchestration 用一套简洁的声明式语法（`.flow`），把多 agent 协作变成可复用、可调度、可恢复的工作流——就像 N8N 之于 API，它是 Claude Code 的工作流引擎。

---

### 它解决什么问题

  日常开发中有大量"多步骤 + 多角度"的任务：排查 bug 需要同时看日志、查代码、翻 commit 记录；上线前需要并行跑安全检查、性能审计、文档更新；日常站会需要汇总 PR、issue、Sentry 告警。这些任务天然适合并行 agent，但手动协调不仅麻烦，还容易丢上下文——比如第 3 步的输出忘了传给第 5 步，或者中间某步挂了整个流程重来。

  claude-orchestration 把编排逻辑从"每次口头描述"变成"一次定义、反复执行"的 `.flow` 文件。支持顺序执行、并行分派、条件分支、人工审批检查点，甚至定时无人值守运行。

### 核心原理

  插件由 7 个自动激活的子 skill 组成（creating-workflows、executing-workflows、managing-agents 等），覆盖工作流全生命周期。当你用自然语言描述任务或提供 `.flow` 语法文件时，编排引擎做四件事：

  1. **解析**：识别顺序（`->`）、并行（`||`）、条件（`~>`）、检查点（`@review`）等控制流
  2. **生成**：自动创建所需临时脚本（Python/Node.js），如爬虫、API 调用、数据处理
  3. **分发**：将每个节点派发给对应的 Agent（内置 Explore/Plan/general-purpose，或你导入的自定义 agent）
  4. **追踪**：保存状态到 `.orchestration/state.json`，中间某步挂了可以断点续跑

  变量通过 `:var` 捕获、`{var}` 引用，在节点间自动传递——不需要手动复制粘贴。

### 安装配置

```bash
# 第一步：添加插件市场
/plugin marketplace add mbruhler/claude-orchestration

# 第二步：安装插件
/plugin install orchestration@mbruhler

# 第三步：验证安装
/help
# 看到 /orchestration:menu, /orchestration:init 等命令即成功
```

  可选：导入你已有的自定义 Agent（`~/.claude/agents/` 下的 agent 文件），让它们在编排中可用：

```bash
/orchestration:init
# 交互式选择要导入的 agent，导入后可在 .flow 中直接引用
```

### 使用指南

#### 场景一：并行 Bug 排查

  线上报了一个 bug，不确定是前端还是后端的问题。与其一个个 agent 串行查，不如三条线并行：

```flow
[
  Explore:"Find error pattern in codebase":code ||
  general-purpose:"Analyze error logs":logs ||
  general-purpose:"Check recent commits":commits
] ->
general-purpose:"Identify root cause from {code}, {logs}, {commits}":cause ->
@review:"Diagnosis correct?" ->
general-purpose:"Implement fix and run tests":fix
```

  三个 agent 同时开始工作——Explore 搜代码、general-purpose 看日志、另一个 general-purpose 查提交记录。汇总后由一个 agent 综合诊断，最后人工确认修复方案。

#### 场景二：TDD 开发流水线

  把红-绿-重构循环固化为可复用模板，每次新功能都走同一套纪律：

```flow
# RED：写失败测试
general-purpose:"Write failing test for the feature":test ->
general-purpose:"Run test suite - verify it fails":red_result ->
@review-coverage:"Test coverage sufficient?" ->

# GREEN：最小实现
general-purpose:"Write minimal code to pass the test":impl ->
general-purpose:"Run test suite - verify it passes":green_result ->
@review:"Code quality OK?" ->

# REFACTOR：清理
general-purpose:"Refactor code and add documentation":refactored ->
general-purpose:"Final test run and commit"
```

  每个阶段有明确的输入（上一阶段的 `:var`）、输出（`:var` 捕获）、检查点（`@review`），不会跳步也不会漏验证。

#### 场景三：每日站会自动汇总

  每天早上 8 点自动收集 PR、issue 和告警，生成站会摘要写入文件：

```flow
@schedule("0 8 * * *")

[
  Explore:"Check recent GitHub PRs":prs ||
  general-purpose:"Check open Sentry issues":bugs
] ->
general-purpose:"Draft daily standup summary from {prs} and {bugs}":summary ->
general-purpose:"Append {summary} to standup_log.md"
```

  `@schedule` 指令配合 Claude Code 的 `/loop` 机制，实现真正的无人值守定时任务。检查点可以设置 `fallback=skip` 或 `fallback=notify`，后台模式不会卡住。

### 进阶技巧

  **语义路由（Semantic Routing）**：不用写死 if-else，让 LLM 动态判断分支走向：

```flow
general-purpose:"Read the latest user support ticket":ticket ->
route({ticket}) => [
  (if "UI, CSS, or layout bug") ~> frontend-agent:"Fix styling" ||
  (if "Database, 500 error, crash") ~> backend-agent:"Fix server"
]
```

  `route()` 把变量内容交给 LLM 判断，自动匹配最合适的分支。比正则匹配灵活得多，适合客服工单路由、代码审查分类等场景。

  **社区工作流注册表**：不想从零写 `.flow`？直接从社区拉取经过验证的模板：

```bash
/orchestration:pull tdd-autopilot
/orchestration:pull username/repo/my-workflow.flow
```

  就像 npm install 一样，社区沉淀的最佳实践可以直接复用。

### 常见踩坑

  **工作流卡住不动**：检查是否有 `@review` 检查点需要人工确认。后台/定时模式下，给检查点加 `fallback=skip` 或 `fallback=notify`，否则流程会一直等待输入。

  **临时脚本执行失败**：编排引擎会自动在 `temp-scripts/` 下生成 Python/Node.js 脚本。如果脚本报错，去这个目录检查生成的代码，通常是 API 凭证缺失或依赖未安装。脚本不会被自动清理，可以直接调试。

  **自定义 Agent 找不到**：内置 agent（Explore、Plan、general-purpose）直接用名字。导入的自定义 agent 需要用 `orchestration:` 前缀。临时 agent 用 `$` 前缀（如 `$scanner:"Scan"`）。

## 今日小结

  如果你的日常工作中经常需要"同时让 Claude 做几件事然后汇总"，或者想把一些固定的多步骤流程（TDD、bug 排查、上线检查）固化下来，claude-orchestration 值得今天花 5 分钟装上。两行命令安装，然后用自然语言描述第一个工作流就能跑起来。随着 `.flow` 文件的积累，你会逐渐拥有一个专属的 Agent 工作流库——复杂任务不再靠记忆和口头描述，而是靠可复用、可追溯的编排文件驱动。
