# BiswaViraj/agent-skills — 三个技能管一天：PR 审查、CI 修复、每日站会

  每天打开 GitHub，你面对的是同一套流程：PR 上堆着 Greptile、CodeRabbit、Copilot 和人类同事的审查意见，CI 红了不知道是 flaky 还是真 bug，站会前五分钟疯狂翻 commit 历史回忆昨天干了什么。这三个场景几乎每个开发者每天都要走一遍，但 Claude Code 默认不会帮你自动化其中任何一个。

  BiswaViraj/agent-skills 是一个打包了三个技能的 Claude Code 插件——reviewloop、ciloop、standup——分别对应「清空 PR 审查」「修绿 CI」「生成站会报告」。一次安装，三个命令，覆盖开发者每天最高频的三个机械化操作。

### 它解决什么问题

  用 Claude Code 写代码的时候，有三件事你几乎每天都要手动做：

  1. **清 PR 审查意见**。一个 PR 上可能有 Greptile 的评分、CodeRabbit 的行内建议、Copilot 的代码审查、以及人类同事的 review。逐一回复、逐一解决、逐一 re-request——这个流程每次至少吃掉 15 分钟。
  2. **修 CI**。CI 红了，你需要打开 GitHub Actions 页面，翻日志，找到真正报错的那一行，在本地复现，修好，push，等 CI 重跑——如果又红了，重复以上。
  3. **写站会报告**。每天早上站会前，你需要回忆昨天 merge 了什么、开了什么 PR、review 了什么。忘了就得翻 GitHub 历史。

  这三个技能分别用一个命令解决这三个场景。它们不是"帮你生成代码"的 AI 技能，而是"帮你操作 GitHub"的自动化技能——核心能力来自 `gh` CLI 的熟练编排，Claude Code 负责理解上下文和做决策。

### 核心原理

  三个技能共享同一个设计原则：**用 `gh` CLI 的 GraphQL/REST API 精确获取 GitHub 数据，Claude Code 做决策和代码修改，循环直到目标达成**。

  **reviewloop 的工作机制：**

  它把 PR 上的所有 reviewer 分成两类——bot（可重触发）和人类（不可控）——然后用不同的策略处理：

  ```
  1. 用 GraphQL 一次性拉取所有 review 和 thread 数据
  2. 按 __typename 分类每个 reviewer（Bot vs User）
  3. 逐条处理 unresolved thread：该修的修，该回复的回复
  4. 批量 resolve 所有 thread
  5. Commit → Push → 重触发所有 bot（Greptile: @greptile review, CodeRabbit: @coderabbitai full review, Copilot: gh pr edit --add-reviewer）
  6. 轮询 bot 完成 → 回到步骤 1，直到所有 bot 绿 + 所有 thread resolved
  7. 对人类 reviewer：修完 → re-request → 记录 "waiting on @某人"，不阻塞循环
  ```

  **ciloop 的工作机制：**

  ciloop 的核心洞察是"不要在 CI 上做 edit-test 循环"——CI 又慢又贵：

  ```
  1. 找到当前分支最新的失败 CI run
  2. 用 --log-failed 拉取真实报错（不是 "job X failed"，是具体哪一行报了什么错）
  3. 把失败分成两类：
     - 可本地复现的（lint、typecheck、单测失败）→ 本地复现 + 修 + 本地跑绿
     - 不可本地复现的（flaky、infra、secrets）→ 记录原因，交还给你
  4. 所有可修复的在本地全部跑绿后，一次性 push
  5. 等新 CI → 绿了就结束，红了回到步骤 2
  ```

  **standup 的工作机制：**

  这个最轻量——一个 shell 脚本，用 `gh search prs` 搜索你在整个 org 范围内的 merged/opened/reviewed PR，按「Done / Today / Blockers」模板输出，周一自动回溯到上周五。

### 安装配置

  三种安装方式，推荐第一种（一次安装三个技能）：

  ```bash
  # 方式一：Claude Code 插件市场（推荐）
  claude plugin marketplace add BiswaViraj/agent-skills
  claude plugin install skills@biswaviraj-skills

  # 方式二：用配套的一键配置工具
  npx @biswaviraj/cc-setup@latest
  # 在交互菜单中勾选 agent-skills

  # 方式三：手动复制单个技能
  git clone https://github.com/BiswaViraj/agent-skills /tmp/agent-skills
  cp -r /tmp/agent-skills/skills/reviewloop ~/.claude/skills/reviewloop
  cp -r /tmp/agent-skills/skills/ciloop ~/.claude/skills/ciloop
  cp -r /tmp/agent-skills/skills/standup ~/.claude/skills/standup
  ```

  前置依赖：`gh` CLI 已认证（`gh auth status` 确认），版本 ≥ 2.88.0（Copilot re-request 功能需要）。仓库使用 GitHub Actions 做 CI。不需要额外 API token——所有操作走 `gh` CLI。

  安装后重启 Claude Code 会话，技能会自动被发现。standup 脚本需要可执行权限（`chmod +x ~/.claude/skills/standup/standup.sh`，Windows 上 Git Bash 环境不需要额外设置）。

### 使用指南

#### 场景一：PR 上堆了 5 个 reviewer 的反馈，一键清空

  你的 PR #5881 同时收到了 Greptile（3/5 分）、CodeRabbit（6 条行内建议）、Copilot（2 条建议）、以及同事小张（requested changes）的反馈。往常你要逐一回复每个人，手动点 resolve，等 bot 重新跑。

  用 reviewloop 一句话搞定：

  ```
  reviewloop this PR
  ```

  Claude Code 会：拉取所有 reviewer → 分类 bot/人类 → 逐条分析每个 thread → 能修的代码直接改，不需要改的写回复 → 批量 resolve → push → 重触发 Greptile/CodeRabbit/Copilot → 轮询新结果 → 如果还有问题，再来一轮。对人类 reviewer 小张，修完他的意见后 re-request 并标注「waiting on @小张」。

  你从「盯着屏幕一个一个点」变成了「发一个命令，等它说 done」。

#### 场景二：CI 红了，但你不想手动翻日志

  你 push 了一个 commit，CI 红了——lint 报 3 个文件、typecheck 报 1 个类型错误、e2e 测试超时（可能是 flaky）。

  ```
  ciloop
  ```

  ciloop 会自动：找到当前分支最新的失败 run → 拉取失败日志 → 分类：lint 和 typecheck 是可修复的，e2e 超时是 hand-back（可能是 infra 问题）→ 本地跑 `pnpm lint` 复现 → 修好 → 本地跑 `pnpm typecheck` 复现 → 修好 → 所有可修复的在本地绿了 → 一次性 commit + push → 等 CI 重跑 → 确认变绿。

  最终输出：
  ```
  Ciloop complete.
    Branch:      feat/jwt-verification
    Iterations:  1
    Fixed:       lint (3 files), typecheck (1 error)
    Handed back: e2e job — timeout on external service, not a code issue
    Final CI:    green ✅
  ```

  e2e 的 flaky 问题它不会瞎改——这正是 ciloop 设计上的克制之处。宁可交还给你，也不推送一个"也许能过"的猜测。

#### 场景三：站会前 30 秒生成报告

  早上 9:29，站会 9:30 开始，你还没想好「昨天干了什么」。

  ```
  write my standup
  ```

  输出：
  ```markdown
  **Standup — Thu 25 Jun** (since Wed 24 Jun)

  **Done**
  - ✅ Merged: myorg/backend#5877 — chore(env): rename the connection URI
  - 🚀 Opened: myorg/backend#5881 — feat: verify JWT inbound and forward bearer
  - 👀 Reviewed: myorg/frontend#1234 — fix: dropdown overflow in settings

  **Today:**
  **Blockers:** none
  ```

  Done 部分完全自动（从 GitHub 拉取你的 merged/opened/reviewed PR），Today 和 Blockers 留空让你自己填——不瞎编你今天的计划。周一自动回溯到上周五，不会漏掉周末前的工作。

### 进阶技巧

  **reviewloop + ciloop 组合技**。PR 审查过程中，reviewloop 修改代码后 push，可能触发 CI 失败。此时直接接一个 ciloop：

  ```
  reviewloop this PR
  # ... reviewloop 修完 push 后 CI 红了 ...
  ciloop
  ```

  两个技能接力，从「清空审查」到「修绿 CI」全自动，你只需要在两个命令之间看一眼 reviewloop 的修改摘要。

  **standup 自定义时间窗口**。除了默认的「上个工作日至今」，standup 支持任意起始日期，适合周报或月度总结：

  ```bash
  ./standup.sh --since 2026-06-16   # 从指定日期开始
  ./standup.sh --no-reviews          # 只看 merged/opened，不列出 reviewed
  ./standup.sh --user some-colleague # 帮同事查（需要同一 org）
  ```

### 常见踩坑

  **Copilot re-request 方式**。很多人以为 Copilot 和 CodeRabbit 一样，发一条 `@copilot review` 评论就能重新触发。实际上 Copilot 没有评论触发机制——必须用 `gh pr edit --add-reviewer @copilot`（需要 gh ≥ 2.88.0）。reviewloop 的 reviewer registry 表格已经硬编码了这个正确方式，不会犯这个错。

  **ciloop 不会修 flaky test**。如果你期望 ciloop 自动重试 flaky test 直到通过，你会失望——它故意不这么做。flaky test 被归类为 hand-back，原样报告给你。这是设计决策：掩盖 flaky 比 CI 红更危险。

  **standup 的 reviewed 列表可能不准**。GitHub 的 `--reviewed-by` 过滤器按 PR 最后更新时间而非你实际 review 的时间筛选，所以你很久以前 review 过的长期 PR 可能冒出来。如果噪音太大，用 `--no-reviews` 关掉。

| 技能 | 安装方式 | 核心特点 | 适用场景 | 热度 |
|------|---------|---------|---------|------|
| **BiswaViraj/agent-skills** | 插件市场一键安装 | 三合一：审查循环 + CI 循环 + 站会、bot/人分离策略 | 日常 GitHub 工作流自动化 | ⭐ 新 |
| chanmuzi/git-claw | 手动复制 | 六合一：commit + PR + issue + review + review-reply + handoff | 全流程 Git 规范管理 | ⭐ 6 |
| joshelser/claude-git-pr-skill | 手动复制 | 专注 PR review 的并行 agent 分析 | 单次 PR 审查 | ⭐ ~10 |

  BiswaViraj/agent-skills 的差异化在于「循环」——reviewloop 和 ciloop 不是一次性执行，而是循环直到目标达成。这比单次执行的 PR review 技能更适合「扔一个命令然后去喝咖啡」的使用习惯。

## 今日小结

  这三个技能适合每天和 GitHub PR/CI 打交道的开发者。安装一分钟，三个命令覆盖你每天至少 30 分钟的机械化操作——清 PR 审查、修 CI、写站会。它们不是帮你写代码的 AI 技能，而是帮你操作 GitHub 的自动化技能。reviewloop 的「bot 循环 + 人类一次 pass」策略设计得很务实，ciloop 的「本地复现优先、绝不猜测 flaky」原则更值得学习——它知道什么时候该住手。
