# react-doctor — AI 写的 React 代码，它来把关

  AI 编程助手一天能吐几百行 React 代码，人根本 review 不过来。react-doctor 是给 AI 写的 React 代码做"体检"的工具——一行命令扫出状态管理、性能、安全、可访问性四个维度的病灶，给出 0-100 的健康评分。13,409 stars，Million Software 团队（React Scan 的作者）维护，20 位贡献者，v0.7.1 刚在 7 月 4 日发布。

---

### 它解决什么问题

  用 AI 写 React 的人都会撞上三个问题：

  1. **AI 生成的 React 代码有系统性缺陷**。Claude、Cursor、Copilot 写出来的代码能跑，但经不起推敲——useEffect 依赖缺失、闭包陷阱、不必要的重渲染、XSS 漏洞。这些问题不是偶尔出现，而是 AI 的"写作习惯"——它倾向于用最短路径完成任务，不考虑长期维护成本。

  2. **人 review 跟不上 AI 的产出速度**。一个熟练的 AI 辅助开发者一天能提交几百行代码。让同事逐行 review 不现实，ESLint 只能检查语法和风格，管不了数据流、架构耦合、Context 滥用这些需要"理解代码意图"才能发现的问题。

  3. **问题积累到线上才暴露**。useEffect 缺少清理函数导致的"内存泄漏"不会在开发环境直接报错；key 用 index 写的列表在你拖拽排序时才出 bug。这些隐患埋在代码里，直到用户反馈"页面越来越卡"时才被发现。

  react-doctor 用三个设计解决了这三个问题：**60+ 检测规则覆盖 14 个类别**（状态、性能、架构、安全、可访问性、包体积、死代码），**基于 Oxlint (Rust) 引擎运行**（比传统 JS linter 快 10-100 倍），**打分制给出可量化的质量基线**（0-100 分，不是"多少个 error"这种抽象数字）。

  知名开源项目的参考分数很能说明问题：tldraw 84 分、excalidraw 84 分、supabase 69 分、sentry 64 分。如果你的项目低于 50 分，就进入了"危急"区间。

### 核心原理

  react-doctor 不是"又一个 lint 工具"。它的工作方式有三个关键区别：

  **1. 规则覆盖语义层面，不限于语法。** 传统的 ESLint 插件检查的是 AST 节点——"这里用了 var 而不是 const""那里少了分号"。react-doctor 的规则理解 React 的数据流模式：它知道 `useEffect` 的依赖数组应该包含哪些变量、能检测出闭包中引用的过期 state、能识别 Context value 变化导致的全树重渲染。这些规则来自 Million 团队在 React 性能优化领域的积累——他们开发 React Scan 时见过几乎所有常见的 React 反模式。

  **2. Rust 引擎驱动的确定性扫描。** react-doctor 基于 Oxlint（Rust 写的 JavaScript linter）构建，不依赖 LLM。这意味着：结果是确定性的——同一份代码扫描两次结果相同；速度极快——大型项目几秒内完成；零 hallucination——不会凭空捏造不存在的问题。

  **3. 按"唯一规则"计分，而不是按"问题实例数"。** 评分公式是 `Score = 100 - (unique_error_rules × 1.5) - (unique_warning_rules × 0.75)`。这意味着如果你有 100 个文件犯了同一个规则（比如都用 index 做 key），只扣一次分。你必须真正消除这个"问题类别"才能涨分，而不是修几个文件就能把分刷上去。

  **4. 从源头教 AI，不只是事后查。** 这是 react-doctor 最特别的能力。`npx react-doctor@latest install` 会把 47+ 条 React 最佳实践规则注入 Claude Code、Cursor、Codex 等 50+ 个 AI 工具的上下文——让 AI 在写代码的时候就避开这些坑，而不是写完再被 lint 报错。

### 安装配置

```bash
# 1. 一键扫描当前项目
npx react-doctor@latest

# 2. 将规则注入 AI 编码助手（Claude Code / Cursor / Codex 等 50+）
npx react-doctor@latest install

# 3. 安装 CI 门禁（GitHub Actions，PR 自动扫描）
npx react-doctor@latest ci install
```

安装到 AI 助手后，可在 `doctor.config.ts` 中按需配置规则：

```typescript
// doctor.config.ts
import { defineConfig } from "react-doctor";

export default defineConfig({
  // 关闭不需要的规则
  rules: {
    "react-doctor/no-array-index-as-key": "error",
    "react-doctor/no-unused-vars": "off",
  },
  // 排除不需要扫描的目录
  exclude: ["**/__tests__/**", "**/*.test.*"],
  // CI 模式下只报告 PR 变更引入的新问题
  ci: {
    diff: "main",
    failOnScore: 60,  // 低于 60 分禁止合并
  },
});
```

### 使用指南

#### 场景一：新项目接入，拿到第一份"体检报告"

  新项目或接手遗留代码，第一步是搞清楚代码质量基线。在项目根目录跑：

```bash
npx react-doctor@latest
```

  输出包含总分、各维度明细、每个问题的文件位置和修复建议。按优先级从 error 开始逐个消除——因为计分公式按"唯一规则"计算，每消除一个规则大类，分数就会有可见的上涨。

#### 场景二：给 Claude Code / Cursor 装上"React 驾驶执照"

  AI 写 React 代码最大的问题是"不知道什么是不好的"。`npx react-doctor@latest install` 会把规则集写入 `SKILL.md` 或 `.cursorrules`，之后 AI 写代码时会自动遵循这些规则——比如不再用 index 做 key、useEffect 自动补上依赖数组、避免在渲染中创建新对象引用。

  实际效果：在 AI 生成代码之前就拦截了最常见的问题，review 阶段少了一半"回去改一下 key"的来回。

#### 场景三：CI 流水线中的质量门禁

  `npx react-doctor@latest ci install` 一键添加 GitHub Actions workflow。每个 PR 自动扫描，只报告**本次变更引入的新问题**（不是全量 backlog），并在 PR 下评论诊断摘要。设置 `failOnScore: 60` 后，低于 60 分自动拦截合并。

  GitLab CI 也支持基础门禁。整套 CI 配置 3 分钟搞定——不需要手写 yaml、不需要配置 runner、不需要调阈值。

### 进阶技巧

  **组合技：react-doctor + react-scan。** react-doctor 做静态分析，react-scan 做运行时性能剖析。先跑 react-doctor 消除代码层面的反模式，再跑 react-scan 捕获动态渲染性能瓶颈——一个在写代码阶段防病，一个在运行时做体检，两个工具互补。

  **与 ESLint 并存而不是替代。** react-doctor 覆盖的是 ESLint 管不了的语义层面（数据流、架构、闭包陷阱）。ESLint 继续负责代码风格和基础语法。两个工具并行跑，各管各的维度。

### 常见踩坑

  **规则误报。** 部分规则（如检测 Context 滥用）依赖静态分析推断数据流，可能误判。遇到误报可以用 `// react-doctor-ignore-next-line` 单行禁用，或在 `doctor.config.ts` 中关闭整条规则。团队建议先跑一遍全量扫描、标记误报、再接入 CI——避免 CI 被误报告警淹没。

  **大型项目首次扫描可能输出几十个 error。** 正常现象。不用一次性修完——CI 模式配了 `diff: "main"` 后，只要求 PR 不引入新问题，存量问题可以慢慢修。把分数从 40 提到 60 只需要消除几个高频规则大类，工作量通常在一两天内。

## 今日小结

  react-doctor 适合所有用 React 且用 AI 辅助编程的开发者。配置只需一行 `npx react-doctor@latest install`，3 分钟后你的 AI 助手就开始写出更规范的 React 代码。它不替代人的 review，但能把 review 的精力从"这里 key 不能用 index"这种机械检查中解放出来，聚焦到架构和业务逻辑上。在一个 AI 一天写几百行代码的时代，有台自动体检机在 CI 里守着，睡得踏实。
