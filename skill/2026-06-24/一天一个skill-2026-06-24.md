# mfranzon/tdd — 把 TDD 纪律写进 AI 编程助手的肌肉记忆

  每次让 AI 帮你写代码，最怕什么？不是它不会写，而是它太会写了——一口气吐出 300 行没有测试兜底的业务逻辑，等你回过神来，已经在调试一个不知道从哪崩掉的函数。mfranzon/tdd 这个 Skill 做的事很直接：在 Claude Code 里强制执行 Red-Green-Refactor 循环，让 AI 在写任何一行实现代码之前，先把测试写好、跑红、等你确认。

![[skill/2026-06-24/cover.png]]

### 它解决什么问题

  Claude Code 在自由模式下写代码，天然倾向于"先实现再补测试"——不是因为偷懒，而是因为它的优化目标就是尽快给你一个看起来能用的结果。等你回过神来要求它补测试，它写出的测试往往只是"验证实现"而非"驱动设计"。

  这个 Skill 把流程反转了。它接管了整个开发节奏：拆需求 → 写失败测试 → 等你确认 → 写最小实现 → 自动跑全量测试 → 重构。你只需要在 RED 阶段确认测试是否合理，剩下的自动化流转。它直接往真实文件里写代码，没有临时文件、没有手动搬运——Git 就是你的撤销键。

### 核心原理

  Skill 本身是一个约 200 行的 SKILL.md 文件，安装在 `~/.claude/skills/tdd/` 下。当你执行 `/tdd "功能描述"` 时，Claude Code 加载这个 Skill，进入一个三段式状态机：

  ```
  SETUP（拆需求、检测语言/框架、生成增量列表）
    ↓
  TDD LOOP（每轮：RED → GREEN → REFACTOR）
    ↓
  WRAP-UP（汇总构建内容、建议后续工作）
  ```

  SETUP 阶段自动检测你的语言和测试框架——Python 配 pytest、TypeScript 配 vitest、Go 配标准库 testing。然后它把需求拆成小的、可测试的增量，列出清单让你审阅。

  TDD LOOP 是核心。每轮只处理一个增量：
  - **RED**：往真实测试文件里写一个失败测试，跑一遍确认变红，然后停下来等你审阅。这是唯一需要你手动确认的步骤。
  - **GREEN**：写刚好能让测试通过的最小代码，跑全量测试套件。绿了就自动进入下一步。
  - **REFACTOR**：改进代码结构、消除重复，每次改动后自动跑测试。绿了就进入下一个增量。

  这个设计很聪明——只在真正需要人类判断的地方（"这个测试测对了吗？"）暂停，其余步骤全自动流转。既保持了纪律，又不拖慢节奏。

### 安装配置

```bash
# 克隆仓库
git clone https://github.com/mfranzon/tdd.git

# 复制到 Claude Code 全局技能目录
cp -r tdd ~/.claude/skills/tdd/

# 或者用 npx skills 通用安装器
npx skills add mfranzon/tdd
```

安装后重启 Claude Code 会话即可使用。Skill 通过 `/tdd` 命令显式调用，不会自动触发——因为 TDD 是一种需要你主动选择的开发模式，不是每个场景都适用。

### 使用指南

#### 场景一：从零开始一个新功能

  假设你要给一个 FastAPI 项目添加用户注册接口。传统做法是让 Claude Code 直接写路由、模型、验证逻辑，代码写完再补测试。用 `/tdd` 的话：

  ```
  /tdd "add user registration endpoint with email and password validation"
  ```

  Skill 会先检测到 Python + pytest 环境，然后拆出类似这样的增量列表：
  1. 密码长度校验（< 8 位拒绝）
  2. 邮箱格式校验
  3. 重复邮箱拒绝
  4. 成功注册返回 201 + 用户数据

  你审阅确认后，它从第 1 条开始：写一个 `test_password_too_short` 测试 → 跑红 → 等你确认 → 写 `if len(password) < 8: raise ValueError` → 跑绿 → 重构（如有需要）→ 进入第 2 条。最终产出一组有测试护航的代码，每一步都有 git 记录可以回溯。

#### 场景二：修 Bug 先复现

  一个经典的场景：用户反馈"订单金额超过 99999 时计算错误"。你不需要自己去查代码猜原因：

  ```
  /tdd "fix order amount calculation for values above 99999"
  ```

  Skill 会先把 Bug 表述转化为一个失败的测试——`test_order_amount_above_99999`——用断言精确描述期望行为。测试跑红（证明 Bug 存在），然后写最小修复，跑绿（证明 Bug 修复）。这比"改一行代码 → 祈祷没引入新问题"的流程安全得多。

#### 场景三：重构遗留代码

  面对一段没有测试覆盖的老代码，直接改是赌博。用 `/tdd` 配合重构意图：

  ```
  /tdd "refactor the payment service to use strategy pattern, keep existing behavior"
  ```

  Skill 会先写表征测试（characterization tests）——捕捉现有行为的测试，不判断对错，只记录"现在是这样的"。然后走 RED-GREEN-REFACTOR：每次重构一个增量，测试始终绿着。这意味着你可以放心地把一个 500 行的 `process_payment` 拆成策略模式，每一步都有安全网。

### 进阶技巧

  **结合 Git 做细粒度提交**。Skill 的每个增量天然适合作为一个 commit。在每轮 REFACTOR 完成后手动 `git commit`，你会得到一组原子化的、提交信息清晰的 commit 历史——每个 commit 对应一个测试驱动的小功能增量。

  **自定义增量粒度**。如果你发现 Skill 拆出的增量太大或太小，可以在首次提出需求时加一句"每个增量控制在 20 行以内"或"按 API 端点拆分"。Skill 会据此调整分解策略。

### 常见踩坑

  **测试文件路径**：Skill 默认把测试文件放在项目已有的测试目录下（如 `tests/`）。如果你的项目结构不标准（比如测试和源码混放），第一次使用时最好先说明测试文件的存放位置，避免 Skill 猜错。

  **Go 项目的模块路径**：Go 项目使用标准库 `testing`，Skill 运行 `go test -v ./...`。如果你的项目有多个 module 或使用了 replace 指令，需要在执行前确认 `go test` 能正常运行。

  **大型增量列表**：如果一次性给的需求太泛（比如"重构整个用户系统"），Skill 拆出的增量列表可能多达 20+ 条。建议第一轮只描述核心路径，让增量控制在 5~8 条以内，后续再追加。

### 同类技能对比

| 技能 | 安装方式 | 核心特点 | 适用场景 | 热度 |
|------|---------|---------|---------|------|
| **mfranzon/tdd** | `npx skills add` 或手动复制 | 轻量、直接写真实文件、最小暂停 | 日常 TDD 开发，追求效率 | ⭐ 141 |
| NTCoding/tdd-process | 手动 symlink | 6 状态机、11 条强制规则、证据驱动 | 极度严格的 TDD 场景 | ⭐ 294 |
| glebis/tdd | 来自 glebis/claude-skills 合集 | 多 Agent 上下文隔离、垂直切片 | 大型项目、需要 Agent 隔离 | ⭐ 274 |
| morodomi/tdd-skills | Claude Code 插件 | 7 阶段、多语言专项插件、Agent Teams | 团队协作 + 多语言 | ⭐ 7 |

  mfranzon/tdd 的定位是"刚好够用的 TDD 纪律"。它不像 NTCoding/tdd-process 那样要求每步提供截图证据，也不像 glebis/tdd 那样启动多个 Agent 做上下文隔离。它适合那些认同 TDD 价值但不想被流程绑架的开发者——纪律足够让 AI 不乱来，灵活足够让你不烦躁。

## 今日小结

  这个 Skill 适合每天用 Claude Code 写业务代码的开发者。安装 10 秒，配置零依赖，上手 30 分钟就能走完第一个 TDD 循环。它解决的不是"AI 不会写代码"——AI 很会写。它解决的是"AI 写太快，你来不及思考它写得对不对"。每次 RED 阶段花 10 秒确认测试，换来每次提交都有绿色测试作为安全网。
