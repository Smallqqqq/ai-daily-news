# grill-me — 写代码之前，让 AI 先把你的方案审到无懈可击

  你用 Claude Code 说了一句话："帮我做一个用户权限管理系统"。agent 二话不说开始建表、写接口、搭页面。二十分钟后你发现它把"管理员能看所有数据"理解成了"所有用户都能看所有数据"——根子在需求阶段，agent 没追问、你没说清楚，代码全白写。

  grill-me 是 Julius Brussee（caveman 作者，82k stars）出品的 8 个个人技能之一。它的核心设计极其克制：**在 agent 动手写代码之前，先像面试官一样一个问题一个问题地质询你的方案**。不问完不罢休，问到你的目标、约束、替代方案、失败模式、回滚路径全都清晰了，才开始实现。109 stars，MIT 协议，一条命令安装。

### 它解决什么问题

  AI 编程助手有一个系统性的行为模式：**你给什么它就做什么，不问为什么。** 这个模式在简单任务上效率极高——"加一个导出 CSV 的按钮"不需要追问。但在复杂任务上，这个模式是灾难性的：

  你说"做一个支付模块"。agent 不会问你：支持哪些支付渠道？退款流程怎么走？幂等性怎么保证？对账失败了怎么补救？它只会按照自己对"支付模块"这四个字的理解，直接开始写 Stripe 集成代码。

  这不是 agent 不够聪明。正相反，它太聪明了——聪明到能用一个模糊的输入生成一千行看似合理的代码。问题出在**对话结构**上：你给了一句描述，它给了一堆代码，中间缺失了一个"方案审查"环节。

  grill-me 填补的就是这个环节。它把一个资深工程师在 code review 之前会做的事——追问需求、挑战假设、穷举边界条件——编码成了一组结构化的对话规则。

  你当然可以直接跟 Claude 说"先问我几个问题"。但一句随口的"先问我几个问题"没有校准机制、没有问题梯子、没有压力适配、没有决策地图。agent 可能随便问两个表面问题就觉得自己"问够了"，然后继续写代码。grill-me 的 300 行 SKILL.md 把这整个流程标准化了——不是"问几个问题"，而是一套完整的方案审查协议。

### 核心原理

  grill-me 的工作方式可以拆成四个阶段，每个阶段解决一个问题：

  **Phase 1：锁定目标。** 如果你说的是一句模糊的"帮我做个 dashboard"，grill-me 会先让你把目标具体化：什么数据？给谁看？刷新频率？权限？然后它用 3-6 个要点总结你的目标，让你确认"对，审的就是这个"。这一步保证了 agent 和你在审的是同一件事。

  **Phase 2：校准压力。** 这是 grill-me 区别于"随便问几个问题"的关键设计。在正式开始质询之前，grill-me 会先问两个问题：你对这个领域了解多少？（新手/能用/专家）你想要多大的压力？（轻松/标准/严厉）然后根据你的回答，调整后续所有问题的深度和语气。

  一个新手说"帮我设计一个数据库"，grill-me 会先花两句话解释"什么是规范化"，再问"你的查询模式是读多还是写多"。一个专家说同样的话，grill-me 会跳过基础，直接问"你打算用 CQRS 还是单库读写？为什么？如果 QPS 翻十倍你的方案哪里先撑不住？"

  **Phase 3：构建决策地图。** 在提问过程中，grill-me 在后台维护一份"决策地图"——目标、用户、约束、替代方案、依赖、风险、验证方式、回滚路径。这份地图不出现在对话中（除非你要求），但决定了每个下一个问题的方向。

  **Phase 4：问题梯子。** 这是整个技能的核心引擎。grill-me 按七个层级逐步深入：

  1. **目标匹配**——这件事做成了是什么样的？什么情况下不值得做？
  2. **约束现实**——什么硬约束不能动？哪个假设如果错了整个方案会垮？
  3. **选项压力**——前两个替代方案是什么？为什么选这个不选那个更无聊的方案？
  4. **执行路径**——最小可用版本是什么？什么可以先不做？
  5. **失败模式**——生产环境里怎么挂？什么边界情况会让这个方案很难看？
  6. **验证方式**——什么测试、指标、截图能证明它真的工作了？
  7. **可逆性**——哪个决策最难回滚？逃生通道在哪？

  每一层只问一个问题，每个问题附带一个推荐答案。你可以直接接受推荐答案，也可以给出自己的答案。类似于一个结构化面试——但被面试的是你的方案。

  一个值得注意的设计细节：**如果答案可以通过读代码、文档、日志、issue 获取，grill-me 会先去读，而不是问你。** 这避免了 agent 问你"这个接口叫什么"这种你也不知道的问题——它会自己去看。

### 安装配置

```bash
# 一行安装（支持 Claude Code / Codex / Cursor / Gemini CLI 等 70+ agent）
npx skills add JuliusBrussee/skills
```

安装后 grill-me 自动注册为 Claude Code skill。触发方式：

```
/grill-me
```

或者用自然语言："grill me on this plan""stress-test my architecture""challenge my design decision"。

### 使用指南

#### 场景一：新功能开工前审查方案

  你准备让 agent 实现一个"多租户数据隔离"功能。在说"开始写代码"之前，你先说一句"grill me on the multi-tenant isolation plan"。

  grill-me 会先让你确认目标："我理解你想让每个租户的数据完全隔离，租户 A 看不到租户 B 的数据。目前的方案是在每张表上加 tenant_id 字段并在查询层统一过滤。对吗？"

  确认后，它会校准你的知识水平。假设你选"Working / Standard"，接下来的问题可能是：

  > "你的查询层过滤是中间件拦截 SQL 还是在 ORM 层自动注入 WHERE tenant_id？推荐答案：ORM 层自动注入，因为中间件拦截在处理原生 SQL 和跨库查询时会失效。"

  > "如果某个租户的数据量是其他租户的 100 倍，你的单表方案会不会让那个租户拖慢其他租户？推荐答案：会。需要考虑按 tenant_id 分区或大租户独立 schema。"

  15 分钟，7 层问题走完，你的方案从"加个 tenant_id 字段"变成了一个考虑了隔离级别、大租户策略、跨租户审计日志、回滚迁移方案的完整设计。然后 agent 才开始写代码。

#### 场景二：技术选型决策

  你在纠结用 PostgreSQL 还是 MongoDB 存某个业务数据。你说"grill me on the database choice for event logging"。

  grill-me 不会直接告诉你"用 PostgreSQL"，而是：

  > "事件日志的查询模式是什么？是固定字段的聚合分析（每小时 error 数量、按用户分组的操作频率），还是半结构化字段的全文搜索？推荐答案：如果你主要做聚合分析和时序查询，PostgreSQL + 分区表比 MongoDB 更适合。"

  > "写入量级是多少？如果每秒万级以上，你打算怎么处理写入瓶颈？推荐答案：万级写入需要异步批量写入 + 缓冲队列，或者用时序数据库如 ClickHouse。"

  每个问题都给你一个推荐的思考方向，但不替你决策。最终选型是你做的，grill-me 只负责确保你没有漏掉重要的考量维度。

#### 场景三：重构风险评估

  你要把一个单体的订单模块拆成微服务。你说"grill me on the order service extraction plan"。

  grill-me 会从目标开始追问：

  > "拆分订单服务的首要目标是什么？独立部署？独立扩容？团队边界？还是三者都有？推荐答案：如果是团队边界（订单组和商品组各自迭代），只需拆服务边界，数据库可以先不拆。如果目标是独立扩容，需要先确认瓶颈确实在订单服务而非数据库。"

  > "订单表和商品表之间有外键关联。拆分后这个关联怎么处理？推荐答案：通过 API 调用替代外键，但要接受最终一致性。需要在订单服务侧缓存商品信息以应对商品服务不可用的情况。"

  > "回滚方案是什么？如果拆分后线上出问题，能在多长时间内切回单体？推荐答案：保留单体部署至少一个迭代周期，用流量灰度逐步切换，出问题一键切回。"

  这些问题是你在动手之前自己也会想的——但 grill-me 确保你一个不落地想了，而且按照正确的顺序想了。

### 进阶技巧

  **组合技：grill-me → junior-to-senior → 实现。** 先用 grill-me 把你的方案审清楚，再用 Julius 的另一个技能 `junior-to-senior` 让 agent 以"资深工程师 review 初级工程师的方案"的视角，对 grilled 之后的方案做一轮对抗性审查。两轮审查过后，方案的质量基线显著高于"直接开始写代码"。

  **动态调压：** 在 grilling 过程中随时可以说"softer""harder""teach more""skip basics"来调整节奏。如果某个领域你特别熟，可以说"harder on the database design but softer on the API layer"——grill-me 会针对性调整不同维度的问题深度。

### 常见踩坑

  **grill 时间过长导致 token 消耗。** 如果方案本身很复杂，grill 过程可能持续 20-30 轮对话。每轮对话都在累计上下文，token 成本不低。建议重要方案在独立 session 中 grill，结束后用 `/compact` 压缩上下文，或把 grilled 结果总结成一份 plan 文档再带回主 session。

  **默认压力可能偏轻。** grill-me 的默认设置是 Knowledge: Working、Pressure: Standard。如果你经验丰富且方案涉及线上核心链路，建议一开始就说"hard"——让 agent 直接进入严厉模式，跳过温和的澄清问题，直接问失败模式和回滚方案。

| **grill-me**（今日推荐）| `npx skills add JuliusBrussee/skills` | 校准压力 + 7 层问题梯子 + 决策地图 | 方案审查、技术选型、重构评估 | ⭐ 109（作者 caveman 82k） |

  grill-me 的差异化在于它只做一件事——**在代码写出来之前把方案审透**。它不是 brainstorming（发散创意），不是 spec-driven（写规范文档），也不是 code review（审查已写好的代码）。它是一个精确地插在"有想法"和"开始写"之间的质量闸门。
