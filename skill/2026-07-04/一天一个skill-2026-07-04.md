# react-doctor — AI 写的 React 代码，它来把关

  Claude Code 写 React 组件的速度快到让人上瘾——但你有没有想过，它生成的 `useEffect` 可能根本没清理定时器？它写的列表渲染可能用了数组索引做 key？它引入的第三方组件可能是一个 200KB 的重型炸弹而你完全没察觉？React Doctor 专门解决这个问题：一条命令扫描整个项目，给代码健康度打出 0-100 的分数，并逐条告诉你问题出在哪、怎么修。

### 它解决什么问题

  AI 编程助手写 React 代码有三个系统性盲区：

  1. **Hooks 语义错误**。Agent 知道 `useEffect` 怎么写，但它经常在 effect 里忘了清副作用、把不需要 effect 的计算逻辑硬塞进 effect、或者写出了依赖缺失导致的过时闭包。ESLint 的 `react-hooks/exhaustive-deps` 规则能抓依赖缺失，但抓不到"这个 effect 压根不该存在"。
  2. **性能退化是静默的**。Agent 加了一个重型依赖、忘了 memo 化计算结果、用索引做列表 key——这些都不会报错，只有用户体验在慢慢变差。
  3. **Agent 跳过了安全检查**。`dangerouslySetInnerHTML`、不安全的 URL 拼接、服务端鉴权函数的误用——Agent 专注于"让功能跑起来"，安全是它最容易跳过的环节。

  React Doctor 用 60+ 条确定性规则把这三个盲区全扫了一遍。它不是"AI 建议"，是**确定性扫描**——同一份代码、同一个版本、同一个结果。这一点对于 CI 门禁尤其关键：你不会想让一个 LLM 的判断去决定 PR 能不能合入。

### 核心原理

  React Doctor 的扫描引擎分两个并行通道：

  **Lint 通道**：60+ 条规则覆盖 5 个类别——Security（安全漏洞）、Bugs（逻辑错误）、Performance（性能退化）、Accessibility（无障碍）、Maintainability（可维护性）。规则是手写的 oxlint 插件，不是 LLM 生成的，所以结果是确定性的。同一份代码跑十次，结果完全一样。

  **Dead Code 通道**：检测未使用的文件、导出、类型定义和重复代码。这个通道在 diff 模式和 staged 模式下自动跳过（全量扫描才有意义）。

  两条通道跑完后，结果按严重程度加权计算出一个 0-100 的健康分数：

  | 分数 | 等级 | 含义 |
  |------|------|------|
  | 75-100 | Great | 代码健康，没有严重问题 |
  | 50-74 | Needs work | 存在需要关注的中等问题 |
  | 0-49 | Critical | 有严重安全或性能隐患 |

  分数的计算逻辑很简单：每个 error 级别的诊断扣 3-5 分，warning 级别扣 1-2 分。Security 和 Bugs 类别的诊断权重最高——一个 `dangerouslySetInnerHTML` 未消毒的 error 比十个"可以用更简洁的写法"的 warning 影响更大。

  **Agent Skill 的工作方式**：`npx react-doctor@latest install` 会往 `.claude/skills/` 目录写入一个 `SKILL.md`，包含完整的诊断修复指引。Claude Code 在工作时会自动加载这套知识：当你写 React 代码时，它知道什么是好的实践；当你要求审查代码时，它能逐条对照那 60+ 条规则给出建议。Skill 文件本身约 300 行，不加载时只占约 100 tokens 的元数据。

  **CI 集成**：GitHub Actions 模式下的 React Doctor 只扫描 PR 改动的文件，报告**新增**的问题——不会因为遗留代码的既有问题把 PR 卡住。这解决了"想引入代码质量门禁但不敢因为老代码把所有人卡死"的经典两难。

### 安装配置

  一行扫描，一行安装 Agent Skill：

```bash
# 1. 先跑一次全量扫描，看看项目现状
npx react-doctor@latest

# 2. 安装 Agent Skill（让 Claude Code 学会规则）
npx react-doctor@latest install
```

  安装 Agent Skill 时会自动检测你使用的 AI 编程助手（Claude Code、Cursor、Codex 等），写入对应的 skills 目录。对于 Claude Code，文件在 `.claude/skills/react-doctor/SKILL.md`。

  可选：添加 GitHub Actions 让每个 PR 自动扫描。`npx react-doctor@latest install` 过程中选择 Yes 即可自动生成 `.github/workflows/react-doctor.yml`。也可以手动创建：

```yaml
name: React Doctor
on:
  pull_request:
    types: [opened, synchronize, reopened, ready_for_review]
  push:
    branches: [main]

permissions:
  contents: read
  pull-requests: write
  issues: write
  statuses: write

concurrency:
  group: react-doctor-${{ github.event.pull_request.number || github.ref }}
  cancel-in-progress: true

jobs:
  react-doctor:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
        with:
          fetch-depth: 0
      - uses: millionco/react-doctor@v2
```

  配置文件 `doctor.config.ts`（或 `doctor.config.json`）：

```ts
import { defineConfig } from "react-doctor/api";

export default defineConfig({
  ignore: {
    rules: ["react-doctor/no-danger"],           // 关闭特定规则
    files: ["src/generated/**"],                  // 排除生成代码
  },
  rules: {
    "react-doctor/no-array-index-as-key": "error", // 升级某条规则到 error
  },
  categories: {
    Maintainability: "warn",                       // 降低某类规则到 warn
  },
});
```

### 使用指南

  #### 场景一：接手一个新项目，先搞清楚代码有多"脏"

  你接手了一个运行了两年的 React 项目，README 里写着"基于 Next.js 14"，`package.json` 里躺着 80 个依赖。你不知道代码质量怎么样，但你知道肯定有坑。

```bash
npx react-doctor@latest
```

  输出类似：

```
  React Doctor v0.5.8
  Framework: Next.js 14.2.3 · React 18.3.1 · TypeScript 5.5

  Security        3 errors, 7 warnings
  Bugs            2 errors, 5 warnings
  Performance     4 errors, 12 warnings
  Accessibility   0 errors, 8 warnings
  Maintainability 1 error, 15 warnings

  Score: 62/100 (Needs work)

  Top issues:
  • react-doctor/no-danger — src/components/Comment.tsx:42
    dangerouslySetInnerHTML without sanitization
  • react-doctor/no-array-index-as-key — src/pages/List.tsx:28
    Using array index as React key
  • react-doctor/effect-no-cleanup — src/hooks/usePolling.ts:15
    setInterval in useEffect without cleanup
```

  十分钟内你就有了一份完整的技术债务清单。不需要通读代码，不需要跑项目，一个命令搞定。

  #### 场景二：AI 写完代码后，先过一遍 Doctor 再提交

  你让 Claude Code 写了一个新的 Dashboard 页面，它写了一个数据轮询的 hook、五个组件、三张图表。代码看起来没问题，跑起来也没报错。

  在你 `git add` 之前，先跑一下：

```bash
npx react-doctor@latest --staged
```

  React Doctor 在 staged 文件上扫出了两个问题：

  - `usePolling` hook 里的 `setInterval` 在组件卸载时没有清理——内存泄漏
  - Dashboard 组件引入了一个 180KB 的图表库，但其实只需要它里面的一个 12KB 的子模块

  第一个问题是 Claude 自己不会发现的（代码能跑，React 不会报错，月复一月内存慢慢涨）。第二个问题是你肉眼根本不会注意的（import 语句看起来完全正常）。两个问题在提交前就被拦截了。

  #### 场景三：PR 审查不必再逐条检查 React 最佳实践

  你团队有 5 个前端开发，每个人对 React 的理解深浅不一。你作为 reviewer，每次 PR 除了看业务逻辑，还要检查：key 用得对不对、effect 清理了没有、有没有不必要的 re-render、服务端数据有没有在前端暴露。

  把 React Doctor 接入 CI 后，这些检查全部自动化。PR 提交后，GitHub Actions 运行 React Doctor，只扫描改动的文件。如果有新增问题，直接在 PR 评论里贴出行号和修复建议，同时给 commit status 标红。

  你现在 review PR 时只需要关注业务逻辑和架构设计。那些"useEffect 没清理"、"数组索引用作 key"之类的机械检查，React Doctor 比你更细心——它永远不会因为赶时间漏掉一个。

  #### 场景四：渐进式治理——从"门禁全关"到"零新债"

  你们项目的历史代码有 200+ 个 Doctor warning。把门禁直接设为 error 级别等于把所有 PR 都卡死。React Doctor 的渐进式策略让你分三步走：

  第一步，全量扫，摸底：

```yaml
- uses: millionco/react-doctor@v2
  with:
    blocking: none    # 只报告，不拦截
    scope: full       # 全量扫描
```

  第二步，只拦截新增问题：

```yaml
- uses: millionco/react-doctor@v2
  with:
    blocking: error   # 新引入的 error 拦截
    scope: changed    # 只扫变动文件（默认）
```

  第三步，收紧到 warning 级别也拦截：

```yaml
- uses: millionco/react-doctor@v2
  with:
    blocking: warning
    scope: changed
```

  每一步之间隔两周，给团队时间清存量。三到四个迭代后，你们的"新债"为零——任何违反规则的新代码都合不进去。

### 进阶技巧

  **和 ESLint/oxlint 组合使用**。React Doctor 提供了独立的 oxlint 和 ESLint 插件，可以把 60+ 条规则直接嵌入到你现有的 lint 流程里。lint 阶段就拦截，不需要等到 CI。配置方式：

```bash
npm install -D @react-doctor/eslint-plugin
```

```js
// eslint.config.js
import reactDoctor from "@react-doctor/eslint-plugin";
export default [
  reactDoctor.configs.recommended,
];
```

  **用 `--fail-on` 控制 CI 行为**。不同的 CI 场景需要不同的失败策略：

```bash
npx react-doctor@latest --fail-on error   # 只有 error 才失败（宽松）
npx react-doctor@latest --fail-on warning # warning 也失败（严格）
npx react-doctor@latest --fail-on none    # 永不失败（纯报告模式）
```

  **用 `--category` 做专项审计**。比如你只想检查性能问题：

```bash
npx react-doctor@latest --category performance --verbose
```

  或者只在 React 18 升 19 时检查废弃 API：

```bash
npx react-doctor@latest --category maintainability --verbose
```

  **离线模式**。CI 环境访问外网受限时，用 `--offline` 跳过分数 API 和分享 URL：

```bash
npx react-doctor@latest --offline
```

### 常见踩坑

  **`fetch-depth: 0` 是必须加的**。GitHub Actions 的 `actions/checkout` 默认只拉取最近一次 commit，React Doctor 的 diff 模式需要知道 PR 的合并基准在哪里。不加 `fetch-depth: 0`，它会扫全量文件而不是只扫改动——结果就是你的 PR 被 200 个历史遗留问题卡住了。这个问题在 setup 文档里有明确说明，但很多人没注意看。

  **不是替代 ESLint，是补 ESLint 的盲区**。React Doctor 的规则和 ESLint 的 React 规则有交集但不重叠。ESLint 擅长语法层面的检查（"你没加 deps"），React Doctor 擅长语义层面的检查（"这个 effect 不需要存在，你应该用计算属性"）。两者并存才是完整的代码质量网。

  **Modified MIT License 的限制**：React Doctor 对个人开发者和大多数公司免费。但如果你的核心业务是 AI 训练或者超大规模商业使用（具体阈值见 LICENSE 文件），需要向 Million Software 申请书面许可。对绝大多数团队来说这不构成限制，但法务审 license 的时候需要知道这一点。

  **Skill 安装路径**：`npx react-doctor@latest install` 会把技能装在**项目根目录**的 `.claude/skills/` 下，不是全局 `~/.claude/skills/`。这意味着每个项目需要单独安装。设计意图是让每个项目的 React Doctor 版本和配置独立管理——不同项目的 React 版本不同，规则基线也不同。

## 今日小结

  React Doctor 适合所有用 AI 写 React 代码的开发者——不管你用 Claude Code、Cursor 还是 Codex。安装一条命令，之后 AI 写的每个 hook、每个组件、每个 import 都会对照 60+ 条规则接受检查。它给了 AI 一份你不一定有耐心逐条口述的 React 生产经验。花两分钟装好，之后 code review 时不用再逐条检查 useEffect 有没有清理、列表 key 用得对不对——那些坑，Million 团队的 20 位贡献者已经帮你编成确定性规则了。
