# SkillOpt — 用训练神经网络的纪律，训练你的 Agent Skill

  你给 Claude Code 装了几个 skill，起初跑得挺好。两个 sprint 之后，代码库变了、API 升级了、你的工作流也变了——那些 skill 开始给出过时的建议，触发越来越不稳定，但你哪有时间一个个去重写？SkillOpt 把这件事自动化了：它把 skill 文档当作"可训练的模型参数"，用 rollout 收集证据、用优化器模型提出修改、用验证集把关——像训练神经网络一样训练你的 skill。11,862 stars，MIT 协议，微软官方维护，30 位贡献者，v0.2.0 刚在 7 月 2 日发布，新增了夜间自动进化引擎 SkillOpt-Sleep。

### 它解决什么问题

  Agent skill 的手工维护有三个死结，每一个用 Claude Code 超过一个月的人都会撞上：

  1. **写完就过时**。skill 是在某个时间点、针对某个代码库状态写的。代码库在变——依赖升级了、API 改了、团队规范调整了——skill 不会自动跟着变。你上个月精心调试的那份 React 编码规范，这个月已经有 30% 的规则不适用了，但你不知道是哪 30%。

  2. **改进全凭感觉**。skill 里加一句话、改一段描述——改了之后真的更好了吗？没有量化反馈。你可能花了半小时"优化"一段 skill，结果 agent 的表现反而更差了，因为新指令和旧指令冲突了。

  3. **一个人调好了，没法复制给团队**。你花了三天调出一套对某个项目有效的 skill 组合，换一个项目、换一个模型、甚至换一个 agent 平台（从 Claude Code 切到 Codex），那些 skill 的效果就大打折扣。

  SkillOpt 用一个核心洞察解决了这三个问题：**skill 文档本质上就是 agent 的"外部可训练状态"**。就像神经网络的权重可以通过反向传播优化，skill 文档也可以通过" rollout → 反思 → 编辑 → 验证"的循环来自动进化。而且优化后的 skill 是一个纯文本文件（通常 300-2000 tokens），部署时零额外推理开销——agent 该用什么模型还是用什么模型。

  实际数据比口号更有说服力。在 6 个 benchmark、7 个目标模型、3 种执行环境（直接对话、Codex CLI、Claude Code CLI）上，SkillOpt 在所有 52 个评测组合中都是最优或并列最优。在 GPT-5.5 上，平均准确率提升 **+23.5 个百分点**（直接对话）、**+24.8 个百分点**（Codex agent 循环）、**+19.1 个百分点**（Claude Code）。

### 核心原理

  SkillOpt 把 skill 训练建模成一个严格的优化循环，而不是"让 LLM 自己随便改"。这个循环包含四个关键组件：

  **1. Rollout（前向执行）**：冻结的目标模型（比如你用的 Claude Sonnet）带着当前的 skill 去执行一批任务，记录完整的轨迹——每一步的消息、工具调用、验证器反馈和最终得分。这个阶段就像深度学习的 forward pass，只不过收集的不是梯度，而是"这个 skill 指导下的 agent 行为表现"。

  **2. Reflect（反思分析）**：一个独立的优化器模型（通常用更强的模型，比如 Opus）阅读 rollout 的轨迹，分析成功和失败的模式。关键设计是：成功和失败的样本分开分析。这样做是为了避免"一刀切"式的修改——成功的经验要保留，只有反复出现的失败模式才触发编辑。

  **3. Edit（受限编辑）**：优化器模型对 skill 文档提出结构化的增/删/改操作。这里有一个核心约束——**文本学习率**（textual learning rate），限制了每轮可以修改的幅度。就像训练神经网络时学习率太大会导致震荡甚至发散，没有预算限制的 skill 编辑会把之前学到的有用规则一并覆盖掉。

  **4. Gate（验证把关）**：候选 skill 必须在留出的验证集上严格优于当前版本才会被接受。这不是"看着差不多就通过"，而是数值上的硬性要求。被拒绝的编辑进入缓冲区，作为负面反馈帮助优化器避免重复犯错。

  整个训练循环结束后，输出的只是一个 `best_skill.md` 文件。部署时不需要优化器模型、不需要验证集、不需要任何额外推理——目标 agent 加载这个文件就跑。

  v0.2.0 新增的 **SkillOpt-Sleep** 把这个循环做成了夜间自动运行的后台引擎：采集（harvest）白天的 agent 会话记录 → 挖掘（mine）高频失败模式 → 重放（replay）关键任务 → 整合（consolidate）验证通过的 skill 更新。第二天早上打开 Claude Code，skill 已经根据昨天的实战反馈自动进化过了。

### 安装配置

```bash
# 安装 SkillOpt（需要 Python 3.10+）
pip install skillopt

# 验证安装
skillopt --help

# 启动 WebUI 监控面板（可选）
pip install -e ".[webui]"
python -m skillopt_webui.app
```

  配置一个训练任务需要指定三个要素：

```yaml
# config.yaml — 训练配置示例
target_model: gpt-5.5          # 被优化的目标模型（冻结，不更新权重）
optimizer_model: claude-opus-4  # 负责提出 skill 编辑的优化器模型
benchmark: searchqa             # 用于训练和验证的 benchmark
backend: claude_code_exec       # 执行后端：openai_chat / claude_chat / codex_exec / claude_code_exec
learning_rate: 4                # 文本学习率：每轮最多 4 次编辑操作
epochs: 10                      # 训练轮数
batch_size: 8                   # 每轮 rollout 的样本数
```

  训练命令：

```bash
# 启动训练
skillopt train --config config.yaml

# 用 SkillOpt-Sleep 启动夜间自动进化
skillopt-sleep --watch-dir ~/.claude/sessions/

# 导出优化后的 skill
skillopt export --run-id <run_id> --output best_skill.md
```

  生成的 `best_skill.md` 直接复制到 `~/.claude/skills/` 或 `.claude/skills/` 目录即可使用，部署零依赖。

### 使用指南

#### 场景一：新接手一个项目，agent 表现稀烂

  你刚加入一个中型 Python 后端项目，Claude Code 写代码的质量忽高忽低——有时候能写出符合项目规范的好代码，有时候连 import 路径都搞错。

  **操作步骤**：

  1. 收集 20-50 个该项目的典型任务（修 bug、加功能、写测试），准备对应的验收标准
  2. 写一份初始 skill（不需要完美，把基本规范写进去就行），作为训练起点
  3. 跑一轮 SkillOpt 训练，让它自动发现哪些规则有效、哪些需要细化、哪些需要补充

  **预期结果**：训练后的 skill 在项目典型任务上的准确率提升 10-20 个百分点。最关键是——它发现了一些你没想到要写进 skill 的规则，比如"这个项目所有 ORM 查询都要加 `.select_related()` 避免 N+1"。

#### 场景二：团队有多个 agent 平台，skill 需要跨平台复用

  你的团队有人用 Claude Code、有人用 Codex、有人用 Cursor——之前你为 Claude Code 调好的 skill，在 Codex 上效果打七折。

  **操作步骤**：

  1. 用 SkillOpt 在一个平台上训练 skill（比如 Claude Code 后端）
  2. 导出 `best_skill.md`，直接在另一个平台加载测试

  SkillOpt 论文验证过：**跨 harness 迁移有效**。在 Codex 上训练的 SpreadsheetBench skill，直接迁移到 Claude Code 里使用，准确率仍然有 +31.8 个百分点的提升。skill 学到的是"解决问题的通用方法"，不是"某个平台的特殊用法"。

#### 场景三：日常使用，让 skill 自己跟着你的工作进化（SkillOpt-Sleep）

  你每天用 Claude Code 写代码、做 code review、重构。你不想专门花时间"训练 skill"，但希望 skill 能自动从你的日常使用中学习。

  **操作步骤**：

  ```bash
  # 配置 SkillOpt-Sleep 监控你的 Claude Code 会话目录
  skillopt-sleep --watch-dir ~/.claude/sessions/ --schedule daily

  # 或者配置为系统定时任务
  # macOS/Linux crontab:
  # 0 3 * * * skillopt-sleep --watch-dir ~/.claude/sessions/
  ```

  **预期结果**：每天早上打开 Claude Code，skill 已经自动根据昨天的实战反馈更新过了。比如你在昨天发现 agent 总是在某个特定的代码模式上犯错，SkillOpt-Sleep 已经在夜间把针对性的规则加进了 skill。整个过程不需要你手动写一行 skill 配置。

### 进阶技巧

  **用更强的优化器模型训练弱模型的 skill**。SkillOpt 的一个反直觉发现：优化器模型不一定需要和被优化的目标模型同等级。用 GPT-5.5 做优化器训练 GPT-5.4-mini 的 skill，效果比自己训练自己好得多——因为更强的优化器更能从 rollout 轨迹中提取出可复用的经验。但即使目标模型自己做自己的优化器，只要编辑被约束、缓冲、验证，也能发现有用的改进（+10.4 个百分点）。

  **skill 可以当"经验包"在团队间共享**。SkillOpt 训练出来的 `best_skill.md` 就是一个纯文本文件。你可以把它提交到团队的 Git 仓库，新同事 checkout 就能用。也可以按场景维护多份 skill："写后端代码时用这套规则"、"写前端组件时用那套规则"、"做 performance review 时用另一套"。skill 之间不冲突——Claude Code 根据上下文自动加载相关的 skill。

### 常见踩坑

  **初始 skill 别写太烂**。SkillOpt 是"优化"而非"从零发明"。如果你给一份完全空白的 skill，训练会非常慢（因为 rollout 阶段 agent 没有有效指引，产生的轨迹大多是噪音）。至少写 100-200 字的初始指令，列出最基本的规范和约束。

  **验证集的代表性决定最终效果**。验证集的任务要和实际使用场景匹配。如果你的验证集全是简单的 CRUD 操作，训练出来的 skill 在处理复杂架构问题时表现会很差。建议从最近一周的真实任务中抽 30% 作为验证集。

  **文本学习率别设太大**。学习率是每轮允许的最大编辑次数。设太大（比如 20），优化器会倾向于"大修大改"，结果往往是把原本有效的规则也覆盖了。作者建议从 4 开始，效果不好再调。

### 同类技能对比

| 技能 | 安装方式 | 核心特点 | 适用场景 | 热度 |
|------|---------|---------|---------|------|
| **SkillOpt**（今日推荐）| `pip install skillopt` | 确定性优化循环 + 验证把关 + 夜间自动进化 | 长期维护 skill、跨平台复用、团队共享 | ⭐ 11.9K |
| EvoSkill | `git clone` + 手动配置 | 遗传算法式进化、测试用例驱动 | 有完整测试套件的项目 | ⭐ 960 |
| SkillAnything | `npx skills add` | 一次性从描述生成 skill | 快速从零创建新 skill | ⭐ 445 |

  SkillOpt 和 EvoSkill 的区别在于方法论根基：EvoSkill 用的是遗传算法思路（生成多个 skill 变体 → 测试淘汰 → 保留最好的），SkillOpt 用的是深度学习优化思路（rollout → 梯度式编辑 → 验证把关）。前者更像是"养蛊"，后者更像是"训练"。两种思路不互斥，但对于需要持续维护的 skill，SkillOpt 的夜间自动进化机制（SkillOpt-Sleep）提供了更省心的长期方案。

## 今日小结

  SkillOpt 适合任何装了超过 3 个 agent skill 的开发者。如果你只有一个 skill，你手动维护就够了。但当 skill 数量增长到 5 个、10 个、20 个——每个都需要随着项目变化而调整——你必然面临"skill 维护债"。SkillOpt 用一套可重复、可验证的优化流程替代手工调参：你今天花 30 分钟配好训练配置，以后每次代码库发生大变化时跑一轮，skill 就能跟上。v0.2.0 的 SkillOpt-Sleep 更是把这件事变成了全自动——你的 skill 在你睡觉的时候自我进化，你醒来的时候直接用新的。
