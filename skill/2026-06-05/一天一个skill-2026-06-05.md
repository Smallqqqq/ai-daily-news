# agents-best-practices — 你的 Agent 是草台班子还是正规军？

  2026 年是 Agent 之年，每天都有无数人在搭建 AI Agent 系统。但大多数人只关注 prompt 写得好不好、模型选哪个，却忽略了 Agent 真正出问题的地方——**运行时架构**。这个 Skill 不是教你写 prompt 的，而是教你设计一个可靠的 Agent 运行系统：模型只负责提议，真正执行、校验、授权、记录的是外层的 harness（控制面）。1669 个 star，5 月 15 号才发布，三周就获得这么高的认可，因为它切中了一个真实痛点。

![[cover.png]]

### 它解决什么问题

  你搭了一个 Agent，让它调用 API、操作数据库、发消息。起初跑得不错，但用着用着就出问题：Agent 陷入死循环烧 token、上下文压缩后忘了自己在干什么、权限过宽导致误操作、出了问题找不到日志排查。这些问题本质上都不是模型能力问题，而是 **你没有给 Agent 设计一个合格的运行时系统**。

  这个 Skill 帮你从零开始设计一个生产级的 Agent harness，覆盖：工具设计与权限分级、规划模式与审批门禁、上下文压缩与状态保持、工作流编排与预算控制、可观测性与安全评估——以及一个可直接套用的 MVP 蓝图模板。适用于 Claude Code、Codex、Gemini CLI 等所有 Agent 平台。

### 核心原理

  这个 Skill 的核心立场只有一句话：**"The harness acts, not the model."** 模型只负责提议下一步做什么，真正的执行、校验、授权、记录全部由 harness（控制面代码）完成。

  架构模型非常简洁：

```
用户/任务
  → 指令和上下文构建器
  → 模型调用
  → 工具/动作提议
  → Schema 校验
  → 权限决策
  → 执行或暂停审批
  → 结构化观测结果
  → 上下文更新
  → 在预算范围内重复或结束
```

  与常见做法最大的区别在于：**权限校验在模型之外**。模型提议调用某个工具，harness 先校验参数是否符合 schema，再检查当前风险等级是否允许执行，高风险操作（外部通信、资金交易、销毁操作等）必须走审批门禁。这就是"draft and commit are separate"——草稿和提交是两回事。

### 使用指南

#### 场景一：从零设计一个领域 Agent 的 MVP 蓝图

  这是最常用的场景。你有一个业务需求（如"做一个客户续约风险分析 Agent"），但不知道从哪开始。

  **触发条件**：向 Agent 描述你的领域需求，如"Build an agent for account renewal risk."

  **预期结果**：Skill 会生成一个完整的 MVP harness 蓝图，包含：
  - 最小可行版本的边界和假设
  - 核心 Agent 循环设计
  - 工具注册表（每个工具的 schema、风险等级、权限策略）
  - 上下文/内存/压缩策略
  - 权限分级矩阵
  - evals 和 launch gate 标准

  示例输出片段：

```
Core loop:
  user/task -> context builder -> model call -> typed tool call
  -> schema validation -> permission check -> execution or pause
  -> structured observation -> next step or final brief

Minimal tools:
  - read_account_profile      risk: read_private_data
  - list_support_tickets      risk: read_private_data
  - fetch_usage_summary       risk: read_private_data
  - draft_customer_email      risk: draft_external_message
  - request_approval          risk: approval_gate

Launch gate:
  20 historical accounts, trace review, no unapproved external
  sends, human acceptance on ≥80% of draft actions.
```

#### 场景二：审计现有 Agent 的架构问题

  你有一个跑了一段时间的 Agent，但它不稳定——有时无限循环、有时压缩上下文后忘了决策、有时输出莫名其妙的结果。

  **触发方式**：描述你的问题，如"Our research agent sometimes runs tools forever and forgets why it made a decision after context compaction. Audit the harness."

  **预期结果**：Skill 会按运行时层面（非 prompt 层面）逐项诊断，给出修复优先级：

```
Failure points (runtime-level, not prompt-level):
  - no hard step/tool/time/cost budget
  - compaction preserves prose but loses active approvals
  - tool results are unbounded and mix trusted/untrusted data
  - no event trace for model output -> tool call -> observation

Fix order:
  1. Add loop budgets and termination reasons.
  2. Store plan, approvals, todos, and artifacts outside the prompt.
  3. Make compaction rehydrate active state, not chat history.
  4. Add evals for injection, missing tool result, timeout, budget exhaustion.
```

#### 场景三：设计工具权限和外部连接器

  当你需要让 Agent 对接真实系统（Slack、Linear、数据库、部署 API 等），但不知道如何安全地暴露这些能力。

  **触发方式**："The ops agent needs Slack, Linear, Google Drive, and our internal deploy API. How should tools and permissions work?"

  **预期结果**：Skill 会输出按风险分级的工具设计策略：

  - 只读操作（scoped reads）：可自主执行
  - 草稿操作（drafts）：可自主执行但需标注
  - 外部通信/写入/部署/资金/特权操作：需要 approval record 在模型之外
  - **不要**暴露 `send_message`、`write_database`、`run_command` 这类宽泛工具，每个操作必须封装为窄类型工具，带结构化结果和确定性权限检查

### 进阶技巧

  **MVP 优先，不要一上来就搞多 Agent 系统。** Skill 明确强调：在单个 Agent 的 loop 被可测量的 evals 证明不够用之前，不要引入多 Agent 架构。大多数场景下一个设计良好的单 Agent harness 就够了，多 Agent 只增加复杂度。

  **上下文压缩时保留"工作状态"而非"聊天记录"。** Skill 提出了一个关键设计：auto-compaction 应该重新水合（rehydrate）活跃状态——当前计划、待审批项、待办列表、已变更的 artifacts——而不是只保留对话摘要。这解决了很多 Agent 压缩上下文后"失忆"的根因。

### 常见踩坑

  **陷阱一：把 prompt 当安全边界。** 很多人以为在 prompt 里写"不要执行危险操作"就安全了。Skill 明确指出：需要代码层面强制执行的规则，不能依赖 prompt 文本。权限校验必须在模型之外。

  **陷阱二：工具设计过宽。** 比如直接给 Agent 一个 `execute_sql` 工具而不是 `read_user_by_id`、`update_order_status` 这样的窄工具。宽工具让模型有太多自由度，既难调试也难审计。Skill 要求每个工具必须有严格 schema、类型化和可审计。

  **陷阱三：把 web 抓取的内容当可信指令。** Skill 特别提醒：不要将从网页、邮件、PDF、日志或连接器获取的描述当作可信指令。这些内容可能包含注入攻击。

## 今日小结

  如果你正在搭建一个 AI Agent——不管是编码 Agent、客服 Agent、金融分析 Agent 还是内部运维 Agent——这个 Skill 值得花 20 分钟装上看一遍。它不会教你写 prompt，而是教你设计 Agent 外面的那层"控制面"：什么时候该审批、什么时候该暂停、怎么防止无限循环、怎么压缩上下文不丢状态、怎么设计安全的工具权限。这些才是 Agent 从 demo 走向生产的关键。配置 2 分钟（一行 npx 命令），但能帮你避开 Agent 开发中最常见的十个坑。今天的 Agent 开发，最缺的不是更强的模型，而是更可靠的运行时架构。
