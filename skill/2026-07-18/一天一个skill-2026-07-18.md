# workflow-orchestrator — 从一句话到一个团队，把复杂任务自动拆解、并行执行、验证收尾

  你用 Claude Code 说了一句"帮我做一个全栈计算器，前端 Next.js 加 Tailwind，后端 Python uv 项目，每个阶段加验证"。如果不加任何编排工具，Claude Code 会怎么做？一个接一个——先搭前端，再写后端，再补测试，再验证。每一步等前一步完成，前端的上下文和后端的上下文搅在一起，token 越堆越多，到最后几轮对话质量直线下降。

  workflow-orchestrator 改变的不只是"快慢"，而是整个执行范式：**把一句话拆成一棵任务树，每个分支派一个专职 agent 并行执行，执行完自动验证、自动收尾。** 75 stars、26 个 release、14 个 hook 覆盖 6 个生命周期节点，v2.1.1 刚在 5 月 21 日发布，支持双模执行（隔离子 agent + 协作 Agent Teams），是 Claude Code 生态中把"多 agent 编排"这件事做得最系统化的插件。

### 它解决什么问题

  Claude Code 的原生 agent 工具（`Task` / `Agent`）已经很强大，但缺少一层"编排器"：你每次都要手动决定：这个任务拆成几步？哪几步可以并行？每步派哪个 agent？什么时候该验证？验证不通过怎么办？

  这个手动决策的过程有三个代价：

  **1. 决策疲劳。** 拆任务是认知负荷最高的事——给一个复杂需求判断哪些子任务有依赖、哪些独立、按什么顺序执行。人是能做，但每次都要做一遍，而且容易漏。

  **2. 串行执行的沉默成本。** 不改编排器的话，大部分 Claude Code 用户会让 agent 串行做完所有事——前端、后端、测试、部署，一件接一件。但其实后端 API 和前端 UI 在约定好接口后完全可以并行开发。串行浪费的不是算力，是你在电脑前等的时间。

  **3. 上下文膨胀。** 一个大 session 里堆积了六个阶段的上下文，越到后面 agent 越"健忘"。而把每个阶段分派给独立子 agent，每个子 agent 只带自己的任务上下文，干净利落。

  workflow-orchestrator 做的是把这三件事自动化：**决策（拆不拆、怎么拆）→ 执行（谁来做、并行还是串行）→ 质量（验证通过才算完）。**

### 核心原理

  它的实现不靠 AI prompt 工程，而是一套**钩子（hook）驱动的执行框架**。14 个 hook 脚本分布在 Claude Code 的 6 个生命周期节点上，构成了一个完整的编排管线：

  **6 个生命周期节点：**

  | 节点 | 钩子数 | 做什么 |
  |------|--------|--------|
  | SessionStart | 1 | 注入编排器存根（~1.1KB），指向 `/workflow-orchestrator:delegate` |
  | PreToolUse | 3 | 软执法：主 agent 绕过委派时逐级升级提醒（静默→提示→警告→强提醒） |
  | PostToolUse | 4 | Python 代码验证（Ruff/Pyright）+ 委派后归零计数器 + 任务图深度检查 |
  | UserPromptSubmit | 1 | 每轮用户输入时重置软执法计数器 |
  | SubagentStop | 2 | 子 agent 停止时提醒更新任务状态 + 触发验证 |
  | Stop | 1 | 会话停止时追踪轮次时长 |

  **两阶段执行管线：**

  **阶段 0：Plan 模式规划。** 主 agent 进入 `EnterPlanMode`，分析任务复杂度（单步 vs 多步）。对多步任务：拆成原子阶段 → 做依赖分析 → 通过关键词匹配分配 8 个专职 agent → 创建并行波次安排 → 用 `TaskCreate` 建立结构化任务图（含波次 ID、agent 分配、依赖关系）。

  **阶段 1：双模执行。** 检测 `TeamCreate` 工具是否可用（环境变量 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 开启），自动选择：
  - **子 agent 模式（默认）：** 隔离的并行 `Agent` 实例，每波次完成后 `DONE|{path}` 传回结果。上下文高效，适合大多数场景。
  - **Team 模式（实验性）：** `TeamCreate` + `Agent(team_name=...)` + `SendMessage`，agent 之间实时通信、共享任务列表、自协调。适合需要 agent 之间来回讨论的复杂协作任务。

  **8 个专职 agent 目录：**

  | Agent | 专长 |
  |-------|------|
  | code-cleaner | 代码清理、格式化、重构 |
  | test-writer | 编写和维护测试用例 |
  | architect | 系统架构、设计模式、技术选型 |
  | devops-engineer | CI/CD、Docker、基础设施配置 |
  | security-auditor | 安全审查、漏洞扫描 |
  | documentation-writer | API 文档、README、注释生成 |
  | code-reviewer | 代码审查、PR 反馈 |
  | implementer | 通用功能实现 |

  **软执法的设计哲学。** 这是 v2.1.1 最值得玩味的设计决策——不硬性阻断主 agent 的工具调用，而是通过**逐级升级的 stderr 提醒**来引导行为。第一跳：静默无视；第二跳：轻声提示；第三跳：明确警告"你正在失去规划、并行化和上下文隔离"；第四跳及以上：保持强提醒。计数器每轮用户输入后归零，委派执行后也归零。这个设计避免了"装了就卡死"的用户体验，同时保持了足够的纪律压力。

### 安装配置

  前提：Python 3.12+、uv、bun、jq（可选）。

  **方式一：插件安装（推荐，30 秒）：**

  ```bash
  # 添加市场
  claude plugin marketplace add barkain/claude-code-workflow-orchestration

  # 安装插件
  claude plugin install workflow-orchestrator@barkain-plugins
  ```

  **方式二：项目级手动安装：**

  ```bash
  git clone https://github.com/barkain/claude-code-workflow-orchestration.git
  cd /path/to/your/project
  /path/to/repo/install.sh
  ```

  装完重启 Claude Code。可选：运行 `/workflow-orchestrator:add-statusline` 启用状态栏显示，或配置 `outputStyle` 为 `technical-adaptive` 以获得优化过的输出风格。

  **开启 Agent Teams 双模（实验性）：**

  ```bash
  export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
  ```

  设置后，编排器自动在 `TeamCreate` 可用时切换到团队模式：agent 之间实时通信、自协调任务。

### 使用指南

  #### 场景一：全栈项目一键编排——从一句话到完整应用

  你对着 Claude Code 说：

  > "create a simple calculator app with basic math operations. add a nice UI and use NextJS/Tailwind to build this out. the backend should be implemented in python as a modern uv project. add verification steps after each phase."

  安装 workflow-orchestrator 后，这里发生的不是串行执行，而是：

  1. 主 agent 进入 Plan 模式，把需求拆成：前端 UI 搭建 → 后端 API 开发 → 前后端联调 → 整体验证
  2. 依赖分析判定前端和后端互不依赖 → 归入同一并行波次
  3. `architect` agent 规划项目结构，`implementer` 写前端，`implementer`（另一个实例）写后端——三个并行跑
  4. 第一波完成后，`test-writer` 验证联调结果，`code-reviewer` 审查代码质量
  5. 全部通过后自动收尾

  你在这期间没做任何事。你说了 55 个单词，编排器替你拆成了 4 个阶段、调度了 5 个 agent、协调了两波并行执行、做完了验证。

  ![[img_team_graph.png]]
  （Plan 模式输出的依赖图和并行波次——一目了然谁在做什么、谁在等谁）

  #### 场景二：大型代码审查——并行派多个审查 agent

  你的 PR 改了 12 个文件，涉及认证逻辑、数据库 schema、API 接口三个纬度。手动审查要花半小时以上，而且不同纬度需要不同的审查思路。

  > "review this PR for security issues, code quality, and test coverage"

  编排器识别出三个独立维度——安全审查、代码质量、测试覆盖——分配给 `security-auditor`、`code-reviewer`、`test-writer` 三个 agent，并行跑。三个 agent 各自带着自己的审查视角和对应的文件子集，互不干扰。五分钟后三份报告汇总成一份——安全问题 2 个（高危 1 个）、代码质量问题 4 个、测试覆盖缺口 3 处。

  #### 场景三：Team 模式——让 agent 之间互相讨论

  有些任务不适合"各干各的然后汇总"。比如架构重构需要 agent 之间来回讨论 trade-off。开启 Team 模式后：

  ```bash
  export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
  ```

  > "重构用户权限系统，从基于角色的 RBAC 迁移到基于属性的 ABAC"

  编排器创建团队：`architect` 负责整体设计，`implementer` 负责代码迁移，`security-auditor` 负责验证权限边界没有遗漏。三个 agent 通过 `SendMessage` 实时通信——`implementer` 遇到不确定的权限判断逻辑，直接发消息问 `architect`，`architect` 回复后 `security-auditor` 同步收到上下文。整个过程像三个开发者在 Slack 上讨论——不需要你当传话筒。

  ### 进阶技巧

  **软执法的开关节奏。** 有时你需要快速验证一个想法，不想走完整的委派流程。`/workflow-orchestrator:bypass` 可以临时关掉软执法：agent 自由调用工具而不触发提醒。验证完再回到委派模式。这比"装了就绕不开"的硬阻断设计灵活得多。

  **token 预算节省来自"上下文隔离"。** 编排器的一个被低估的价值：每个子 agent 只带自己的任务上下文启动，看不到其他阶段的细节。一个 8 阶段的复杂任务如果全在主 session 里串行跑，上下文中会堆积所有 8 个阶段的中间产物——代码片段、错误日志、讨论记录。子 agent 模式让每个 agent 只看到自己需要的那一小块上下文，token 使用量线性增长而非指数膨胀。

  **结合 `CLAUDE_TOKEN_EFFICIENCY=1` 进一步瘦身。** 编排器内置了 token 高效的 CLI 指南（通过 `inject_all.py` 的 `token_efficient_cli.md` 注入），教 agent 用紧凑标志和命令模式。配合 workflow-orchestrator 的上下文隔离，总 token 消耗可以进一步压缩。

  ### 常见踩坑

  **大量并行 agent 会导致上下文限制耗尽。** 这是目前已知的最大限制。如果你让编排器一次派 15 个 agent 并行审查代码，每个 agent 都会在短时间内消耗完上下文限制，导致只有部分任务完成。barkain 在 Issue #22 中确认了这个问题，并提出了三个改进方向：规划器预先评估数据量、限制单波次 agent 数量、子 agent 输出写入文件而非返回主会话。目前的实用解法：用 `CLAUDE_MAX_CONCURRENT=8`（默认值，可调低到 4 或 5）控制并行度，或者手动把大任务拆成更小的波次。

  **Team 模式是实验性的。** `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 需要 Claude Code 的实验性功能支持，可能存在不稳定的情况。对于大多数工作流，默认的子 agent 模式（`subagent`）已经足够高效。只在需要 agent 之间实时协商的复杂任务上才开启 Team 模式。

  **装完后第一次委派可能有"冷启动"开销。** 编排器的完整逻辑（~7.5KB）在首次 `/workflow-orchestrator:delegate` 调用时才加载。会话启动时只注入一个 ~1.1KB 的路由存根。这个设计省了 ~6.6K token 的会话启动开销，但第一次委派会比后续调用稍慢——正常的。

  workflow-orchestrator 的定位不是替代 Claude Code 内置的动态 Workflow（那是另一个层次的工具，用于大规模确定性编排），也不是替代 LangGraph（那是独立的多 agent 框架）。它是**让 Claude Code 的日常使用自动进入"多 agent 模式"**的一层薄壳：不需要学新语法、不需要配外部服务、不需要写编排脚本。装完自动生效，用自然语言驱动。

  ## 今日小结

  workflow-orchestrator 适合所有用 Claude Code 做复杂开发的工程师。它不是让你学一套新工具，而是把"拆任务、派 agent、并行跑、验证收尾"这个你每次都手动做一遍的决策流程，固化成了一套安装即生效的钩子系统。14 个 hook 在你每次打开 Claude Code 时自动运行，8 个专职 agent 在你每次说出复杂需求时自动待命。从"帮我做一个全栈计算器"到 agent swarm 跑完收工——你说了 55 个单词，编排器替你做了剩下的一切。75 个 star 说明它还没有被大规模发现，但 26 个 release 的迭代密度说明作者在认真打磨。如果你受够了手动在脑子里拆任务、盯着串行执行等结果、在上下文膨胀中看着 agent 越来越"健忘"——花 30 秒装上，下次复杂任务直接交给它编排。
