# test-driven-development — 把测试纪律写进 AI 编程助手的肌肉记忆

  Addy Osmani 出品的 Agent Skills 系列中最硬核的一篇——不是"让 AI 帮你写测试"，而是"让 AI 用 TDD 的方式写代码"。红→绿→重构的循环被编码成 AI 不可绕过的工作流，Bug 修复必须先写复现测试。

  这个技能来自 `addyosmani/agent-skills`（77.2k stars，Google 工程大佬 Addy Osmani 维护），是 24 个生产级工程技能中的测试基石。它不教你"怎么用测试框架"，而是把 TDD 的纪律写成 AI 必须遵守的规则。

---

### 它解决什么问题

  AI 写代码最大的问题是"看起来对，跑起来就挂"。你让它修一个 Bug，它直接改代码，改完告诉你"已修复"——没有测试，没有证据。你让它加新功能，它跳过测试直接写实现，下次重构时全崩。

  这个技能强制 AI 走 TDD 全流程：先写会失败的测试 → 用最少代码让测试通过 → 重构清理。对 Bug 修复则强制执行"复现测试先于修复"的 Prove-It 模式。代码写得对，测试绿了才算数；"看起来对"不算。

### 核心原理

  技能本身是一份 `SKILL.md` 文件，约 300 行 Markdown。它不是脚本或工具，而是一套结构化的**工作流约束**：

  - **描述触发**：YAML frontmatter 的 `description` 字段声明了触发条件——"implementing logic, fixing bugs, changing behavior"。当你的对话匹配这些场景，Claude Code 自动加载技能全文。
  - **TDD 循环编码**：文档将 RED → GREEN → REFACTOR 三步写成带代码示例的标准流程，每个步骤有明确的输入/输出约定。
  - **反偷懒检查表**：文末的 anti-rationalization table 列出了 AI 常见的自欺借口（"这太简单了不需要测试"、"我手动验证过了"）和对应的反驳逻辑。
  - **测试金字塔强制**：80% 单元测试 / 15% 集成测试 / 5% E2E 的比例不是建议，是默认要求。

  技能加载后，Claude Code 的规划阶段会优先考虑测试策略。你不需要说"请写测试"，AI 会主动问"先写哪个级别的测试？"

### 使用指南

#### 场景一：新功能开发——从测试开始，而非从实现开始

  你说："给 TaskService 加一个 completeTask 方法，完成任务时设置 completedAt 时间戳。"

  AI 不会直接写 `completeTask` 函数。它会先写一个测试：

  ```typescript
  it('sets completedAt when task is completed', async () => {
    const task = await taskService.createTask({ title: 'Test' });
    const completed = await taskService.completeTask(task.id);
    expect(completed.status).toBe('completed');
    expect(completed.completedAt).toBeInstanceOf(Date);
  });
  ```

  测试运行——红灯。然后 AI 写最小实现让测试通过。最后重构清理。你得到的不只是功能代码，还有保护这段代码的测试。

#### 场景二：Bug 修复——Prove-It 模式

  你说："完成任务的 Bug：completedAt 时间戳没有更新。"

  关键区别：AI **不会直接改代码**。Prove-It 模式要求：

  1. 先写一个复现 Bug 的测试（应该失败）
  2. 确认测试确实失败了（证明 Bug 存在）
  3. 修改代码
  4. 确认测试通过（证明 Bug 修复）
  5. 跑全量测试套件（证明没有引入回归）

  每一步都有明确的验证门，不存在"我猜问题出在这里"式的盲修。

#### 场景三：重构保护——先有测试网，再动代码

  你说："这段 `TaskService` 代码太乱了，帮我重构一下。"

  AI 首先检查现有测试覆盖率。如果不满足测试金字塔要求（80/15/5），它会拒绝重构，而是建议先补测试。只有当测试网足够密时，才会开始重构——每改一步跑一次测试。这就是 Beyonce Rule："如果你喜欢这段代码，就应该给它写测试。"

### 进阶技巧

  **TDD + Chrome DevTools MCP 组合技**：对于浏览器端代码，`test-driven-development` 技能可与 `browser-testing-with-devtools` 技能联动。AI 会先用 Chrome DevTools MCP 获取运行时数据（DOM 结构、网络请求、控制台日志），再针对实际运行时行为写测试，而不是凭猜测写 mock。

  **自定义测试金字塔比例**：如果你的项目是 E2E 重型（如移动端自动化），在 `CLAUDE.md` 中声明测试策略偏好（如 20/30/50），技能会读取项目级约定而非死守默认值。

### 常见踩坑

  **技能不自动触发**：如果你说"帮我实现这个功能"但 AI 没有走 TDD 流程，检查技能是否正确安装——`ls ~/.claude/skills/test-driven-development/SKILL.md`。如果文件存在但不触发，在请求中明确说"用 TDD 方式实现"。

  **AI 跳过测试直接写代码**：这是 Anti-Rationalization Table 的真正用途。当 AI 说"这个太简单不需要测试"时，技能会提醒它检查表里的对应条目——没有例外，即使是单行代码也要有测试。

  **测试写得太慢影响效率**：对于纯配置修改（JSON/YAML/环境变量）、文档更新、静态资源替换，TDD 反而浪费时间。技能内置了"何时不适用"清单，这些场景 AI 会跳过测试流程。

## 今日小结

  如果你受够了 AI 写代码时的"自信满满，一跑就挂"，这个技能是最低成本的解法——安装只需一行命令，不需要任何依赖，不需要改项目结构。它把 TDD 的纪律变成了 AI 的系统提示词，让你不需要每次都说"请先写测试"。

  花 1 分钟安装，换来的是每次 AI 写代码时自动走的红→绿→重构循环。Bug 修复有了证据链，重构有了安全网，新增功能有了回归保护。不是银弹，但确实是把 AI 编程质量从"碰运气"拉到"可预期"的最简单方法。
