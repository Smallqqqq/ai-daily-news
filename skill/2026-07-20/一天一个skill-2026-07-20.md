# debug-mode — AI 不再瞎猜 Bug，它先假设、再验证、后修复

  AI 编程助手修 Bug 有一个经典模式：你描述症状，它扫一眼代码，直接改——没有证据，没有验证，只有"应该可以了"。如果改对了，皆大欢喜；如果没对，你再描述一遍、它再猜一遍，三四个回合下来，token 烧了不少，Bug 还在原地。debug-mode 做的就是把 Cursor Debug Mode 那套"假设→探针→复现→验证→清理"的结构化调试流程搬到 Claude Code/Codex/Gemini CLI 里，强制 agent 在动手改代码之前先收集运行时证据。100 stars，MIT 协议，2026 年 6 月仍在更新。

---

### 它解决什么问题

  用 AI 助手修 Bug 时，有两个系统性问题在反复上演：

  第一，**agent 跳过诊断直接开药方**。你说"登录页点提交没反应"，它看到 `handleSubmit` 里有个 `try/catch` 没写 `catch` 体，于是补了一行 `console.error(err)` 就告诉你修好了。但真正的原因可能是 API 返回了 302 重定向、或者 CSRF token 过期、或者 CORS 头没配。agent 看到什么改什么，不追根因。

  第二，**修复缺乏可验证的证据链**。agent 改完代码说"已修复"，你手动打开浏览器试了一下——确实好了。但三天后另一个场景下 Bug 又回来了，因为根因没找到，只是恰好把表面症状盖住了。没有日志、没有复现记录、没有假设验证过程。

  debug-mode 用一套不可跳过的八步工作流解决了这两个问题：agent 必须先理解 Bug → 列出 3-5 个可验证的假设 → 在代码中插入探针日志 → 让你复现 Bug → 从日志中确认或推翻每个假设 → 才动手修 → 修完再验证 → 最后清理所有探针代码。每一步都有证据，每一步都不可跳过。

### 核心原理

  debug-mode 的核心设计可以浓缩为三样东西：**假设标签、区域探针、人机循环**。

  **假设标签**：agent 生成的每个假设都有编号（H1、H2、H3…），插入的日志全部带标签。比如假设 H1 是"缓存返回了过期数据"，对应的日志就是 `[DEBUG H1] cache_key=user:123 hit=true age=3600s`。复现 Bug 后，你只需要把 `.claude/debug.log` 的内容丢回给 agent，它能直接看出 H1 确认、H2 推翻、H3 确认——因为每条日志都标注了出自哪个假设。

  **区域探针**：探针代码用 `#region DEBUG` / `#endregion DEBUG` 包裹，而不是散落在代码里。这确保了调试结束后，agent 可以一键清除所有探针，不留残留代码。支持 JS/TS/Java/C#/Go/Rust/C/C++ (`// #region DEBUG`)、Python/Ruby/Shell/YAML (`# #region DEBUG`)、HTML/Vue/Svelte (`<!-- #region DEBUG -->`)、Lua (`-- #region DEBUG`) 等所有主流语言。

  **人机循环**：agent 不能自己宣布"修复完成"。整个流程中有两个关键节点必须由你来触发：复现 Bug（你去操作、去触发，agent 等着日志）和验证修复（你去确认 Bug 不再出现）。agent 的角色是提出假设、插入探针、分析日志、写修复；你的角色是触发真实的运行时行为。没有你的确认，agent 不会删除探针代码，也不会声称 Bug 已修复。

  日志输出到 `.claude/debug.log`（绝对路径）而非 stdout/stderr——这样终端不会被日志洪流淹没，agent 的上下文窗口也不会被几万行日志撑爆。

### 安装配置

  Claude Code 安装（注意目录层级，此步最容易踩坑）：

```bash
# 克隆仓库到 skills 目录
mkdir -p ~/.claude/skills && git clone https://github.com/doraemonkeys/claude-code-debug-mode.git ~/.claude/skills/claude-code-debug-mode

# ⚠️ 关键一步：Claude Code 只扫描 ~/.claude/skills/ 的直接子目录，
# 不会递归。必须把内层 debug-mode/ 提升一层
mv ~/.claude/skills/claude-code-debug-mode/debug-mode ~/.claude/skills/debug-mode
rm -rf ~/.claude/skills/claude-code-debug-mode

# 验证
ls ~/.claude/skills/debug-mode/SKILL.md
```

  Codex 安装（Codex 递归扫描 skills/，无需额外步骤）：

```bash
mkdir -p ~/.codex/skills && git clone https://github.com/doraemonkeys/claude-code-debug-mode.git ~/.codex/skills/claude-code-debug-mode
```

  Gemini CLI 安装：

```bash
mkdir -p ~/.gemini/skills && git clone https://github.com/doraemonkeys/claude-code-debug-mode.git ~/.gemini/skills/claude-code-debug-mode
# 或用内置命令
gemini skills install https://github.com/doraemonkeys/claude-code-debug-mode.git
```

  无需额外配置，重启 Claude Code / Codex / Gemini CLI 即可。安装后 agent 会自动在遇到 Bug 描述时触发此技能。

### 使用指南

#### 场景一：API 返回了空数据，但你不知道是前端、后端还是网关的锅

  最常见的"薛定谔的 Bug"——数据有时候正常，有时候空，没规律。agent 会生成 3-5 个假设：H1 是 API 请求根本没发出去（前端拦截器吞了错误）、H2 是后端返回了 200 但 body 为空（数据库连接池耗尽）、H3 是网关超时返回了默认空 JSON。

  agent 会在前端请求拦截器中插入 `[DEBUG H1]` 探针（记录请求是否发出、响应状态码），在后端路由处理器中插入 `[DEBUG H2]` 探针（记录数据库查询结果行数），在网关配置中插入 `[DEBUG H3]` 探针（记录上游响应时间）。

  你触发一次请求，把 `.claude/debug.log` 丢给 agent。日志显示 H1 探针记录了 `status=200 bodyLength=0`，H3 探针记录了 `upstreamLatency=30001ms timeout=30000ms`。agent 确认 H3（网关超时），修复网关超时配置，你再验证一次——确认修复后 agent 清理全部探针。

#### 场景二：间歇性 Bug，"有时候好有时候不行"

  这是最难排查的一类 Bug——它不总是出现，等你准备好了复现条件它又消失了。debug-mode 提供了"预埋探针"模式：让 agent 先把探针全部插入，你不用当场复现，正常使用你的应用。探针就一直待在那里，等 Bug 某天再次触发时，所有日志已经自动收集好了。

  比如一个 WebSocket 连接在运行 2 小时后断开的 Bug——你不可能坐着等 2 小时。agent 在 WebSocket 心跳、重连逻辑、连接池管理中插入探针，你回去继续工作。两小时后 Bug 出现，你取回日志文件交给 agent，它从日志时间线中定位到：心跳在第 1h58m 开始延迟，第 2h03m 超时，但重连逻辑在心跳未超时之前就触发了——是一个竞态条件。

#### 场景三：跨语言项目，前端 Vue + 后端 Python + 脚本 Shell

  大多数调试工具绑定特定语言。debug-mode 的 `#region DEBUG` 标记覆盖了所有主流语言的注释语法，同一个 Bug 跨三个语言层也能统一追踪。

  比如一个文件上传失败——前端 Vue 组件 (`<!-- #region DEBUG -->`)、后端 Python Flask 路由 (`# #region DEBUG`)、Shell 脚本做文件转码 (`# #region DEBUG`)。三层的探针日志都带上统一的假设标签 `[DEBUG H1]`。agent 一次性分析三层日志，发现前端传了正确的文件、Python 路由收到了文件、但 Shell 转码脚本因为文件超过 500MB 而 OOM 被 kill。根因在 Shell 脚本那层，agent 只需改 Shell 脚本加流式处理。

### 进阶技巧

  **灵活适配——核心是"证据"，不是"规程"**。技能默认把日志写到本地 `.claude/debug.log`，但如果你的代码跑在手机上、远程设备上、或者 Docker 容器里，写不到本地文件怎么办？没关系——让探针写到那个设备上的任意位置，复现后把日志拷贝/粘贴回给 agent。传输方式变了，但核心两步没变：复现 Bug → 把证据交给 agent。

  **与其他 skill 组合**。debug-mode 定位到根因后，修复阶段可以结合 TDD skill——agent 先写一个能复现 Bug 的失败测试，再修代码让测试通过，最后清理探针。这样你的代码库里不仅修了 Bug，还留下了回归测试。

### 常见踩坑

  **Claude Code 安装后 skill 不生效**：这是最高频的问题。Claude Code 只扫描 `~/.claude/skills/` 的直接子目录，不递归。clone 下来的仓库结构是 `claude-code-debug-mode/debug-mode/SKILL.md`，Claude Code 看不到。必须手动把内层 `debug-mode/` 移动到 `~/.claude/skills/debug-mode/`。如果没做这一步，skill 静默不加载，没有任何报错提示，非常容易忽略。

  **探针日志太多撑爆上下文**：如果你的 Bug 在高频路径上（比如每个请求都经过的中间件），一次复现可能产生几万条日志。agent 不需要全量日志——告诉它"日志大概有 2 万行，你先 grep [DEBUG H1] 看看结果"。如果 H1 已经被推翻，相关日志可以直接跳过，聚焦到确认中的假设线上。

### 同类技能对比

| 技能 | 安装方式 | 核心特点 | 适用场景 | 热度 |
|------|---------|---------|---------|------|
| **debug-mode**（今日推荐）| `git clone` + 目录调整 | 假设驱动、日志探针、人机协同、跨语言 | 所有需要运行时证据的 Bug | ⭐ 100 |
| Cursor Debug Mode | Cursor 内置 | 原生集成、自动复现、AI 辅助诊断 | Cursor IDE 用户 | ⭐ 内置 |
| AlmogBaku/debug-skill | plugin marketplace | DAP 真调试器协议、断点、单步执行 | 本地可运行、需要逐行调试的场景 | ⭐ 未验证 |

  debug-mode 的独特价值在于**不绑定特定 IDE，不依赖调试器协议**——只要代码里有注释语法就能插探针。对于远程设备、移动端、Docker 容器、生产环境等无法挂调试器的场景，探针模式是唯一可行的选择。Cursor Debug Mode 虽强大但限定在 Cursor IDE 内，debug-skill 需要 DAP 兼容的运行环境。

## 今日小结

  今天这个技能适合每个被"AI 修了三次没修好"折磨过的开发者。安装只要两条命令，配置为零，但它改变的不只是调试效率——它改变的是你和 agent 之间的协作模式：从"你说症状它猜修复"变成"你说症状它提假设，你去复现它看日志，确认根因后再修"。花五分钟装上它，下一次遇到 Bug 时，你得到的不是一个"应该能用了"的 patch，而是一个有日志证据支撑的根因修复。
