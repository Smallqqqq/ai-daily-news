# agent-teams-playbook — 一个 Skill 指挥一群 Agent 协同作战

  2026 年是 AI Agent 之年，单个 Agent 的能力已经很强了。但当你面对一个需要多角度审视、多模块并行推进的复杂任务时，单靠一个 Agent 反复切换上下文、逐个处理子任务，效率瓶颈非常明显。agent-teams-playbook 就是为这个痛点而生——它不是又一个"怎么写好 prompt"的教程，而是一套**跨平台的多 Agent 编排协议**，让你的 Claude Code 化身"项目经理"，按需组建 Agent 团队，制定执行计划，分工协作，最终汇总交付。

---

### 它解决什么问题

  当你对 Claude Code 说"帮我重构整个项目"或"帮我做一次全面的安全审计"时，一个 Agent 得先读代码、再分析、再出方案、再执行、再验证——每一步都要切上下文。更糟糕的是，代码审查这种需要多维度审视的任务，一个 Agent 看过的视角天然有盲区。

  agent-teams-playbook 的做法是：收到任务后，先不急着动手，而是走一遍"规划准备 → 能力发现 → 团队组建 → 并行执行 → 质量把关 → 结果交付"的六阶段流程。它让 Claude Code 先评估任务复杂度和本地可用能力（skills、agents、MCP 工具），再决定是用单人模式、直接复用已有 Skill、还是组建团队——一切都是**自适应决策**，不是写死的配置。

### 核心原理

  这个 Skill 的核心是一套"编排合同"（orchestration contract）：**先澄清目标 → 再能力发现 → 再团队蓝图 → 再执行/评审/验证**。具体工具名由运行时适配层决定，不硬编码。

  六个阶段的运作逻辑：

  1. **阶段 0（规划准备）**：优先调用 `planning-with-files` 建立 task_plan.md、findings.md、progress.md，把"要做什么"结构化为文件
  2. **阶段 1（能力发现）**：扫描本地已有的 skills / agents / MCP 工具，如果不够用再尝试外部搜索，确定可用武器库
  3. **阶段 2（团队组建）**：根据任务复杂度和场景匹配，确定走哪种编排模式——单人增强、直接复用 Skill、计划+评审、Leader-Member、还是复合编排
  4. **阶段 3（并行执行）**：派出子 Agent 并行处理各自任务，通过 Task / TeamCreate / SendMessage 协作
  5. **阶段 4（质量把关）**：关键里程碑设门控，产出不符合预期就回滚或重试
  6. **阶段 5（结果交付）**：汇总所有子 Agent 的输出，压缩上下文后交还给用户

  这套流程特别强调**降级策略**——如果某个依赖的 Skill 不存在、某个平台的工具不支持，它不会报错或者"假装已调用"，而是明确说明降级方式，用主线程或其他可用工具替代。

### 安装配置

```bash
# 克隆仓库
git clone https://github.com/KimYx0207/agent-teams-playbook.git

# 安装到 Claude Code
cd agent-teams-playbook
chmod +x scripts/install.sh
./scripts/install.sh

# 或一次性装到所有支持的平台
./scripts/install.sh --target all --from-github
```

手动安装也极其简单，只需要把 SKILL.md 放到对应目录：

```bash
# Claude Code
mkdir -p ~/.claude/skills/agent-teams-playbook
cp SKILL.md ~/.claude/skills/agent-teams-playbook/
cp README.md ~/.claude/skills/agent-teams-playbook/
```

安装完成后，用自然语言即可触发——说"帮我拉个 agent 团队来审查这个重构方案"，Claude Code 就会自动加载这个 Skill。

### 使用指南

#### 场景一：多维度代码审查

  假设你刚完成一个涉及安全、性能、架构三方面的大改动，手动审查很容易遗漏。

  触发方式：直接对 Claude Code 说"帮我组建一个 agent 团队来审查这次改动，覆盖安全、性能、架构三个维度"。

  Skill 会自动进入场景 3（计划+评审模式）：先制定审查计划，然后并行派出 3 个子 Agent——一个安全审计、一个性能分析、一个架构一致性检查——各自独立完成审查后汇总结果。你看到的不是 3 份独立报告，而是一份**合并后的质量闸门摘要**，标注每个维度的通过/警告/阻断状态。

#### 场景二：大型重构的并行实施

  需要同时重构三个独立模块（比如用户系统、支付系统、通知系统），每个模块的改动互相不耦合。

  触发方式："帮我并行重构用户模块、支付模块和通知模块，各自独立实现，最后汇总"。

  Skill 检测到任务可并行拆分，走场景 4（Lead-Member 模式）：Leader 负责整体协调，三个 Member Agent 各自负责一个模块的重构。Leader 设定统一的编码规范和接口约定，Member 在各自 worktree 中并行开发，完成后再由 Leader 汇总、处理合并冲突、跑集成测试。

#### 场景三：从零到一的技术调研 + 方案设计

  接到一个新需求——"调研市面上主流消息队列方案，给出推荐并在项目中搭建 PoC"。

  触发方式："帮我做一次完整的技术选型：调研 Kafka、RabbitMQ、Pulsar、NATS，出对比报告，然后在项目里搭一个推荐方案的 PoC"。

  Skill 走场景 5（复合编排）：阶段 1 先并行派出 4 个 explore agent 各自调研一个消息队列；阶段 2 由 Leader 汇总生成对比报告；阶段 3 让你确认选型后，再派出 implementer agent 搭建 PoC；阶段 4 由 reviewer agent 验证 PoC 是否符合预期。

### 进阶技巧

  **组合技：agent-teams-playbook + planning-with-files**。agent-teams-playbook 本身就推荐搭配 `planning-with-files` 使用。当你启动一个团队任务时，Skill 会自动创建 task_plan.md（任务拆分）、findings.md（各 Agent 的发现汇总）、progress.md（进度追踪）。这些文件在整个任务周期内持续更新，即使你中途切到别的事情、第二天再回来看，也能无缝接上进度。

  **fork 模式隔离**：如果你不想让编排过程占用主会话的上下文窗口，可以在 SKILL.md frontmatter 里加一行 `context: fork`。这样整个六阶段流程在隔离上下文中运行，你的主会话保持清爽，最终只收到一份汇总交付物。适合大规模、长时间运行的编排任务。

### 常见踩坑

  **子 Agent 数量不要贪多**。Skill 的设计原则是"并行 Agent 建议不超过 5 个"，这不是保守，是务实——每个子 Agent 都有独立的上下文窗口和 token 消耗，5 个以上边际效益骤降且成本飙升。如果任务确实需要更多并行度，考虑拆成多个 Phase 分批执行。

  **跨平台时注意工具名差异**。这个 Skill 支持 Claude Code、Codex、OpenClaw、Cursor 四个平台，但各平台的工具名和能力不完全等价。比如 `TeamCreate` 和 `SendMessage` 是 Claude Code 专属的——在 Codex 上，Skill 会自动降级为"多个独立 subagent + 主线程协调"。如果你在两个平台间切换，注意输出格式可能会不同，但核心的"编排合同"逻辑是一致的。
