# 一天一个 Skill · 2026-06-03

  AI 编程 Agent 写 React 代码的质量参差不齐——变量 stale closure、useEffect 死循环、props 下钻太深、缺少 memoization。今天这个 Skill 像一个专看 React 代码的医生，用确定性规则扫描你的代码库，在 Agent 写的烂代码合入之前把它拦下来。

---

## react-doctor — AI Agent 写的 React 代码，它来把关

### 它解决什么问题

  很多人让 Claude Code、Cursor、Codex 帮忙写 React 组件时都有一个共同体验：功能能跑，但代码质量一言难尽。useEffect 缺少依赖、state 直接 mutate、props 层层下钻不 memo、异步操作中的 stale closure……这些问题在 Agent 生成的代码里高频出现，人工 review 很容易漏掉。

  react-doctor 是一个确定性的 React 代码扫描器，专门针对 AI 编码 Agent 产出的代码做质量审计。它不依赖 LLM 判断（避免了 AI review AI 的幻觉链），而是用静态分析规则检查五大维度：状态与副作用、性能、架构、安全性、可访问性。每次扫描输出一个 0-100 的代码健康分。

### 核心原理

  react-doctor 是一个 TypeScript 写就的规则引擎，内置了一套针对 React 代码的静态分析规则集。它和 ESLint 类似，但规则更聚焦 React 特有的反模式——很多是 AI 生成代码中的高频问题。

  它的工作方式分三层：

  **第一层：npx 快速扫描。** 在项目根目录跑 `npx react-doctor@latest`，它会扫描所有 React 文件，按五大类别输出问题列表和健康评分。

  **第二层：Agent Skill 安装。** 跑 `npx react-doctor@latest install` 会把规则集安装为你的 AI 编码 Agent 的 Skill。之后 Agent 写 React 代码时，Skill 里的规则会被加载到上下文中，Agent 写代码前就知道该避免什么。

  **第三层：CI 自动门控。** GitHub Action 在 PR 上自动扫描变更文件，在 diff 行上直接标注问题，还能设置评分门槛阻止劣化 PR 合并。

  它的 playbook 机制（`https://www.react.doctor/prompts/react-doctor-agent.md`）是一个可远程更新的修复指南，Agent 每次跑 `/doctor` 都会拉取最新版，确保修复逻辑始终和规则引擎同步。

### 安装配置

**第一步：跑一次审计**

```bash
npx react-doctor@latest
```

这个命令会扫描整个项目，输出得分和问题列表。加 `--verbose` 可以看到具体文件和行号。

**第二步：安装为 Agent Skill**

```bash
npx react-doctor@latest install
```

安装后 Claude Code、Cursor、Codex 等 Agent 在执行 React 编码任务时会自动加载规则。你也可以直接打 `/doctor` 触发修复流程。

**第三步（可选）：配置**

在项目根目录创建 `react-doctor.config.json`：

```json
{
  "$schema": "https://react.doctor/schema/config.json",
  "rules": {
    "effect/no-derived-state": "error",
    "performance/prop-drilling": "warn",
    "security/no-dangerously-set-innerhtml": "error"
  },
  "ignore": ["**/*.test.tsx", "**/vendor/**"],
  "score": {
    "threshold": 80
  }
}
```

`$schema` 会提供字段补全和悬停文档。

### 使用指南

#### 场景一：Agent 写完后自查

  让 AI Agent 写完一个 React 组件后，在对话里输入 `/doctor`。Agent 会拉取最新的 playbook，扫描刚才的改动，逐个修复诊断出的问题，然后重新扫描确认评分没有下降。

  典型对话：
  ```
  你：/doctor
  Agent：扫描到 3 个问题——
  1. useCallback 缺少依赖数组（effect/exhaustive-deps）
  2. 列表渲染缺少 key（correctness/missing-key）
  3. 大量 props 未 memo 导致不必要的重渲染（performance/prop-drilling）
  修复后重新扫描，评分从 72 → 91。
  ```

#### 场景二：只检查 diff，不扫全量

  在功能分支上，只关注你改动的文件：

  ```bash
  npx react-doctor@latest --verbose --diff
  ```

  这只会扫描相对于 main 分支的变更文件。PR 阶段用这个最快，不用等全量扫描。

#### 场景三：CI 自动拦截劣化

  用 GitHub Actions 在每次 PR 上自动运行：

  ```yaml
  - uses: millionco/react-doctor@v1
    with:
      fail-on: error
      annotations: true
  ```

  效果：PR 页面上出现 inline annotations 标注具体行的问题，还有一个 sticky comment 显示评分和问题摘要。只有解决所有 error 级别问题的 PR 才能合并（`fail-on: error`）。

### 进阶技巧

  **和 ESLint / oxlint 组合使用：** react-doctor 提供了 `eslint-plugin-react-doctor` 和 `oxlint` 插件，可以把部分规则直接集成到已有的 lint 流程里。这样你不需要单独跑 react-doctor，ESLint 保存时就提示了。

  **playbook 远程更新：** Agent 的 playbook（`https://www.react.doctor/prompts/react-doctor-agent.md`）和每条规则的修复指南（`https://www.react.doctor/prompts/rules/{category}/{rule}.md`）是远程托管、独立版本化的。react-doctor 更新规则后，所有 Agent 只需要重新 `/doctor` 就能拿到最新修复策略，不用重新安装 Skill。

### 常见踩坑

  **问题一：扫描太慢。** 如果项目很大，首次全量扫描可能需要几十秒。用 `--diff` 只扫描变更文件可以大幅加速。如果 CI 中需要更快，可以配合 `--offline` 跳过在线评分 API。

  **问题二：死代码检测已移除。** v0.2 版本移除了 dead-code 检测（官方建议用 `npx knip` 替代），不要期待 react-doctor 会报告未使用的变量或导入。

### 同类技能对比

| 技能 | 安装方式 | 核心特点 | 适用场景 | 热度 |
|------|---------|---------|---------|------|
| **react-doctor**（今日推荐）| `npx react-doctor@latest install` | 确定性规则引擎，非 LLM 判断，0-100 健康分 | AI Agent 写的 React 代码质量把关 | ⭐ 10918 |
| ESLint + React 插件 | `npm i -D eslint` | 通用 JS/TS lint，规则覆盖广但不专门针对 React 反模式 | 所有 JS/TS 项目 | ⭐ 25000+ |
| `/code-review`（Claude Code 内置）| 内置 | LLM 驱动的通用代码审查 | 非 React 特异性 | 内置 |
| biome | `npm i -D @biomejs/biome` | 统一 formatter + linter | 追求速度的替代方案 | ⭐ 16000+ |

  推荐理由：react-doctor 不替代 ESLint，而是在 AI 编码 Agent 场景下补 ESLint 比较弱的领域——尤其是 effect 类的运行时陷阱、AI 生成代码的高频 anti-pattern。它专门针对 "Agent 写的 React 代码" 这个场景设计，playbook 机制让 Agent 不仅知道什么是错的，还知道怎么改。

## 今日小结

  react-doctor 适合所有让 AI 帮忙写 React 代码的开发者。安装只要两条 `npx` 命令，之后每次 Agent 写 React 组件都能得到确定性的质量反馈——不是"AI 审查 AI"的幻觉链，而是可复现的静态规则。特别是 `/doctor` 命令配合 playbook 自动修复流程：扫描 → 修问题 → 重扫确认，把 React 代码质量的反馈循环从"等 PR review"变成了"写完就自查"。今天花 5 分钟装上，从下一行 Agent 写的 React 代码开始受益。
