# playwright-skill — 给你的 AI 编程助手装上一个真正的浏览器

  Claude Code 能写代码、能查文档、能跑终端命令，但有一件事它天生做不到：打开一个浏览器，真正看一眼你的网页长什么样。你让 agent 写了个登录页，它说"写好了"，但你不知道按钮有没有对齐、表单提交后有没有跳转、移动端布局有没有炸。playwright-skill 填补了这个空缺——它给 Claude Code 装了一个真实的 Chromium 浏览器，agent 可以自己打开网页、填表单、点按钮、截图、跑端到端测试。2,879 stars，MIT 协议，一条命令安装。

---

### 它解决什么问题

  AI 编程助手写前端代码的速度越来越快，但验证代码是否真的"能用"仍然靠人。典型场景：agent 写完一个注册页面，你打开浏览器手动输入邮箱密码点提交，看看有没有报错、有没有跳转。这个过程每次少则 30 秒，多则几分钟，一天重复几十次。

  playwright-skill 让 agent 自己来做这件事。它不是让 agent 猜测页面行为，而是真正启动一个 Chromium 浏览器，执行你描述的操作，然后告诉你结果——包括截图和控制台输出。

  具体来说，它解决三个问题：

  1. **写完即验**：agent 写完页面代码后，自动打开浏览器验证功能是否正常，不用切到浏览器手动操作。
  2. **回归测试自动化**：每次改完代码，跑一遍端到端流程，确保没有引入新问题。
  3. **跨视口验证**：桌面端、平板、手机三种尺寸一键截图，不用手动调 Chrome DevTools。

### 核心原理

  playwright-skill 的结构很简洁：一个 SKILL.md 描述文件 + 一个通用执行器 `run.js` + Playwright 的 API 参考文档。没有复杂的抽象层。

  **工作流程**：

  1. 你告诉 agent 要做什么（"测试登录流程"、"给首页截个图"、"检查有没有死链"）
  2. agent 先自动检测本地是否有正在运行的 dev server（通过 `detectDevServers()` 扫描端口），有的话直接用，没有就问你要 URL
  3. agent 现场写一段 Playwright 脚本，放到 `/tmp/playwright-test-*.js`
  4. agent 通过 `run.js` 执行这段脚本，浏览器窗口弹出（默认可见模式）
  5. 脚本运行完，agent 报告结果——控制台输出、截图路径、报错信息

  **设计上的几个关键决策**：

  - **脚本写到 /tmp，不污染项目目录**。测试脚本用完即扔，操作系统会自动清理 /tmp。
  - **浏览器默认可见**（`headless: false`）。你可以亲眼看到浏览器在做什么，调试时这比 headless 模式直观得多。
  - **渐进式加载文档**。SKILL.md 保持精炼，完整的 Playwright API 参考放在 `API_REFERENCE.md` 里，agent 只在需要时加载。节省 token。
  - **URL 参数化**。目标 URL 永远放在脚本顶部的 `TARGET_URL` 常量里，改一个地方就能切换环境。

### 安装配置

  前提：Node.js ≥ 18，已安装 Claude Code。

  **方式一：插件安装（推荐）**：

  ```
  /plugin marketplace add lackeyjb/playwright-skill
  /plugin install playwright-skill@playwright-skill
  ```

  然后进入插件目录执行一次性初始化：

  ```bash
  cd ~/.claude/plugins/marketplaces/playwright-skill/skills/playwright-skill
  npm run setup
  ```

  **方式二：手动安装**：

  ```bash
  git clone https://github.com/lackeyjb/playwright-skill.git /tmp/playwright-skill-temp
  mkdir -p ~/.claude/skills
  cp -r /tmp/playwright-skill-temp/skills/playwright-skill ~/.claude/skills/
  cd ~/.claude/skills/playwright-skill
  npm run setup
  rm -rf /tmp/playwright-skill-temp
  ```

  `npm run setup` 会安装 Playwright 和 Chromium 浏览器，只需执行一次。装完重启 Claude Code，`/help` 能看到 skill 即表示成功。

### 使用指南

  #### 场景一：写完页面，一键验证

  你让 agent 写了一个联系表单页面。以前你需要切到浏览器、打开 `localhost:3000/contact`、填名字、填邮箱、填留言、点提交、看有没有成功提示。现在一句：

  > "测试联系表单的提交流程"

  agent 自动检测 dev server，写一段 Playwright 脚本，打开浏览器，填表单，点提交，等待成功提示出现，然后告诉你：

  ```
  ✅ Form submitted successfully
  📸 Screenshot saved to /tmp/screenshot.png
  ```

  十秒完成，不需要你动手。

  #### 场景二：响应式设计检查

  你的页面在桌面端看起来完美，但移动端可能布局错乱。每次手动调 Chrome DevTools 的设备模拟器，切换三种尺寸截图——重复劳动。

  > "检查首页在桌面、平板、手机三种尺寸下的显示效果"

  agent 写一个循环脚本，依次设置 1920×1080、768×1024、375×667 三种视口，每种截一张全页图。三张截图排在一起，差异一目了然。

  #### 场景三：死链扫描

  你的文档站有 200 个外链，不知道哪些已经 404。手动点一遍不现实。

  > "扫描所有外链，找出返回 404 的"

  agent 写脚本提取页面上所有 `a[href^="http"]` 链接，逐个发 HEAD 请求，汇总返回非 200 的链接。五分钟扫完，输出一个死链清单。

  #### 场景四：登录流程端到端测试

  用户认证是最容易出 bug 的地方——token 过期、重定向错误、session 丢失。每次改认证逻辑都应该跑一遍完整流程。

  > "测试完整的登录流程：访问登录页 → 输入账号密码 → 提交 → 验证是否跳转到 dashboard"

  agent 执行每一步并验证中间状态——确认登录页加载、确认表单可以填写、确认提交后有重定向、确认 dashboard 页面包含用户名。任何一步失败都会有明确的报错信息。

### 进阶技巧

  **并行跑多个测试**。如果你有多个独立页面需要验证，可以同时启动多个 agent，每个负责一个页面。playwright-skill 的脚本写到 `/tmp` 且用不同的文件名，不会互相干扰。

  **结合 agent 的 code review 流程**。在 code review 阶段，让 agent 先跑一遍 playwright-skill 验证变更页面的功能，确认通过后再进入代码审查。等于在 CI 之前就加了一层浏览器级别的冒烟测试。

  **自定义 helper 函数**。`lib/helpers.js` 里有一些实用的工具函数（比如 `detectDevServers`），你可以往里加自己的常用操作——比如统一的登录函数、截图命名规则、自定义的等待策略。改一次，所有后续的测试脚本都能复用。

### 常见踩坑

  **第一次运行报 `Playwright not found`**。说明没执行 `npm run setup`。这个命令会安装 Playwright npm 包 + 下载 Chromium 浏览器。装完就好了。

  **dev server 检测不到**。`detectDevServers()` 扫描的是常见端口（3000、3001、5173、8080 等）。如果你的项目跑在非标准端口，直接告诉 agent URL 即可，比如"测试 http://localhost:4200 的首页"。

  **浏览器窗口没弹出来**。检查是不是在无图形界面的环境（SSH、CI、Docker）中运行。这种情况需要 agent 主动加 `headless: true`，或者你明确说"用 headless 模式"。

  **脚本报模块找不到**。一定要通过 `run.js` 执行脚本（`cd $SKILL_DIR && node run.js /tmp/xxx.js`），不要直接 `node /tmp/xxx.js`。`run.js` 会确保 Playwright 模块路径正确解析。

  playwright-skill 的独特价值在于**灵活性**。它不是给你一套预定义的浏览器操作工具，而是让 agent 现场写 Playwright 代码。Playwright 能做什么，它就能做什么——从简单的页面截图到复杂的多步流程、网络请求拦截、性能追踪。MCP 方式封装了有限的工具调用，agent-browser 侧重自然语言驱动。playwright-skill 给了 agent 完整编程能力，代价是 agent 需要理解 Playwright API。对于已经有 Playwright 经验的团队，这是零学习成本的选项。
